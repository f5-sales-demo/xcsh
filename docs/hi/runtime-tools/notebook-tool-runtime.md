# नोटबुक टूल रनटाइम आंतरिक संरचना

यह दस्तावेज़ वर्तमान `notebook` टूल कार्यान्वयन और कर्नल-समर्थित Python रनटाइम के साथ इसके संबंध का वर्णन करता है।

महत्वपूर्ण भेद: **`notebook` एक JSON नोटबुक एडिटर है, नोटबुक executor नहीं**। यह `.ipynb` सेल सोर्स को सीधे संपादित करता है; यह Python कर्नल को शुरू नहीं करता या उससे संवाद नहीं करता।

## कार्यान्वयन फाइलें

- [`src/tools/notebook.ts`](../../packages/coding-agent/src/tools/notebook.ts)
- [`src/ipy/executor.ts`](../../packages/coding-agent/src/ipy/executor.ts)

## 1) रनटाइम सीमा: संपादन बनाम निष्पादन

## `notebook` टूल (`src/tools/notebook.ts`)

- `.ipynb` फाइल पर `action: edit | insert | delete` का समर्थन करता है।
- सेशन CWD (`resolveToCwd`) के सापेक्ष पथ को हल करता है।
- नोटबुक JSON लोड करता है, `cells` array को मान्य करता है, `cell_index` सीमाओं को मान्य करता है।
- इन-मेमोरी में सोर्स संपादन लागू करता है और `JSON.stringify(notebook, null, 1)` के साथ पूर्ण नोटबुक JSON वापस लिखता है।
- टेक्स्ट सारांश + संरचित `details` (`action`, `cellIndex`, `cellType`, `totalCells`, `cellSource`) लौटाता है।

इस टूल में कोई कर्नल जीवनचक्र मौजूद नहीं है:

- कोई गेटवे अधिग्रहण नहीं
- कोई कर्नल सेशन ID नहीं
- कोई `execute_request` नहीं
- कर्नल चैनलों से कोई स्ट्रीम चंक नहीं
- कोई रिच डिस्प्ले कैप्चर नहीं (`image/png`, JSON डिस्प्ले, स्टेटस MIME)

## नोटबुक-जैसा निष्पादन पथ (`src/tools/python.ts` + `src/ipy/*`)

जब एजेंट को सेल-शैली Python कोड चलाने की आवश्यकता होती है (क्रमिक सेल, स्थायी स्थिति, रिच डिस्प्ले), तो यह **`python` टूल** के माध्यम से जाता है, `notebook` के माध्यम से नहीं।

वह पथ वह जगह है जहां कर्नल मोड, रीस्टार्ट/कैंसल व्यवहार, चंक स्ट्रीमिंग, और आउटपुट आर्टिफैक्ट ट्रंकेशन रहते हैं।

## 2) नोटबुक सेल हैंडलिंग शब्दार्थ (`notebook` टूल)

## सोर्स नॉर्मलाइज़ेशन

`content` को न्यूलाइन संरक्षण के साथ `source: string[]` में विभाजित किया जाता है:

- प्रत्येक गैर-अंतिम पंक्ति अनुगामी `\n` रखती है
- अंतिम पंक्ति में कोई बाध्यकारी अनुगामी न्यूलाइन नहीं होती

यह नोटबुक JSON परंपराओं को दर्शाता है और बाद के संपादनों पर आकस्मिक पंक्ति संयोजन से बचाता है।

## क्रिया व्यवहार

- `edit`
  - `cells[cell_index].source` को प्रतिस्थापित करता है
  - मौजूदा `cell_type` को संरक्षित करता है
- `insert`
  - `[0..cellCount]` पर सम्मिलित करता है
  - `cell_type` डिफ़ॉल्ट रूप से `code` होता है
  - कोड सेल `execution_count: null` और `outputs: []` को आरंभ करते हैं
  - मार्कडाउन सेल केवल `metadata` + `source` को आरंभ करते हैं
- `delete`
  - `cells[cell_index]` को हटाता है
  - रेंडरर प्रीव्यू के लिए विवरण में हटाए गए `source` को लौटाता है

## त्रुटि सतहें

कठोर विफलताएं इनके लिए फेंकी जाती हैं:

- गायब नोटबुक फाइल
- अमान्य JSON
- गायब/गैर-array `cells`
- सीमा से बाहर इंडेक्स (insert और गैर-insert की अलग-अलग मान्य सीमाएं हैं)
- `edit`/`insert` के लिए गायब `content`

ये अपस्ट्रीम में `Error:` टूल प्रतिक्रियाएं बन जाती हैं; रेंडरर नोटबुक पथ + स्वरूपित त्रुटि टेक्स्ट का उपयोग करता है।

## 3) कर्नल सेशन शब्दार्थ (जहां वे वास्तव में मौजूद हैं)

कर्नल शब्दार्थ `executePython` / `PythonKernel` में कार्यान्वित हैं और `python` टूल पर लागू होते हैं।

## मोड

`PythonKernelMode`:

- `session` (डिफ़ॉल्ट)
  - कर्नल `kernelSessions` मैप में कैश किए जाते हैं
  - अधिकतम 4 सेशन; ओवरफ्लो पर सबसे पुराना हटाया जाता है
  - हर 30 सेकंड में निष्क्रिय/मृत सफाई, 5 मिनट बाद टाइमआउट
  - प्रति-सेशन कतार निष्पादन को क्रमबद्ध करती है (`session.queue`)
- `per-call`
  - अनुरोध के लिए कर्नल बनाता है
  - निष्पादित करता है
  - `finally` में हमेशा कर्नल को बंद करता है

## रीसेट व्यवहार

`python` टूल `reset` केवल मल्टी-सेल कॉल में पहली सेल के लिए पास करता है; बाद की सेल हमेशा `reset: false` के साथ चलती हैं।

## कर्नल मृत्यु / रीस्टार्ट / पुनः प्रयास

सेशन मोड (`withKernelSession`) में:

- मृत कर्नल हार्टबीट (`kernel.isAlive()` हर 5 सेकंड जांच) या निष्पादन विफलता द्वारा पहचाना जाता है।
- पूर्व-रन मृत स्थिति `restartKernelSession` को ट्रिगर करती है।
- निष्पादन-समय क्रैश पथ एक बार पुनः प्रयास करता है: कर्नल रीस्टार्ट, हैंडलर पुनः चलाना।
- उसी सेशन में `restartCount > 1` होने पर `Python kernel restarted too many times in this session` फेंकता है।

स्टार्टअप पुनः प्रयास व्यवहार:

- साझा गेटवे कर्नल निर्माण HTTP 5xx के साथ `SharedGatewayCreateError` पर एक बार पुनः प्रयास करता है।

संसाधन थकावट पुनर्प्राप्ति:

- `EMFILE`/`ENFILE`/"Too many open files" शैली की विफलताओं का पता लगाता है
- ट्रैक किए गए सेशन साफ़ करता है
- `shutdownSharedGateway()` कॉल करता है
- कर्नल सेशन निर्माण का एक बार पुनः प्रयास करता है

## 4) पर्यावरण/सेशन वेरिएबल इंजेक्शन

कर्नल स्टार्टअप executor से वैकल्पिक env मैप प्राप्त करता है:

- `PI_SESSION_FILE` (सेशन स्टेट फाइल पथ)
- `ARTIFACTS` (आर्टिफैक्ट डायरेक्टरी)

`PythonKernel.#initializeKernelEnvironment(...)` फिर कर्नल के अंदर init स्क्रिप्ट चलाता है:

- `os.chdir(cwd)`
- `os.environ` में env एंट्री इंजेक्ट करना
- यदि गायब है तो `sys.path` में cwd को पहले जोड़ना

निहितार्थ:

- प्रीलूड हेल्पर जो सेशन या आर्टिफैक्ट संदर्भ पढ़ते हैं, Python प्रोसेस स्थिति में इन env vars पर निर्भर करते हैं।

## 5) स्ट्रीमिंग/चंक और डिस्प्ले हैंडलिंग (कर्नल-समर्थित पथ)

कर्नल क्लाइंट प्रति निष्पादन Jupyter प्रोटोकॉल संदेशों को प्रोसेस करता है:

- `stream` -> `onChunk` को टेक्स्ट चंक
- `execute_result` / `display_data` ->
  - MIME प्राथमिकता द्वारा डिस्प्ले टेक्स्ट चुना जाता है: `text/markdown` > `text/plain` > परिवर्तित `text/html`
  - संरचित आउटपुट अलग से कैप्चर किए जाते हैं:
    - `application/json` -> `{ type: "json" }`
    - `image/png` -> `{ type: "image" }`
    - `application/x-xcsh-status` -> `{ type: "status" }` (कोई टेक्स्ट उत्सर्जन नहीं)
- `error` -> ट्रेसबैक टेक्स्ट चंक स्ट्रीम में पुश किया जाता है + संरचित त्रुटि मेटाडेटा
- `input_request` -> stdin चेतावनी टेक्स्ट उत्सर्जित करता है, खाली `input_reply` भेजता है, stdin अनुरोधित चिह्नित करता है
- पूर्णता `execute_reply` और कर्नल `status=idle` दोनों की प्रतीक्षा करती है

रद्दीकरण/टाइमआउट:

- abort सिग्नल `interrupt()` ट्रिगर करता है (REST `/interrupt` + control-channel `interrupt_request`)
- परिणाम `cancelled=true` चिह्नित करता है
- टाइमआउट पथ आउटपुट को `Command timed out after <n> seconds` के साथ एनोटेट करता है

## 6) ट्रंकेशन और आर्टिफैक्ट व्यवहार

`src/session/streaming-output.ts` में `OutputSink` कर्नल निष्पादन पथों (`executeWithKernel`) द्वारा उपयोग किया जाता है:

- प्रत्येक चंक को सैनिटाइज़ करता है (`sanitizeText`)
- कुल/आउटपुट पंक्तियों और बाइट्स को ट्रैक करता है
- वैकल्पिक आर्टिफैक्ट स्पिल फाइल (`artifactPath`, `artifactId`)
- जब इन-मेमोरी बफर थ्रेशोल्ड से अधिक हो जाता है (`DEFAULT_MAX_BYTES` जब तक ओवरराइड न हो):
  - ट्रंकेटेड चिह्नित करता है
  - टेल बाइट्स मेमोरी में रखता है (UTF-8 सुरक्षित सीमा)
  - पूर्ण स्ट्रीम को आर्टिफैक्ट सिंक में स्पिल कर सकता है

`dump()` लौटाता है:

- दृश्य आउटपुट टेक्स्ट (संभवतः टेल-ट्रंकेटेड)
- ट्रंकेशन फ्लैग + गणनाएं
- आर्टिफैक्ट ID (`artifact://<id>` संदर्भों के लिए)

`python` टूल इस मेटाडेटा को परिणाम ट्रंकेशन सूचनाओं और TUI चेतावनियों में परिवर्तित करता है।

`notebook` टूल `OutputSink` का उपयोग **नहीं** करता; इसके पास कोई स्ट्रीम/आर्टिफैक्ट ट्रंकेशन पाइपलाइन नहीं है क्योंकि यह कोड निष्पादित नहीं करता।

## 7) रेंडरर धारणाएं और स्वरूपण

## नोटबुक रेंडरर (`notebookToolRenderer`)

- कॉल दृश्य: क्रिया + नोटबुक पथ + सेल/प्रकार मेटाडेटा के साथ स्थिति पंक्ति
- परिणाम दृश्य:
  - `details` से प्राप्त सफलता सारांश
  - `cellSource` को `renderCodeCell` के माध्यम से रेंडर किया जाता है
  - मार्कडाउन सेल भाषा संकेत `markdown` सेट करते हैं; अन्य सेल में कोई स्पष्ट भाषा ओवरराइड नहीं है
  - संक्षिप्त कोड प्रीव्यू सीमा `PREVIEW_LIMITS.COLLAPSED_LINES * 2` है
  - साझा रेंडर विकल्पों के माध्यम से विस्तृत मोड का समर्थन करता है
  - चौड़ाई + विस्तृत स्थिति द्वारा कुंजीबद्ध रेंडर कैश का उपयोग करता है

त्रुटि रेंडरिंग धारणा:

- यदि पहली टेक्स्ट सामग्री `Error:` से शुरू होती है, तो रेंडरर नोटबुक त्रुटि ब्लॉक के रूप में स्वरूपित करता है।

## Python रेंडरर (वास्तविक निष्पादन आउटपुट के लिए)

कर्नल-समर्थित निष्पादन रेंडरिंग अपेक्षा करता है:

- प्रति-सेल स्थिति संक्रमण (`pending/running/complete/error`)
- वैकल्पिक संरचित स्थिति इवेंट अनुभाग
- वैकल्पिक JSON आउटपुट ट्री
- ट्रंकेशन चेतावनियां + वैकल्पिक `artifact://<id>` पॉइंटर

यह रेंडरर व्यवहार `notebook` JSON संपादन परिणामों से असंबंधित है सिवाय इसके कि दोनों साझा TUI प्रिमिटिव का पुन: उपयोग करते हैं।

## 8) सादे Python टूल व्यवहार से विचलन

यदि "सादा Python टूल" का अर्थ `python` निष्पादन पथ है:

- `python` कर्नल में कोड निष्पादित करता है, मोड द्वारा स्थिति बनाए रखता है, चंक स्ट्रीम करता है, रिच डिस्प्ले कैप्चर करता है, इंटरप्ट/टाइमआउट संभालता है, और आउटपुट ट्रंकेशन/आर्टिफैक्ट का समर्थन करता है।
- `notebook` केवल नियतात्मक नोटबुक JSON म्यूटेशन करता है; कोई निष्पादन नहीं, कोई कर्नल स्थिति नहीं, कोई चंक स्ट्रीम नहीं, कोई डिस्प्ले आउटपुट नहीं, कोई आर्टिफैक्ट पाइपलाइन नहीं।

यदि किसी वर्कफ़्लो को दोनों की आवश्यकता है:

1. `notebook` के साथ नोटबुक सोर्स संपादित करें
2. `python` के माध्यम से कोड सेल निष्पादित करें (मैन्युअल रूप से कोड पास करते हुए), `notebook` के माध्यम से नहीं

वर्तमान कार्यान्वयन एक एकल टूल प्रदान नहीं करता जो `.ipynb` को म्यूटेट भी करे और कर्नल संदर्भ के माध्यम से नोटबुक सेल निष्पादित भी करे।
