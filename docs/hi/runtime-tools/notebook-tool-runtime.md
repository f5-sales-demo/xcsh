# नोटबुक टूल रनटाइम आंतरिक संरचना

यह दस्तावेज़ वर्तमान `notebook` टूल कार्यान्वयन और कर्नेल-समर्थित Python रनटाइम के साथ इसके संबंध का वर्णन करता है।

महत्वपूर्ण अंतर: **`notebook` एक JSON नोटबुक संपादक है, नोटबुक निष्पादक नहीं**। यह `.ipynb` सेल स्रोतों को सीधे संपादित करता है; यह Python कर्नेल शुरू नहीं करता और न ही उससे संवाद करता है।

## कार्यान्वयन फ़ाइलें

- [`src/tools/notebook.ts`](../../packages/coding-agent/src/tools/notebook.ts)
- [`src/ipy/executor.ts`](../../packages/coding-agent/src/ipy/executor.ts)
- [`src/ipy/kernel.ts`](../../packages/coding-agent/src/ipy/kernel.ts)
- [`src/session/streaming-output.ts`](../../packages/coding-agent/src/session/streaming-output.ts)
- [`src/tools/python.ts`](../../packages/coding-agent/src/tools/python.ts)

## 1) रनटाइम सीमा: संपादन बनाम निष्पादन

## `notebook` टूल (`src/tools/notebook.ts`)

- `.ipynb` फ़ाइल पर `action: edit | insert | delete` का समर्थन करता है।
- सत्र CWD (`resolveToCwd`) के सापेक्ष पथ हल करता है।
- नोटबुक JSON लोड करता है, `cells` ऐरे को मान्य करता है, `cell_index` सीमाओं को मान्य करता है।
- इन-मेमोरी स्रोत संपादन लागू करता है और `JSON.stringify(notebook, null, 1)` के साथ पूर्ण नोटबुक JSON वापस लिखता है।
- टेक्स्ट सारांश + संरचित `details` (`action`, `cellIndex`, `cellType`, `totalCells`, `cellSource`) लौटाता है।

इस टूल में कोई कर्नेल जीवनचक्र मौजूद नहीं है:

- कोई गेटवे अधिग्रहण नहीं
- कोई कर्नेल सत्र ID नहीं
- कोई `execute_request` नहीं
- कर्नेल चैनलों से कोई स्ट्रीम चंक नहीं
- कोई रिच डिस्प्ले कैप्चर नहीं (`image/png`, JSON डिस्प्ले, स्टेटस MIME)

## नोटबुक-जैसा निष्पादन पथ (`src/tools/python.ts` + `src/ipy/*`)

जब एजेंट को सेल-शैली Python कोड चलाने की आवश्यकता होती है (अनुक्रमिक सेल, स्थायी स्थिति, रिच डिस्प्ले), तो वह **`python` टूल** के माध्यम से जाता है, `notebook` के माध्यम से नहीं।

वह पथ वह है जहाँ कर्नेल मोड, रीस्टार्ट/कैंसल व्यवहार, चंक स्ट्रीमिंग, और आउटपुट आर्टिफैक्ट ट्रंकेशन रहते हैं।

## 2) नोटबुक सेल हैंडलिंग सिमेंटिक्स (`notebook` टूल)

## स्रोत सामान्यीकरण

`content` को न्यूलाइन संरक्षण के साथ `source: string[]` में विभाजित किया जाता है:

- प्रत्येक गैर-अंतिम पंक्ति अनुगामी `\n` रखती है
- अंतिम पंक्ति में कोई बाध्यकारी अनुगामी न्यूलाइन नहीं होती

यह नोटबुक JSON कन्वेंशन को प्रतिबिंबित करता है और बाद के संपादनों पर आकस्मिक पंक्ति संयोजन से बचाता है।

## क्रिया व्यवहार

- `edit`
  - `cells[cell_index].source` को प्रतिस्थापित करता है
  - मौजूदा `cell_type` को संरक्षित रखता है
- `insert`
  - `[0..cellCount]` पर सम्मिलित करता है
  - `cell_type` डिफ़ॉल्ट रूप से `code` होता है
  - कोड सेल `execution_count: null` और `outputs: []` को आरंभ करते हैं
  - मार्कडाउन सेल केवल `metadata` + `source` को आरंभ करते हैं
- `delete`
  - `cells[cell_index]` को हटाता है
  - रेंडरर पूर्वावलोकन के लिए विवरण में हटाया गया `source` लौटाता है

## त्रुटि सतहें

कठोर विफलताएँ इनके लिए फेंकी जाती हैं:

- गुम नोटबुक फ़ाइल
- अमान्य JSON
- गुम/गैर-ऐरे `cells`
- सीमा-बाह्य इंडेक्स (insert और गैर-insert की अलग-अलग मान्य श्रेणियाँ हैं)
- `edit`/`insert` के लिए गुम `content`

ये अपस्ट्रीम `Error:` टूल प्रतिक्रियाएँ बनती हैं; रेंडरर नोटबुक पथ + स्वरूपित त्रुटि टेक्स्ट का उपयोग करता है।

## 3) कर्नेल सत्र सिमेंटिक्स (जहाँ वे वास्तव में मौजूद हैं)

कर्नेल सिमेंटिक्स `executePython` / `PythonKernel` में कार्यान्वित हैं और `python` टूल पर लागू होते हैं।

## मोड

`PythonKernelMode`:

- `session` (डिफ़ॉल्ट)
  - कर्नेल `kernelSessions` मैप में कैश किए जाते हैं
  - अधिकतम 4 सत्र; ओवरफ़्लो पर सबसे पुराना निष्कासित
  - हर 30 सेकंड निष्क्रिय/मृत सफाई, 5 मिनट के बाद टाइमआउट
  - प्रति-सत्र कतार निष्पादन को क्रमबद्ध करती है (`session.queue`)
- `per-call`
  - अनुरोध के लिए कर्नेल बनाता है
  - निष्पादित करता है
  - `finally` में हमेशा कर्नेल बंद करता है

## रीसेट व्यवहार

`python` टूल मल्टी-सेल कॉल में केवल पहले सेल के लिए `reset` पास करता है; बाद के सेल हमेशा `reset: false` के साथ चलते हैं।

## कर्नेल मृत्यु / पुनरारंभ / पुनर्प्रयास

सत्र मोड (`withKernelSession`) में:

- मृत कर्नेल का पता हार्टबीट (`kernel.isAlive()` हर 5 सेकंड जाँच) या निष्पादन विफलता द्वारा लगाया जाता है।
- चलाने से पहले मृत स्थिति `restartKernelSession` को ट्रिगर करती है।
- निष्पादन-समय क्रैश पथ एक बार पुनर्प्रयास करता है: कर्नेल पुनरारंभ, हैंडलर पुनः चलाएँ।
- उसी सत्र में `restartCount > 1` `Python kernel restarted too many times in this session` फेंकता है।

स्टार्टअप पुनर्प्रयास व्यवहार:

- साझा गेटवे कर्नेल निर्माण HTTP 5xx के साथ `SharedGatewayCreateError` पर एक बार पुनर्प्रयास करता है।

संसाधन थकान पुनर्प्राप्ति:

- `EMFILE`/`ENFILE`/"Too many open files" शैली विफलताओं का पता लगाता है
- ट्रैक किए गए सत्रों को साफ़ करता है
- `shutdownSharedGateway()` कॉल करता है
- कर्नेल सत्र निर्माण एक बार पुनर्प्रयास करता है

## 4) पर्यावरण/सत्र चर इंजेक्शन

कर्नेल स्टार्टअप को निष्पादक से वैकल्पिक env मैप प्राप्त होता है:

- `PI_SESSION_FILE` (सत्र स्थिति फ़ाइल पथ)
- `ARTIFACTS` (आर्टिफैक्ट डायरेक्टरी)

`PythonKernel.#initializeKernelEnvironment(...)` फिर कर्नेल के अंदर init स्क्रिप्ट चलाता है:

- `os.chdir(cwd)`
- `os.environ` में env प्रविष्टियाँ इंजेक्ट करता है
- यदि गुम हो तो `sys.path` में cwd को प्रारंभ में जोड़ता है

निहितार्थ:

- सत्र या आर्टिफैक्ट संदर्भ पढ़ने वाले प्रस्तावना सहायक Python प्रक्रिया स्थिति में इन env वेरिएबल्स पर निर्भर करते हैं।

## 5) स्ट्रीमिंग/चंक और डिस्प्ले हैंडलिंग (कर्नेल-समर्थित पथ)

कर्नेल क्लाइंट प्रति निष्पादन Jupyter प्रोटोकॉल संदेशों को संसाधित करता है:

- `stream` -> `onChunk` को टेक्स्ट चंक
- `execute_result` / `display_data` ->
  - MIME प्राथमिकता द्वारा डिस्प्ले टेक्स्ट चुना जाता है: `text/markdown` > `text/plain` > परिवर्तित `text/html`
  - संरचित आउटपुट अलग से कैप्चर किए जाते हैं:
    - `application/json` -> `{ type: "json" }`
    - `image/png` -> `{ type: "image" }`
    - `application/x-xcsh-status` -> `{ type: "status" }` (कोई टेक्स्ट उत्सर्जन नहीं)
- `error` -> ट्रेसबैक टेक्स्ट चंक स्ट्रीम में धकेला जाता है + संरचित त्रुटि मेटाडेटा
- `input_request` -> stdin चेतावनी टेक्स्ट उत्सर्जित करता है, खाली `input_reply` भेजता है, stdin अनुरोधित चिह्नित करता है
- पूर्णता `execute_reply` और कर्नेल `status=idle` दोनों की प्रतीक्षा करती है

रद्दीकरण/टाइमआउट:

- एबॉर्ट सिग्नल `interrupt()` को ट्रिगर करता है (REST `/interrupt` + कंट्रोल-चैनल `interrupt_request`)
- परिणाम `cancelled=true` चिह्नित करता है
- टाइमआउट पथ आउटपुट को `Command timed out after <n> seconds` के साथ एनोटेट करता है

## 6) ट्रंकेशन और आर्टिफैक्ट व्यवहार

`src/session/streaming-output.ts` में `OutputSink` कर्नेल निष्पादन पथों (`executeWithKernel`) द्वारा उपयोग किया जाता है:

- प्रत्येक चंक को सैनिटाइज़ करता है (`sanitizeText`)
- कुल/आउटपुट पंक्तियों और बाइट्स को ट्रैक करता है
- वैकल्पिक आर्टिफैक्ट स्पिल फ़ाइल (`artifactPath`, `artifactId`)
- जब इन-मेमोरी बफर सीमा से अधिक हो जाता है (जब तक ओवरराइड न हो `DEFAULT_MAX_BYTES`):
  - ट्रंकेटेड चिह्नित करता है
  - मेमोरी में टेल बाइट्स रखता है (UTF-8 सुरक्षित सीमा)
  - पूर्ण स्ट्रीम को आर्टिफैक्ट सिंक में स्पिल कर सकता है

`dump()` लौटाता है:

- दृश्यमान आउटपुट टेक्स्ट (संभवतः टेल-ट्रंकेटेड)
- ट्रंकेशन फ़्लैग + काउंट
- आर्टिफैक्ट ID (`artifact://<id>` संदर्भों के लिए)

`python` टूल इस मेटाडेटा को परिणाम ट्रंकेशन सूचनाओं और TUI चेतावनियों में परिवर्तित करता है।

`notebook` टूल `OutputSink` का उपयोग **नहीं** करता; इसके पास कोई स्ट्रीम/आर्टिफैक्ट ट्रंकेशन पाइपलाइन नहीं है क्योंकि यह कोड निष्पादित नहीं करता।

## 7) रेंडरर धारणाएँ और स्वरूपण

## नोटबुक रेंडरर (`notebookToolRenderer`)

- कॉल दृश्य: क्रिया + नोटबुक पथ + सेल/प्रकार मेटाडेटा के साथ स्थिति पंक्ति
- परिणाम दृश्य:
  - `details` से प्राप्त सफलता सारांश
  - `cellSource` `renderCodeCell` के माध्यम से रेंडर किया जाता है
  - मार्कडाउन सेल भाषा संकेत `markdown` सेट करते हैं; अन्य सेल में कोई स्पष्ट भाषा ओवरराइड नहीं
  - संक्षिप्त कोड पूर्वावलोकन सीमा `PREVIEW_LIMITS.COLLAPSED_LINES * 2` है
  - साझा रेंडर विकल्पों के माध्यम से विस्तारित मोड का समर्थन करता है
  - चौड़ाई + विस्तारित स्थिति द्वारा कुंजीकृत रेंडर कैश का उपयोग करता है

त्रुटि रेंडरिंग धारणा:

- यदि पहला टेक्स्ट कंटेंट `Error:` से शुरू होता है, तो रेंडरर नोटबुक त्रुटि ब्लॉक के रूप में स्वरूपित करता है।

## Python रेंडरर (वास्तविक निष्पादन आउटपुट के लिए)

कर्नेल-समर्थित निष्पादन रेंडरिंग अपेक्षा करता है:

- प्रति-सेल स्थिति संक्रमण (`pending/running/complete/error`)
- वैकल्पिक संरचित स्थिति ईवेंट अनुभाग
- वैकल्पिक JSON आउटपुट ट्री
- ट्रंकेशन चेतावनियाँ + वैकल्पिक `artifact://<id>` पॉइंटर

यह रेंडरर व्यवहार `notebook` JSON संपादन परिणामों से असंबंधित है सिवाय इसके कि दोनों साझा TUI प्रिमिटिव का पुनः उपयोग करते हैं।

## 8) सादे Python टूल व्यवहार से विचलन

यदि "सादा Python टूल" का अर्थ `python` निष्पादन पथ है:

- `python` कर्नेल में कोड निष्पादित करता है, मोड द्वारा स्थिति बनाए रखता है, चंक स्ट्रीम करता है, रिच डिस्प्ले कैप्चर करता है, इंटरप्ट/टाइमआउट संभालता है, और आउटपुट ट्रंकेशन/आर्टिफैक्ट का समर्थन करता है।
- `notebook` केवल नियतात्मक नोटबुक JSON म्यूटेशन करता है; कोई निष्पादन नहीं, कोई कर्नेल स्थिति नहीं, कोई चंक स्ट्रीम नहीं, कोई डिस्प्ले आउटपुट नहीं, कोई आर्टिफैक्ट पाइपलाइन नहीं।

यदि किसी वर्कफ़्लो को दोनों की आवश्यकता है:

1. `notebook` के साथ नोटबुक स्रोत संपादित करें
2. `python` के माध्यम से कोड सेल निष्पादित करें (मैन्युअल रूप से कोड पास करके), `notebook` के माध्यम से नहीं

वर्तमान कार्यान्वयन एक एकल टूल प्रदान नहीं करता जो `.ipynb` को म्यूटेट भी करे और कर्नेल संदर्भ के माध्यम से नोटबुक सेल निष्पादित भी करे।
