# नोटबुक टूल रनटाइम आंतरिक संरचना

यह दस्तावेज़ वर्तमान `notebook` टूल के कार्यान्वयन और कर्नेल-समर्थित Python रनटाइम के साथ इसके संबंध का वर्णन करता है।

महत्वपूर्ण अंतर: **`notebook` एक JSON नोटबुक संपादक है, न कि नोटबुक निष्पादक**। यह `.ipynb` सेल स्रोतों को सीधे संपादित करता है; यह Python कर्नेल को आरंभ नहीं करता और न ही उससे संवाद करता है।

## कार्यान्वयन फ़ाइलें

- [`src/tools/notebook.ts`](../../packages/coding-agent/src/tools/notebook.ts)
- [`src/ipy/executor.ts`](../../packages/coding-agent/src/ipy/executor.ts)
- [`src/ipy/kernel.ts`](../../packages/coding-agent/src/ipy/kernel.ts)
- [`src/session/streaming-output.ts`](../../packages/coding-agent/src/session/streaming-output.ts)
- [`src/tools/python.ts`](../../packages/coding-agent/src/tools/python.ts)

## 1) रनटाइम सीमा: संपादन बनाम निष्पादन

## `notebook` टूल (`src/tools/notebook.ts`)

- `.ipynb` फ़ाइल पर `action: edit | insert | delete` का समर्थन करता है।
- सेशन CWD (`resolveToCwd`) के सापेक्ष पथ को हल करता है।
- नोटबुक JSON लोड करता है, `cells` सरणी को सत्यापित करता है, `cell_index` सीमाओं को सत्यापित करता है।
- स्मृति में स्रोत संपादन लागू करता है और `JSON.stringify(notebook, null, 1)` के साथ पूर्ण नोटबुक JSON वापस लिखता है।
- पाठ्य सारांश + संरचित `details` (`action`, `cellIndex`, `cellType`, `totalCells`, `cellSource`) लौटाता है।

इस टूल में कोई कर्नेल जीवनचक्र मौजूद नहीं है:

- कोई गेटवे अधिग्रहण नहीं
- कोई कर्नेल सेशन ID नहीं
- कोई `execute_request` नहीं
- कर्नेल चैनलों से कोई स्ट्रीम खंड नहीं
- कोई रिच डिस्प्ले कैप्चर नहीं (`image/png`, JSON डिस्प्ले, status MIME)

## नोटबुक-जैसा निष्पादन पथ (`src/tools/python.ts` + `src/ipy/*`)

जब एजेंट को सेल-स्टाइल Python कोड चलाने की आवश्यकता होती है (अनुक्रमिक सेल, स्थायी स्थिति, रिच डिस्प्ले), तो वह **`python` टूल** के माध्यम से जाता है, न कि `notebook` के।

कर्नेल मोड, restart/cancel व्यवहार, खंड स्ट्रीमिंग और आउटपुट आर्टिफैक्ट ट्रंकेशन उसी पथ में रहते हैं।

## 2) नोटबुक सेल हैंडलिंग की अर्थार्थता (`notebook` टूल)

## स्रोत सामान्यीकरण

`content` को `source: string[]` में विभाजित किया जाता है, जिसमें नई पंक्तियाँ संरक्षित रहती हैं:

- प्रत्येक गैर-अंतिम पंक्ति में अनुगामी `\n` बनी रहती है
- अंतिम पंक्ति में कोई बाध्य अनुगामी नई पंक्ति नहीं होती

यह नोटबुक JSON परंपराओं को प्रतिबिंबित करता है और बाद के संपादनों में आकस्मिक पंक्ति संयोजन से बचाता है।

## क्रिया व्यवहार

- `edit`
  - `cells[cell_index].source` को प्रतिस्थापित करता है
  - मौजूदा `cell_type` को संरक्षित करता है
- `insert`
  - `[0..cellCount]` पर सम्मिलित करता है
  - `cell_type` का डिफ़ॉल्ट `code` होता है
  - कोड सेल `execution_count: null` और `outputs: []` से आरंभ होते हैं
  - मार्कडाउन सेल केवल `metadata` + `source` से आरंभ होते हैं
- `delete`
  - `cells[cell_index]` हटाता है
  - रेंडरर पूर्वावलोकन के लिए विवरण में हटाए गए `source` को लौटाता है

## त्रुटि सतहें

निम्नलिखित के लिए हार्ड विफलताएँ फेंकी जाती हैं:

- नोटबुक फ़ाइल अनुपस्थित
- अमान्य JSON
- अनुपस्थित/गैर-सरणी `cells`
- सीमा से बाहर इंडेक्स (insert और गैर-insert के लिए अलग-अलग वैध श्रेणियाँ)
- `edit`/`insert` के लिए `content` अनुपस्थित

ये अपस्ट्रीम `Error:` टूल प्रतिक्रियाएँ बन जाती हैं; रेंडरर नोटबुक पथ + स्वरूपित त्रुटि पाठ का उपयोग करता है।

## 3) कर्नेल सेशन की अर्थार्थता (जहाँ वे वास्तव में मौजूद हैं)

कर्नेल अर्थार्थता `executePython` / `PythonKernel` में कार्यान्वित की जाती है और `python` टूल पर लागू होती है।

## मोड

`PythonKernelMode`:

- `session` (डिफ़ॉल्ट)
  - कर्नेल `kernelSessions` मैप में कैश किए जाते हैं
  - अधिकतम 4 सेशन; अतिप्रवाह पर सबसे पुराना निष्कासित
  - हर 30 सेकंड में idle/dead सफाई, 5 मिनट के बाद टाइमआउट
  - प्रति-सेशन कतार निष्पादन को क्रमबद्ध करती है (`session.queue`)
- `per-call`
  - अनुरोध के लिए कर्नेल बनाता है
  - निष्पादित करता है
  - `finally` में कर्नेल को हमेशा बंद करता है

## रीसेट व्यवहार

`python` टूल `reset` केवल बहु-सेल कॉल में पहले सेल के लिए पास करता है; बाद के सेल हमेशा `reset: false` के साथ चलते हैं।

## कर्नेल की मृत्यु / पुनरारंभ / पुनः प्रयास

सेशन मोड में (`withKernelSession`):

- मृत कर्नेल का पता हार्टबीट (`kernel.isAlive()` हर 5 सेकंड में जाँच) या निष्पादन विफलता से होता है।
- पूर्व-चलाव मृत अवस्था `restartKernelSession` को ट्रिगर करती है।
- निष्पादन-समय क्रैश पथ एक बार पुनः प्रयास करता है: कर्नेल पुनरारंभ करें, हैंडलर पुनः चलाएँ।
- एक ही सेशन में `restartCount > 1` `Python kernel restarted too many times in this session` फेंकता है।

स्टार्टअप पुनः प्रयास व्यवहार:

- साझा गेटवे कर्नेल निर्माण HTTP 5xx के साथ `SharedGatewayCreateError` पर एक बार पुनः प्रयास करता है।

संसाधन थकावट पुनर्प्राप्ति:

- `EMFILE`/`ENFILE`/"Too many open files" शैली की विफलताओं का पता लगाता है
- ट्रैक किए गए सेशन साफ़ करता है
- `shutdownSharedGateway()` कॉल करता है
- कर्नेल सेशन निर्माण एक बार पुनः प्रयास करता है

## 4) पर्यावरण/सेशन चर इंजेक्शन

कर्नेल स्टार्टअप एक्ज़ीक्यूटर से वैकल्पिक env मैप प्राप्त करता है:

- `PI_SESSION_FILE` (सेशन स्थिति फ़ाइल पथ)
- `ARTIFACTS` (आर्टिफैक्ट डायरेक्टरी)

`PythonKernel.#initializeKernelEnvironment(...)` फिर कर्नेल के अंदर init स्क्रिप्ट चलाता है:

- `os.chdir(cwd)`
- `os.environ` में env प्रविष्टियाँ इंजेक्ट करें
- यदि अनुपस्थित हो तो cwd को `sys.path` में प्रारंभ में जोड़ें

निहितार्थ:

- सेशन या आर्टिफैक्ट संदर्भ पढ़ने वाले प्रस्तावना सहायक Python प्रक्रिया स्थिति में इन env vars पर निर्भर करते हैं।

## 5) स्ट्रीमिंग/खंड और डिस्प्ले हैंडलिंग (कर्नेल-समर्थित पथ)

कर्नेल क्लाइंट प्रति निष्पादन Jupyter प्रोटोकॉल संदेशों को संसाधित करता है:

- `stream` -> `onChunk` को पाठ खंड
- `execute_result` / `display_data` ->
  - MIME प्राथमिकता द्वारा चुना गया डिस्प्ले पाठ: `text/markdown` > `text/plain` > परिवर्तित `text/html`
  - संरचित आउटपुट अलग से कैप्चर किए जाते हैं:
    - `application/json` -> `{ type: "json" }`
    - `image/png` -> `{ type: "image" }`
    - `application/x-xcsh-status` -> `{ type: "status" }` (कोई पाठ उत्सर्जन नहीं)
- `error` -> ट्रेसबैक पाठ खंड स्ट्रीम में पुश + संरचित त्रुटि मेटाडेटा
- `input_request` -> stdin चेतावनी पाठ उत्सर्जित करता है, खाली `input_reply` भेजता है, stdin अनुरोधित चिह्नित करता है
- पूर्णता `execute_reply` और कर्नेल `status=idle` दोनों की प्रतीक्षा करती है

रद्दीकरण/टाइमआउट:

- abort सिग्नल `interrupt()` को ट्रिगर करता है (REST `/interrupt` + control-channel `interrupt_request`)
- परिणाम `cancelled=true` चिह्नित करता है
- टाइमआउट पथ आउटपुट को `Command timed out after <n> seconds` के साथ अनुसूचित करता है

## 6) ट्रंकेशन और आर्टिफैक्ट व्यवहार

`src/session/streaming-output.ts` में `OutputSink` का उपयोग कर्नेल निष्पादन पथों (`executeWithKernel`) द्वारा किया जाता है:

- प्रत्येक खंड को स्वच्छ करता है (`sanitizeText`)
- कुल/आउटपुट पंक्तियाँ और बाइट्स ट्रैक करता है
- वैकल्पिक आर्टिफैक्ट स्पिल फ़ाइल (`artifactPath`, `artifactId`)
- जब इन-मेमोरी बफर सीमा से अधिक हो (`DEFAULT_MAX_BYTES` जब तक अधिलेखित न हो):
  - ट्रंकेटेड चिह्नित करता है
  - मेमोरी में टेल बाइट्स रखता है (UTF-8 सुरक्षित सीमा)
  - पूर्ण स्ट्रीम को आर्टिफैक्ट सिंक पर स्पिल कर सकता है

`dump()` लौटाता है:

- दृश्यमान आउटपुट पाठ (संभवतः टेल-ट्रंकेटेड)
- ट्रंकेशन ध्वज + गणना
- आर्टिफैक्ट ID (`artifact://<id>` संदर्भों के लिए)

`python` टूल इस मेटाडेटा को परिणाम ट्रंकेशन सूचनाओं और TUI चेतावनियों में बदलता है।

`notebook` टूल `OutputSink` का उपयोग **नहीं** करता; इसमें कोई स्ट्रीम/आर्टिफैक्ट ट्रंकेशन पाइपलाइन नहीं है क्योंकि यह कोड निष्पादित नहीं करता।

## 7) रेंडरर मान्यताएँ और स्वरूपण

## नोटबुक रेंडरर (`notebookToolRenderer`)

- कॉल दृश्य: action + नोटबुक पथ + सेल/प्रकार मेटाडेटा के साथ स्थिति पंक्ति
- परिणाम दृश्य:
  - `details` से प्राप्त सफलता सारांश
  - `cellSource` को `renderCodeCell` के माध्यम से प्रस्तुत किया गया
  - मार्कडाउन सेल भाषा संकेत `markdown` सेट करते हैं; अन्य सेल में कोई स्पष्ट भाषा ओवरराइड नहीं
  - संकुचित कोड पूर्वावलोकन सीमा `PREVIEW_LIMITS.COLLAPSED_LINES * 2` है
  - साझा रेंडर विकल्पों के माध्यम से विस्तारित मोड का समर्थन करता है
  - चौड़ाई + विस्तारित स्थिति द्वारा कुंजीबद्ध रेंडर कैश का उपयोग करता है

त्रुटि रेंडरिंग मान्यता:

- यदि पहली पाठ सामग्री `Error:` से शुरू होती है, तो रेंडरर नोटबुक त्रुटि ब्लॉक के रूप में स्वरूपित करता है।

## Python रेंडरर (वास्तविक निष्पादन आउटपुट के लिए)

कर्नेल-समर्थित निष्पादन रेंडरिंग की अपेक्षाएँ:

- प्रति-सेल स्थिति संक्रमण (`pending/running/complete/error`)
- वैकल्पिक संरचित स्थिति इवेंट अनुभाग
- वैकल्पिक JSON आउटपुट ट्री
- ट्रंकेशन चेतावनियाँ + वैकल्पिक `artifact://<id>` पॉइंटर

यह रेंडरर व्यवहार `notebook` JSON संपादन परिणामों से असंबंधित है, सिवाय इसके कि दोनों साझा TUI प्रिमिटिव का पुनः उपयोग करते हैं।

## 8) सादे Python टूल व्यवहार से विचलन

यदि "सादे Python टूल" का अर्थ `python` निष्पादन पथ है:

- `python` एक कर्नेल में कोड निष्पादित करता है, मोड के अनुसार स्थिति बनाए रखता है, खंड स्ट्रीम करता है, रिच डिस्प्ले कैप्चर करता है, interrupts/timeouts संभालता है, और आउटपुट ट्रंकेशन/आर्टिफैक्ट का समर्थन करता है।
- `notebook` केवल निर्धारक नोटबुक JSON म्यूटेशन करता है; कोई निष्पादन नहीं, कोई कर्नेल स्थिति नहीं, कोई खंड स्ट्रीम नहीं, कोई डिस्प्ले आउटपुट नहीं, कोई आर्टिफैक्ट पाइपलाइन नहीं।

यदि किसी वर्कफ़्लो को दोनों की आवश्यकता है:

1. `notebook` के साथ नोटबुक स्रोत संपादित करें
2. `python` के माध्यम से कोड सेल निष्पादित करें (मैन्युअल रूप से कोड पास करते हुए), `notebook` के माध्यम से नहीं

वर्तमान कार्यान्वयन एक ऐसा एकल टूल प्रदान नहीं करता जो `.ipynb` को म्यूटेट भी करे और कर्नेल संदर्भ के माध्यम से नोटबुक सेल निष्पादित भी करे।
