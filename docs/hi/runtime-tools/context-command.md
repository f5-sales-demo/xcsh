# F5 XC Contexts

xcsh **contexts** के माध्यम से F5 Distributed Cloud से कनेक्ट होता है -- ये नामित क्रेडेंशियल सेट हैं जो एक टेनेंट URL, API टोकन और नेमस्पेस को बाइंड करते हैं। यदि आपने `kubectl config use-context` या `kubectx` का उपयोग किया है, तो वर्कफ़्लो बिल्कुल वही है: एक कॉन्टेक्स्ट बनाएं, नाम से उनके बीच स्विच करें, और वापस फ़्लिप करने के लिए `-` का उपयोग करें।

## शुरू करना

### 1. अपना पहला कॉन्टेक्स्ट बनाएं

आपको अपने F5 XC कंसोल से तीन चीज़ें चाहिए: टेनेंट URL, एक API टोकन, और वैकल्पिक रूप से एक नेमस्पेस।

```
/context create production https://acme.console.ves.volterra.io p12k3-your-api-token
```

```
Context 'production' created. Use /context activate production to switch to it.
```

या यदि आप चरण-दर-चरण प्रॉम्प्ट पसंद करते हैं तो गाइडेड विज़ार्ड का उपयोग करें:

```
/context wizard
```

### 2. इसे एक्टिवेट करें

```
/context production
```

```
╭─ production ─────────────────────────────────────────────────╮
│ F5XC_TENANT     acme                                         │
│ F5XC_API_URL    https://acme.console.ves.volterra.io         │
│ F5XC_API_TOKEN  ...oken                                      │
│ Status          Connected (312ms)                            │
├─ Environment ────────────────────────────────────────────────┤
│ F5XC_NAMESPACE  default                                      │
╰──────────────────────────────────────────────────────────────╯
```

एक्टिवेट होने के बाद, xcsh आपके सेशन में टेनेंट क्रेडेंशियल्स इंजेक्ट करता है। एजेंट अब F5 XC API कॉल कर सकता है, और स्टेटस लाइन सक्रिय कॉन्टेक्स्ट दिखाती है।

### 3. और कॉन्टेक्स्ट जोड़ें और उनके बीच स्विच करें

```
/context create staging https://staging.console.ves.volterra.io p12k3-staging-token
```

नाम से स्विच करें -- किसी सबकमांड वर्ब की ज़रूरत नहीं:

```
/context staging
```

पिछले कॉन्टेक्स्ट पर वापस स्विच करें (`cd -` शैली):

```
/context -
```

`/context -` को दो बार कॉल करने पर आप वापस वहीं पहुँच जाते हैं जहाँ से शुरू किया था।

### 4. देखें आपके पास क्या है

```
/context
```

```
  production           https://acme.console.ves.volterra.io
* staging              https://staging.console.ves.volterra.io
```

`*` सक्रिय कॉन्टेक्स्ट को चिह्नित करता है।

## रोज़मर्रा के कमांड

| कमांड | यह क्या करता है |
|---|---|
| `/context` | सभी कॉन्टेक्स्ट की सूची दिखाएं |
| `/context <name>` | किसी कॉन्टेक्स्ट पर स्विच करें |
| `/context -` | पिछले कॉन्टेक्स्ट पर स्विच करें |
| `/context show` | सक्रिय कॉन्टेक्स्ट का विवरण दिखाएं (टोकन मास्क किए हुए) |
| `/context status` | वर्तमान ऑथ स्टेटस दिखाएं |

## कॉन्टेक्स्ट जीवनचक्र

| कमांड | यह क्या करता है |
|---|---|
| `/context create <name> <url> <token> [namespace]` | कॉन्टेक्स्ट बनाएं |
| `/context delete <name> --confirm` | कॉन्टेक्स्ट हटाएं (`--confirm` आवश्यक है) |
| `/context rename <old> <new>` | कॉन्टेक्स्ट का नाम बदलें |
| `/context validate <name>` | स्विच किए बिना क्रेडेंशियल्स का परीक्षण करें |
| `/context export [name] [--include-token]` | JSON के रूप में एक्सपोर्ट करें (डिफ़ॉल्ट रूप से टोकन मास्क किए हुए) |
| `/context import <path-or-json> [--overwrite]` | फ़ाइल या इनलाइन JSON से इम्पोर्ट करें |
| `/context wizard` | गाइडेड इंटरैक्टिव सेटअप |

## नेमस्पेस स्विच करना

प्रत्येक कॉन्टेक्स्ट का एक डिफ़ॉल्ट नेमस्पेस होता है। कॉन्टेक्स्ट बदले बिना इसे स्विच करें:

```
/context namespace system
```

टैब कम्प्लीशन सक्रिय टेनेंट से नेमस्पेस नाम प्रदान करता है।

## कॉन्टेक्स्ट पर एनवायरनमेंट वेरिएबल

कॉन्टेक्स्ट अतिरिक्त एनवायरनमेंट वेरिएबल ले जा सकते हैं जो एक्टिवेशन पर आपके सेशन में इंजेक्ट किए जाते हैं। प्रति-टेनेंट कॉन्फ़िगरेशन के लिए उपयोगी जो क्रेडेंशियल सेट का हिस्सा नहीं है।

```
/context set CUSTOM_HEADER=x-acme-trace
/context set LOG_LEVEL=debug
/context env list
/context unset LOG_LEVEL
```

उपनाम: `add` = `set`, `remove`/`clear` = `unset`।

## टैब कम्प्लीशन

`/context ` टाइप करें और Tab दबाएं। ड्रॉपडाउन दिखाता है:

1. **कॉन्टेक्स्ट नाम** -- टेनेंट URL संकेतों के साथ, ताकि आप टेनेंट्स को अलग-अलग पहचान सकें
2. **`-`** -- तब दिखाई देता है जब आपने पहले स्विच किया हो, दिखाता है कि आप किस कॉन्टेक्स्ट पर फ़्लिप होंगे
3. **सबकमांड** -- `list`, `create`, `delete`, आदि।

कॉन्टेक्स्ट नाम पहले दिखाई देते हैं क्योंकि स्विच करना सबसे सामान्य क्रिया है।

सबकमांड-स्तरीय कम्प्लीशन भी काम करते हैं: `/context activate <Tab>` कॉन्टेक्स्ट नाम पूरे करता है, `/context namespace <Tab>` नेमस्पेस पूरे करता है, `/context unset <Tab>` ज्ञात env var कुंजियों को पूरा करता है।

## नामकरण नियम

कॉन्टेक्स्ट नाम 1-64 अक्षर होने चाहिए: अक्षर, अंक, हाइफ़न, अंडरस्कोर।

जो नाम सबकमांड से टकराते हैं, उन्हें अस्वीकार कर दिया जाता है:

```
/context create list https://example.com tok
```

```
Error: Context name 'list' conflicts with a /context subcommand. Choose a different name.
```

पूर्ण आरक्षित सेट: `list`, `show`, `status`, `create`, `delete`, `rename`, `namespace`, `env`, `set`, `unset`, `add`, `remove`, `clear`, `activate`, `validate`, `export`, `import`, `wizard`, `help`। तुलना केस-इनसेंसिटिव है।

## एनवायरनमेंट वेरिएबल ओवरराइड

यदि xcsh लॉन्च करने से पहले आपके शेल एनवायरनमेंट में `F5XC_API_URL` और `F5XC_API_TOKEN` सेट हैं, तो वे किसी भी कॉन्टेक्स्ट पर प्राथमिकता लेते हैं। यह CI/CD पाइपलाइन या एक-बार के सेशन के लिए उपयोगी है जहाँ आप एक स्थायी कॉन्टेक्स्ट नहीं बनाना चाहते।

इस मोड में चलने पर, `/context` एनवायरनमेंट-सोर्स्ड क्रेडेंशियल्स को `(via env vars)` लेबल के साथ दिखाता है।

## पिछले कॉन्टेक्स्ट का व्यवहार

- **सेशन-स्कोप्ड**: जब आप xcsh रीस्टार्ट करते हैं तो पिछला कॉन्टेक्स्ट रीसेट हो जाता है। इसे डिस्क पर पर्सिस्ट नहीं किया जाता।
- **पिंग-पोंग**: `/context -` दो बार आपको वहीं वापस लाता है जहाँ से शुरू किया था।
- **म्यूटेशन में सुरक्षित**: यदि आप पिछले कॉन्टेक्स्ट को हटाते हैं, तो पॉइंटर साफ़ हो जाता है। यदि आप इसका नाम बदलते हैं, तो पॉइंटर नए नाम का अनुसरण करता है।
- **पुनः-एक्टिवेशन नो-ऑप है**: जब पहले से `production` पर हों तो `/context production` पिछले पॉइंटर को रीसेट नहीं करता।

## डिज़ाइन सिद्धांत

`/context` UX इनका अनुसरण करता है:

- **kubectx**: स्विच करने के लिए `kubectx <name>`, पिछले के लिए `kubectx -`, सूची के लिए बिना तर्क `kubectx`
- **kubectl**: स्पष्ट रूप के लिए `kubectl config use-context`
- **Shell**: पिछली-डायरेक्टरी ट्रैकिंग के लिए `cd -` / `OLDPWD`
