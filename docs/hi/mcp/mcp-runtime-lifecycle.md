# MCP रनटाइम जीवनचक्र

यह दस्तावेज़ वर्णन करता है कि MCP सर्वर कोडिंग-एजेंट रनटाइम में कैसे खोजे, कनेक्ट, टूल के रूप में उजागर, रिफ्रेश और बंद किए जाते हैं।

## जीवनचक्र एक नज़र में

1. **SDK स्टार्टअप** `discoverAndLoadMCPTools()` को कॉल करता है (जब तक MCP अक्षम न हो)।
2. **खोज** (`loadAllMCPConfigs`) कैपेबिलिटी स्रोतों से MCP सर्वर कॉन्फिग हल करता है, अक्षम/प्रोजेक्ट/Exa प्रविष्टियों को फ़िल्टर करता है, और स्रोत मेटाडेटा संरक्षित करता है।
3. **मैनेजर कनेक्ट चरण** (`MCPManager.connectServers`) प्रति-सर्वर कनेक्ट + `tools/list` को समानांतर में शुरू करता है।
4. **फास्ट स्टार्टअप गेट** अधिकतम 250ms प्रतीक्षा करता है, फिर लौटा सकता है:
   - पूर्ण रूप से लोड `MCPTool`s,
   - प्रति सर्वर विफलताएं,
   - या अभी-भी-लंबित सर्वरों के लिए कैश्ड `DeferredMCPTool`s।
5. **SDK वायरिंग** MCP टूल्स को सत्र के रनटाइम टूल रजिस्ट्री में मर्ज करता है।
6. **लाइव सत्र** `/mcp` फ्लो (`disconnectAll` + पुनर्खोज + `session.refreshMCPTools`) के माध्यम से MCP टूल्स को रिफ्रेश कर सकता है।
7. **टियरडाउन** तब होता है जब कॉलर `disconnectServer`/`disconnectAll` को इनवोक करते हैं; मैनेजर डिस्कनेक्ट किए गए सर्वरों के MCP टूल पंजीकरण भी साफ़ करता है।

## खोज और लोड चरण

### SDK से प्रवेश पथ

`src/sdk.ts` में `createAgentSession()` MCP स्टार्टअप तब करता है जब `enableMCP` true हो (डिफ़ॉल्ट):

- `discoverAndLoadMCPTools(cwd, { ... })` को कॉल करता है,
- `authStorage`, कैश स्टोरेज, और `mcp.enableProjectConfig` सेटिंग पास करता है,
- हमेशा `filterExa: true` सेट करता है,
- प्रति-सर्वर लोड/कनेक्ट त्रुटियों को लॉग करता है,
- लौटाए गए मैनेजर को `toolSession.mcpManager` और सत्र परिणाम में संग्रहीत करता है।

यदि `enableMCP` false है, तो MCP खोज पूरी तरह से छोड़ दी जाती है।

### कॉन्फिग खोज और फ़िल्टरिंग

`loadAllMCPConfigs()` (`src/mcp/config.ts`) कैपेबिलिटी खोज के माध्यम से कैनोनिकल MCP सर्वर आइटम लोड करता है, फिर लेगेसी `MCPServerConfig` में परिवर्तित करता है।

फ़िल्टरिंग व्यवहार:

- `enableProjectConfig: false` प्रोजेक्ट-स्तर की प्रविष्टियों (`_source.level === "project"`) को हटाता है।
- `enabled: false` सर्वरों को कनेक्ट प्रयासों से पहले छोड़ दिया जाता है।
- Exa सर्वरों को डिफ़ॉल्ट रूप से फ़िल्टर किया जाता है और नेटिव Exa टूल एकीकरण के लिए API कुंजियाँ निकाली जाती हैं।

परिणाम में `configs` और `sources` (मेटाडेटा जो बाद में प्रदाता लेबलिंग के लिए उपयोग होता है) दोनों शामिल होते हैं।

### खोज-स्तर विफलता व्यवहार

`discoverAndLoadMCPTools()` दो विफलता वर्गों में अंतर करता है:

- **खोज कठिन विफलता** (`manager.discoverAndConnect` से अपवाद, आमतौर पर कॉन्फिग खोज से): एक खाली टूल सेट और एक सिंथेटिक त्रुटि `{ path: ".mcp.json", error }` लौटाता है।
- **प्रति-सर्वर रनटाइम/कनेक्ट विफलता**: मैनेजर `errors` मैप के साथ आंशिक सफलता लौटाता है; अन्य सर्वर जारी रहते हैं।

इसलिए जब व्यक्तिगत MCP सर्वर विफल होते हैं तो स्टार्टअप पूरे एजेंट सत्र को विफल नहीं करता।

## मैनेजर स्टेट मॉडल

`MCPManager` अलग-अलग रजिस्ट्रियों के साथ रनटाइम जीवनचक्र को ट्रैक करता है:

- `#connections: Map<string, MCPServerConnection>` — पूर्ण रूप से कनेक्टेड सर्वर।
- `#pendingConnections: Map<string, Promise<MCPServerConnection>>` — हैंडशेक प्रगति में।
- `#pendingToolLoads: Map<string, Promise<{ connection, serverTools }>>` — कनेक्टेड लेकिन टूल अभी भी लोड हो रहे हैं।
- `#tools: CustomTool[]` — कॉलर्स को उजागर वर्तमान MCP टूल दृश्य।
- `#sources: Map<string, SourceMeta>` — कनेक्ट पूर्ण होने से पहले भी प्रदाता/स्रोत मेटाडेटा।

`getConnectionStatus(name)` इन मैप्स से स्थिति व्युत्पन्न करता है:

- `connected` यदि `#connections` में है,
- `connecting` यदि लंबित कनेक्ट या लंबित टूल लोड में है,
- अन्यथा `disconnected`।

## कनेक्शन स्थापना और स्टार्टअप समय

## प्रति-सर्वर कनेक्ट पाइपलाइन

`connectServers()` में प्रत्येक खोजे गए सर्वर के लिए:

1. स्रोत मेटाडेटा संग्रहीत/अपडेट करें,
2. यदि पहले से कनेक्टेड/लंबित है तो छोड़ें,
3. ट्रांसपोर्ट फ़ील्ड सत्यापित करें (`validateServerConfig`),
4. auth/शेल प्रतिस्थापन हल करें (`#resolveAuthConfig`),
5. `connectToServer(name, resolvedConfig)` कॉल करें,
6. `listTools(connection)` कॉल करें,
7. टूल परिभाषाएं (`MCPToolCache.set`) सर्वोत्तम-प्रयास कैश करें।

`connectToServer()` व्यवहार (`src/mcp/client.ts`):

- stdio या HTTP/SSE ट्रांसपोर्ट बनाता है,
- MCP `initialize` + `notifications/initialized` करता है,
- टाइमआउट (`config.timeout` या 30s डिफ़ॉल्ट) का उपयोग करता है,
- init विफलता पर ट्रांसपोर्ट बंद करता है।

### फास्ट स्टार्टअप गेट + डिफ़र्ड फ़ॉलबैक

`connectServers()` इनके बीच एक रेस पर प्रतीक्षा करता है:

- सभी कनेक्ट/टूल-लोड कार्य पूर्ण, और
- `STARTUP_TIMEOUT_MS = 250`।

250ms के बाद:

- पूर्ण हुए कार्य लाइव `MCPTool`s बनते हैं,
- अस्वीकृत कार्य प्रति-सर्वर त्रुटियाँ उत्पन्न करते हैं,
- अभी-भी-लंबित कार्य:
  - यदि उपलब्ध हो तो कैश्ड टूल परिभाषाओं (`MCPToolCache.get`) का उपयोग करके `DeferredMCPTool`s बनाते हैं,
  - अन्यथा उन लंबित कार्यों के पूर्ण होने तक प्रतीक्षा करते हैं।

यह एक हाइब्रिड स्टार्टअप मॉडल है: जब कैश उपलब्ध हो तो तेज़ वापसी, जब कैश न हो तो शुद्धता के लिए प्रतीक्षा।

### पृष्ठभूमि पूर्णता व्यवहार

प्रत्येक लंबित `toolsPromise` में एक पृष्ठभूमि निरंतरता भी होती है जो अंततः:

- `#replaceServerTools` के माध्यम से मैनेजर स्टेट में उस सर्वर के टूल स्लाइस को प्रतिस्थापित करती है,
- कैश लिखती है,
- स्टार्टअप के बाद ही देर से विफलताओं को लॉग करती है (`allowBackgroundLogging`)।

## टूल उजागर करना और लाइव-सत्र उपलब्धता

### स्टार्टअप पंजीकरण

`discoverAndLoadMCPTools()` मैनेजर टूल्स को `LoadedCustomTool[]` में परिवर्तित करता है और पथों को सजाता है (जब ज्ञात हो तो `mcp:<server> via <providerName>`)।

`createAgentSession()` फिर इन टूल्स को `customTools` में पुश करता है, जो `mcp_<server>_<tool>` जैसे नामों के साथ रैप और रनटाइम टूल रजिस्ट्री में जोड़े जाते हैं।

### टूल कॉल

- `MCPTool` पहले से कनेक्टेड `MCPServerConnection` के माध्यम से टूल कॉल करता है।
- `DeferredMCPTool` कॉल करने से पहले `waitForConnection(server)` की प्रतीक्षा करता है; यह कनेक्शन तैयार होने से पहले कैश्ड टूल्स के अस्तित्व की अनुमति देता है।

दोनों संरचित टूल आउटपुट लौटाते हैं और ट्रांसपोर्ट/टूल त्रुटियों को `MCP error: ...` टूल सामग्री में परिवर्तित करते हैं (एबॉर्ट एबॉर्ट ही रहता है)।

## रिफ्रेश/रीलोड पथ (स्टार्टअप बनाम लाइव रीलोड)

### प्रारंभिक स्टार्टअप पथ

- `sdk.ts` में एक बार की खोज/लोड,
- टूल प्रारंभिक सत्र टूल रजिस्ट्री में पंजीकृत होते हैं।

### इंटरैक्टिव रीलोड पथ

`/mcp reload` पथ (`src/modes/controllers/mcp-command-controller.ts`) करता है:

1. `mcpManager.disconnectAll()`,
2. `mcpManager.discoverAndConnect()`,
3. `session.refreshMCPTools(mcpManager.getTools())`।

`session.refreshMCPTools()` (`src/session/agent-session.ts`) सभी `mcp_` टूल्स को हटाता है, नवीनतम MCP टूल्स को पुनः रैप करता है, और टूल सेट को पुनः सक्रिय करता है ताकि MCP परिवर्तन सत्र पुनः आरंभ किए बिना लागू हों।

देर से कनेक्शन के लिए एक अनुवर्ती पथ भी है: किसी विशिष्ट सर्वर की प्रतीक्षा के बाद, यदि स्थिति `connected` हो जाती है, तो यह `session.refreshMCPTools(...)` पुनः चलाता है ताकि नव उपलब्ध टूल्स सत्र में पुनः बाइंड हो जाएं।

## स्वास्थ्य, पुनः कनेक्ट, और आंशिक विफलता व्यवहार

वर्तमान रनटाइम व्यवहार जानबूझकर न्यूनतम है:

- मैनेजर/क्लाइंट में **कोई स्वायत्त स्वास्थ्य मॉनिटर नहीं**।
- जब ट्रांसपोर्ट गिरता है तो **कोई स्वचालित पुनः कनेक्ट लूप नहीं**।
- मैनेजर ट्रांसपोर्ट `onClose`/`onError` की सदस्यता नहीं लेता; स्थिति रजिस्ट्री-संचालित है।
- पुनः कनेक्ट स्पष्ट है: रीलोड फ्लो या सीधा `connectServers()` इनवोकेशन।

संचालन की दृष्टि से:

- एक सर्वर विफल होने से स्वस्थ सर्वरों से टूल्स नहीं हटते,
- कनेक्ट/लिस्ट विफलताएं प्रति सर्वर पृथक हैं,
- टूल कैश और पृष्ठभूमि अपडेट सर्वोत्तम-प्रयास हैं (चेतावनियाँ/त्रुटियाँ लॉग होती हैं, कोई कठिन रोक नहीं)।

## टियरडाउन शब्दार्थ

### सर्वर-स्तर टियरडाउन

`disconnectServer(name)`:

- लंबित प्रविष्टियाँ/स्रोत मेटाडेटा हटाता है,
- यदि कनेक्टेड है तो ट्रांसपोर्ट बंद करता है,
- मैनेजर स्टेट से उस सर्वर के `mcp_` टूल्स हटाता है।

### ग्लोबल टियरडाउन

`disconnectAll()`:

- `Promise.allSettled` के साथ सभी सक्रिय ट्रांसपोर्ट बंद करता है,
- लंबित मैप, स्रोत, कनेक्शन, और मैनेजर टूल सूची साफ़ करता है।

वर्तमान वायरिंग में, स्पष्ट टियरडाउन MCP कमांड फ्लो (रीलोड/रिमूव/डिसेबल के लिए) में उपयोग होता है। स्टार्टअप पथ में कोई अलग स्वचालित मैनेजर डिस्पोज़ल हुक नहीं है; कॉलर्स ज़िम्मेदार हैं कि जब उन्हें निर्धारक MCP शटडाउन की आवश्यकता हो तो मैनेजर डिस्कनेक्ट मेथड्स को इनवोक करें।

## सार्वजनिक API सतह

`src/mcp/index.ts` बाहरी कॉलर्स के लिए लोडर/मैनेजर/क्लाइंट API पुनः निर्यात करता है। `src/sdk.ts` समान लोडर परिणाम आकार लौटाने वाले एक सुविधा रैपर के रूप में `discoverMCPServers()` उजागर करता है।

- [`src/mcp/manager.ts`](../../packages/coding-agent/src/mcp/manager.ts) — जीवनचक्र स्टेट रजिस्ट्रियाँ, समानांतर कनेक्ट/लिस्ट फ्लो, रिफ्रेश/डिस्कनेक्ट।
- [`src/mcp/client.ts`](../../packages/coding-agent/src/mcp/client.ts) — ट्रांसपोर्ट सेटअप, इनिशियलाइज़ हैंडशेक, लिस्ट/कॉल/डिस्कनेक्ट।
- [`src/mcp/index.ts`](../../packages/coding-agent/src/mcp/index.ts) — MCP मॉड्यूल API निर्यात।
- [`src/sdk.ts`](../../packages/coding-agent/src/sdk.ts) — सत्र/टूल रजिस्ट्री में स्टार्टअप वायरिंग।
- [`src/mcp/config.ts`](../../packages/coding-agent/src/mcp/config.ts) — मैनेजर द्वारा उपयोग की जाने वाली कॉन्फिग खोज/फ़िल्टरिंग/सत्यापन।
- [`src/mcp/tool-bridge.ts`](../../packages/coding-agent/src/mcp/tool-bridge.ts) — `MCPTool` और `DeferredMCPTool` रनटाइम व्यवहार।
- [`src/session/agent-session.ts`](../../packages/coding-agent/src/session/agent-session.ts) — `refreshMCPTools` लाइव रीबाइंडिंग।
- [`src/modes/controllers/mcp-command-controller.ts`](../../packages/coding-agent/src/modes/controllers/mcp-command-controller.ts) — इंटरैक्टिव रीलोड/पुनः कनेक्ट फ्लो।
- [`src/task/executor.ts`](../../packages/coding-agent/src/task/executor.ts) — पैरेंट मैनेजर कनेक्शन के माध्यम से सबएजेंट MCP प्रॉक्सीइंग।
