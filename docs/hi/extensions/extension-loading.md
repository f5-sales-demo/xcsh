# एक्सटेंशन लोडिंग (TypeScript/JavaScript मॉड्यूल)

यह दस्तावेज़ बताता है कि कोडिंग एजेंट स्टार्टअप के समय **एक्सटेंशन मॉड्यूल** (`.ts`/`.js`) को कैसे खोजता और लोड करता है।

यह `gemini-extension.json` मैनिफेस्ट एक्सटेंशन को **कवर नहीं** करता (जो अलग से प्रलेखित है)।

## यह सबसिस्टम क्या करता है

एक्सटेंशन लोडिंग मॉड्यूल एंट्री फाइलों की एक सूची बनाता है, Bun के साथ प्रत्येक मॉड्यूल को इम्पोर्ट करता है, इसकी फैक्ट्री को एक्सीक्यूट करता है, और यह लौटाता है:

- लोड किए गए एक्सटेंशन डेफिनिशन
- प्रति-पाथ लोड त्रुटियाँ (पूरे लोड को निरस्त किए बिना)
- एक साझा एक्सटेंशन रनटाइम ऑब्जेक्ट जो बाद में `ExtensionRunner` द्वारा उपयोग किया जाता है

## प्राथमिक कार्यान्वयन फाइलें

- `src/extensibility/extensions/loader.ts` — पाथ खोज + इम्पोर्ट/एक्सीक्यूशन
- `src/extensibility/extensions/index.ts` — सार्वजनिक एक्सपोर्ट
- `src/extensibility/extensions/runner.ts` — लोड के बाद रनटाइम/इवेंट एक्सीक्यूशन
- `src/discovery/builtin.ts` — एक्सटेंशन मॉड्यूल के लिए नेटिव ऑटो-डिस्कवरी प्रोवाइडर
- `src/config/settings.ts` — मर्ज किए गए `extensions` / `disabledExtensions` सेटिंग्स लोड करता है

## एक्सटेंशन लोडिंग के इनपुट

### 1) ऑटो-डिस्कवर किए गए नेटिव एक्सटेंशन मॉड्यूल

`discoverAndLoadExtensions()` पहले डिस्कवरी प्रोवाइडर्स से `extension-module` कैपेबिलिटी आइटम माँगता है, फिर केवल प्रोवाइडर `native` आइटम रखता है।

प्रभावी नेटिव स्थान:

- प्रोजेक्ट: `<cwd>/.xcsh/extensions`
- उपयोगकर्ता: `~/.xcsh/agent/extensions`

पाथ रूट नेटिव प्रोवाइडर (`SOURCE_PATHS.native`) से आते हैं।

नोट्स:

- नेटिव ऑटो-डिस्कवरी वर्तमान में `.xcsh` आधारित है।
- लेगेसी `.pi` अभी भी `package.json` मैनिफेस्ट कीज़ (`pi.extensions`) में स्वीकार किया जाता है, लेकिन यहाँ नेटिव रूट के रूप में नहीं।

### 2) स्पष्ट रूप से कॉन्फ़िगर किए गए पाथ

ऑटो-डिस्कवरी के बाद, कॉन्फ़िगर किए गए पाथ जोड़े और रिज़ॉल्व किए जाते हैं।

मुख्य सेशन स्टार्टअप पाथ (`sdk.ts`) में कॉन्फ़िगर किए गए पाथ स्रोत:

1. CLI द्वारा प्रदान किए गए पाथ (`--extension/-e`, और `--hook` को भी एक्सटेंशन पाथ के रूप में माना जाता है)
2. सेटिंग्स `extensions` ऐरे (मर्ज किए गए ग्लोबल + प्रोजेक्ट सेटिंग्स)

ग्लोबल सेटिंग्स फाइल:

- `~/.xcsh/agent/config.yml` (या `PI_CODING_AGENT_DIR` के माध्यम से कस्टम एजेंट डायरेक्टरी)

प्रोजेक्ट सेटिंग्स फाइल:

- `<cwd>/.xcsh/settings.json`

उदाहरण:

```yaml
# ~/.xcsh/agent/config.yml
extensions:
  - ~/my-exts/safety.ts
  - ./local/ext-pack
```

```json
{
  "extensions": ["./.xcsh/extensions/my-extra"]
}
```

## सक्षम/अक्षम नियंत्रण

### डिस्कवरी अक्षम करें

- CLI: `--no-extensions`
- SDK विकल्प: `disableExtensionDiscovery`

व्यवहार विभाजन:

- SDK: जब `disableExtensionDiscovery=true` हो, तो यह अभी भी `loadExtensions()` के माध्यम से `additionalExtensionPaths` लोड करता है।
- CLI पाथ बिल्डिंग (`main.ts`) वर्तमान में `--no-extensions` सेट होने पर CLI एक्सटेंशन पाथ क्लियर करता है, इसलिए स्पष्ट `-e/--hook` उस मोड में फॉरवर्ड नहीं किए जाते।

### विशिष्ट एक्सटेंशन मॉड्यूल अक्षम करें

`disabledExtensions` सेटिंग एक्सटेंशन id फॉर्मेट द्वारा फ़िल्टर करती है:

- `extension-module:<derivedName>`

`derivedName` एंट्री पाथ (`getExtensionNameFromPath`) पर आधारित है, उदाहरण के लिए:

- `/x/foo.ts` -> `foo`
- `/x/bar/index.ts` -> `bar`

उदाहरण:

```yaml
disabledExtensions:
  - extension-module:foo
```

## पाथ और एंट्री रिज़ॉल्यूशन

### पाथ नॉर्मलाइज़ेशन

कॉन्फ़िगर किए गए पाथ के लिए:

1. यूनिकोड स्पेस नॉर्मलाइज़ करें
2. `~` को विस्तृत करें
3. यदि रिलेटिव है, तो वर्तमान `cwd` के विरुद्ध रिज़ॉल्व करें

### यदि कॉन्फ़िगर किया गया पाथ एक फाइल है

इसे सीधे मॉड्यूल एंट्री कैंडिडेट के रूप में उपयोग किया जाता है।

### यदि कॉन्फ़िगर किया गया पाथ एक डायरेक्टरी है

रिज़ॉल्यूशन क्रम:

1. उस डायरेक्टरी में `package.json` जिसमें `xcsh.extensions` (या लेगेसी `pi.extensions`) हो -> घोषित एंट्रीज़ का उपयोग करें
2. `index.ts`
3. `index.js`
4. अन्यथा एक्सटेंशन एंट्रीज़ के लिए एक स्तर स्कैन करें:
   - सीधे `*.ts` / `*.js`
   - सबडायरेक्टरी `index.ts` / `index.js`
   - सबडायरेक्टरी `package.json` जिसमें `xcsh.extensions` / `pi.extensions` हो

नियम और बाधाएँ:

- एक सबडायरेक्टरी स्तर से परे कोई रिकर्सिव डिस्कवरी नहीं
- घोषित `extensions` मैनिफेस्ट एंट्रीज़ उस पैकेज डायरेक्टरी के सापेक्ष रिज़ॉल्व की जाती हैं
- घोषित एंट्रीज़ केवल तभी शामिल की जाती हैं जब फाइल मौजूद हो/एक्सेस की अनुमति हो
- `*/index.{ts,js}` जोड़ों में, TypeScript को JavaScript पर वरीयता दी जाती है
- सिमलिंक को पात्र फाइलों/डायरेक्टरीज़ के रूप में माना जाता है

### इग्नोर व्यवहार स्रोत के अनुसार भिन्न होता है

- नेटिव ऑटो-डिस्कवरी (डिस्कवरी हेल्पर्स में `discoverExtensionModulePaths`) नेटिव glob का उपयोग `gitignore: true` और `hidden: false` के साथ करता है।
- `loader.ts` में स्पष्ट कॉन्फ़िगर डायरेक्टरी स्कैनिंग `readdir` नियमों का उपयोग करती है और gitignore फ़िल्टरिंग **लागू नहीं** करती।

## लोड क्रम और प्राथमिकता

`discoverAndLoadExtensions()` एक क्रमित सूची बनाता है और फिर `loadExtensions()` कॉल करता है।

क्रम:

1. नेटिव ऑटो-डिस्कवर किए गए मॉड्यूल
2. स्पष्ट कॉन्फ़िगर किए गए पाथ (प्रदान किए गए क्रम में)

`sdk.ts` में, कॉन्फ़िगर किया गया क्रम है:

1. CLI अतिरिक्त पाथ
2. सेटिंग्स `extensions`

डी-डुप्लीकेशन:

- एब्सोल्यूट पाथ आधारित
- पहले देखा गया पाथ जीतता है
- बाद के डुप्लिकेट अनदेखा किए जाते हैं

निहितार्थ: यदि वही मॉड्यूल पाथ ऑटो-डिस्कवर और स्पष्ट रूप से कॉन्फ़िगर दोनों है, तो इसे पहली स्थिति (ऑटो-डिस्कवर चरण) पर एक बार लोड किया जाता है।

## मॉड्यूल इम्पोर्ट और फैक्ट्री अनुबंध

प्रत्येक कैंडिडेट पाथ डायनामिक इम्पोर्ट से लोड किया जाता है:

- `await import(resolvedPath)`
- फैक्ट्री `module.default ?? module` है
- फैक्ट्री एक फंक्शन (`ExtensionFactory`) होनी चाहिए

यदि एक्सपोर्ट फंक्शन नहीं है, तो वह पाथ एक स्ट्रक्चर्ड एरर के साथ विफल हो जाता है और लोडिंग जारी रहती है।

## विफलता हैंडलिंग और आइसोलेशन

### लोडिंग के दौरान

प्रति एक्सटेंशन पाथ, विफलताओं को `{ path, error }` के रूप में कैप्चर किया जाता है और अन्य पाथ को लोड होने से नहीं रोकती हैं।

सामान्य मामले:

- इम्पोर्ट विफलता / गायब फाइल
- अमान्य फैक्ट्री एक्सपोर्ट (नॉन-फंक्शन)
- फैक्ट्री एक्सीक्यूट करते समय फेंका गया अपवाद

### रनटाइम आइसोलेशन मॉडल

- एक्सटेंशन **सैंडबॉक्स नहीं** हैं (समान प्रोसेस/रनटाइम)।
- वे एक `EventBus` और एक `ExtensionRuntime` इंस्टेंस साझा करते हैं।
- लोड के दौरान, रनटाइम एक्शन मेथड्स जानबूझकर `ExtensionRuntimeNotInitializedError` फेंकती हैं; एक्शन वायरिंग बाद में `ExtensionRunner.initialize()` में होती है।

### लोडिंग के बाद

जब इवेंट `ExtensionRunner` के माध्यम से चलते हैं, तो हैंडलर अपवाद पकड़े जाते हैं और रनर लूप को क्रैश करने के बजाय एक्सटेंशन एरर के रूप में एमिट किए जाते हैं।

## न्यूनतम उपयोगकर्ता/प्रोजेक्ट लेआउट उदाहरण

### उपयोगकर्ता-स्तर

```text
~/.xcsh/agent/
  config.yml
  extensions/
    guardrails.ts
    audit/
      index.ts
```

### प्रोजेक्ट-स्तर

```text
<repo>/
  .xcsh/
    settings.json
    extensions/
      checks/
        package.json
      lint-gates.ts
```

`checks/package.json`:

```json
{
  "xcsh": {
    "extensions": ["./src/check-a.ts", "./src/check-b.js"]
  }
}
```

लेगेसी मैनिफेस्ट कुंजी अभी भी स्वीकृत:

```json
{
  "pi": {
    "extensions": ["./index.ts"]
  }
}
```
