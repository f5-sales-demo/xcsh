# Python टूल और IPython रनटाइम

यह दस्तावेज़ `packages/coding-agent` में वर्तमान Python निष्पादन स्टैक का वर्णन करता है।
इसमें टूल व्यवहार, कर्नेल/गेटवे जीवनचक्र, पर्यावरण प्रबंधन, निष्पादन सेमेंटिक्स, आउटपुट रेंडरिंग और परिचालन विफलता मोड शामिल हैं।

## कार्यक्षेत्र और प्रमुख फ़ाइलें

- टूल सर्फेस: `src/tools/python.ts`
- सत्र/प्रति-कॉल कर्नेल ऑर्केस्ट्रेशन: `src/ipy/executor.ts`
- कर्नेल प्रोटोकॉल + गेटवे एकीकरण: `src/ipy/kernel.ts`
- साझा स्थानीय गेटवे समन्वयक: `src/ipy/gateway-coordinator.ts`
- उपयोगकर्ता-ट्रिगर Python रन के लिए इंटरैक्टिव-मोड रेंडरर: `src/modes/components/python-execution.ts`
- रनटाइम/पर्यावरण फ़िल्टरिंग और Python रिज़ॉल्यूशन: `src/ipy/runtime.ts`

## Python टूल क्या है

`python` टूल एक या अधिक Python सेल को Jupyter Kernel Gateway-समर्थित कर्नेल के माध्यम से निष्पादित करता है (प्रति सेल `python -c` सीधे स्पॉन करके नहीं)।

टूल पैरामीटर:

```ts
{
  cells: Array<{ code: string; title?: string }>;
  timeout?: number; // सेकंड, 1..600 तक सीमित, डिफ़ॉल्ट 30
  cwd?: string;
  reset?: boolean; // केवल पहले सेल से पहले कर्नेल रीसेट करें
}
```

यह टूल एक सत्र के लिए `concurrency = "exclusive"` है, इसलिए कॉल ओवरलैप नहीं होते।

## गेटवे जीवनचक्र

### मोड

दो गेटवे पथ हैं:

1. **बाह्य गेटवे** (`PI_PYTHON_GATEWAY_URL` सेट होने पर)
   - कॉन्फ़िगर किए गए URL का सीधे उपयोग करता है।
   - `PI_PYTHON_GATEWAY_TOKEN` के साथ वैकल्पिक प्रमाणीकरण।
   - कोई स्थानीय गेटवे प्रक्रिया स्पॉन या प्रबंधित नहीं की जाती।

2. **स्थानीय साझा गेटवे** (डिफ़ॉल्ट पथ)
   - `~/.xcsh/agent/python-gateway` के अंतर्गत समन्वित एकल साझा प्रक्रिया का उपयोग करता है।
   - मेटाडेटा फ़ाइल: `gateway.json`
   - लॉक फ़ाइल: `gateway.lock`
   - स्पॉन कमांड:
     - `python -m kernel_gateway`
     - `127.0.0.1:<allocated-port>` से बाइंड
     - स्टार्टअप हेल्थ चेक: `GET /api/kernelspecs`

### स्थानीय साझा गेटवे समन्वय

`acquireSharedGateway()`:

- हार्टबीट के साथ फ़ाइल लॉक (`gateway.lock`) लेता है।
- यदि PID जीवित है और हेल्थ चेक पास होता है तो `gateway.json` का पुनः उपयोग करता है।
- आवश्यकता पड़ने पर पुरानी जानकारी/PID को साफ करता है।
- जब कोई स्वस्थ गेटवे मौजूद नहीं होता तो नया गेटवे शुरू करता है।

`releaseSharedGateway()` वर्तमान में एक no-op है (कर्नेल शटडाउन साझा गेटवे को बंद नहीं करता)।

`shutdownSharedGateway()` साझा प्रक्रिया को स्पष्ट रूप से समाप्त करता है और गेटवे मेटाडेटा साफ करता है।

### महत्वपूर्ण बाधा

कर्नेल स्टार्ट पर `python.sharedGateway=false` अस्वीकार किया जाता है:

- त्रुटि: `Shared Python gateway required; local gateways are disabled`
- कोई प्रति-प्रक्रिया गैर-साझा स्थानीय गेटवे मोड नहीं है।

## कर्नेल जीवनचक्र

प्रत्येक निष्पादन में चुने गए गेटवे पर `POST /api/kernels` के माध्यम से बनाए गए कर्नेल का उपयोग होता है।

कर्नेल स्टार्टअप अनुक्रम:

1. उपलब्धता जाँच (`checkPythonKernelAvailability`)
2. कर्नेल बनाएं (`/api/kernels`)
3. वेबसॉकेट खोलें (`/api/kernels/:id/channels`)
4. कर्नेल पर्यावरण प्रारंभ करें (`cwd`, env vars, `sys.path`)
5. `PYTHON_PRELUDE` निष्पादित करें
6. निम्नलिखित से एक्सटेंशन मॉड्यूल लोड करें:
   - उपयोगकर्ता: `~/.xcsh/agent/modules/*.py`
   - प्रोजेक्ट: `<cwd>/.xcsh/modules/*.py` (समान नाम के उपयोगकर्ता मॉड्यूल को ओवरराइड करता है)

कर्नेल शटडाउन:

- `DELETE /api/kernels/:id` के माध्यम से रिमोट कर्नेल हटाता है
- वेबसॉकेट बंद करता है
- साझा गेटवे रिलीज़ हुक कॉल करता है (आज no-op है)

## सत्र स्थायित्व सेमेंटिक्स

`python.kernelMode` कर्नेल पुनः उपयोग नियंत्रित करता है:

- `session` (डिफ़ॉल्ट)
  - सत्र पहचान + cwd द्वारा कुंजीकृत कर्नेल सत्रों का पुनः उपयोग करता है।
  - निष्पादन को कतार के माध्यम से प्रति सत्र क्रमबद्ध किया जाता है।
  - निष्क्रिय सत्र 5 मिनट बाद हटा दिए जाते हैं।
  - अधिकतम 4 सत्र; ओवरफ्लो पर सबसे पुराना हटाया जाता है।
  - हार्टबीट जाँच मृत कर्नेल का पता लगाती है।
  - एक बार ऑटो-रीस्टार्ट की अनुमति; बार-बार क्रैश => हार्ड विफलता।

- `per-call`
  - प्रत्येक निष्पादन अनुरोध के लिए नया कर्नेल बनाता है।
  - अनुरोध के बाद कर्नेल बंद करता है।
  - कोई क्रॉस-कॉल स्थिति स्थायित्व नहीं।

### एकल टूल कॉल में मल्टी-सेल व्यवहार

उस टूल कॉल के लिए सेल एक ही कर्नेल इंस्टेंस में क्रमिक रूप से चलते हैं।

यदि कोई मध्यवर्ती सेल विफल हो जाता है:

- पहले के सेल की स्थिति मेमोरी में बनी रहती है।
- टूल एक लक्षित त्रुटि लौटाता है जो बताता है कि कौन सा सेल विफल हुआ।
- बाद के सेल निष्पादित नहीं होते।

`reset=true` केवल उस कॉल में पहले सेल निष्पादन पर लागू होता है।

## पर्यावरण फ़िल्टरिंग और रनटाइम रिज़ॉल्यूशन

गेटवे/कर्नेल रनटाइम लॉन्च करने से पहले पर्यावरण फ़िल्टर किया जाता है:

- अनुमति सूची में `PATH`, `HOME`, लोकेल वेरिएबल, `VIRTUAL_ENV`, `PYTHONPATH` आदि जैसे मूल वेरिएबल शामिल हैं।
- अनुमति-उपसर्ग: `LC_`, `XDG_`, `PI_`
- निषेध सूची सामान्य API कुंजियाँ हटाती है (OpenAI/Anthropic/Gemini/आदि)

रनटाइम चयन क्रम:

1. सक्रिय/स्थित venv (`VIRTUAL_ENV`, फिर `<cwd>/.venv`, `<cwd>/venv`)
2. `~/.xcsh/python-env` पर प्रबंधित venv
3. PATH पर `python` या `python3`

जब कोई venv चुना जाता है, तो उसका bin/Scripts पथ `PATH` में पहले जोड़ा जाता है।

Python के अंदर कर्नेल पर्यावरण प्रारंभीकरण भी:

- `os.chdir(cwd)`
- प्रदान किए गए env map को `os.environ` में इंजेक्ट करता है
- सुनिश्चित करता है कि cwd `sys.path` में है

## टूल उपलब्धता और मोड चयन

`python.toolMode` (डिफ़ॉल्ट `both`) + वैकल्पिक `PI_PY` ओवरराइड एक्सपोज़र नियंत्रित करता है:

- `ipy-only`
- `bash-only`
- `both`

`PI_PY` स्वीकृत मान:

- `0` / `bash` -> `bash-only`
- `1` / `py` -> `ipy-only`
- `mix` / `both` -> `both`

यदि Python प्रीफ्लाइट विफल होता है, तो उस सत्र के लिए टूल निर्माण bash-only में डिग्रेड हो जाता है।

## निष्पादन प्रवाह और रद्दीकरण/टाइमआउट

### टूल-स्तर टाइमआउट

`python` टूल टाइमआउट सेकंड में है, डिफ़ॉल्ट 30, `1..600` तक सीमित।

टूल इन्हें संयुक्त करता है:

- कॉलर एबॉर्ट सिग्नल
- टाइमआउट एबॉर्ट सिग्नल

`AbortSignal.any(...)` के साथ।

### कर्नेल निष्पादन रद्दीकरण

एबॉर्ट/टाइमआउट पर:

- निष्पादन को रद्द किया गया चिह्नित किया जाता है।
- REST (`POST /interrupt`) और कंट्रोल-चैनल `interrupt_request` के माध्यम से कर्नेल इंटरप्ट का प्रयास किया जाता है।
- परिणाम में `cancelled=true` शामिल होता है।
- टाइमआउट पथ आउटपुट को `Command timed out after <n> seconds` के रूप में एनोटेट करता है।

### stdin व्यवहार

इंटरैक्टिव stdin समर्थित नहीं है।

यदि कर्नेल `input_request` भेजता है:

- टूल `stdinRequested=true` रिकॉर्ड करता है
- व्याख्यात्मक टेक्स्ट भेजता है
- खाली `input_reply` भेजता है
- निष्पादन को executor परत पर विफलता माना जाता है

## आउटपुट कैप्चर और रेंडरिंग

### कैप्चर किए गए आउटपुट वर्ग

कर्नेल संदेशों से:

- `stream` -> सादा टेक्स्ट खंड
- `display_data`/`execute_result` -> रिच डिस्प्ले हैंडलिंग
- `error` -> ट्रेसबैक टेक्स्ट
- कस्टम MIME `application/x-xcsh-status` -> संरचित स्थिति इवेंट

डिस्प्ले MIME प्राथमिकता:

1. `text/markdown`
2. `text/plain`
3. `text/html` (बेसिक markdown में परिवर्तित)

संरचित आउटपुट के रूप में अतिरिक्त कैप्चर:

- `application/json` -> JSON ट्री डेटा
- `image/png` -> इमेज पेलोड
- `application/x-xcsh-status` -> स्थिति इवेंट

### भंडारण और ट्रंकेशन

आउटपुट `OutputSink` के माध्यम से स्ट्रीम किया जाता है और आर्टिफैक्ट स्टोरेज में सहेजा जा सकता है।

टूल परिणामों में ट्रंकेशन मेटाडेटा और पूर्ण आउटपुट पुनर्प्राप्ति के लिए `artifact://<id>` शामिल हो सकता है।

### रेंडरर व्यवहार

- टूल रेंडरर (`python.ts`):
  - प्रति-सेल स्थिति के साथ कोड-सेल ब्लॉक दिखाता है
  - संक्षिप्त पूर्वावलोकन डिफ़ॉल्ट 10 पंक्तियाँ
  - पूर्ण आउटपुट और समृद्ध स्थिति विवरण के लिए विस्तारित मोड का समर्थन करता है
- इंटरैक्टिव रेंडरर (`python-execution.ts`):
  - TUI में उपयोगकर्ता-ट्रिगर Python निष्पादन के लिए उपयोग किया जाता है
  - संक्षिप्त पूर्वावलोकन डिफ़ॉल्ट 20 पंक्तियाँ
  - डिस्प्ले सुरक्षा के लिए बहुत लंबी व्यक्तिगत पंक्तियों को 4000 वर्णों तक सीमित करता है
  - रद्दीकरण/त्रुटि/ट्रंकेशन नोटिस दिखाता है

## बाह्य गेटवे समर्थन

सेट करें:

```bash
export PI_PYTHON_GATEWAY_URL="http://127.0.0.1:8888"
# वैकल्पिक:
export PI_PYTHON_GATEWAY_TOKEN="..."
```

स्थानीय साझा गेटवे से व्यवहार अंतर:

- कोई स्थानीय गेटवे लॉक/इन्फो फ़ाइलें नहीं
- कोई स्थानीय प्रक्रिया स्पॉन/समाप्ति नहीं
- हेल्थ चेक और कर्नेल CRUD बाह्य एंडपॉइंट के विरुद्ध चलते हैं
- प्रमाणीकरण विफलताएं स्पष्ट टोकन मार्गदर्शन के साथ प्रदर्शित होती हैं

## परिचालन समस्या निवारण (वर्तमान विफलता मोड)

- **Python टूल उपलब्ध नहीं**
  - `python.toolMode` / `PI_PY` जाँचें।
  - यदि प्रीफ्लाइट विफल होता है, तो रनटाइम bash-only पर वापस आ जाता है।

- **कर्नेल उपलब्धता त्रुटियाँ**
  - स्थानीय मोड के लिए रिज़ॉल्व किए गए Python रनटाइम में `kernel_gateway` और `ipykernel` दोनों आयात करने योग्य होने चाहिए।
  - इनस्टॉल करें:

    ```bash
    python -m pip install jupyter_kernel_gateway ipykernel
    ```

- **`python.sharedGateway=false` स्टार्टअप विफलता का कारण बनता है**
  - यह वर्तमान कार्यान्वयन के साथ अपेक्षित है।

- **बाह्य गेटवे प्रमाणीकरण/पहुँच विफलताएं**
  - 401/403 -> `PI_PYTHON_GATEWAY_TOKEN` सेट करें।
  - टाइमआउट/अनुपलब्ध -> URL/नेटवर्क और गेटवे हेल्थ सत्यापित करें।

- **निष्पादन रुकता है फिर टाइमआउट होता है**
  - यदि कार्यभार वैध है तो टूल `timeout` बढ़ाएं (अधिकतम 600s)।
  - अटके हुए कोड के लिए, रद्दीकरण कर्नेल इंटरप्ट ट्रिगर करता है लेकिन उपयोगकर्ता कोड को फिर भी रिफैक्टर करने की आवश्यकता हो सकती है।

- **Python कोड में stdin/input प्रॉम्प्ट**
  - `input()` इस रनटाइम पथ में इंटरैक्टिव रूप से समर्थित नहीं है; डेटा प्रोग्रामेटिक रूप से पास करें।

- **संसाधन समाप्ति (`EMFILE` / बहुत अधिक खुली फ़ाइलें)**
  - सत्र प्रबंधक साझा-गेटवे पुनर्प्राप्ति ट्रिगर करता है (सत्र टियरडाउन + साझा गेटवे रीस्टार्ट)।

- **कार्यशील निर्देशिका त्रुटियाँ**
  - टूल निष्पादन से पहले सत्यापित करता है कि `cwd` मौजूद है और एक निर्देशिका है।

## प्रासंगिक पर्यावरण वेरिएबल

- `PI_PY` — टूल एक्सपोज़र ओवरराइड (ऊपर दिया गया `bash-only`/`ipy-only`/`both` मैपिंग)
- `PI_PYTHON_GATEWAY_URL` — बाह्य गेटवे का उपयोग करें
- `PI_PYTHON_GATEWAY_TOKEN` — वैकल्पिक बाह्य गेटवे प्रमाणीकरण टोकन
- `PI_PYTHON_SKIP_CHECK=1` — Python प्रीफ्लाइट/वार्म चेक बाईपास करें
- `PI_PYTHON_IPC_TRACE=1` — कर्नेल IPC send/receive ट्रेस लॉग करें
- `PI_DEBUG_STARTUP=1` — स्टार्टअप-स्तर डीबग मार्कर भेजें
