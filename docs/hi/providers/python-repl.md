# Python Tool और IPython Runtime

यह दस्तावेज़ `packages/coding-agent` में वर्तमान Python निष्पादन स्टैक का वर्णन करता है।
इसमें टूल व्यवहार, कर्नेल/गेटवे जीवनचक्र, एनवायरनमेंट हैंडलिंग, निष्पादन सिमेंटिक्स, आउटपुट रेंडरिंग, और परिचालन विफलता मोड शामिल हैं।

## कार्यक्षेत्र और प्रमुख फ़ाइलें

- टूल सरफेस: `src/tools/python.ts`
- सत्र/प्रति-कॉल कर्नेल ऑर्केस्ट्रेशन: `src/ipy/executor.ts`
- कर्नेल प्रोटोकॉल + गेटवे इंटीग्रेशन: `src/ipy/kernel.ts`
- साझा लोकल गेटवे कोऑर्डिनेटर: `src/ipy/gateway-coordinator.ts`
- उपयोगकर्ता द्वारा ट्रिगर किए गए Python रन के लिए इंटरैक्टिव-मोड रेंडरर: `src/modes/components/python-execution.ts`
- रनटाइम/env फ़िल्टरिंग और Python रिज़ॉल्यूशन: `src/ipy/runtime.ts`

## Python टूल क्या है

`python` टूल एक या अधिक Python सेल्स को Jupyter Kernel Gateway-समर्थित कर्नेल के माध्यम से निष्पादित करता है (प्रति सेल सीधे `python -c` स्पॉन करके नहीं)।

टूल पैरामीटर:

```ts
{
  cells: Array<{ code: string; title?: string }>;
  timeout?: number; // seconds, clamped to 1..600, default 30
  cwd?: string;
  reset?: boolean; // reset kernel before first cell only
}
```

टूल एक सत्र के लिए `concurrency = "exclusive"` है, इसलिए कॉल ओवरलैप नहीं होते।

## गेटवे जीवनचक्र

### मोड

दो गेटवे पथ हैं:

1. **बाहरी गेटवे** (`PI_PYTHON_GATEWAY_URL` सेट)
   - कॉन्फ़िगर किए गए URL का सीधे उपयोग करता है।
   - `PI_PYTHON_GATEWAY_TOKEN` के साथ वैकल्पिक प्रमाणीकरण।
   - कोई लोकल गेटवे प्रक्रिया स्पॉन या प्रबंधित नहीं की जाती।

2. **लोकल साझा गेटवे** (डिफ़ॉल्ट पथ)
   - `~/.xcsh/agent/python-gateway` के अंतर्गत समन्वित एकल साझा प्रक्रिया का उपयोग करता है।
   - मेटाडेटा फ़ाइल: `gateway.json`
   - लॉक फ़ाइल: `gateway.lock`
   - स्पॉन कमांड:
     - `python -m kernel_gateway`
     - `127.0.0.1:<allocated-port>` पर बाइंड
     - स्टार्टअप हेल्थ चेक: `GET /api/kernelspecs`

### लोकल साझा गेटवे समन्वय

`acquireSharedGateway()`:

- हार्टबीट के साथ फ़ाइल लॉक (`gateway.lock`) लेता है।
- यदि PID सक्रिय है और हेल्थ चेक पास होता है तो `gateway.json` का पुन: उपयोग करता है।
- आवश्यकता पड़ने पर पुरानी info/PIDs को साफ़ करता है।
- जब कोई स्वस्थ गेटवे मौजूद नहीं होता तो नया गेटवे शुरू करता है।

`releaseSharedGateway()` वर्तमान में एक no-op है (कर्नेल शटडाउन साझा गेटवे को बंद नहीं करता)।

`shutdownSharedGateway()` स्पष्ट रूप से साझा प्रक्रिया को समाप्त करता है और गेटवे मेटाडेटा साफ़ करता है।

### महत्वपूर्ण बाधा

`python.sharedGateway=false` कर्नेल स्टार्ट पर अस्वीकृत होता है:

- त्रुटि: `Shared Python gateway required; local gateways are disabled`
- कोई प्रति-प्रक्रिया गैर-साझा लोकल गेटवे मोड नहीं है।

## कर्नेल जीवनचक्र

प्रत्येक निष्पादन चयनित गेटवे पर `POST /api/kernels` के माध्यम से बनाए गए कर्नेल का उपयोग करता है।

कर्नेल स्टार्टअप क्रम:

1. उपलब्धता जांच (`checkPythonKernelAvailability`)
2. कर्नेल बनाना (`/api/kernels`)
3. वेबसॉकेट खोलना (`/api/kernels/:id/channels`)
4. कर्नेल env प्रारंभ करना (`cwd`, env vars, `sys.path`)
5. `PYTHON_PRELUDE` निष्पादित करना
6. एक्सटेंशन मॉड्यूल लोड करना:
   - उपयोगकर्ता: `~/.xcsh/agent/modules/*.py`
   - प्रोजेक्ट: `<cwd>/.xcsh/modules/*.py` (समान-नाम उपयोगकर्ता मॉड्यूल को ओवरराइड करता है)

कर्नेल शटडाउन:

- `DELETE /api/kernels/:id` के माध्यम से रिमोट कर्नेल हटाता है
- वेबसॉकेट बंद करता है
- साझा गेटवे रिलीज़ हुक कॉल करता है (आज no-op)

## सत्र स्थायित्व सिमेंटिक्स

`python.kernelMode` कर्नेल पुन: उपयोग को नियंत्रित करता है:

- `session` (डिफ़ॉल्ट)
  - सत्र पहचान + cwd द्वारा कुंजीबद्ध कर्नेल सत्रों का पुन: उपयोग करता है।
  - प्रति सत्र एक कतार के माध्यम से निष्पादन क्रमबद्ध होता है।
  - निष्क्रिय सत्र 5 मिनट के बाद हटा दिए जाते हैं।
  - अधिकतम 4 सत्र; ओवरफ़्लो पर सबसे पुराना हटा दिया जाता है।
  - हार्टबीट जांच मृत कर्नेल का पता लगाती है।
  - ऑटो-रिस्टार्ट एक बार अनुमत; बार-बार क्रैश => गंभीर विफलता।

- `per-call`
  - प्रत्येक निष्पादन अनुरोध के लिए एक नया कर्नेल बनाता है।
  - अनुरोध के बाद कर्नेल बंद कर देता है।
  - कोई क्रॉस-कॉल स्थिति स्थायित्व नहीं।

### एकल टूल कॉल में बहु-सेल व्यवहार

सेल्स उस टूल कॉल के लिए एक ही कर्नेल इंस्टेंस में क्रमिक रूप से चलते हैं।

यदि कोई मध्यवर्ती सेल विफल होता है:

- पहले के सेल की स्थिति मेमोरी में बनी रहती है।
- टूल एक लक्षित त्रुटि लौटाता है जो बताती है कि कौन सा सेल विफल हुआ।
- बाद के सेल निष्पादित नहीं होते।

`reset=true` केवल उस कॉल में पहले सेल निष्पादन पर लागू होता है।

## एनवायरनमेंट फ़िल्टरिंग और रनटाइम रिज़ॉल्यूशन

गेटवे/कर्नेल रनटाइम लॉन्च करने से पहले एनवायरनमेंट फ़िल्टर किया जाता है:

- अनुमति सूची में `PATH`, `HOME`, लोकेल vars, `VIRTUAL_ENV`, `PYTHONPATH`, आदि जैसे मुख्य vars शामिल हैं।
- अनुमति-उपसर्ग: `LC_`, `XDG_`, `PI_`
- अस्वीकृति सूची सामान्य API कुंजियों (OpenAI/Anthropic/Gemini/आदि) को हटाती है।

रनटाइम चयन क्रम:

1. सक्रिय/स्थित venv (`VIRTUAL_ENV`, फिर `<cwd>/.venv`, `<cwd>/venv`)
2. `~/.xcsh/python-env` पर प्रबंधित venv
3. PATH पर `python` या `python3`

जब कोई venv चुना जाता है, तो उसका bin/Scripts पथ `PATH` के आगे जोड़ दिया जाता है।

Python के अंदर कर्नेल env प्रारंभीकरण भी:

- `os.chdir(cwd)`
- प्रदत्त env मैप को `os.environ` में इंजेक्ट करता है
- सुनिश्चित करता है कि cwd `sys.path` में है

## टूल उपलब्धता और मोड चयन

`python.toolMode` (डिफ़ॉल्ट `both`) + वैकल्पिक `PI_PY` ओवरराइड एक्सपोज़र को नियंत्रित करता है:

- `ipy-only`
- `bash-only`
- `both`

`PI_PY` स्वीकृत मान:

- `0` / `bash` -> `bash-only`
- `1` / `py` -> `ipy-only`
- `mix` / `both` -> `both`

यदि Python प्रीफ़्लाइट विफल होता है, तो उस सत्र के लिए टूल निर्माण bash-only में अवनत हो जाता है।

## निष्पादन प्रवाह और रद्दीकरण/टाइमआउट

### टूल-स्तरीय टाइमआउट

`python` टूल टाइमआउट सेकंड में है, डिफ़ॉल्ट 30, `1..600` तक सीमित।

टूल संयोजित करता है:

- कॉलर abort सिग्नल
- टाइमआउट abort सिग्नल

`AbortSignal.any(...)` के साथ।

### कर्नेल निष्पादन रद्दीकरण

abort/टाइमआउट पर:

- निष्पादन रद्द के रूप में चिह्नित होता है।
- REST (`POST /interrupt`) और control-channel `interrupt_request` के माध्यम से कर्नेल इंटरप्ट का प्रयास किया जाता है।
- परिणाम में `cancelled=true` शामिल होता है।
- टाइमआउट पथ आउटपुट को `Command timed out after <n> seconds` के रूप में एनोटेट करता है।

### stdin व्यवहार

इंटरैक्टिव stdin समर्थित नहीं है।

यदि कर्नेल `input_request` उत्सर्जित करता है:

- टूल `stdinRequested=true` रिकॉर्ड करता है
- व्याख्यात्मक टेक्स्ट उत्सर्जित करता है
- खाली `input_reply` भेजता है
- निष्पादन को executor परत पर विफलता के रूप में माना जाता है

## आउटपुट कैप्चर और रेंडरिंग

### कैप्चर किए गए आउटपुट वर्ग

कर्नेल संदेशों से:

- `stream` -> सादा टेक्स्ट चंक
- `display_data`/`execute_result` -> रिच डिस्प्ले हैंडलिंग
- `error` -> ट्रेसबैक टेक्स्ट
- कस्टम MIME `application/x-xcsh-status` -> संरचित स्थिति इवेंट

डिस्प्ले MIME प्राथमिकता:

1. `text/markdown`
2. `text/plain`
3. `text/html` (बेसिक markdown में परिवर्तित)

अतिरिक्त रूप से संरचित आउटपुट के रूप में कैप्चर:

- `application/json` -> JSON ट्री डेटा
- `image/png` -> इमेज पेलोड
- `application/x-xcsh-status` -> स्थिति इवेंट

### स्टोरेज और ट्रंकेशन

आउटपुट `OutputSink` के माध्यम से स्ट्रीम किया जाता है और आर्टिफ़ैक्ट स्टोरेज में स्थायी किया जा सकता है।

टूल परिणामों में ट्रंकेशन मेटाडेटा और पूर्ण आउटपुट पुनर्प्राप्ति के लिए `artifact://<id>` शामिल हो सकता है।

### रेंडरर व्यवहार

- टूल रेंडरर (`python.ts`):
  - प्रति-सेल स्थिति के साथ कोड-सेल ब्लॉक दिखाता है
  - संक्षिप्त पूर्वावलोकन डिफ़ॉल्ट रूप से 10 पंक्तियों का होता है
  - पूर्ण आउटपुट और समृद्ध स्थिति विवरण के लिए विस्तारित मोड का समर्थन करता है
- इंटरैक्टिव रेंडरर (`python-execution.ts`):
  - TUI में उपयोगकर्ता द्वारा ट्रिगर किए गए Python निष्पादन के लिए उपयोग किया जाता है
  - संक्षिप्त पूर्वावलोकन डिफ़ॉल्ट रूप से 20 पंक्तियों का होता है
  - डिस्प्ले सुरक्षा के लिए बहुत लंबी व्यक्तिगत पंक्तियों को 4000 वर्णों तक सीमित करता है
  - रद्दीकरण/त्रुटि/ट्रंकेशन सूचनाएं दिखाता है

## बाहरी गेटवे समर्थन

सेट करें:

```bash
export PI_PYTHON_GATEWAY_URL="http://127.0.0.1:8888"
# Optional:
export PI_PYTHON_GATEWAY_TOKEN="..."
```

लोकल साझा गेटवे से व्यवहार अंतर:

- कोई लोकल गेटवे लॉक/info फ़ाइलें नहीं
- कोई लोकल प्रक्रिया स्पॉन/समाप्ति नहीं
- हेल्थ चेक और कर्नेल CRUD बाहरी एंडपॉइंट के विरुद्ध चलते हैं
- प्रमाणीकरण विफलताएं स्पष्ट टोकन मार्गदर्शन के साथ प्रदर्शित होती हैं

## परिचालन समस्या निवारण (वर्तमान विफलता मोड)

- **Python टूल उपलब्ध नहीं**
  - `python.toolMode` / `PI_PY` जांचें।
  - यदि प्रीफ़्लाइट विफल होता है, तो रनटाइम bash-only पर वापस आ जाता है।

- **कर्नेल उपलब्धता त्रुटियां**
  - लोकल मोड में रिज़ॉल्व किए गए Python रनटाइम में `kernel_gateway` और `ipykernel` दोनों इम्पोर्ट करने योग्य होने चाहिए।
  - इससे इंस्टॉल करें:

    ```bash
    python -m pip install jupyter_kernel_gateway ipykernel
    ```

- **`python.sharedGateway=false` स्टार्टअप विफलता का कारण बनता है**
  - वर्तमान कार्यान्वयन के साथ यह अपेक्षित है।

- **बाहरी गेटवे प्रमाणीकरण/पहुंच विफलताएं**
  - 401/403 -> `PI_PYTHON_GATEWAY_TOKEN` सेट करें।
  - टाइमआउट/अप्राप्य -> URL/नेटवर्क और गेटवे हेल्थ सत्यापित करें।

- **निष्पादन अटक जाता है फिर टाइमआउट होता है**
  - यदि कार्यभार वैध है तो टूल `timeout` बढ़ाएं (अधिकतम 600s)।
  - अटके हुए कोड के लिए, रद्दीकरण कर्नेल इंटरप्ट ट्रिगर करता है लेकिन उपयोगकर्ता कोड को अभी भी रिफैक्टर करने की आवश्यकता हो सकती है।

- **Python कोड में stdin/input प्रॉम्प्ट**
  - इस रनटाइम पथ में `input()` इंटरैक्टिव रूप से समर्थित नहीं है; डेटा प्रोग्रामेटिक रूप से पास करें।

- **संसाधन थकान (`EMFILE` / बहुत अधिक खुली फ़ाइलें)**
  - सत्र प्रबंधक साझा-गेटवे पुनर्प्राप्ति ट्रिगर करता है (सत्र विघटन + साझा गेटवे पुनरारंभ)।

- **कार्य निर्देशिका त्रुटियां**
  - टूल निष्पादन से पहले सत्यापित करता है कि `cwd` मौजूद है और एक निर्देशिका है।

## प्रासंगिक एनवायरनमेंट वेरिएबल

- `PI_PY` — टूल एक्सपोज़र ओवरराइड (ऊपर `bash-only`/`ipy-only`/`both` मैपिंग)
- `PI_PYTHON_GATEWAY_URL` — बाहरी गेटवे का उपयोग करें
- `PI_PYTHON_GATEWAY_TOKEN` — वैकल्पिक बाहरी गेटवे प्रमाणीकरण टोकन
- `PI_PYTHON_SKIP_CHECK=1` — Python प्रीफ़्लाइट/वार्म जांच बायपास करें
- `PI_PYTHON_IPC_TRACE=1` — कर्नेल IPC सेंड/रिसीव ट्रेस लॉग करें
- `PI_DEBUG_STARTUP=1` — स्टार्टअप-चरण डीबग मार्कर उत्सर्जित करें
