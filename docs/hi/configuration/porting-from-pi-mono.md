## 0) स्कोप परिभाषित करें

- अपस्ट्रीम संदर्भ (कमिट, टैग, या PR) की पहचान करें।
- उन पैकेजों या फ़ोल्डरों की सूची बनाएं जिन्हें आप संशोधित करने की योजना बना रहे हैं।

## 1) कोड को सुरक्षित रूप से लाएं

- थोक कॉपी के बजाय एक स्वच्छ, केंद्रित diff को प्राथमिकता दें।
- बिल्ट आर्टिफैक्ट्स या जनरेटेड फ़ाइलों को कॉपी करने से बचें।
- यदि अपस्ट्रीम ने नई फ़ाइलें जोड़ी हैं, तो उन्हें स्पष्ट रूप से जोड़ें और सामग्री की समीक्षा करें।

## 2) इम्पोर्ट एक्सटेंशन कन्वेंशन का मिलान करें

अधिकांश रनटाइम TypeScript स्रोत आंतरिक इम्पोर्ट में `.js` को छोड़ देते हैं, लेकिन कुछ test/bench एंट्रीपॉइंट ESM
रनटाइम संगतता के लिए `.js` रखते हैं। स्थानीय पैकेज की मौजूदा शैली का पालन करें; बिना सोचे-समझे एक्सटेंशन न हटाएं।

- `packages/coding-agent` रनटाइम स्रोतों में, जब तक नॉन-TS एसेट्स इम्पोर्ट नहीं कर रहे हों, आंतरिक इम्पोर्ट को एक्सटेंशन रहित रखें।
- `packages/tui/test` और `packages/natives/bench` में, जहां आसपास की फ़ाइलें पहले से `.js` का उपयोग करती हैं, वहां `.js` रखें।
- जब टूलिंग द्वारा आवश्यक हो तो वास्तविक फ़ाइल एक्सटेंशन रखें (जैसे, `.json`, `.css`, `.md` टेक्स्ट एम्बेड)।
- उदाहरण: `import { x } from "./foo.js";` → `import { x } from "./foo";` (केवल तब जब पैकेज कन्वेंशन एक्सटेंशन रहित हो)।

## 4) जहां Bun API बेहतर हों, वहां उनका उपयोग करें

हम Bun पर चलते हैं। Node API को केवल तभी बदलें जब Bun बेहतर विकल्प प्रदान करता हो।

**बदलें:**

- प्रोसेस स्पॉनिंग: `child_process.spawn` → सरल कमांड के लिए Bun Shell `$`, स्ट्रीमिंग या लंबे समय तक चलने वाले कार्य के लिए `Bun.spawn`/`Bun.spawnSync`
- फ़ाइल I/O: `fs.readFileSync` → `Bun.file().text()` / `Bun.write()`
- HTTP क्लाइंट: `node-fetch`, `axios` → नेटिव `fetch`
- क्रिप्टो हैशिंग: `node:crypto` → Web Crypto या `Bun.hash`
- SQLite: `better-sqlite3` → `bun:sqlite`
- Env लोडिंग: `dotenv` → Bun स्वचालित रूप से `.env` लोड करता है

**न बदलें (ये Bun में ठीक काम करते हैं):**

- `os.homedir()` — `Bun.env.HOME`, `Bun.env.HOME`, या शाब्दिक `"~"` से न बदलें
- `os.tmpdir()` — `Bun.env.TMPDIR || "/tmp"` या हार्डकोडेड पथों से न बदलें
- `fs.mkdtempSync()` — मैन्युअल पथ निर्माण से न बदलें
- `path.join()`, `path.resolve()`, आदि — ये ठीक हैं

**इम्पोर्ट शैली:** `node:` प्रीफ़िक्स का उपयोग केवल नेमस्पेस इम्पोर्ट के साथ करें (कोई नेम्ड इम्पोर्ट `node:fs` या `node:path` से नहीं)।

**अतिरिक्त Bun कन्वेंशन:**

- छोटे, नॉन-स्ट्रीमिंग कमांड के लिए Bun Shell `$` को प्राथमिकता दें; `Bun.spawn` का उपयोग केवल तभी करें जब आपको स्ट्रीमिंग I/O या प्रोसेस कंट्रोल की आवश्यकता हो।
- फ़ाइलों के लिए `Bun.file()`/`Bun.write()` और डायरेक्टरीज के लिए `node:fs/promises` का उपयोग करें।
- `Bun.file().exists()` जांच से बचें; try/catch में `isEnoent` हैंडलिंग का उपयोग करें।
- `setTimeout` रैपर की बजाय `Bun.sleep(ms)` को प्राथमिकता दें।

**गलत:**

```typescript
// BROKEN: env vars may be undefined, "~" is not expanded
const home = Bun.env.HOME || "~";
const tmp = Bun.env.TMPDIR || "/tmp";
```

**सही:**

```typescript
import * as os from "node:os";
import * as fs from "node:fs";
import * as path from "node:path";

const configDir = path.join(os.homedir(), ".config", "myapp");
const tempDir = fs.mkdtempSync(path.join(os.tmpdir(), "myapp-"));
```

## 5) Bun एम्बेड को प्राथमिकता दें (कॉपी नहीं)

बिल्ड समय पर रनटाइम एसेट्स या वेंडर फ़ाइलों को कॉपी न करें।

- यदि अपस्ट्रीम एसेट्स को dist फ़ोल्डर में कॉपी करता है, तो Bun-अनुकूल एम्बेड से बदलें।
- प्रॉम्प्ट स्थिर `.md` फ़ाइलें हैं; इनलाइन प्रॉम्प्ट स्ट्रिंग्स के बजाय Bun टेक्स्ट इम्पोर्ट (`with { type: "text" }`) और Handlebars का उपयोग करें।
- आसपास के नॉन-टेक्स्ट रिसोर्स लोड करने के लिए `import.meta.dir` + `Bun.file` का उपयोग करें।
- एसेट्स को इन-रेपो रखें और बंडलर को उन्हें शामिल करने दें।
- जब तक उपयोगकर्ता स्पष्ट रूप से अनुरोध नहीं करता, कॉपी स्क्रिप्ट हटा दें।
- यदि अपस्ट्रीम रनटाइम पर बंडल्ड फ़ॉलबैक फ़ाइल पढ़ता है, तो फ़ाइलसिस्टम रीड्स को Bun टेक्स्ट एम्बेड इम्पोर्ट से बदलें।
  - उदाहरण (Codex निर्देश फ़ॉलबैक):
    - `const FALLBACK_PROMPT_PATH = join(import.meta.dir, "codex-instructions.md");` -> हटाया गया
    - `import FALLBACK_INSTRUCTIONS from "./codex-instructions.md" with { type: "text" };`
    - `readFileSync(FALLBACK_PROMPT_PATH, "utf8")` के बजाय `return FALLBACK_INSTRUCTIONS;` का उपयोग करें

## 6) `package.json` को सावधानी से पोर्ट करें

`package.json` को एक अनुबंध मानें। जानबूझकर मर्ज करें।

- मौजूदा `name`, `version`, `type`, `exports`, और `bin` को तब तक बनाए रखें जब तक पोर्ट को परिवर्तन की आवश्यकता न हो।
- npm/node स्क्रिप्ट को Bun समकक्षों से बदलें (जैसे, `bun check`, `bun test`)।
- सुनिश्चित करें कि डिपेंडेंसीज सही स्कोप का उपयोग करती हैं।
- टाइप एरर ठीक करने के लिए डिपेंडेंसीज को डाउनग्रेड न करें; इसके बजाय अपग्रेड करें।
- वर्कस्पेस पैकेज लिंक और `peerDependencies` को सत्यापित करें।

## 7) कोड शैली और टूलिंग को संरेखित करें

- मौजूदा फ़ॉर्मेटिंग कन्वेंशन बनाए रखें।
- जब तक आवश्यक न हो `any` न लाएं।
- डायनामिक इम्पोर्ट और इनलाइन टाइप इम्पोर्ट से बचें; केवल टॉप-लेवल इम्पोर्ट का उपयोग करें।
- कोड में कभी प्रॉम्प्ट न बनाएं; प्रॉम्प्ट Handlebars के साथ रेंडर की गई स्थिर `.md` फ़ाइलें हैं।
- coding-agent में, कभी `console.log`/`console.warn`/`console.error` का उपयोग न करें; `@f5xc-salesdemos/pi-utils` से `logger` का उपयोग करें।
- `new Promise((resolve, reject) => ...)` के बजाय `Promise.withResolvers()` का उपयोग करें।
- **क्लास फ़ील्ड या मेथड पर `private`/`protected`/`public` कीवर्ड न लगाएं।** एनकैप्सुलेशन के लिए ES `#` प्राइवेट फ़ील्ड का उपयोग करें; एक्सेसिबल सदस्यों को बिना कीवर्ड के रखें। एकमात्र अपवाद कंस्ट्रक्टर पैरामीटर प्रॉपर्टीज़ (`constructor(private readonly x: T)`) हैं, जहां TypeScript द्वारा कीवर्ड आवश्यक है। अपस्ट्रीम कोड जो `private foo` या `protected bar` का उपयोग करता है उसे पोर्ट करते समय, `#foo` (प्राइवेट) या बेयर `bar` (एक्सेसिबल) में बदलें।
- नए एड-हॉक कोड की बजाय मौजूदा हेल्पर और यूटिलिटीज को प्राथमिकता दें।
- इस रेपो में पहले से किए गए Bun-फ़र्स्ट इंफ्रास्ट्रक्चर परिवर्तनों को संरक्षित करें:
  - रनटाइम Bun है (कोई Node एंट्री पॉइंट नहीं)।
  - पैकेज मैनेजर Bun है (कोई npm लॉकफ़ाइल नहीं)।
  - भारी Node API (`child_process`, `readline`) को Bun समकक्षों से बदला गया है।
  - हल्के Node API (`os.homedir`, `os.tmpdir`, `fs.mkdtempSync`, `path.*`) को रखा गया है।
  - CLI शीबैंग `bun` का उपयोग करते हैं (`node` नहीं, `tsx` नहीं)।
  - पैकेज सीधे सोर्स फ़ाइलों का उपयोग करते हैं (कोई TypeScript बिल्ड स्टेप नहीं)।
  - CI वर्कफ़्लो install/check/test के लिए Bun चलाते हैं।

## 10) पोर्ट को सत्यापित करें

परिवर्तनों के बाद मानक जांच चलाएं:

- `bun check`

यदि रेपो में पहले से ही आपके परिवर्तनों से असंबंधित विफल जांच हैं, तो उसे बताएं।
टेस्ट Bun के रनर का उपयोग करते हैं (Vitest नहीं), लेकिन `bun test` केवल तभी चलाएं जब स्पष्ट रूप से अनुरोध किया जाए।

## 11) सुधारी गई सुविधाओं की रक्षा करें (रिग्रेशन ट्रैप सूची)

यदि आपने पहले से स्थानीय रूप से व्यवहार में सुधार किया है, तो उन्हें **अ-समझौता योग्य** मानें। पोर्टिंग से पहले,
सुधारों को लिख लें और स्पष्ट जांच जोड़ें ताकि वे मर्ज में खो न जाएं।

- **अपेक्षित व्यवहार को फ़्रीज़ करें**: प्रत्येक सुधार के लिए एक संक्षिप्त "पहले/बाद" नोट जोड़ें (इनपुट, आउटपुट,
  डिफ़ॉल्ट, एज केस)। यह मौन रोलबैक को रोकता है।
- **पुराने → नए API का मैप बनाएं**: यदि अपस्ट्रीम ने अवधारणाओं का नाम बदला (hooks → extensions, custom tools → tools, आदि),
  तो सुनिश्चित करें कि हर पुराना एंट्री पॉइंट अभी भी जुड़ा है। एक छूटा हुआ फ़्लैग या एक्सपोर्ट = खोई हुई कार्यक्षमता।
- **एक्सपोर्ट सत्यापित करें**: `package.json` `exports`, पब्लिक टाइप, और बैरल फ़ाइलों की जांच करें। अपस्ट्रीम पोर्ट
  अक्सर स्थानीय परिवर्धन को री-एक्सपोर्ट करना भूल जाते हैं।
- **नॉन-हैप्पी पथों को कवर करें**: यदि आपने एरर हैंडलिंग, टाइमआउट, या फ़ॉलबैक लॉजिक ठीक किया है, तो एक टेस्ट जोड़ें या
  कम से कम एक मैन्युअल चेकलिस्ट बनाएं जो उन पथों का परीक्षण करे।
- **डिफ़ॉल्ट और कॉन्फ़िग मर्ज क्रम की जांच करें**: सुधार अक्सर डिफ़ॉल्ट में रहते हैं। पुष्टि करें कि नए डिफ़ॉल्ट
  वापस नहीं गए (जैसे, नई कॉन्फ़िग प्राथमिकता, अक्षम सुविधाएं, टूल सूचियां)।
- **env/shell व्यवहार का ऑडिट करें**: यदि आपने निष्पादन या सैंडबॉक्सिंग ठीक की है, तो सत्यापित करें कि नया पथ अभी भी आपके
  सैनिटाइज़्ड env का उपयोग करता है और alias/function ओवरराइड को फिर से नहीं लाता।
- **लक्षित सैंपल फिर से चलाएं**: "ज्ञात अच्छे" उदाहरणों का एक न्यूनतम सेट रखें और पोर्ट के बाद उन्हें चलाएं
  (CLI फ़्लैग, एक्सटेंशन पंजीकरण, टूल निष्पादन)।

## 12) रीवर्क किए गए कोड का पता लगाएं और संभालें

फ़ाइल पोर्ट करने से पहले, जांचें कि क्या अपस्ट्रीम ने इसे महत्वपूर्ण रूप से रिफ़ैक्टर किया है:

```bash
# Compare the file you're about to port against what you have locally
git diff HEAD upstream/main -- path/to/file.ts
```

यदि diff दिखाता है कि फ़ाइल **रीवर्क** की गई थी (केवल पैच नहीं):

- नई एब्स्ट्रैक्शन, नाम बदली गई अवधारणाएं, मर्ज किए गए मॉड्यूल, बदला हुआ डेटा फ़्लो

तब आपको पोर्टिंग से पहले **नए कार्यान्वयन को पूरी तरह से पढ़ना होगा**। रीवर्क किए गए कोड का अंधा मर्ज कार्यक्षमता खो देता है क्योंकि:

नोट: इंटरैक्टिव मोड को हाल ही में controllers/utils/types में विभाजित किया गया था। संबंधित परिवर्तनों को बैकपोर्ट करते समय, हमारे द्वारा बनाई गई व्यक्तिगत फ़ाइलों में अपडेट पोर्ट करें और सुनिश्चित करें कि `interactive-mode.ts` वायरिंग सिंक में रहे।

1. **डिफ़ॉल्ट चुपचाप बदल जाते हैं** - एक नया वेरिएबल `defaultFoo = [a, b]` एक पुराने `getAllFoo()` को बदल सकता है जो `[a, b, c, d, e]` लौटाता था।

2. **API विकल्प छूट जाते हैं** - जब सिस्टम मर्ज होते हैं (जैसे, `hooks` + `customTools` → `extensions`), पुराने विकल्प नए कार्यान्वयन तक नहीं पहुंच सकते।

3. **कोड पथ बासी हो जाते हैं** - एक नाम बदली गई अवधारणा (जैसे, `hookMessage` → `custom`) को हर switch स्टेटमेंट, टाइप गार्ड, और हैंडलर में अपडेट की आवश्यकता होती है—केवल परिभाषा में नहीं।

4. **संदर्भ/क्षमताएं सिकुड़ जाती हैं** - पुराने API `{ logger, typebox, pi }` एक्सपोज़ कर सकते थे जिन्हें नए API ने शामिल करना भूल दिया।

### सिमैंटिक पोर्टिंग प्रक्रिया

जब अपस्ट्रीम ने एक मॉड्यूल को रीवर्क किया:

1. **पुराना कार्यान्वयन पढ़ें** - समझें कि यह क्या करता था, कौन से विकल्प स्वीकार करता था, क्या एक्सपोज़ करता था।

2. **नया कार्यान्वयन पढ़ें** - नई एब्स्ट्रैक्शन को समझें और वे पुराने व्यवहार से कैसे मैप होती हैं।

3. **फ़ीचर समानता सत्यापित करें** - पुराने कोड की प्रत्येक क्षमता के लिए, पुष्टि करें कि नया कोड इसे संरक्षित करता है या स्पष्ट रूप से हटाता है।

4. **अवशेषों के लिए grep करें** - पुराने नामों/अवधारणाओं की खोज करें जो switch स्टेटमेंट, हैंडलर, UI कंपोनेंट में छूट गए हो सकते हैं।

5. **सीमाओं का परीक्षण करें** - CLI फ़्लैग, SDK विकल्प, इवेंट हैंडलर, डिफ़ॉल्ट मान—ये वे जगहें हैं जहां रिग्रेशन छिपते हैं।

### त्वरित जांच

```bash
# Find all uses of an old concept that may need updating
rg "oldConceptName" --type ts

# Compare default values between versions
git show upstream/main:path/to/file.ts | rg "default|DEFAULT"

# Check if all enum/union values have handlers
rg "case \"" path/to/file.ts
```

## 14) कमिट संदेश प्रारूप

बैकपोर्ट कमिट करते समय, रेपो प्रारूप `<type>(scope): <past-tense description>` का पालन करें और कमिट
रेंज को शीर्षक में रखें।

```
fix(coding-agent): backported pi-mono changes (<from>..<to>)

packages/<package>:
- <type>: <description>
- <type>: <description> (#<issue> by @<contributor>)

packages/<other-package>:
- <type>: <description>
```

**उदाहरण:**

```
fix(coding-agent): backported pi-mono changes (9f3eef65f..52532c7c0)

packages/ai:
- fix: handle "sensitive" stop reason from Anthropic API
- fix: normalize tool call IDs with special characters for Responses API
- fix: add overflow detection for Bedrock, MiniMax, Kimi providers
- fix: 429 status is rate limiting, not context overflow

packages/tui:
- fix: refactored autocomplete state tracking
- fix: file autocomplete should not trigger on empty text
- fix: configurable autocomplete max visible items
- fix: improved table column width calculation with word-aware wrapping

packages/coding-agent:
- fix: preserve external config.yml edits on save (#1046 by @nicobailonMD)
- fix: resolve macOS NFD and curly quote variants in file paths
```

**नियम:**

- परिवर्तनों को पैकेज के अनुसार समूहित करें
- पारंपरिक कमिट प्रकारों का उपयोग करें (`fix`, `feat`, `refactor`, `perf`, `docs`)
- बाहरी योगदान के लिए अपस्ट्रीम issue/PR नंबर और योगदानकर्ता विशेषता शामिल करें
- शीर्षक में कमिट रेंज सिंक बिंदुओं को ट्रैक करने में मदद करती है
