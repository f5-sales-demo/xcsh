# pi-mono से पोर्टिंग: एक व्यावहारिक मर्ज गाइड

## 0) स्कोप परिभाषित करें

- अपस्ट्रीम संदर्भ (कमिट, टैग, या PR) की पहचान करें।
- उन पैकेज या फ़ोल्डर्स की सूची बनाएं जिन्हें आप बदलना चाहते हैं।
- तय करें कि कौन से फ़ीचर स्कोप में हैं और कौन से जानबूझकर छोड़े गए हैं।

## 1) कोड सुरक्षित रूप से लाएं

- थोक कॉपी के बजाय एक साफ, फोकस्ड डिफ को प्राथमिकता दें।
- बिल्ट आर्टिफैक्ट्स या जनरेटेड फ़ाइलों को कॉपी करने से बचें।
- यदि अपस्ट्रीम ने नई फ़ाइलें जोड़ी हैं, तो उन्हें स्पष्ट रूप से जोड़ें और सामग्री की समीक्षा करें।

## 2) इम्पोर्ट एक्सटेंशन कन्वेंशन का मिलान करें

अधिकांश रनटाइम TypeScript स्रोत आंतरिक इम्पोर्ट्स में `.js` छोड़ देते हैं, लेकिन कुछ test/bench एंट्रीपॉइंट्स ESM
रनटाइम संगतता के लिए `.js` रखते हैं। लोकल पैकेज की मौजूदा शैली का पालन करें; एक्सटेंशन को अंधाधुंध न हटाएं।

- `packages/coding-agent` रनटाइम स्रोतों में, आंतरिक इम्पोर्ट्स को बिना एक्सटेंशन रखें जब तक कि नॉन-TS एसेट्स इम्पोर्ट न कर रहे हों।
- `packages/tui/test` और `packages/natives/bench` में, `.js` वहां रखें जहां आसपास की फ़ाइलें पहले से इसका उपयोग करती हैं।
- जब टूलिंग द्वारा आवश्यक हो तो वास्तविक फ़ाइल एक्सटेंशन रखें (जैसे, `.json`, `.css`, `.md` टेक्स्ट एम्बेड्स)।
- उदाहरण: `import { x } from "./foo.js";` → `import { x } from "./foo";` (केवल तब जब पैकेज कन्वेंशन एक्सटेंशनलेस हो)।

## 4) जहां Bun APIs Node से बेहतर हों वहां उनका उपयोग करें

हम Bun पर चलते हैं। Node APIs को केवल तभी बदलें जब Bun बेहतर विकल्प प्रदान करता हो।

**बदलें:**

- प्रोसेस स्पॉनिंग: `child_process.spawn` → सरल कमांड के लिए Bun Shell `$`, स्ट्रीमिंग या लंबे समय तक चलने वाले कार्य के लिए `Bun.spawn`/`Bun.spawnSync`
- फ़ाइल I/O: `fs.readFileSync` → `Bun.file().text()` / `Bun.write()`
- HTTP क्लाइंट: `node-fetch`, `axios` → नेटिव `fetch`
- क्रिप्टो हैशिंग: `node:crypto` → Web Crypto या `Bun.hash`
- SQLite: `better-sqlite3` → `bun:sqlite`
- Env लोडिंग: `dotenv` → Bun स्वचालित रूप से `.env` लोड करता है

**न बदलें (ये Bun में ठीक काम करते हैं):**

- `os.homedir()` — `Bun.env.HOME`, `Bun.env.HOME`, या लिटरल `"~"` से न बदलें
- `os.tmpdir()` — `Bun.env.TMPDIR || "/tmp"` या हार्डकोडेड पथ से न बदलें
- `fs.mkdtempSync()` — मैनुअल पथ निर्माण से न बदलें
- `path.join()`, `path.resolve()`, आदि — ये ठीक हैं

**इम्पोर्ट शैली:** `node:` प्रीफिक्स का उपयोग केवल नेमस्पेस इम्पोर्ट्स के साथ करें (कोई नेम्ड इम्पोर्ट्स `node:fs` या `node:path` से नहीं)।

**अतिरिक्त Bun कन्वेंशन:**

- छोटे, नॉन-स्ट्रीमिंग कमांड के लिए Bun Shell `$` को प्राथमिकता दें; `Bun.spawn` का उपयोग केवल तभी करें जब आपको स्ट्रीमिंग I/O या प्रोसेस कंट्रोल की आवश्यकता हो।
- फ़ाइलों के लिए `Bun.file()`/`Bun.write()` और डायरेक्ट्रीज़ के लिए `node:fs/promises` का उपयोग करें।
- `Bun.file().exists()` जांच से बचें; try/catch में `isEnoent` हैंडलिंग का उपयोग करें।
- `setTimeout` रैपर्स के बजाय `Bun.sleep(ms)` को प्राथमिकता दें।

**गलत:**

```typescript
// BROKEN: env vars may be undefined, "~" is not expanded
const home = Bun.env.HOME || "~";
const tmp = Bun.env.TMPDIR || "/tmp";
```

**सही:**

```typescript
import * as os from "node:os";
import * as fs from "node:fs";
import * as path from "node:path";

const configDir = path.join(os.homedir(), ".config", "myapp");
const tempDir = fs.mkdtempSync(path.join(os.tmpdir(), "myapp-"));
```

## 5) Bun एम्बेड्स को प्राथमिकता दें (कॉपी नहीं)

बिल्ड समय पर रनटाइम एसेट्स या वेंडर फ़ाइलों को कॉपी न करें।

- यदि अपस्ट्रीम एसेट्स को dist फ़ोल्डर में कॉपी करता है, तो Bun-अनुकूल एम्बेड्स से बदलें।
- प्रॉम्प्ट्स स्थिर `.md` फ़ाइलें हैं; इनलाइन प्रॉम्प्ट स्ट्रिंग्स के बजाय Bun टेक्स्ट इम्पोर्ट्स (`with { type: "text" }`) और Handlebars का उपयोग करें।
- आसन्न नॉन-टेक्स्ट संसाधनों को लोड करने के लिए `import.meta.dir` + `Bun.file` का उपयोग करें।
- एसेट्स को रेपो में रखें और बंडलर को उन्हें शामिल करने दें।
- जब तक उपयोगकर्ता स्पष्ट रूप से अनुरोध न करे, कॉपी स्क्रिप्ट हटा दें।
- यदि अपस्ट्रीम रनटाइम पर एक बंडल्ड फ़ॉलबैक फ़ाइल पढ़ता है, तो फ़ाइलसिस्टम रीड्स को Bun टेक्स्ट एम्बेड इम्पोर्ट से बदलें।
  - उदाहरण (Codex instructions फ़ॉलबैक):
    - `const FALLBACK_PROMPT_PATH = join(import.meta.dir, "codex-instructions.md");` -> हटाया गया
    - `import FALLBACK_INSTRUCTIONS from "./codex-instructions.md" with { type: "text" };`
    - `readFileSync(FALLBACK_PROMPT_PATH, "utf8")` के बजाय `return FALLBACK_INSTRUCTIONS;` का उपयोग करें

## 6) `package.json` सावधानी से पोर्ट करें

`package.json` को एक अनुबंध के रूप में मानें। जानबूझकर मर्ज करें।

- मौजूदा `name`, `version`, `type`, `exports`, और `bin` रखें जब तक कि पोर्ट में परिवर्तन आवश्यक न हों।
- npm/node स्क्रिप्ट्स को Bun समकक्षों से बदलें (जैसे, `bun check`, `bun test`)।
- सुनिश्चित करें कि डिपेंडेंसीज़ सही स्कोप का उपयोग करती हैं।
- टाइप एरर ठीक करने के लिए डिपेंडेंसीज़ को डाउनग्रेड न करें; बल्कि अपग्रेड करें।
- वर्कस्पेस पैकेज लिंक और `peerDependencies` को सत्यापित करें।

## 7) कोड शैली और टूलिंग को संरेखित करें

- मौजूदा फ़ॉर्मेटिंग कन्वेंशन बनाए रखें।
- जब तक आवश्यक न हो `any` न लाएं।
- डायनामिक इम्पोर्ट्स और इनलाइन टाइप इम्पोर्ट्स से बचें; केवल टॉप-लेवल इम्पोर्ट्स का उपयोग करें।
- कोड में प्रॉम्प्ट्स कभी न बनाएं; प्रॉम्प्ट्स स्थिर `.md` फ़ाइलें हैं जो Handlebars के साथ रेंडर होती हैं।
- coding-agent में, `console.log`/`console.warn`/`console.error` कभी न उपयोग करें; `@f5xc-salesdemos/pi-utils` से `logger` का उपयोग करें।
- `new Promise((resolve, reject) => ...)` के बजाय `Promise.withResolvers()` का उपयोग करें।
- **क्लास फ़ील्ड्स या मेथड्स पर `private`/`protected`/`public` कीवर्ड नहीं।** इनकैप्सुलेशन के लिए ES `#` प्राइवेट फ़ील्ड्स का उपयोग करें; एक्सेसिबल सदस्यों को बिना किसी कीवर्ड के रखें। एकमात्र अपवाद कंस्ट्रक्टर पैरामीटर प्रॉपर्टीज़ (`constructor(private readonly x: T)`) हैं, जहां TypeScript द्वारा कीवर्ड आवश्यक है। अपस्ट्रीम कोड पोर्ट करते समय जो `private foo` या `protected bar` का उपयोग करता है, `#foo` (प्राइवेट) या बेयर `bar` (एक्सेसिबल) में बदलें।
- नए एड-हॉक कोड के बजाय मौजूदा हेल्पर्स और यूटिलिटीज़ को प्राथमिकता दें।
- इस रेपो में पहले से किए गए Bun-फर्स्ट इन्फ्रास्ट्रक्चर परिवर्तनों को संरक्षित रखें:
  - रनटाइम Bun है (कोई Node एंट्री पॉइंट नहीं)।
  - पैकेज मैनेजर Bun है (कोई npm लॉकफ़ाइल नहीं)।
  - भारी Node APIs (`child_process`, `readline`) को Bun समकक्षों से बदला गया है।
  - हल्के Node APIs (`os.homedir`, `os.tmpdir`, `fs.mkdtempSync`, `path.*`) रखे गए हैं।
  - CLI shebangs `bun` का उपयोग करते हैं (`node` नहीं, `tsx` नहीं)।
  - पैकेज सीधे सोर्स फ़ाइलों का उपयोग करते हैं (कोई TypeScript बिल्ड स्टेप नहीं)।
  - CI वर्कफ़्लो install/check/test के लिए Bun चलाते हैं।

## 10) पोर्ट को मान्य करें

परिवर्तनों के बाद मानक जांच चलाएं:

- `bun check`

यदि रेपो में पहले से आपके परिवर्तनों से असंबंधित विफल जांच हैं, तो उसे इंगित करें।
टेस्ट Bun के रनर (Vitest नहीं) का उपयोग करते हैं, लेकिन `bun test` केवल तभी चलाएं जब स्पष्ट रूप से अनुरोध किया जाए।

## 11) सुधारे गए फ़ीचर्स की रक्षा करें (रिग्रेशन ट्रैप सूची)

यदि आपने पहले से स्थानीय रूप से व्यवहार में सुधार किया है, तो उन्हें **गैर-समझौता योग्य** मानें। पोर्ट करने से पहले,
सुधारों को लिख लें और स्पष्ट जांच जोड़ें ताकि वे मर्ज में खो न जाएं।

- **अपेक्षित व्यवहार को फ्रीज करें**: प्रत्येक सुधार के लिए एक संक्षिप्त "पहले/बाद" नोट जोड़ें (इनपुट, आउटपुट,
  डिफ़ॉल्ट, एज केस)। यह मौन रोलबैक को रोकता है।
- **पुराने → नए APIs का मानचित्र बनाएं**: यदि अपस्ट्रीम ने अवधारणाओं का नाम बदला है (hooks → extensions, custom tools → tools, आदि),
  तो सुनिश्चित करें कि प्रत्येक पुराना एंट्री पॉइंट अभी भी जुड़ा हुआ है। एक छूटा हुआ फ्लैग या एक्सपोर्ट का मतलब खोई हुई कार्यक्षमता है।
- **एक्सपोर्ट्स सत्यापित करें**: `package.json` `exports`, पब्लिक टाइप्स, और बैरल फ़ाइलों की जांच करें। अपस्ट्रीम पोर्ट अक्सर
  स्थानीय जोड़ को री-एक्सपोर्ट करना भूल जाते हैं।
- **नॉन-हैप्पी पाथ कवर करें**: यदि आपने एरर हैंडलिंग, टाइमआउट, या फ़ॉलबैक लॉजिक ठीक किया है, तो एक टेस्ट या कम से कम
  एक मैनुअल चेकलिस्ट जोड़ें जो उन पथों का परीक्षण करे।
- **डिफ़ॉल्ट और कॉन्फ़िग मर्ज ऑर्डर की जांच करें**: सुधार अक्सर डिफ़ॉल्ट में रहते हैं। पुष्टि करें कि नए डिफ़ॉल्ट
  वापस नहीं गए (जैसे, नई कॉन्फ़िग प्राथमिकता, अक्षम फ़ीचर्स, टूल सूचियां)।
- **env/shell व्यवहार की जांच करें**: यदि आपने एक्ज़ीक्यूशन या सैंडबॉक्सिंग को ठीक किया है, तो सत्यापित करें कि नया पथ अभी भी आपके
  सैनिटाइज़्ड env का उपयोग करता है और alias/function ओवरराइड को पुनः प्रस्तुत नहीं करता।
- **लक्षित सैंपल पुनः चलाएं**: "ज्ञात अच्छे" उदाहरणों का एक न्यूनतम सेट रखें और पोर्ट के बाद उन्हें चलाएं
  (CLI फ्लैग्स, एक्सटेंशन रजिस्ट्रेशन, टूल एक्ज़ीक्यूशन)।

## 12) पुनर्निर्मित कोड का पता लगाएं और संभालें

फ़ाइल पोर्ट करने से पहले, जांचें कि क्या अपस्ट्रीम ने इसे महत्वपूर्ण रूप से पुनर्निर्मित किया है:

```bash
# Compare the file you're about to port against what you have locally
git diff HEAD upstream/main -- path/to/file.ts
```

यदि डिफ दिखाता है कि फ़ाइल **पुनर्निर्मित** (केवल पैच नहीं) की गई थी:

- नई एब्स्ट्रैक्शन, नामित अवधारणाएं, मर्ज किए गए मॉड्यूल, बदला हुआ डेटा प्रवाह

तो आपको पोर्ट करने से पहले **नए कार्यान्वयन को अच्छी तरह पढ़ना** होगा। पुनर्निर्मित कोड का अंधा मर्ज कार्यक्षमता खो देता है क्योंकि:

नोट: इंटरैक्टिव मोड को हाल ही में controllers/utils/types में विभाजित किया गया था। संबंधित परिवर्तनों को बैकपोर्ट करते समय, अपडेट को हमारी बनाई गई अलग-अलग फ़ाइलों में पोर्ट करें और सुनिश्चित करें कि `interactive-mode.ts` वायरिंग सिंक में रहे।

1. **डिफ़ॉल्ट मौन रूप से बदलते हैं** - एक नया वैरिएबल `defaultFoo = [a, b]` पुराने `getAllFoo()` को बदल सकता है जो `[a, b, c, d, e]` लौटाता था।

2. **API विकल्प छूट जाते हैं** - जब सिस्टम मर्ज होते हैं (जैसे, `hooks` + `customTools` → `extensions`), पुराने विकल्प नए कार्यान्वयन में वायर नहीं हो सकते।

3. **कोड पथ पुराने हो जाते हैं** - एक नामित अवधारणा (जैसे, `hookMessage` → `custom`) को हर switch स्टेटमेंट, टाइप गार्ड, और हैंडलर में अपडेट की आवश्यकता होती है—केवल परिभाषा में नहीं।

4. **संदर्भ/क्षमताएं सिकुड़ती हैं** - पुराने APIs ने `{ logger, typebox, pi }` एक्सपोज़ किया हो सकता है जो नए APIs शामिल करना भूल गए।

### सिमेंटिक पोर्टिंग प्रक्रिया

जब अपस्ट्रीम ने एक मॉड्यूल को पुनर्निर्मित किया:

1. **पुराना कार्यान्वयन पढ़ें** - समझें कि यह क्या करता था, कौन से विकल्प स्वीकार करता था, क्या एक्सपोज़ करता था।

2. **नया कार्यान्वयन पढ़ें** - नई एब्स्ट्रैक्शन को समझें और वे पुराने व्यवहार से कैसे मैप होती हैं।

3. **फ़ीचर समानता सत्यापित करें** - पुराने कोड में प्रत्येक क्षमता के लिए, पुष्टि करें कि नया कोड इसे संरक्षित करता है या स्पष्ट रूप से हटाता है।

4. **बचे हुए के लिए खोजें** - पुराने नामों/अवधारणाओं की खोज करें जो switch स्टेटमेंट्स, हैंडलर्स, UI कंपोनेंट्स में छूट गए हो सकते हैं।

5. **सीमाओं का परीक्षण करें** - CLI फ्लैग्स, SDK विकल्प, इवेंट हैंडलर्स, डिफ़ॉल्ट मान—यहीं रिग्रेशन छिपते हैं।

### त्वरित जांच

```bash
# Find all uses of an old concept that may need updating
rg "oldConceptName" --type ts

# Compare default values between versions
git show upstream/main:path/to/file.ts | rg "default|DEFAULT"

# Check if all enum/union values have handlers
rg "case \"" path/to/file.ts
```

## 14) कमिट संदेश प्रारूप

बैकपोर्ट कमिट करते समय, रेपो प्रारूप `<type>(scope): <past-tense description>` का पालन करें और कमिट

```
- <type>: <description>
```

**उदाहरण:**

```
- fix: normalize tool call IDs with special characters for Responses API

packages/tui:
- fix: refactored autocomplete state tracking
- fix: file autocomplete should not trigger on empty text
- fix: configurable autocomplete max visible items
- fix: improved table column width calculation with word-aware wrapping

packages/coding-agent:
- fix: preserve external config.yml edits on save (#1046 by @nicobailonMD)
- fix: resolve macOS NFD and curly quote variants in file paths
```

**नियम:**

- परिवर्तनों को पैकेज के अनुसार समूहित करें
- पारंपरिक कमिट टाइप्स का उपयोग करें (`fix`, `feat`, `refactor`, `perf`, `docs`)
- बाहरी योगदानों के लिए अपस्ट्रीम issue/PR नंबर और योगदानकर्ता एट्रिब्यूशन शामिल करें
- शीर्षक में कमिट रेंज सिंक पॉइंट्स को ट्रैक करने में मदद करती है
