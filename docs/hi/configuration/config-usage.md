# कॉन्फ़िगरेशन डिस्कवरी और रिज़ॉल्यूशन

यह दस्तावेज़ वर्णन करता है कि coding-agent आज कॉन्फ़िगरेशन को कैसे रिज़ॉल्व करता है: कौन से रूट्स स्कैन किए जाते हैं, प्रीसीडेंस कैसे काम करता है, और रिज़ॉल्व किया गया कॉन्फ़िग सेटिंग्स, स्किल्स, हुक्स, टूल्स, और एक्सटेंशन्स द्वारा कैसे उपभोग किया जाता है।

## दायरा

प्राथमिक इम्प्लीमेंटेशन:

- `src/config.ts`
- `src/config/settings.ts`
- `src/config/settings-schema.ts`
- `src/discovery/builtin.ts`
- `src/discovery/helpers.ts`

मुख्य इंटीग्रेशन पॉइंट्स:

- `src/capability/index.ts`
- `src/discovery/index.ts`
- `src/extensibility/skills.ts`
- `src/extensibility/hooks/loader.ts`
- `src/extensibility/custom-tools/loader.ts`
- `src/extensibility/extensions/loader.ts`

---

## रिज़ॉल्यूशन प्रवाह (विज़ुअल)

```text
         Config roots (ordered)
┌───────────────────────────────────────┐
│ 1) ~/.xcsh/agent + <cwd>/.xcsh          │
│ 2) ~/.claude   + <cwd>/.claude        │
│ 3) ~/.codex    + <cwd>/.codex         │
│ 4) ~/.gemini   + <cwd>/.gemini        │
└───────────────────────────────────────┘
                    │
                    ▼
        config.ts helper resolution
  (getConfigDirs/findConfigFile/findNearest...)
                    │
                    ▼
       capability providers enumerate items
 (native, claude, codex, gemini, agents, etc.)
                    │
                    ▼
      priority sort + per-capability dedup
                    │
                    ▼
          subsystem-specific consumption
   (settings, skills, hooks, tools, extensions)
```

## 1) कॉन्फ़िग रूट्स और सोर्स क्रम

## कैनोनिकल रूट्स

`src/config.ts` एक निश्चित सोर्स प्राथमिकता सूची परिभाषित करता है:

1. `.xcsh` (नेटिव)
2. `.claude`
3. `.codex`
4. `.gemini`

उपयोगकर्ता-स्तर बेस:

- `~/.xcsh/agent`
- `~/.claude`
- `~/.codex`
- `~/.gemini`

प्रोजेक्ट-स्तर बेस:

- `<cwd>/.xcsh`
- `<cwd>/.claude`
- `<cwd>/.codex`
- `<cwd>/.gemini`

`CONFIG_DIR_NAME` `.xcsh` है (`packages/utils/src/dirs.ts`)।

## महत्वपूर्ण प्रतिबंध

`src/config.ts` में जेनेरिक हेल्पर्स सोर्स डिस्कवरी क्रम में `.pi` को **शामिल नहीं** करते।

## 2) कोर डिस्कवरी हेल्पर्स (`src/config.ts`)

## `getConfigDirs(subpath, options)`

क्रमित एंट्रीज़ लौटाता है:

- पहले उपयोगकर्ता-स्तर एंट्रीज़ (सोर्स प्राथमिकता के अनुसार)
- फिर प्रोजेक्ट-स्तर एंट्रीज़ (उसी सोर्स प्राथमिकता के अनुसार)

विकल्प:

- `user` (डिफ़ॉल्ट `true`)
- `project` (डिफ़ॉल्ट `true`)
- `cwd` (डिफ़ॉल्ट `getProjectDir()`)
- `existingOnly` (डिफ़ॉल्ट `false`)

यह API डायरेक्टरी-आधारित कॉन्फ़िग लुकअप (कमांड्स, हुक्स, टूल्स, एजेंट्स, आदि) के लिए उपयोग किया जाता है।

## `findConfigFile(subpath, options)` / `findConfigFileWithMeta(...)`

क्रमित बेस में पहली मौजूद फ़ाइल खोजता है, पहला मैच लौटाता है (केवल-पथ या पथ+मेटाडेटा)।

## `findAllNearestProjectConfigDirs(subpath, cwd)`

पैरेंट डायरेक्टरीज़ को ऊपर की ओर ट्रैवर्स करता है और **प्रत्येक सोर्स बेस** (`.xcsh`, `.claude`, `.codex`, `.gemini`) के लिए **निकटतम मौजूद डायरेक्टरी** लौटाता है, फिर परिणामों को सोर्स प्राथमिकता के अनुसार सॉर्ट करता है।

इसका उपयोग तब करें जब प्रोजेक्ट कॉन्फ़िग को पूर्वज डायरेक्टरीज़ से इनहेरिट किया जाना चाहिए (मोनोरेपो/नेस्टेड वर्कस्पेस व्यवहार)।

## 3) फ़ाइल कॉन्फ़िग रैपर (`ConfigFile<T>` `src/config.ts` में)

`ConfigFile<T>` एकल कॉन्फ़िग फ़ाइलों के लिए स्कीमा-वैलिडेटेड लोडर है।

समर्थित फ़ॉर्मेट:

- `.yml` / `.yaml`
- `.json` / `.jsonc`

व्यवहार:

- AJV के साथ प्रदान किए गए TypeBox स्कीमा के विरुद्ध पार्स किए गए डेटा को वैलिडेट करता है।
- `invalidate()` तक लोड परिणाम को कैश करता है।
- `tryLoad()` के माध्यम से त्रि-स्थिति परिणाम लौटाता है:
  - `ok`
  - `not-found`
  - `error` (स्कीमा/पार्स संदर्भ के साथ `ConfigError`)

लेगेसी माइग्रेशन अभी भी समर्थित है:

- यदि लक्ष्य पथ `.yml`/`.yaml` है, तो एक सिबलिंग `.json` एक बार ऑटो-माइग्रेट किया जाता है (`migrateJsonToYml`)।

## 4) सेटिंग्स रिज़ॉल्यूशन मॉडल (`src/config/settings.ts`)

रनटाइम सेटिंग्स मॉडल लेयर्ड है:

1. ग्लोबल सेटिंग्स: `~/.xcsh/agent/config.yml`
2. प्रोजेक्ट सेटिंग्स: सेटिंग्स कैपेबिलिटी के माध्यम से खोजी गई (प्रोवाइडर्स से `settings.json`)
3. रनटाइम ओवरराइड्स: इन-मेमोरी, नॉन-पर्सिस्टेंट
4. स्कीमा डिफ़ॉल्ट्स: `SETTINGS_SCHEMA` से

प्रभावी रीड पथ:

`defaults <- global <- project <- overrides`

राइट व्यवहार:

- `settings.set(...)` **ग्लोबल** लेयर (`config.yml`) में लिखता है और बैकग्राउंड सेव को क्यू करता है।
- प्रोजेक्ट सेटिंग्स कैपेबिलिटी डिस्कवरी से रीड-ओनली हैं।

## माइग्रेशन व्यवहार अभी भी सक्रिय है

स्टार्टअप पर, यदि `config.yml` अनुपस्थित है:

1. `~/.xcsh/agent/settings.json` से माइग्रेट करें (सफलता पर `.bak` में नाम बदलें)
2. `agent.db` से लेगेसी DB सेटिंग्स के साथ मर्ज करें
3. मर्ज किए गए परिणाम को `config.yml` में लिखें

`#migrateRawSettings` में फ़ील्ड-स्तर माइग्रेशन:

- `queueMode` -> `steeringMode`
- `ask.timeout` मिलीसेकंड -> सेकंड जब पुराना मान ms जैसा दिखता है (`> 1000`)
- लेगेसी फ्लैट `theme: "..."` -> `theme.dark/theme.light` संरचना

## 5) कैपेबिलिटी/डिस्कवरी इंटीग्रेशन

अधिकांश नॉन-कोर कॉन्फ़िग लोडिंग कैपेबिलिटी रजिस्ट्री (`src/capability/index.ts` + `src/discovery/index.ts`) के माध्यम से होती है।

## प्रोवाइडर क्रम

प्रोवाइडर्स को न्यूमेरिक प्राथमिकता के अनुसार सॉर्ट किया जाता है (उच्च पहले)। उदाहरण प्राथमिकताएँ:

- नेटिव OMP (`builtin.ts`): `100`
- Claude: `80`
- Codex / agents / Claude marketplace: `70`
- Gemini: `60`

```text
Provider precedence (higher wins)

native (.xcsh)          priority 100
claude                 priority  80
codex / agents / ...   priority  70
gemini                 priority  60
```

## डीडुप सिमेंटिक्स

कैपेबिलिटीज़ एक `key(item)` परिभाषित करती हैं:

- समान key => पहला आइटम जीतता है (उच्च-प्राथमिकता/पहले-लोडेड आइटम)
- कोई key नहीं (`undefined`) => कोई डीडुप नहीं, सभी आइटम बनाए रखे जाते हैं

प्रासंगिक keys:

- स्किल्स: `name`
- टूल्स: `name`
- हुक्स: `${type}:${tool}:${name}`
- एक्सटेंशन मॉड्यूल्स: `name`
- एक्सटेंशन्स: `name`
- सेटिंग्स: कोई डीडुप नहीं (सभी आइटम संरक्षित)

## 6) नेटिव `.xcsh` प्रोवाइडर व्यवहार (`src/discovery/builtin.ts`)

नेटिव प्रोवाइडर (`id: native`) यहाँ से पढ़ता है:

- प्रोजेक्ट: `<cwd>/.xcsh/...`
- उपयोगकर्ता: `~/.xcsh/agent/...`

### डायरेक्टरी प्रवेश नियम

`builtin.ts` एक कॉन्फ़िग रूट को तभी शामिल करता है जब डायरेक्टरी मौजूद हो **और नॉन-एम्प्टी** हो (`ifNonEmptyDir`)।

### स्कोप-विशिष्ट लोडिंग

- स्किल्स: `skills/*/SKILL.md`
- स्लैश कमांड्स: `commands/*.md`
- रूल्स: `rules/*.{md,mdc}`
- प्रॉम्प्ट्स: `prompts/*.md`
- इंस्ट्रक्शन्स: `instructions/*.md`
- हुक्स: `hooks/pre/*`, `hooks/post/*`
- टूल्स: `tools/*.json|*.md` और `tools/<name>/index.ts`
- एक्सटेंशन मॉड्यूल्स: `extensions/` के अंतर्गत खोजे जाते हैं (+ लेगेसी `settings.json.extensions` स्ट्रिंग ऐरे)
- एक्सटेंशन्स: `extensions/<name>/gemini-extension.json`
- सेटिंग्स कैपेबिलिटी: `settings.json`

### निकटतम-प्रोजेक्ट लुकअप की बारीकी

`SYSTEM.md` और `AGENTS.md` के लिए, नेटिव प्रोवाइडर निकटतम-पूर्वज प्रोजेक्ट `.xcsh` डायरेक्टरी सर्च (वॉक-अप) का उपयोग करता है लेकिन फिर भी `.xcsh` डायरेक्टरी का नॉन-एम्प्टी होना आवश्यक है।

## 7) प्रमुख सबसिस्टम कॉन्फ़िग का उपभोग कैसे करते हैं

## सेटिंग्स सबसिस्टम

- `Settings.init()` ग्लोबल `config.yml` + खोजे गए प्रोजेक्ट `settings.json` कैपेबिलिटी आइटम्स को लोड करता है।
- केवल `level === "project"` वाले कैपेबिलिटी आइटम्स को प्रोजेक्ट लेयर में मर्ज किया जाता है।

## स्किल्स सबसिस्टम

- `extensibility/skills.ts` `loadCapability(skillCapability.id, { cwd })` के माध्यम से लोड करता है।
- सोर्स टॉगल्स और फ़िल्टर्स (`ignoredSkills`, `includeSkills`, कस्टम dirs) लागू करता है।
- लेगेसी-नामित टॉगल्स अभी भी मौजूद हैं (`skills.enablePiUser`, `skills.enablePiProject`) लेकिन वे नेटिव प्रोवाइडर (`provider === "native"`) को गेट करते हैं।

## हुक्स सबसिस्टम

- `discoverAndLoadHooks()` हुक कैपेबिलिटी + स्पष्ट रूप से कॉन्फ़िगर किए गए पथों से हुक पथों को रिज़ॉल्व करता है।
- फिर Bun import के माध्यम से मॉड्यूल्स लोड करता है।

## टूल्स सबसिस्टम

- `discoverAndLoadCustomTools()` टूल कैपेबिलिटी + प्लगइन टूल पथों + स्पष्ट रूप से कॉन्फ़िगर किए गए पथों से टूल पथों को रिज़ॉल्व करता है।
- डिक्लेरेटिव `.md/.json` टूल फ़ाइलें केवल मेटाडेटा हैं; एक्ज़ीक्यूटेबल लोडिंग कोड मॉड्यूल्स की अपेक्षा करती है।

## एक्सटेंशन्स सबसिस्टम

- `discoverAndLoadExtensions()` एक्सटेंशन-मॉड्यूल कैपेबिलिटी प्लस स्पष्ट पथों से एक्सटेंशन मॉड्यूल्स को रिज़ॉल्व करता है।
- वर्तमान इम्प्लीमेंटेशन जानबूझकर लोडिंग से पहले केवल `_source.provider === "native"` वाले कैपेबिलिटी आइटम्स रखता है।

## 8) भरोसा करने योग्य प्रीसीडेंस नियम

इस मानसिक मॉडल का उपयोग करें:

1. `config.ts` से सोर्स डायरेक्टरी क्रम उम्मीदवार पथ क्रम निर्धारित करता है।
2. कैपेबिलिटी प्रोवाइडर प्राथमिकता क्रॉस-प्रोवाइडर प्रीसीडेंस निर्धारित करती है।
3. कैपेबिलिटी key डीडुप टकराव व्यवहार निर्धारित करता है (keyed कैपेबिलिटीज़ के लिए पहला जीतता है)।
4. सबसिस्टम-विशिष्ट मर्ज लॉजिक प्रभावी प्रीसीडेंस को और बदल सकता है (विशेषकर सेटिंग्स)।

### सेटिंग्स-विशिष्ट चेतावनी

सेटिंग्स कैपेबिलिटी आइटम्स डीडुप्लिकेट नहीं किए जाते; `Settings.#loadProjectSettings()` लौटाए गए क्रम में प्रोजेक्ट आइटम्स को डीप-मर्ज करता है। चूँकि मर्ज बाद के आइटम मानों को पहले के मानों पर लागू करता है, प्रभावी ओवरराइड व्यवहार प्रोवाइडर एमिशन क्रम पर निर्भर करता है, न कि केवल कैपेबिलिटी key सिमेंटिक्स पर।

## 9) लेगेसी/संगतता व्यवहार अभी भी मौजूद हैं

- YAML-लक्षित फ़ाइलों के लिए `ConfigFile` JSON -> YAML माइग्रेशन।
- `settings.json` और `agent.db` से `config.yml` में सेटिंग्स माइग्रेशन।
- सेटिंग्स key माइग्रेशन (`queueMode`, `ask.timeout`, फ्लैट `theme`)।
- एक्सटेंशन मैनिफ़ेस्ट संगतता: लोडर `package.json.xcsh` और `package.json.pi` दोनों मैनिफ़ेस्ट सेक्शन स्वीकार करता है।
- लेगेसी सेटिंग नाम `skills.enablePiUser` / `skills.enablePiProject` अभी भी नेटिव स्किल सोर्स के लिए सक्रिय गेट हैं।

यदि ये संगतता पथ कोड में हटाए जाते हैं, तो इस दस्तावेज़ को तुरंत अपडेट करें; कई रनटाइम व्यवहार आज भी उन पर निर्भर करते हैं।
