# सीक्रेट ऑबफस्केशन

संवेदनशील मानों (API कुंजियाँ, टोकन, पासवर्ड) को LLM प्रोवाइडर्स को भेजे जाने से रोकता है। सक्षम होने पर, सीक्रेट को प्रक्रिया छोड़ने से पहले निर्धारणात्मक प्लेसहोल्डर से बदल दिया जाता है, और मॉडल द्वारा लौटाए गए टूल कॉल आर्गुमेंट में पुनर्स्थापित किया जाता है।

## सक्षम करना

डिफ़ॉल्ट रूप से सक्षम। `/settings` UI के माध्यम से या सीधे `config.yml` में टॉगल करें:

```yaml
secrets:
  enabled: false
```

## यह कैसे काम करता है

1. सेशन स्टार्टअप पर, सीक्रेट दो स्रोतों से एकत्र किए जाते हैं:
   - **एनवायरनमेंट वेरिएबल** जो सामान्य सीक्रेट पैटर्न (`*_KEY`, `*_SECRET`, `*_TOKEN`, `*_PASSWORD`, आदि) से मेल खाते हैं और जिनके मान >= 8 वर्ण हैं
   - **`secrets.yml` फ़ाइलें** (नीचे देखें)

2. LLM को भेजे जाने वाले आउटबाउंड संदेशों में सभी सीक्रेट मानों को `<<$env:S0>>`, `<<$env:S1>>`, आदि जैसे प्लेसहोल्डर से बदल दिया जाता है।

3. मॉडल द्वारा लौटाए गए टूल कॉल आर्गुमेंट को गहराई से स्कैन किया जाता है और प्लेसहोल्डर को निष्पादन से पहले मूल मानों में पुनर्स्थापित किया जाता है।

दो मोड नियंत्रित करते हैं कि प्रत्येक सीक्रेट के साथ क्या होता है:

| मोड | व्यवहार | प्रतिवर्तनीय |
|---|---|---|
| `obfuscate` (डिफ़ॉल्ट) | इंडेक्स्ड प्लेसहोल्डर `<<$env:SN>>` से बदला गया | हाँ (टूल आर्गुमेंट में डीऑबफस्केट किया गया) |
| `replace` | निर्धारणात्मक समान-लंबाई वाली स्ट्रिंग से बदला गया | नहीं (एकतरफा) |

## secrets.yml

YAML में कस्टम सीक्रेट प्रविष्टियाँ परिभाषित करें। दो स्थानों की जाँच की जाती है:

| स्तर | पथ | उद्देश्य |
|---|---|---|
| वैश्विक | `~/.xcsh/agent/secrets.yml` | सभी प्रोजेक्ट के लिए सीक्रेट |
| प्रोजेक्ट | `<cwd>/.xcsh/secrets.yml` | प्रोजेक्ट-विशिष्ट सीक्रेट |

प्रोजेक्ट प्रविष्टियाँ मिलते-जुलते `content` वाली वैश्विक प्रविष्टियों को ओवरराइड करती हैं।

### स्कीमा

सरणी में प्रत्येक प्रविष्टि में ये फ़ील्ड होते हैं:

| फ़ील्ड | प्रकार | आवश्यक | विवरण |
|---|---|---|---|
| `type` | `"plain"` या `"regex"` | हाँ | मिलान रणनीति |
| `content` | स्ट्रिंग | हाँ | सीक्रेट मान (plain) या regex पैटर्न (regex) |
| `mode` | `"obfuscate"` या `"replace"` | नहीं | डिफ़ॉल्ट: `"obfuscate"` |
| `replacement` | स्ट्रिंग | नहीं | कस्टम प्रतिस्थापन (केवल replace मोड) |
| `flags` | स्ट्रिंग | नहीं | Regex फ्लैग (केवल regex प्रकार) |

### उदाहरण

#### Plain सीक्रेट

```yaml
# एक विशिष्ट API कुंजी को ऑबफस्केट करें (डिफ़ॉल्ट मोड)
- type: plain
  content: sk-proj-abc123def456

# एक डेटाबेस पासवर्ड को एक निश्चित स्ट्रिंग से बदलें
- type: plain
  content: hunter2
  mode: replace
  replacement: "********"
```

#### Regex सीक्रेट

```yaml
# किसी भी AWS-स्टाइल कुंजी को ऑबफस्केट करें
- type: regex
  content: "AKIA[0-9A-Z]{16}"

# स्पष्ट फ्लैग के साथ केस-असंवेदनशील मिलान
- type: regex
  content: "api[_-]?key\\s*=\\s*\\w+"
  flags: "i"

# Regex लिटरल सिंटैक्स (पैटर्न और फ्लैग एक स्ट्रिंग में)
- type: regex
  content: "/bearer\\s+[a-zA-Z0-9._~+\\/=-]+/i"
```

Regex प्रविष्टियाँ हमेशा वैश्विक रूप से स्कैन करती हैं (`g` फ्लैग स्वचालित रूप से लागू होता है)। `/pattern/flags` Regex लिटरल सिंटैक्स को अलग `content` + `flags` फ़ील्ड के विकल्प के रूप में समर्थित किया जाता है। पैटर्न के भीतर एस्केप्ड स्लैश (`\\/`) को सही तरीके से संभाला जाता है।

#### Replace मोड के साथ Regex

```yaml
# कनेक्शन स्ट्रिंग को एकतरफा बदलें (प्रतिवर्तनीय नहीं)
- type: regex
  content: "postgres://[^\\s]+"
  mode: replace
  replacement: "postgres://***"
```

## एनवायरनमेंट वेरिएबल डिटेक्शन के साथ इंटरैक्शन

एनवायरनमेंट वेरिएबल हमेशा पहले एकत्र किए जाते हैं। फ़ाइल-परिभाषित प्रविष्टियाँ बाद में जोड़ी जाती हैं, इसलिए फ़ाइल प्रविष्टियाँ उन सीक्रेट को कवर कर सकती हैं जो एनवायरनमेंट वेरिएबल में नहीं हैं (कॉन्फिग फ़ाइलें, हार्डकोडेड मान, आदि)। यदि एक ही मान दोनों में दिखाई देता है, तो फ़ाइल प्रविष्टि का मोड प्राथमिकता लेता है।

## मुख्य फ़ाइलें

- `src/secrets/index.ts` -- लोडिंग, मर्जिंग, एनवायरनमेंट वेरिएबल संग्रह
- `src/secrets/obfuscator.ts` -- `SecretObfuscator` क्लास, प्लेसहोल्डर जनरेशन, संदेश ऑबफस्केशन
- `src/secrets/regex.ts` -- regex लिटरल पार्सिंग और संकलन
- `src/config/settings-schema.ts` -- `secrets.enabled` सेटिंग परिभाषा
