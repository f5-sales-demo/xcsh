# नेटिव्स बिल्ड, रिलीज़, और डिबगिंग रनबुक

यह रनबुक वर्णन करती है कि कैसे `@f5xc-salesdemos/pi-natives` बिल्ड पाइपलाइन `.node` एडऑन उत्पन्न करती है, कैसे संकलित वितरण उन्हें लोड करते हैं, और कैसे लोडर/बिल्ड विफलताओं को डिबग किया जाए।

यह `docs/natives-architecture.md` से आर्किटेक्चर शब्दावली का अनुसरण करती है:

- **बिल्ड-टाइम आर्टिफैक्ट उत्पादन** (`scripts/build-native.ts`)
- **एम्बेडेड एडऑन मैनिफ़ेस्ट जनरेशन** (`scripts/embed-native.ts`)
- **रनटाइम एडऑन लोडिंग + वैलिडेशन गेट** (`src/native.ts`)

## कार्यान्वयन फ़ाइलें

- `packages/natives/scripts/build-native.ts`
- `packages/natives/scripts/embed-native.ts`
- `packages/natives/package.json`
- `packages/natives/src/native.ts`
- `crates/pi-natives/Cargo.toml`

## बिल्ड पाइपलाइन अवलोकन

### 1) बिल्ड एंट्रीपॉइंट्स

`packages/natives/package.json` स्क्रिप्ट्स:

- `bun scripts/build-native.ts` (`build`) → रिलीज़ बिल्ड
- `bun scripts/build-native.ts --dev` (`dev:native`) → डिबग/डेव प्रोफाइल बिल्ड (समान आउटपुट नामकरण)
- `bun scripts/embed-native.ts` (`embed:native`) → बिल्ट फ़ाइलों से `src/embedded-addon.ts` जनरेट करें

### 2) Rust आर्टिफैक्ट बिल्ड

`build-native.ts` `crates/pi-natives` में Cargo चलाता है:

- बेस कमांड: `cargo build`
- रिलीज़ मोड `--release` जोड़ता है जब तक `--dev` पास नहीं किया जाता
- क्रॉस टारगेट `--target <CROSS_TARGET>` जोड़ता है

`crates/pi-natives/Cargo.toml` `crate-type = ["cdylib"]` घोषित करता है, इसलिए Cargo एक शेयर्ड लाइब्रेरी (`.so`/`.dylib`/`.dll`) उत्सर्जित करता है जिसे फिर `.node` एडऑन फ़ाइलनाम में कॉपी/रीनेम किया जाता है।

### 3) आर्टिफैक्ट खोज और इंस्टॉल

Cargo पूर्ण होने के बाद, `build-native.ts` क्रम में कैंडिडेट आउटपुट डायरेक्टरीज़ स्कैन करता है:

1. `${CARGO_TARGET_DIR}` (यदि सेट है)
2. `<repo>/target`
3. `crates/pi-natives/target`

प्रत्येक रूट के लिए यह प्रोफाइल डायरेक्टरीज़ जाँचता है:

- क्रॉस बिल्ड: `<root>/<crossTarget>/<profile>` फिर `<root>/<profile>`
- नेटिव बिल्ड: `<root>/<profile>`

फिर यह इनमें से एक खोजता है:

- `libpi_natives.so`
- `libpi_natives.dylib`
- `pi_natives.dll`
- `libpi_natives.dll`

मिलने पर, यह टेम्प-फ़ाइल + रीनेम सिमेंटिक्स के साथ `packages/natives/native/` में एटॉमिक रूप से इंस्टॉल करता है (Windows फ़ॉलबैक लॉक्ड DLL रिप्लेसमेंट विफलताओं को स्पष्ट रूप से संभालता है)।

## टारगेट/वेरिएंट मॉडल और नामकरण परंपराएं

## प्लेटफ़ॉर्म टैग

बिल्ड और रनटाइम दोनों प्लेटफ़ॉर्म टैग का उपयोग करते हैं:

`<platform>-<arch>` (उदाहरण: `darwin-arm64`, `linux-x64`)

## वेरिएंट मॉडल (केवल x64)

x64 CPU वेरिएंट्स का समर्थन करता है:

- `modern` (AVX2-सक्षम पथ)
- `baseline` (फ़ॉलबैक)

गैर-x64 एकल डिफ़ॉल्ट आर्टिफैक्ट का उपयोग करता है (कोई वेरिएंट प्रत्यय नहीं)।

### आउटपुट फ़ाइलनाम

रिलीज़ बिल्ड:

- x64: `pi_natives.<platform>-<arch>-modern.node` या `...-baseline.node`
- गैर-x64: `pi_natives.<platform>-<arch>.node`

डेव बिल्ड (`--dev`):

- डिबग प्रोफाइल फ़्लैग्स का उपयोग करता है लेकिन मानक प्लेटफ़ॉर्म-टैग्ड आउटपुट नामकरण रखता है

`native.ts` में रनटाइम लोडर कैंडिडेट क्रम:

- रिलीज़ कैंडिडेट्स
- कंपाइल्ड मोड पैकेज-लोकल फ़ाइलों से पहले एक्सट्रैक्टेड/कैश कैंडिडेट्स जोड़ता है

## एनवायरनमेंट फ़्लैग्स और बिल्ड विकल्प

## रनटाइम फ़्लैग्स

- `PI_DEV` (लोडर व्यवहार): लोडर डायग्नोस्टिक्स सक्षम करें
- `PI_NATIVE_VARIANT` (लोडर व्यवहार, केवल x64): रनटाइम पर `modern` या `baseline` चयन को बाध्य करें
- `PI_COMPILED` (लोडर व्यवहार): कंपाइल्ड-बाइनरी कैंडिडेट/एक्सट्रैक्शन व्यवहार सक्षम करें

## बिल्ड-टाइम फ़्लैग्स/विकल्प

- `--dev` (स्क्रिप्ट आर्ग): डिबग प्रोफाइल बिल्ड करें
- `CROSS_TARGET`: Cargo `--target` को पास किया जाता है
- `TARGET_PLATFORM`: आउटपुट प्लेटफ़ॉर्म टैग नामकरण ओवरराइड करें
- `TARGET_ARCH`: आउटपुट आर्क नामकरण ओवरराइड करें
- `TARGET_VARIANT` (केवल x64): आउटपुट फ़ाइलनाम और RUSTFLAGS नीति के लिए `modern` या `baseline` बाध्य करें
- `CARGO_TARGET_DIR`: Cargo आउटपुट खोजते समय अतिरिक्त रूट
- `RUSTFLAGS`:
  - यदि अनसेट है और क्रॉस-कंपाइलिंग नहीं कर रहे, तो स्क्रिप्ट सेट करती है:
    - modern: `-C target-cpu=x86-64-v3`
    - baseline: `-C target-cpu=x86-64-v2`
    - गैर-x64 / कोई वेरिएंट नहीं: `-C target-cpu=native`
  - यदि पहले से सेट है, तो स्क्रिप्ट ओवरराइड नहीं करती

## बिल्ड स्थिति/जीवनचक्र ट्रांज़िशन

### बिल्ड जीवनचक्र (`build-native.ts`)

1. **Init**: आर्ग्स/एनव पार्स करें (`--dev`, टारगेट ओवरराइड्स, क्रॉस फ़्लैग्स)
2. **वेरिएंट रिज़ॉल्व**:
   - गैर-x64 → कोई वेरिएंट नहीं
   - x64 + `TARGET_VARIANT` → स्पष्ट वेरिएंट
   - x64 क्रॉस-बिल्ड बिना `TARGET_VARIANT` → हार्ड एरर
   - x64 लोकल बिल्ड बिना ओवरराइड → होस्ट AVX2 डिटेक्ट करें
3. **कंपाइल**: रिज़ॉल्व्ड प्रोफाइल/टारगेट के साथ Cargo चलाएं
4. **आर्टिफैक्ट लोकेट करें**: टारगेट रूट्स/प्रोफाइल डिर/लाइब्रेरी नाम स्कैन करें
5. **इंस्टॉल**: `packages/natives/native` में कॉपी + एटॉमिक रीनेम
6. **पूर्ण**: एडऑन लोडर कैंडिडेट्स के लिए तैयार

किसी भी चरण पर स्पष्ट एरर टेक्स्ट के साथ विफलता एक्ज़िट होती है (अमान्य वेरिएंट, विफल cargo बिल्ड, गुम आउटपुट लाइब्रेरी, इंस्टॉल/रीनेम विफलता)।

### एम्बेड जीवनचक्र (`embed-native.ts`)

1. **Init**: `TARGET_PLATFORM`/`TARGET_ARCH` या होस्ट मानों से प्लेटफ़ॉर्म टैग गणना करें
2. **कैंडिडेट सेट**:
   - x64 `modern` और `baseline` दोनों की अपेक्षा करता है
   - गैर-x64 एक डिफ़ॉल्ट फ़ाइल की अपेक्षा करता है
3. `packages/natives/native` में **उपलब्धता मान्य करें**
4. Bun `file` इम्पोर्ट्स और पैकेज वर्शन के साथ **मैनिफ़ेस्ट जनरेट करें** (`src/embedded-addon.ts`)
5. कंपाइल्ड मोड के लिए **रनटाइम एक्सट्रैक्शन तैयार**

`--reset` वैलिडेशन को बायपास करता है और एक नल मैनिफ़ेस्ट स्टब (`embeddedAddon = null`) लिखता है।

## डेव वर्कफ़्लो बनाम शिप्ड/कंपाइल्ड व्यवहार

## स्थानीय विकास वर्कफ़्लो

विशिष्ट स्थानीय लूप:

1. एडऑन बिल्ड करें:
   - रिलीज़: `bun --cwd=packages/natives run build`
   - डिबग प्रोफाइल: `bun --cwd=packages/natives run dev:native`
2. लोडर डायग्नोस्टिक्स परीक्षण करते समय `PI_DEV=1` सेट करें
3. `native.ts` में लोडर पैकेज-लोकल `native/` (और एक्जीक्यूटेबल-डिर फ़ॉलबैक) कैंडिडेट्स रिज़ॉल्व करता है
4. `validateNative` रैपर्स द्वारा बाइंडिंग उपयोग करने से पहले एक्सपोर्ट संगतता लागू करता है

## शिप्ड/कंपाइल्ड बाइनरी वर्कफ़्लो

कंपाइल्ड मोड में (`PI_COMPILED` या Bun एम्बेडेड मार्कर्स):

1. लोडर वर्शन्ड कैश डिर गणना करता है: `<getNativesDir()>/<packageVersion>` (व्यावहारिक रूप से `~/.xcsh/natives/<version>`)
2. यदि एम्बेडेड मैनिफ़ेस्ट वर्तमान प्लेटफ़ॉर्म+वर्शन से मेल खाता है, तो लोडर चयनित एम्बेडेड फ़ाइल को उस वर्शन्ड डिर में एक्सट्रैक्ट कर सकता है
3. रनटाइम कैंडिडेट क्रम में शामिल हैं:
   - वर्शन्ड कैश डिर
   - लिगेसी कंपाइल्ड-बाइनरी डिर (Windows पर `%LOCALAPPDATA%/xcsh`, अन्यथा `~/.local/bin`)
   - पैकेज/एक्जीक्यूटेबल डायरेक्टरीज़
4. पहले सफलतापूर्वक लोड किए गए एडऑन को अभी भी `validateNative` पास करना होगा

इसीलिए पैकेजिंग + रनटाइम लोडर अपेक्षाओं का संरेखित होना आवश्यक है: फ़ाइलनाम, प्लेटफ़ॉर्म टैग, और एक्सपोर्टेड सिंबल्स को `native.ts` जो प्रोब और वैलिडेट करता है उससे मेल खाना चाहिए।

## JS API ↔ Rust एक्सपोर्ट मैपिंग (वैलिडेशन गेट सबसेट)

`native.ts` को लोड किए गए एडऑन पर इन JS-विज़िबल एक्सपोर्ट्स के मौजूद होने की आवश्यकता है। ये `crates/pi-natives/src` में Rust N-API एक्सपोर्ट्स से मैप होते हैं:

| `validateNative` द्वारा आवश्यक JS नाम | Rust एक्सपोर्ट घोषणा | Rust सोर्स फ़ाइल |
| --- | --- | --- |
| `glob` | `#[napi] pub fn glob(...)` | `crates/pi-natives/src/glob.rs` |
| `grep` | `#[napi] pub fn grep(...)` | `crates/pi-natives/src/grep.rs` |
| `search` | `#[napi] pub fn search(...)` | `crates/pi-natives/src/grep.rs` |
| `highlightCode` | `#[napi] pub fn highlight_code(...)` | `crates/pi-natives/src/highlight.rs` |
| `getSystemInfo` | `#[napi] pub fn get_system_info(...)` | `crates/pi-natives/src/system_info.rs` |
| `getWorkProfile` | `#[napi] pub fn get_work_profile(...)` (कैमल-केस्ड एक्सपोर्ट) | `crates/pi-natives/src/prof.rs` |
| `invalidateFsScanCache` | `#[napi] pub fn invalidate_fs_scan_cache(...)` | `crates/pi-natives/src/fs_cache.rs` |

यदि कोई आवश्यक सिंबल गुम है, तो लोडर रीबिल्ड संकेत के साथ तुरंत विफल हो जाता है।

## विफलता व्यवहार और डायग्नोस्टिक्स

## बिल्ड-टाइम विफलताएं

- अमान्य वेरिएंट कॉन्फ़िगरेशन:
  - गैर-x64 पर `TARGET_VARIANT` सेट → तत्काल एरर
  - स्पष्ट `TARGET_VARIANT` के बिना x64 क्रॉस-बिल्ड → तत्काल एरर
- Cargo बिल्ड विफलता:
  - स्क्रिप्ट नॉन-ज़ीरो एक्ज़िट और stderr दिखाती है
- आर्टिफैक्ट नहीं मिला:
  - स्क्रिप्ट प्रत्येक जाँची गई प्रोफाइल डायरेक्टरी प्रिंट करती है
- इंस्टॉल विफलता:
  - स्पष्ट संदेश; Windows में लॉक्ड-फ़ाइल संकेत शामिल है

## रनटाइम लोडर विफलताएं (`native.ts`)

- असमर्थित प्लेटफ़ॉर्म टैग:
  - समर्थित प्लेटफ़ॉर्म सूची के साथ throw करता है
- कोई कैंडिडेट लोड नहीं हो सका:
  - पूर्ण कैंडिडेट एरर सूची और मोड-विशिष्ट उपचार संकेतों के साथ throw करता है
- गुम एक्सपोर्ट्स:
  - सटीक गुम सिंबल नाम और रीबिल्ड कमांड के साथ throw करता है
- एम्बेडेड एक्सट्रैक्शन समस्याएं:
  - एक्सट्रैक्शन mkdir/write एरर रिकॉर्ड किए जाते हैं और अंतिम डायग्नोस्टिक्स में शामिल किए जाते हैं

## समस्या निवारण मैट्रिक्स

| लक्षण | संभावित कारण | सत्यापित करें | समाधान |
| --- | --- | --- | --- |
| `Native addon missing exports ... ## संचालन कमांड

```bash
# वर्तमान होस्ट के लिए रिलीज़ आर्टिफैक्ट
bun --cwd=packages/natives run build

# डिबग प्रोफाइल आर्टिफैक्ट बिल्ड
bun --cwd=packages/natives run dev:native

# स्पष्ट x64 वेरिएंट्स बिल्ड करें
TARGET_VARIANT=modern bun --cwd=packages/natives run build
TARGET_VARIANT=baseline bun --cwd=packages/natives run build

# बिल्ट नेटिव फ़ाइलों से एम्बेडेड एडऑन मैनिफ़ेस्ट जनरेट करें
bun --cwd=packages/natives run embed:native

# एम्बेडेड मैनिफ़ेस्ट को नल स्टब पर रीसेट करें
bun --cwd=packages/natives run embed:native -- --reset
```
