# TUI ランタイム内部構造

このドキュメントでは、インタラクティブモードにおけるターミナル入力からレンダリング出力までの、テーマ以外のランタイムパスをマッピングします。`packages/tui` での動作と、`packages/coding-agent` コントローラーからの統合に焦点を当てています。

## ランタイムレイヤーと所有権

- **`packages/tui` エンジン**: ターミナルライフサイクル、stdin 正規化、フォーカスルーティング、レンダースケジューリング、差分描画、オーバーレイ合成、ハードウェアカーソル配置。
- **`packages/coding-agent` インタラクティブモード**: コンポーネントツリーの構築、エディタコールバックとキーマップのバインド、エージェント/セッションイベントへの応答、ドメイン状態（ストリーミング、ツール実行、リトライ、プランモード）の UI コンポーネントへの変換。

境界ルール: TUI エンジンはメッセージに依存しません。`Component.render(width)`、`handleInput(data)`、フォーカス、オーバーレイのみを認識します。エージェントのセマンティクスはインタラクティブコントローラーに留まります。

## 起動とコンポーネントツリーの組み立て

`InteractiveMode` は `TUI(new ProcessTerminal(), showHardwareCursor)` を構築し、永続コンテナを作成します:

- `chatContainer`
- `pendingMessagesContainer`
- `statusContainer`
- `todoContainer`
- `statusLine`
- `editorContainer`（`CustomEditor` を保持）

`init()` はこの順序でツリーを接続し、エディタにフォーカスを設定し、`InputController` 経由で入力ハンドラーを登録し、TUI を開始して、強制レンダーをリクエストします。

強制レンダー（`requestRender(true)`）は、再描画前に前回の行キャッシュとカーソルの管理情報をリセットします。

## ターミナルライフサイクルと stdin 正規化

`ProcessTerminal.start()`:

1. rawモードとブラケットペーストを有効にします。
2. リサイズハンドラーをアタッチします。
3. 部分的なエスケープチャンクを完全なシーケンスに分割するための `StdinBuffer` を作成します。
4. Kitty キーボードプロトコルのサポート（`CSI ? u`）を照会し、サポートされている場合はプロトコルフラグを有効にします。
5. Windows では、`kernel32` モードフラグを介して VT 入力の有効化を試みます。

`StdinBuffer` の動作:

- 断片化されたエスケープシーケンス（CSI/OSC/DCS/APC/SS3）をバッファリングします。
- シーケンスが完了するかタイムアウトでフラッシュされた場合のみ `data` を発行します。
- ブラケットペーストを検出し、生のペーストテキストとともに `paste` イベントを発行します。

これにより、部分的なエスケープチャンクが通常のキー入力として誤解釈されることを防ぎます。

## 入力ルーティングとフォーカスモデル

入力パス:

`stdin -> ProcessTerminal -> StdinBuffer -> TUI.#handleInput -> focusedComponent.handleInput`

ルーティングの詳細:

1. TUI は最初に登録済み入力リスナー（`addInputListener`）を実行し、消費/変換動作を可能にします。
2. TUI はコンポーネントディスパッチの前にグローバルデバッグショートカット（`shift+ctrl+d`）を処理します。
3. フォーカスされたコンポーネントが現在非表示/不可視のオーバーレイに属している場合、TUI は次の可視オーバーレイまたは保存されたオーバーレイ前のフォーカスにフォーカスを再割り当てします。
4. フォーカスされたコンポーネントが `wantsKeyRelease = true` を設定していない限り、キーリリースイベントはフィルタリングされます。
5. ディスパッチ後、TUI はレンダーをスケジュールします。

`setFocus()` は `Focusable.focused` も切り替え、コンポーネントがハードウェアカーソル配置のために `CURSOR_MARKER` を発行するかどうかを制御します。

## キー処理の分割: エディタ vs コントローラー

`CustomEditor` は最初に高優先度のコンボ（escape、ctrl-c/d/z、ctrl-v、ctrl-p バリアント、ctrl-t、alt-up、拡張カスタムキー）をインターセプトし、残りをベースの `Editor` 動作（テキスト編集、履歴、オートコンプリート、カーソル移動）に委譲します。

`InputController.setupKeyHandlers()` は次にエディタコールバックをモードアクションにバインドします:

- `Escape` でのキャンセル/モード終了
- ダブル `Ctrl+C` または空エディタでの `Ctrl+D` によるシャットダウン
- `Ctrl+Z` でのサスペンド/レジューム
- スラッシュコマンドとセレクターホットキー
- フォローアップ/デキュートグルと展開トグル

これにより、キー解析/エディタメカニクスは `packages/tui` に、モードセマンティクスは coding-agent コントローラーに保持されます。

## レンダーループと差分戦略

`TUI.requestRender()` は `process.nextTick` を使用して1ティックあたり1レンダーにデバウンスされます。同じターン内の複数の状態変更は統合されます。

`#doRender()` パイプライン:

1. ルートコンポーネントツリーを `newLines` にレンダリングします。
2. 可視オーバーレイがあれば合成します。
3. 可視ビューポート行から `CURSOR_MARKER` を抽出・除去します。
4. 非画像行にセグメントリセットサフィックスを付加します。
5. 全体再描画と差分パッチのどちらを行うか選択します:
   - 最初のフレーム
   - 幅の変更
   - `clearOnShrink` が有効でオーバーレイがない場合の縮小
   - 前回のビューポートより上での編集
6. 差分更新の場合、変更された行範囲のみをパッチし、必要に応じて古い末尾行をクリアします。
7. IME サポートのためにハードウェアカーソルを再配置します。

レンダー書き込みは同期出力モード（`CSI ? 2026 h/l`）を使用して、ちらつき/ティアリングを軽減します。

## レンダー安全性の制約

`TUI` における重要な安全性チェック:

- 非画像のレンダリング行はターミナル幅を超えてはなりません。オーバーフローはスローされ、クラッシュ診断情報が書き込まれます。
- オーバーレイ合成には防御的な切り詰めと、合成後の幅検証が含まれます。
- 幅の変更はラッピングセマンティクスが変わるため、全体再描画を強制します。
- カーソル位置は移動前にクランプされます。

これらの制約は単なる慣例ではなく、ランタイムでの強制です。

## リサイズ処理

リサイズイベントは `ProcessTerminal` から `TUI.requestRender()` へイベント駆動で伝達されます。

影響:

- 幅の変更は全体再描画をトリガーします。
- ビューポート/トップ追跡（`#previousViewportTop`、`#maxLinesRendered`）は、コンテンツやターミナルサイズが変更された際の無効な相対カーソル計算を回避します。
- オーバーレイの可視性はターミナルの寸法に依存する場合があります（`OverlayOptions.visible`）。リサイズ後にオーバーレイが非可視になった場合、フォーカスが修正されます。

## ストリーミングとインクリメンタル UI 更新

`EventController` は `AgentSessionEvent` をサブスクライブし、UI をインクリメンタルに更新します:

- `agent_start`: `statusContainer` でローダーを開始します。
- `message_start` assistant: `streamingComponent` を作成してマウントします。
- `message_update`: ストリーミング中のアシスタントコンテンツを更新し、ツールコールが出現するとツール実行コンポーネントを作成/更新します。
- `tool_execution_update/end`: ツール結果コンポーネントと完了状態を更新します。
- `message_end`: アシスタントストリームを確定し、中断/エラーのアノテーションを処理し、正常停止時に保留中のツール引数を完了としてマークします。
- `agent_end`: ローダーを停止し、一時的なストリーム状態をクリアし、遅延モデル切り替えをフラッシュし、バックグラウンドの場合は完了通知を発行します。

読み取りツールのグループ化は意図的にステートフル（`#lastReadGroup`）であり、非読み取りのブレークが発生するまで連続する読み取りツールコールを1つのビジュアルブロックに統合します。

## ステータスとローダーのオーケストレーション

ステータスレーンの所有権:

- `statusContainer` は一時的なローダー（`loadingAnimation`、`autoCompactionLoader`、`retryLoader`）を保持します。
- `statusLine` は永続的なステータス/フック/プランインジケーターをレンダリングし、エディタの上部ボーダー更新を駆動します。

ローダーの動作:

- `Loader` はインターバルにより80msごとに更新し、各フレームでレンダーをリクエストします。
- 自動コンパクションや自動リトライ中は、それらの操作をキャンセルするためにエスケープハンドラーが一時的にオーバーライドされます。
- 終了/キャンセルパスでは、コントローラーは以前のエスケープハンドラーを復元し、ローダーコンポーネントを停止/クリアします。

## モード遷移とバックグラウンド化

### Bash/Python 入力モード

入力テキストのプレフィックスによってエディタのボーダーモードフラグが切り替わります:

- `!` -> bash モード
- `$`（テンプレートリテラルプレフィックス以外）-> python モード

Escape は非アクティブモードではエディタテキストをクリアしてボーダーカラーを復元して終了します。実行がアクティブな場合は、代わりに実行中のタスクを中断します。

### プランモード

`InteractiveMode` はプランモードフラグ、ステータスライン状態、アクティブツール、モデル切り替えを追跡します。開始/終了はセッションモードエントリとステータス/UI 状態を更新し、ストリーミングがアクティブな場合は遅延モデル切り替えを含みます。

### サスペンド/レジューム（`Ctrl+Z`）

`InputController.handleCtrlZ()`:

1. TUI を再起動して強制レンダーするためのワンショット `SIGCONT` ハンドラーを登録します。
2. サスペンド前に TUI を停止します。
3. プロセスグループに `SIGTSTP` を送信します。

### バックグラウンドモード（`/background` または `/bg`）

`handleBackgroundCommand()`:

- アイドル時は拒否します。
- ツール UI コンテキストを非インタラクティブ（`hasUI=false`）に切り替え、インタラクティブ UI ツールが即座に失敗するようにします。
- ローダー/ステータスラインを停止し、フォアグラウンドイベントハンドラーをアンサブスクライブします。
- バックグラウンドイベントハンドラーをサブスクライブします（主に `agent_end` を待機）。
- TUI を停止し、`SIGTSTP` を送信します（POSIX ジョブ制御パス）。

キューに入った作業がない状態でバックグラウンドで `agent_end` が発生すると、コントローラーは完了通知を送信してシャットダウンします。

## キャンセルパス

主要なキャンセル入力:

- アクティブなストリームローダー中の `Escape`: キューに入ったメッセージをエディタに復元し、エージェントを中断します。
- bash/python 実行中の `Escape`: 実行中のコマンドを中断します。
- 自動コンパクション/リトライ中の `Escape`: 一時的なエスケープハンドラーを通じて専用の中断メソッドを呼び出します。
- `Ctrl+C` シングルプレス: エディタをクリア。500ms 以内のダブルプレス: シャットダウン。

キャンセルは状態条件付きです。同じキーでも、ランタイム状態に応じて中断、モード終了、セレクタートリガー、または何もしないを意味する場合があります。

## イベント駆動 vs スロットル動作

イベント駆動の更新:

- エージェントセッションイベント（`EventController`）
- キー入力コールバック（`InputController`）
- ターミナルリサイズコールバック
- `InteractiveMode` のテーマ/ブランチウォッチャー

スロットル/デバウンスされるパス:

- TUI レンダリングはティックデバウンスされます（`requestRender` の統合）。
- ローダーアニメーションは固定インターバル（80ms）で、各フレームがレンダーをリクエストします。
- エディタのオートコンプリート更新（`Editor` 内部）はデバウンスタイマーを使用し、タイピング中の再計算チャーンを削減します。

したがって、このランタイムはイベント駆動の状態遷移と制限付きレンダーケイデンスを組み合わせ、再描画の嵐を起こすことなくインタラクティビティのレスポンシブ性を維持しています。
