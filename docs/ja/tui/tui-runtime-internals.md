# TUIランタイム内部構造

このドキュメントは、インタラクティブモードにおけるターミナル入力からレンダリング出力までの非テーマランタイムパスをマッピングします。`packages/tui` 内の動作と `packages/coding-agent` コントローラーからの統合に焦点を当てています。

## ランタイムレイヤーと責務

- **`packages/tui` エンジン**: ターミナルライフサイクル、stdin正規化、フォーカスルーティング、レンダースケジューリング、差分描画、オーバーレイ合成、ハードウェアカーソル配置。
- **`packages/coding-agent` インタラクティブモード**: コンポーネントツリーの構築、エディタコールバックとキーマップのバインド、エージェント/セッションイベントへの反応、ドメイン状態（ストリーミング、ツール実行、リトライ、プランモード）のUIコンポーネントへの変換。

境界ルール: TUIエンジンはメッセージ非依存です。`Component.render(width)`、`handleInput(data)`、フォーカス、およびオーバーレイのみを認識します。エージェントのセマンティクスはインタラクティブコントローラーに留まります。

## 起動とコンポーネントツリーの組み立て

`InteractiveMode` は `TUI(new ProcessTerminal(), showHardwareCursor)` を構築し、永続コンテナを作成します:

- `chatContainer`
- `pendingMessagesContainer`
- `statusContainer`
- `todoContainer`
- `statusLine`
- `editorContainer`（`CustomEditor` を保持）

`init()` はこの順序でツリーを接続し、エディタにフォーカスし、`InputController` 経由で入力ハンドラを登録し、TUIを開始して、強制レンダーを要求します。

強制レンダー（`requestRender(true)`）は、再描画前に前回の行キャッシュとカーソルの追跡情報をリセットします。

## ターミナルライフサイクルとstdin正規化

`ProcessTerminal.start()`:

1. rawモードとブラケットペーストを有効化します。
2. リサイズハンドラをアタッチします。
3. 部分的なエスケープチャンクを完全なシーケンスに分割するために `StdinBuffer` を作成します。
4. Kittyキーボードプロトコルのサポートを照会し（`CSI ? u`）、サポートされている場合はプロトコルフラグを有効にします。
5. Windowsでは、`kernel32` モードフラグによるVT入力の有効化を試みます。

`StdinBuffer` の動作:

- 断片化されたエスケープシーケンス（CSI/OSC/DCS/APC/SS3）をバッファリングします。
- シーケンスが完了するかタイムアウトでフラッシュされた場合にのみ `data` を発行します。
- ブラケットペーストを検出し、生のペーストテキストとともに `paste` イベントを発行します。

これにより、部分的なエスケープチャンクが通常のキー入力として誤解釈されるのを防ぎます。

## 入力ルーティングとフォーカスモデル

入力パス:

`stdin -> ProcessTerminal -> StdinBuffer -> TUI.#handleInput -> focusedComponent.handleInput`

ルーティングの詳細:

1. TUIは最初に登録済みの入力リスナー（`addInputListener`）を実行し、消費/変換動作を可能にします。
2. TUIはコンポーネントディスパッチの前にグローバルデバッグショートカット（`shift+ctrl+d`）を処理します。
3. フォーカスされたコンポーネントが非表示/不可視になったオーバーレイに属している場合、TUIは次の可視オーバーレイまたは保存されたオーバーレイ前のフォーカスにフォーカスを再割り当てします。
4. フォーカスされたコンポーネントが `wantsKeyRelease = true` を設定していない限り、キーリリースイベントはフィルタリングされます。
5. ディスパッチ後、TUIはレンダーをスケジュールします。

`setFocus()` は `Focusable.focused` も切り替え、コンポーネントがハードウェアカーソル配置のために `CURSOR_MARKER` を発行するかどうかを制御します。

## キー処理の分割: エディタ vs コントローラー

`CustomEditor` は高優先度のコンボ（escape、ctrl-c/d/z、ctrl-v、ctrl-pバリアント、ctrl-t、alt-up、拡張カスタムキー）を最初にインターセプトし、残りをベースの `Editor` 動作（テキスト編集、履歴、オートコンプリート、カーソル移動）に委譲します。

`InputController.setupKeyHandlers()` はその後、エディタコールバックをモードアクションにバインドします:

- `Escape` でのキャンセル/モード終了
- ダブル `Ctrl+C` またはエディタ空の `Ctrl+D` でのシャットダウン
- `Ctrl+Z` でのサスペンド/レジューム
- スラッシュコマンドとセレクターホットキー
- フォローアップ/デキュートグルと展開トグル

これにより、キー解析/エディタメカニクスは `packages/tui` に、モードセマンティクスはcoding-agentコントローラーに保持されます。

## レンダーループと差分戦略

`TUI.requestRender()` は `process.nextTick` を使用して1ティックあたり1レンダーにデバウンスされます。同一ターン内の複数の状態変更は統合されます。

`#doRender()` パイプライン:

1. ルートコンポーネントツリーを `newLines` にレンダリングします。
2. 可視オーバーレイ（存在する場合）を合成します。
3. 可視ビューポート行から `CURSOR_MARKER` を抽出して除去します。
4. 非画像行にセグメントリセットサフィックスを追加します。
5. フル再描画か差分パッチかを選択します:
   - 最初のフレーム
   - 幅の変更
   - `clearOnShrink` が有効でオーバーレイがない場合の縮小
   - 前のビューポートより上での編集
6. 差分更新の場合、変更された行範囲のみをパッチし、必要に応じて古い末尾行をクリアします。
7. IMEサポートのためにハードウェアカーソルを再配置します。

レンダー書き込みは同期出力モード（`CSI ? 2026 h/l`）を使用してフリッカー/テアリングを軽減します。

## レンダーの安全性制約

`TUI` における重要な安全性チェック:

- 非画像のレンダリング行はターミナル幅を超えてはなりません。オーバーフローはスローを発生させ、クラッシュ診断を書き込みます。
- オーバーレイ合成には防御的な切り詰めと合成後の幅検証が含まれます。
- 幅の変更はラッピングセマンティクスが変わるためフル再描画を強制します。
- カーソル位置は移動前にクランプされます。

これらの制約は単なる慣例ではなく、ランタイムでの強制です。

## リサイズ処理

リサイズイベントは `ProcessTerminal` から `TUI.requestRender()` へイベント駆動で伝達されます。

効果:

- 幅の変更はフル再描画をトリガーします。
- ビューポート/トップの追跡（`#previousViewportTop`、`#maxLinesRendered`）は、コンテンツやターミナルサイズの変更時に無効な相対カーソル計算を回避します。
- オーバーレイの可視性はターミナルの寸法に依存する場合があります（`OverlayOptions.visible`）。リサイズ後にオーバーレイが非可視になった場合、フォーカスが修正されます。

## ストリーミングとインクリメンタルUI更新

`EventController` は `AgentSessionEvent` をサブスクライブし、UIをインクリメンタルに更新します:

- `agent_start`: `statusContainer` でローダーを開始します。
- `message_start` assistant: `streamingComponent` を作成してマウントします。
- `message_update`: ストリーミングアシスタントコンテンツを更新し、ツールコールの出現に応じてツール実行コンポーネントを作成/更新します。
- `tool_execution_update/end`: ツール結果コンポーネントと完了状態を更新します。
- `message_end`: アシスタントストリームをファイナライズし、中断/エラーアノテーションを処理し、通常停止時に保留中のツール引数を完了とマークします。
- `agent_end`: ローダーを停止し、一時的なストリーム状態をクリアし、遅延モデル切り替えをフラッシュし、バックグラウンド中の場合は完了通知を発行します。

読み取りツールのグループ化は意図的にステートフル（`#lastReadGroup`）であり、非読み取りの中断が発生するまで連続する読み取りツールコールを1つのビジュアルブロックに統合します。

## ステータスとローダーのオーケストレーション

ステータスレーンの責務:

- `statusContainer` は一時的なローダー（`loadingAnimation`、`autoCompactionLoader`、`retryLoader`）を保持します。
- `statusLine` は永続的なステータス/フック/プランインジケーターをレンダリングし、エディタの上部ボーダー更新を駆動します。

ローダーの動作:

- `Loader` は80msごとにインターバルで更新し、各フレームでレンダーを要求します。
- 自動コンパクションと自動リトライ中は、それらの操作をキャンセルするためにエスケープハンドラが一時的にオーバーライドされます。
- 終了/キャンセルパスでは、コントローラーは以前のエスケープハンドラを復元し、ローダーコンポーネントを停止/クリアします。

## モード遷移とバックグラウンド化

### Bash/Python入力モード

入力テキストのプレフィックスがエディタのボーダーモードフラグを切り替えます:

- `!` -> bashモード
- `$`（テンプレートリテラルプレフィックスではない場合） -> pythonモード

Escapeは非アクティブモードを終了し、エディタテキストをクリアしてボーダー色を復元します。実行がアクティブな場合、Escapeは代わりに実行中のタスクを中断します。

### プランモード

`InteractiveMode` はプランモードフラグ、ステータスライン状態、アクティブツール、およびモデル切り替えを追跡します。開始/終了はセッションモードエントリとステータス/UI状態を更新し、ストリーミングがアクティブな場合は遅延モデル切り替えを含みます。

### サスペンド/レジューム（`Ctrl+Z`）

`InputController.handleCtrlZ()`:

1. TUIを再起動してフォースレンダーするためのワンショット `SIGCONT` ハンドラを登録します。
2. サスペンド前にTUIを停止します。
3. プロセスグループに `SIGTSTP` を送信します。

### バックグラウンドモード（`/background` または `/bg`）

`handleBackgroundCommand()`:

- アイドル時は拒否します。
- ツールUIコンテキストを非インタラクティブ（`hasUI=false`）に切り替え、インタラクティブUIツールが即座に失敗するようにします。
- ローダー/ステータスラインを停止し、フォアグラウンドイベントハンドラをアンサブスクライブします。
- バックグラウンドイベントハンドラ（主に `agent_end` を待機）をサブスクライブします。
- TUIを停止し、`SIGTSTP` を送信します（POSIXジョブコントロールパス）。

キューに入った作業がない状態でバックグラウンドで `agent_end` が発生すると、コントローラーは完了通知を送信してシャットダウンします。

## キャンセルパス

主なキャンセル入力:

- アクティブストリームローダー中の `Escape`: キューに入ったメッセージをエディタに復元し、エージェントを中断します。
- bash/python実行中の `Escape`: 実行中のコマンドを中断します。
- 自動コンパクション/リトライ中の `Escape`: 一時的なエスケープハンドラを通じて専用の中断メソッドを呼び出します。
- `Ctrl+C` シングルプレス: エディタをクリア。500ms以内のダブルプレス: シャットダウン。

キャンセルは状態条件付きです。同じキーがランタイム状態に応じて、中断、モード終了、セレクタートリガー、またはノーオペレーションを意味する場合があります。

## イベント駆動 vs スロットル動作

イベント駆動の更新:

- エージェントセッションイベント（`EventController`）
- キー入力コールバック（`InputController`）
- ターミナルリサイズコールバック
- `InteractiveMode` のテーマ/ブランチウォッチャー

スロットル/デバウンスパス:

- TUIレンダリングはティックデバウンスされます（`requestRender` 統合）。
- ローダーアニメーションは固定インターバル（80ms）で、各フレームがレンダーを要求します。
- エディタのオートコンプリート更新（`Editor` 内部）はデバウンスタイマーを使用し、タイピング中の再計算チャーンを削減します。

したがって、このランタイムはイベント駆動の状態遷移と制限されたレンダーケイデンスを混合し、再描画ストームなしにインタラクティブ性を維持します。
