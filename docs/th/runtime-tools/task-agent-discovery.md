# การค้นพบและการเลือกตัวแทนงาน

เอกสารนี้อธิบายวิธีที่ระบบย่อยงานค้นพบนิยามตัวแทน รวมแหล่งข้อมูลหลายแหล่งเข้าด้วยกัน และแก้ไขตัวแทนที่ร้องขอในเวลาดำเนินการ

ครอบคลุมพฤติกรรมรันไทม์ตามที่ได้นำไปใช้งานในปัจจุบัน รวมถึงลำดับความสำคัญ การจัดการนิยามที่ไม่ถูกต้อง และข้อจำกัดการสร้าง/ความลึกที่อาจทำให้ตัวแทนไม่พร้อมใช้งานได้

## รูปแบบนิยามตัวแทน

ตัวแทนงานจะถูกนอร์มอลไลซ์ไปยัง `AgentDefinition` (`src/task/types.ts`):

- `name`, `description`, `systemPrompt` (จำเป็นสำหรับตัวแทนที่โหลดได้อย่างถูกต้อง)
- ตัวเลือก: `tools`, `spawns`, `model`, `thinkingLevel`, `output`
- `source`: `"bundled" | "user" | "project"`
- ตัวเลือก: `filePath`

การแปลงข้อมูลมาจาก frontmatter ผ่าน `parseAgentFields()` (`src/discovery/helpers.ts`):

- หาก `name` หรือ `description` หายไป => ไม่ถูกต้อง (`null`) ผู้เรียกใช้จะถือว่าเป็นความล้มเหลวในการแปลง
- `tools` รับค่าเป็น CSV หรืออาร์เรย์ หากระบุไว้ `submit_result` จะถูกเพิ่มโดยอัตโนมัติ
- `spawns` รับค่าเป็น `*`, CSV, หรืออาร์เรย์
- พฤติกรรมความเข้ากันได้แบบย้อนหลัง: หาก `spawns` ไม่มีแต่ `tools` มี `task` อยู่ `spawns` จะกลายเป็น `*`
- `output` จะถูกส่งผ่านไปเป็นข้อมูล schema แบบทึบ

## ตัวแทนที่รวมมาด้วย

ตัวแทนที่รวมมาด้วยจะถูกฝังในเวลาสร้าง (`src/task/agents.ts`) โดยใช้การนำเข้าข้อความ

`EMBEDDED_AGENT_DEFS` กำหนด:

- `explore`, `plan`, `designer`, `reviewer` จากไฟล์ prompt
- `task` และ `quick_task` จากเนื้อหา `task.md` ที่แชร์ร่วมกัน พร้อม frontmatter ที่ถูกฉีดเข้ามา

เส้นทางการโหลด:

1. `loadBundledAgents()` แปลง markdown ที่ฝังมาด้วย `parseAgent(..., "bundled", "fatal")`
2. ผลลัพธ์จะถูกแคชไว้ในหน่วยความจำ (`bundledAgentsCache`)
3. `clearBundledAgentsCache()` ใช้สำหรับรีเซ็ตแคชในการทดสอบเท่านั้น

เนื่องจากการแปลงตัวแทนที่รวมมาด้วยใช้ `level: "fatal"` frontmatter ที่มีรูปแบบไม่ถูกต้องจะโยนข้อยกเว้นและอาจทำให้การค้นพบล้มเหลวทั้งหมด

## การค้นพบจากระบบไฟล์และปลั๊กอิน

`discoverAgents(cwd, home)` (`src/task/discovery.ts`) รวมตัวแทนจากหลายแหล่งก่อนที่จะผนวกนิยามตัวแทนที่รวมมาด้วย

### แหล่งข้อมูลสำหรับการค้นพบ

1. ไดเรกทอรีตัวแทนจากการตั้งค่าผู้ใช้ผ่าน `getConfigDirs("agents", { project: false })`
2. ไดเรกทอรีตัวแทนของโปรเจกต์ที่ใกล้ที่สุดจาก `findAllNearestProjectConfigDirs("agents", cwd)`
3. รากปลั๊กอิน Claude (`listClaudePluginRoots(home)`) พร้อมไดเรกทอรีย่อย `agents/`
4. ตัวแทนที่รวมมาด้วย (`loadBundledAgents()`)

### ลำดับแหล่งข้อมูลที่แท้จริง

ลำดับของกลุ่มแหล่งข้อมูลมาจาก `getConfigDirs("", { project: false })` ซึ่งมาจาก `priorityList` ใน `src/config.ts`:

1. `.xcsh`
2. `.claude`
3. `.codex`
4. `.gemini`

สำหรับแต่ละกลุ่มแหล่งข้อมูล ลำดับการค้นพบคือ:

1. ไดเรกทอรีโปรเจกต์ที่ใกล้ที่สุดสำหรับแหล่งข้อมูลนั้น (ถ้าพบ)
2. ไดเรกทอรีผู้ใช้สำหรับแหล่งข้อมูลนั้น

หลังจากไดเรกทอรีของกลุ่มแหล่งข้อมูลทั้งหมด ไดเรกทอรี `agents/` ของปลั๊กอินจะถูกผนวกต่อท้าย (ปลั๊กอินระดับโปรเจกต์ก่อน แล้วตามด้วยระดับผู้ใช้)

ตัวแทนที่รวมมาด้วยจะถูกผนวกเป็นลำดับสุดท้าย

### ข้อควรระวังสำคัญ: ความไม่สอดคล้องระหว่างความคิดเห็นเก่าและโค้ดปัจจุบัน

ความคิดเห็นส่วนหัวของ `discovery.ts` ยังคงกล่าวถึง `.pi` และไม่ได้กล่าวถึง `.codex`/`.gemini` ลำดับรันไทม์จริงถูกขับเคลื่อนโดย `src/config.ts` และปัจจุบันใช้ `.xcsh`, `.claude`, `.codex`, `.gemini`

## กฎการรวมและการชนกัน

การค้นพบใช้การลบรายการซ้ำแบบชนะก่อนโดยจับคู่ `agent.name` ที่ตรงกันทุกอักษร:

- ใช้ `Set<string>` เพื่อติดตามชื่อที่เห็นแล้ว
- ตัวแทนที่โหลดมาจะถูกแผ่ออกตามลำดับไดเรกทอรีและเก็บไว้เฉพาะเมื่อชื่อยังไม่เคยเห็น
- ตัวแทนที่รวมมาด้วยจะถูกกรองเทียบกับชุดเดียวกันและเพิ่มเข้าเฉพาะเมื่อยังไม่เคยเห็น

ผลกระทบ:

- โปรเจกต์จะแทนที่ผู้ใช้สำหรับกลุ่มแหล่งข้อมูลเดียวกัน
- กลุ่มแหล่งข้อมูลที่มีลำดับความสำคัญสูงกว่าจะแทนที่ลำดับที่ต่ำกว่า (`.xcsh` ก่อน `.claude` เป็นต้น)
- ตัวแทนที่ไม่ได้รวมมาจะแทนที่ตัวแทนที่รวมมาด้วยซึ่งมีชื่อเดียวกัน
- การจับคู่ชื่อคำนึงถึงตัวพิมพ์ (`Task` และ `task` ถือเป็นคนละตัว)
- ภายในไดเรกทอรีเดียว ไฟล์ markdown จะถูกอ่านตามลำดับชื่อไฟล์แบบ lexicographic ก่อนการลบรายการซ้ำ

## พฤติกรรมเมื่อไฟล์ตัวแทนไม่ถูกต้องหรือหายไป

สำหรับแต่ละไดเรกทอรี (`loadAgentsFromDir`):

- ไดเรกทอรีที่อ่านไม่ได้หรือหายไป: ถือว่าว่างเปล่า (`readdir(...).catch(() => [])`)
- ความล้มเหลวในการอ่านหรือแปลงไฟล์: บันทึกคำเตือน ข้ามไฟล์นั้น
- เส้นทางการแปลงใช้ `parseAgent(..., level: "warn")`

พฤติกรรมเมื่อ frontmatter ล้มเหลวมาจาก `parseFrontmatter`:

- ข้อผิดพลาดในการแปลงที่ระดับ `warn` จะบันทึกคำเตือน
- ตัวแปลงจะสำรองไปใช้ตัวแปลง `key: value` แบบง่าย
- หากยังคงขาดฟิลด์ที่จำเป็น `parseAgentFields` จะล้มเหลว จากนั้น `AgentParsingError` จะถูกโยนและถูกดักจับโดยผู้เรียก (ไฟล์จะถูกข้าม)

ผลสุทธิ: ไฟล์ตัวแทนแบบกำหนดเองที่เสียหายหนึ่งไฟล์จะไม่ยุติการค้นพบไฟล์อื่น

## การค้นหาและการเลือกตัวแทน

การค้นหาเป็นการค้นหาเชิงเส้นตามชื่อที่ตรงกันทุกอักษร:

- `getAgent(agents, name)` => `agents.find(a => a.name === name)`

ในการดำเนินการงาน (`TaskTool.execute`):

1. ตัวแทนจะถูกค้นพบใหม่อีกครั้งในเวลาเรียกใช้ (`discoverAgents(this.session.cwd)`)
2. `params.agent` ที่ร้องขอจะถูกแก้ไขผ่าน `getAgent`
3. ตัวแทนที่ไม่พบจะคืนค่าการตอบสนองของเครื่องมือทันที:
   - `Unknown agent "...". Available: ...`
   - ไม่มีการรันกระบวนการย่อย

### ความแตกต่างระหว่างการค้นพบในเวลาอธิบายและเวลาดำเนินการ

`TaskTool.create()` สร้างคำอธิบายเครื่องมือจากผลลัพธ์การค้นพบในเวลาเริ่มต้น (`buildDescription`)

`execute()` จะค้นพบตัวแทนใหม่อีกครั้ง ดังนั้นชุดรันไทม์อาจแตกต่างจากสิ่งที่แสดงอยู่ในคำอธิบายเครื่องมือก่อนหน้านี้หากไฟล์ตัวแทนมีการเปลี่ยนแปลงระหว่างเซสชัน

## การป้องกันข้อมูลเอาต์พุตแบบมีโครงสร้างและลำดับความสำคัญของ schema

ลำดับความสำคัญของ schema เอาต์พุตรันไทม์ใน `TaskTool.execute`:

1. `output` จาก frontmatter ของตัวแทน
2. `params.schema` จากการเรียกงาน
3. `outputSchema` ของเซสชันหลัก

(`effectiveOutputSchema = effectiveAgent.output ?? outputSchema ?? this.session.outputSchema`)

ข้อความป้องกันในเวลา prompt ใน `src/prompts/tools/task.md` เตือนเกี่ยวกับพฤติกรรมที่ไม่สอดคล้องกันสำหรับตัวแทนที่มีเอาต์พุตแบบมีโครงสร้าง (`explore`, `reviewer`): คำสั่งรูปแบบเอาต์พุตในข้อความธรรมดาอาจขัดแย้งกับ schema ที่ฝังมาด้วยและให้ผลลัพธ์เป็น `null`

นี่คือคำแนะนำ ไม่ใช่ลอจิกการตรวจสอบรันไทม์แบบเข้มงวดใน `discoverAgents`

## การโต้ตอบกับการค้นพบคำสั่ง

`src/task/commands.ts` เป็นโครงสร้างพื้นฐานคู่ขนานสำหรับคำสั่งเวิร์กโฟลว์ (ไม่ใช่นิยามตัวแทน) แต่ปฏิบัติตามรูปแบบโดยรวมเดียวกัน:

- ค้นพบจากผู้ให้ความสามารถก่อน
- ลบรายการซ้ำตามชื่อโดยใช้หลักชนะก่อน
- ผนวกคำสั่งที่รวมมาด้วยหากยังไม่เคยเห็น
- ค้นหาตามชื่อที่ตรงกันทุกอักษรผ่าน `getCommand`

ใน `src/task/index.ts` ตัวช่วยคำสั่งจะถูกส่งออกใหม่พร้อมกับตัวช่วยการค้นพบตัวแทน การค้นพบตัวแทนเองไม่ได้ขึ้นอยู่กับการค้นพบคำสั่งในรันไทม์

## ข้อจำกัดความพร้อมใช้งานนอกเหนือจากการค้นพบ

ตัวแทนอาจค้นพบได้แต่ยังไม่พร้อมใช้งานเนื่องจากการป้องกันในการดำเนินการ

### นโยบายการสร้างของเซสชันหลัก

`TaskTool.execute` ตรวจสอบ `session.getSessionSpawns()`:

- `"*"` => อนุญาตทั้งหมด
- `""` => ปฏิเสธทั้งหมด
- รายการ CSV => อนุญาตเฉพาะชื่อที่ระบุ

หากถูกปฏิเสธ: การตอบสนอง `Cannot spawn '...'. Allowed: ...` จะส่งคืนทันที

### การป้องกันการเรียกซ้ำด้วย env

`PI_BLOCKED_AGENT` ถูกอ่านในเวลาสร้างเครื่องมือ หากคำขอตรงกัน การดำเนินการจะถูกปฏิเสธพร้อมข้อความป้องกันการเรียกซ้ำ

### การจำกัดความลึกของการเรียกซ้ำ (ความพร้อมใช้งานของเครื่องมืองานภายในเซสชันลูก)

ใน `runSubprocess` (`src/task/executor.ts`):

- ความลึกคำนวณจาก `taskDepth`
- `task.maxRecursionDepth` ควบคุมจุดตัด
- เมื่อถึงความลึกสูงสุด:
  - เครื่องมือ `task` จะถูกลบออกจากรายการเครื่องมือของลูก
  - `spawns` env ของลูกจะถูกตั้งค่าให้ว่างเปล่า

ดังนั้นระดับที่ลึกกว่าจะไม่สามารถสร้างงานเพิ่มเติมได้แม้ว่านิยามตัวแทนจะมี `spawns` อยู่ก็ตาม

## ข้อควรระวังเกี่ยวกับโหมดแผน (การนำไปใช้งานปัจจุบัน)

`TaskTool.execute` คำนวณ `effectiveAgent` สำหรับโหมดแผน (เพิ่ม prompt โหมดแผนไว้ข้างหน้า บังคับใช้ชุดเครื่องมือแบบอ่านอย่างเดียว ล้าง spawns) แต่ `runSubprocess` ถูกเรียกด้วย `agent` แทน `effectiveAgent`

ผลกระทบปัจจุบัน:

- การแทนที่โมเดล / ระดับการคิด / schema เอาต์พุต มาจาก `effectiveAgent`
- prompt ระบบ และการจำกัดเครื่องมือ/spawn จาก `effectiveAgent` ไม่ถูกส่งผ่านในเส้นทางการเรียกนี้

นี่คือข้อควรระวังในการนำไปใช้งานที่ควรทราบเมื่ออ่านความคาดหวังพฤติกรรมของโหมดแผน
