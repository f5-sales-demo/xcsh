# รันไทม์เครื่องมือ Bash

เอกสารนี้อธิบายเส้นทางรันไทม์ของ**เครื่องมือ `bash`** ที่ใช้โดยการเรียกเครื่องมือของเอเจนต์ ตั้งแต่การปรับมาตรฐานคำสั่งไปจนถึงการดำเนินการ การตัดทอน/อาร์ทิแฟกต์ และการเรนเดอร์

นอกจากนี้ยังระบุจุดที่พฤติกรรมแตกต่างกันใน TUI แบบโต้ตอบ โหมดพิมพ์ โหมด RPC และการดำเนินการเชลล์แบบ bang (`!`) ที่ผู้ใช้เริ่มต้น

## ขอบเขตและพื้นผิวรันไทม์

มีพื้นผิวการดำเนินการ bash สองแบบที่แตกต่างกันใน coding-agent:

1. **พื้นผิวการเรียกเครื่องมือ** (`toolName: "bash"`): ใช้เมื่อโมเดลเรียกเครื่องมือ bash
   - จุดเข้า: `BashTool.execute()`
2. **พื้นผิวคำสั่ง bang ของผู้ใช้** (`!cmd` จากอินพุตแบบโต้ตอบหรือคำสั่ง RPC `bash`): เส้นทางตัวช่วยระดับเซสชัน
   - จุดเข้า: `AgentSession.executeBash()`

ทั้งสองใช้ `executeBash()` ใน `src/exec/bash-executor.ts` สำหรับการดำเนินการแบบไม่ใช้ PTY ในที่สุด แต่เฉพาะเส้นทางการเรียกเครื่องมือเท่านั้นที่รันลอจิกการปรับมาตรฐาน/การสกัดกั้นและตัวเรนเดอร์เครื่องมือ

## ไปป์ไลน์การเรียกเครื่องมือแบบครบวงจร

## 1) การปรับมาตรฐานอินพุตและการรวมพารามิเตอร์

`BashTool.execute()` ปรับมาตรฐานคำสั่งดิบก่อนผ่าน `normalizeBashCommand()`:

- แยก `| head -n N`, `| head -N`, `| tail -n N`, `| tail -N` ที่ต่อท้ายออกเป็นขีดจำกัดแบบมีโครงสร้าง
- ตัดช่องว่างที่ต่อท้าย/นำหน้า
- คงช่องว่างภายในไว้

จากนั้นรวมขีดจำกัดที่แยกออกมากับอาร์กิวเมนต์เครื่องมือที่ระบุชัดเจน:

- อาร์กิวเมนต์ `head`/`tail` ที่ระบุชัดเจนจะแทนที่ค่าที่แยกออกมา
- ค่าที่แยกออกมาเป็นค่าสำรองเท่านั้น

### ข้อควรระวัง

ความคิดเห็นใน `bash-normalize.ts` กล่าวถึงการลบ `2>&1` แต่การใช้งานปัจจุบันไม่ได้ลบออก พฤติกรรมรันไทม์ยังคงถูกต้อง (stdout/stderr ถูกรวมแล้ว) แต่พฤติกรรมการปรับมาตรฐานแคบกว่าที่ความคิดเห็นแนะนำ

## 2) การสกัดกั้นแบบเลือกได้ (เส้นทางคำสั่งที่ถูกบล็อก)

หาก `bashInterceptor.enabled` เป็น true `BashTool` จะโหลดกฎจากการตั้งค่าและรัน `checkBashInterception()` กับคำสั่งที่ปรับมาตรฐานแล้ว

พฤติกรรมการสกัดกั้น:

- คำสั่งจะถูกบล็อก**เฉพาะเมื่อ**:
  - กฎ regex ตรงกัน และ
  - เครื่องมือที่แนะนำมีอยู่ใน `ctx.toolNames`
- กฎ regex ที่ไม่ถูกต้องจะถูกข้ามโดยไม่แจ้ง
- เมื่อถูกบล็อก `BashTool` จะโยน `ToolError` พร้อมข้อความ:
  - `Blocked: ...`
  - รวมคำสั่งดั้งเดิม

รูปแบบกฎเริ่มต้น (กำหนดในโค้ด) กำหนดเป้าหมายการใช้งานผิดที่พบบ่อย:

- ตัวอ่านไฟล์ (`cat`, `head`, `tail`, ...)
- เครื่องมือค้นหา (`grep`, `rg`, ...)
- ตัวค้นหาไฟล์ (`find`, `fd`, ...)
- ตัวแก้ไขแบบ in-place (`sed -i`, `perl -i`, `awk -i inplace`)
- การเขียนด้วยการเปลี่ยนเส้นทางเชลล์ (`echo ... > file`, heredoc redirection)

### ข้อควรระวัง

`InterceptionResult` รวม `suggestedTool` แต่ `BashTool` ปัจจุบันแสดงเฉพาะข้อความ (ไม่มีฟิลด์เครื่องมือแนะนำแบบมีโครงสร้างใน `details`)

## 3) การตรวจสอบ CWD และการจำกัดค่าหมดเวลา

`cwd` ถูกแก้ไขโดยอ้างอิงจาก cwd ของเซสชัน (`resolveToCwd`) จากนั้นตรวจสอบผ่าน `stat`:

- เส้นทางไม่มีอยู่ -> `ToolError("Working directory does not exist: ...")`
- ไม่ใช่ไดเรกทอรี -> `ToolError("Working directory is not a directory: ...")`

ค่าหมดเวลาถูกจำกัดไว้ที่ `[1, 3600]` วินาทีและแปลงเป็นมิลลิวินาที

## 4) การจัดสรรอาร์ทิแฟกต์

ก่อนการดำเนินการ เครื่องมือจะจัดสรรเส้นทาง/id ของอาร์ทิแฟกต์ (แบบ best-effort) สำหรับการจัดเก็บเอาต์พุตที่ถูกตัดทอน

- ความล้มเหลวในการจัดสรรอาร์ทิแฟกต์ไม่ร้ายแรง (การดำเนินการดำเนินต่อไปโดยไม่มีไฟล์ spill ของอาร์ทิแฟกต์)
- id/เส้นทางของอาร์ทิแฟกต์จะถูกส่งเข้าไปในเส้นทางการดำเนินการสำหรับการคงอยู่ของเอาต์พุตแบบเต็มเมื่อถูกตัดทอน

## 5) การเลือกการดำเนินการแบบ PTY กับไม่ใช่ PTY

`BashTool` เลือกการดำเนินการแบบ PTY เฉพาะเมื่อเงื่อนไขทั้งหมดเป็นจริง:

- `bash.virtualTerminal === "on"`
- `PI_NO_PTY !== "1"`
- บริบทเครื่องมือมี UI (`ctx.hasUI === true` และ `ctx.ui` ถูกตั้งค่า)

มิฉะนั้นจะใช้ `executeBash()` แบบไม่โต้ตอบ

นั่นหมายความว่าโหมดพิมพ์และบริบท RPC/เครื่องมือที่ไม่มี UI จะใช้แบบไม่ใช่ PTY เสมอ

## เอนจินการดำเนินการแบบไม่โต้ตอบ (`executeBash`)

## โมเดลการนำเซสชันเชลล์กลับมาใช้ซ้ำ

`executeBash()` แคชอินสแตนซ์ `Shell` เนทีฟในแมประดับโปรเซสส์ทั่วโลกโดยใช้คีย์ตาม:

- เส้นทางเชลล์
- คำนำหน้าคำสั่งที่กำหนดค่า
- เส้นทางสแนปชอต
- shell env ที่ซีเรียลไลซ์
- คีย์เซสชันเอเจนต์แบบเลือกได้

สำหรับการดำเนินการระดับเซสชัน `AgentSession.executeBash()` ส่ง `sessionKey: this.sessionId` เพื่อแยกการนำกลับมาใช้ซ้ำต่อเซสชัน

เส้นทางการเรียกเครื่องมือ**ไม่**ส่ง `sessionKey` ดังนั้นขอบเขตการนำกลับมาใช้ซ้ำจะขึ้นอยู่กับ shell config/snapshot/env

## การกำหนดค่าเชลล์และพฤติกรรมสแนปชอต

ในการเรียกแต่ละครั้ง executor จะโหลดการกำหนดค่าเชลล์จากการตั้งค่า (`shell`, `env`, `prefix` แบบเลือกได้)

หากเชลล์ที่เลือกรวม `bash` จะพยายามใช้ `getOrCreateSnapshot()`:

- สแนปชอตบันทึก aliases/functions/options จาก rc ของผู้ใช้
- การสร้างสแนปชอตเป็นแบบ best-effort
- ความล้มเหลวจะถอยกลับไปใช้แบบไม่มีสแนปชอต

หากกำหนดค่า `prefix` คำสั่งจะกลายเป็น:

```text
<prefix> <command>
```

## การสตรีมและการยกเลิก

`Shell.run()` สตรีมชิ้นข้อมูลไปยัง callback Executor ส่งแต่ละชิ้นข้อมูลเข้าไปใน `OutputSink` และ callback `onChunk` แบบเลือกได้

การยกเลิก:

- สัญญาณ aborted จะทริกเกอร์ `shellSession.abort(...)`
- การหมดเวลาจากผลลัพธ์เนทีฟจะถูกแมปเป็น `cancelled: true` + ข้อความคำอธิบายประกอบ
- การยกเลิกแบบชัดเจนก็คืนค่า `cancelled: true` + คำอธิบายประกอบเช่นกัน

ไม่มีข้อยกเว้นถูกโยนภายใน executor สำหรับการหมดเวลา/ยกเลิก; มันคืนค่า `BashResult` แบบมีโครงสร้างและปล่อยให้ผู้เรียกแมปความหมายของข้อผิดพลาด

## เส้นทาง PTY แบบโต้ตอบ (`runInteractiveBashPty`)

เมื่อเปิดใช้งาน PTY เครื่องมือจะรัน `runInteractiveBashPty()` ซึ่งเปิดคอมโพเนนต์คอนโซลแบบโอเวอร์เลย์และขับเคลื่อน `PtySession` เนทีฟ

จุดเด่นของพฤติกรรม:

- เทอร์มินัลเสมือน xterm-headless เรนเดอร์ viewport ในโอเวอร์เลย์
- อินพุตแป้นพิมพ์ถูกปรับมาตรฐาน (รวมถึงลำดับ Kitty และการจัดการโหมดเคอร์เซอร์แอปพลิเคชัน)
- `esc` ขณะทำงานจะฆ่าเซสชัน PTY
- การปรับขนาดเทอร์มินัลจะถูกส่งต่อไปยัง PTY (`session.resize(cols, rows)`)

ค่าเริ่มต้นการเสริมความปลอดภัยของสภาพแวดล้อมจะถูกฉีดเข้าสำหรับการรันแบบไม่ต้องดูแล:

- ปิดใช้งาน pager (`PAGER=cat`, `GIT_PAGER=cat` เป็นต้น)
- ปิดใช้งานพรอมต์ตัวแก้ไข (`GIT_EDITOR=true`, `EDITOR=true` ...)
- ลดพรอมต์เทอร์มินัล/การยืนยันตัวตน (`GIT_TERMINAL_PROMPT=0`, `SSH_ASKPASS=/usr/bin/false`, `CI=1`)
- แฟล็กอัตโนมัติของตัวจัดการแพ็กเกจ/เครื่องมือสำหรับพฤติกรรมแบบไม่โต้ตอบ

เอาต์พุต PTY ถูกปรับมาตรฐาน (`CRLF`/`CR` เป็น `LF`, `sanitizeText`) และเขียนเข้าไปใน `OutputSink` รวมถึงรองรับ artifact spill

เมื่อเกิดข้อผิดพลาดในการเริ่มต้น/รันไทม์ PTY sink จะรับบรรทัด `PTY error: ...` และคำสั่งจะจบลงด้วย exit code ที่ไม่ได้กำหนด

## การจัดการเอาต์พุต: การสตรีม การตัดทอน artifact spill

ทั้งเส้นทาง PTY และไม่ใช่ PTY ใช้ `OutputSink`

## ความหมายของ OutputSink

- เก็บบัฟเฟอร์ tail ในหน่วยความจำที่ปลอดภัย UTF-8 (`DEFAULT_MAX_BYTES` ปัจจุบัน 50KB)
- ติดตามจำนวนไบต์/บรรทัดทั้งหมดที่เห็น
- หากมีเส้นทางอาร์ทิแฟกต์และเอาต์พุตล้น (หรือไฟล์ทำงานอยู่แล้ว) จะเขียนสตรีมทั้งหมดไปยังไฟล์อาร์ทิแฟกต์
- เมื่อเกณฑ์หน่วยความจำล้น จะตัดบัฟเฟอร์ในหน่วยความจำเป็น tail (ปลอดภัยต่อขอบเขต UTF-8)
- ทำเครื่องหมาย `truncated` เมื่อเกิดการล้น/file spill

`dump()` คืนค่า:

- `output` (คำนำหน้าที่อาจมีคำอธิบายประกอบ)
- `truncated`
- `totalLines/totalBytes`
- `outputLines/outputBytes`
- `artifactId` หากไฟล์อาร์ทิแฟกต์ทำงานอยู่

### ข้อควรระวังเรื่องเอาต์พุตยาว

การตัดทอนรันไทม์ใช้เกณฑ์ไบต์ใน `OutputSink` (ค่าเริ่มต้น 50KB) ไม่ได้บังคับขีดจำกัด 2000 บรรทัดแบบตายตัวในเส้นทางโค้ดนี้

## การอัปเดตเครื่องมือแบบสด

สำหรับการดำเนินการแบบไม่ใช่ PTY `BashTool` ใช้ `TailBuffer` แยกต่างหากสำหรับการอัปเดตบางส่วนและปล่อยสแนปชอต `onUpdate` ขณะคำสั่งกำลังทำงาน

สำหรับการดำเนินการแบบ PTY การเรนเดอร์แบบสดจะจัดการโดยโอเวอร์เลย์ UI แบบกำหนดเอง ไม่ใช่โดยชิ้นข้อความ `onUpdate`

## การจัดรูปแบบผลลัพธ์ เมตาดาต้า และการแมปข้อผิดพลาด

หลังการดำเนินการ:

1. การจัดการ `cancelled`:
   - หากสัญญาณ abort ถูกยกเลิก -> โยน `ToolAbortError` (ความหมายของการยกเลิก)
   - มิฉะนั้น -> โยน `ToolError` (ถือว่าเป็นความล้มเหลวของเครื่องมือ)
2. PTY `timedOut` -> โยน `ToolError`
3. ใช้ตัวกรอง head/tail กับข้อความเอาต์พุตสุดท้าย (`applyHeadTail`, head ก่อนแล้ว tail)
4. เอาต์พุตว่างจะกลายเป็น `(no output)`
5. แนบเมตาดาต้าการตัดทอนผ่าน `toolResult(...).truncationFromSummary(result, { direction: "tail" })`
6. การแมป exit-code:
   - ไม่มี exit code -> `ToolError("... missing exit status")`
   - exit ไม่ใช่ศูนย์ -> `ToolError("... Command exited with code N")`
   - exit เป็นศูนย์ -> ผลลัพธ์สำเร็จ

โครงสร้างเพย์โหลดสำเร็จ:

- `content`: เอาต์พุตข้อความ
- `details.meta.truncation` เมื่อถูกตัดทอน รวมถึง:
  - `direction`, `truncatedBy`, จำนวนบรรทัด+ไบต์ ทั้งหมด/เอาต์พุต
  - `shownRange`
  - `artifactId` เมื่อมี

เนื่องจากเครื่องมือในตัวถูกห่อด้วย `wrapToolWithMetaNotice()` ข้อความแจ้งการตัดทอนจะถูกต่อท้ายเนื้อหาข้อความสุดท้ายโดยอัตโนมัติ (เช่น: `Full: artifact://<id>`)

## เส้นทางการเรนเดอร์

## ตัวเรนเดอร์การเรียกเครื่องมือ (`bashToolRenderer`)

`bashToolRenderer` ใช้สำหรับข้อความการเรียกเครื่องมือ (`toolCall` / `toolResult`):

- โหมดยุบจะแสดงตัวอย่างที่ตัดทอนตามบรรทัดที่มองเห็น
- โหมดขยายจะแสดงข้อความเอาต์พุตทั้งหมดที่มีอยู่ในปัจจุบัน
- บรรทัดคำเตือนรวมเหตุผลการตัดทอนและ `artifact://<id>` เมื่อถูกตัดทอน
- ค่าหมดเวลา (จากอาร์กิวเมนต์) จะแสดงในบรรทัดเมตาดาต้าส่วนท้าย

### ข้อควรระวัง: การขยายอาร์ทิแฟกต์แบบเต็ม

`BashRenderContext` มี `isFullOutput` แต่ตัวสร้างบริบทเรนเดอร์ปัจจุบันไม่ได้ตั้งค่าให้สำหรับผลลัพธ์เครื่องมือ bash มุมมองแบบขยายยังคงใช้ข้อความที่มีอยู่แล้วในเนื้อหาผลลัพธ์ (เอาต์พุต tail/ที่ถูกตัดทอน) เว้นแต่ผู้เรียกรายอื่นจะให้เนื้อหาอาร์ทิแฟกต์แบบเต็ม

## คอมโพเนนต์คำสั่ง bang ของผู้ใช้ (`BashExecutionComponent`)

`BashExecutionComponent` สำหรับคำสั่ง `!` ของผู้ใช้ในโหมดโต้ตอบ (ไม่ใช่การเรียกเครื่องมือของโมเดล):

- สตรีมชิ้นข้อมูลแบบสด
- ตัวอย่างแบบยุบจะเก็บ 20 บรรทัดลอจิคัลสุดท้าย
- จำกัดบรรทัดที่ 4000 อักขระต่อบรรทัด
- แสดงคำเตือนการตัดทอน + อาร์ทิแฟกต์เมื่อมีเมตาดาต้า
- ทำเครื่องหมายสถานะยกเลิก/ข้อผิดพลาด/ออกแยกกัน

คอมโพเนนต์นี้ถูกเชื่อมต่อโดย `CommandController.handleBashCommand()` และรับข้อมูลจาก `AgentSession.executeBash()`

## ความแตกต่างของพฤติกรรมตามโหมด

| พื้นผิว                        | เส้นทางเข้า                                          | มีสิทธิ์ใช้ PTY                                                     | UX เอาต์พุตแบบสด                                                        | การแสดงข้อผิดพลาด                               |
| ------------------------------ | ----------------------------------------------------- | -------------------------------------------------------------------- | ------------------------------------------------------------------------ | ------------------------------------------------ |
| การเรียกเครื่องมือแบบโต้ตอบ    | `BashTool.execute`                                    | ใช่ เมื่อ `bash.virtualTerminal=on` และมี UI และ `PI_NO_PTY!=1`     | โอเวอร์เลย์ PTY (โต้ตอบ) หรือการอัปเดต tail แบบสตรีม                    | ข้อผิดพลาดเครื่องมือกลายเป็น `toolResult.isError` |
| การเรียกเครื่องมือโหมดพิมพ์    | `BashTool.execute`                                    | ไม่ (ไม่มีบริบท UI)                                                 | ไม่มีโอเวอร์เลย์ TUI; เอาต์พุตปรากฏในสตรีมเหตุการณ์/ข้อความผู้ช่วยสุดท้าย | การแมปข้อผิดพลาดเครื่องมือเดียวกัน              |
| การเรียกเครื่องมือ RPC (เครื่องมือเอเจนต์) | `BashTool.execute`                              | ปกติไม่มี UI -> ไม่ใช่ PTY                                          | เหตุการณ์/ผลลัพธ์เครื่องมือแบบมีโครงสร้าง                                | การแมปข้อผิดพลาดเครื่องมือเดียวกัน              |
| คำสั่ง bang แบบโต้ตอบ (`!`)    | `AgentSession.executeBash` + `BashExecutionComponent` | ไม่ (ใช้ executor โดยตรง)                                           | คอมโพเนนต์การดำเนินการ bash เฉพาะ                                        | Controller ดักจับข้อยกเว้นและแสดงข้อผิดพลาด UI  |
| คำสั่ง RPC `bash`              | `rpc-mode` -> `session.executeBash`                   | ไม่                                                                  | คืนค่า `BashResult` โดยตรง                                               | ผู้ใช้จัดการฟิลด์ที่คืนค่า                      |

## ข้อควรระวังในการดำเนินงาน

- Interceptor จะบล็อกคำสั่งเฉพาะเมื่อเครื่องมือที่แนะนำมีอยู่ในบริบทปัจจุบัน
- หากการจัดสรรอาร์ทิแฟกต์ล้มเหลว การตัดทอนยังคงเกิดขึ้นแต่ไม่มีการอ้างอิงกลับ `artifact://`
- แคชเซสชันเชลล์ไม่มีการลบอย่างชัดเจนในโมดูลนี้; อายุการใช้งานอยู่ในขอบเขตระดับโปรเซสส์
- พื้นผิวการหมดเวลาของ PTY และไม่ใช่ PTY แตกต่างกัน:
  - PTY แสดงฟิลด์ผลลัพธ์ `timedOut` อย่างชัดเจน
  - ไม่ใช่ PTY แมปการหมดเวลาเป็นสรุป `cancelled + annotation`
