## 1) โมเดลการค้นพบ

Slash command เป็นความสามารถ (`id: "slash-commands"`) ที่มีคีย์ตามชื่อคำสั่ง (`key: cmd => cmd.name`)

รีจิสทรีความสามารถโหลดผู้ให้บริการที่ลงทะเบียนทั้งหมด เรียงลำดับตามลำดับความสำคัญของผู้ให้บริการจากมากไปน้อย และลบรายการซ้ำตามคีย์ด้วยหลักการ **ผู้มาก่อนชนะ**

### ลำดับความสำคัญของผู้ให้บริการ

ผู้ให้บริการ slash command ในปัจจุบันและลำดับความสำคัญ:

1. `native` (OMP) — ลำดับความสำคัญ `100`
2. `claude` — ลำดับความสำคัญ `80`
3. `claude-plugins` — ลำดับความสำคัญ `70`
4. `codex` — ลำดับความสำคัญ `70`

พฤติกรรมเมื่อเสมอกัน: ผู้ให้บริการที่มีลำดับความสำคัญเท่ากันจะรักษาลำดับการลงทะเบียน ลำดับการนำเข้าในปัจจุบันลงทะเบียน `claude-plugins` ก่อน `codex` ดังนั้นคำสั่งปลั๊กอินจะชนะเหนือคำสั่ง codex เมื่อชื่อชนกัน

### พฤติกรรมเมื่อชื่อชนกัน

สำหรับ `slash-commands` การชนกันจะถูกแก้ไขโดยการลบรายการซ้ำของความสามารถอย่างเคร่งครัด:

- รายการที่มีลำดับความสำคัญสูงสุดจะถูกเก็บไว้ใน `result.items`
- รายการซ้ำที่มีลำดับความสำคัญต่ำกว่าจะอยู่เฉพาะใน `result.all` และถูกทำเครื่องหมาย `_shadowed = true`

สิ่งนี้ใช้กับทั้งผู้ให้บริการต่างๆ และภายในผู้ให้บริการเดียวกันหากมีการส่งคืนชื่อที่ซ้ำกัน

### พฤติกรรมการสแกนไฟล์

ผู้ให้บริการส่วนใหญ่ใช้ `loadFilesFromDir(...)` ซึ่งในปัจจุบัน:

- ค่าเริ่มต้นคือการจับคู่แบบไม่เรียกซ้ำ (`*.md`)
- ใช้ native glob พร้อม `gitignore: true`, `hidden: false`
- อ่านแต่ละไฟล์ที่จับคู่ได้และแปลงเป็น `SlashCommand`

ดังนั้นไฟล์/ไดเรกทอรีที่ซ่อนอยู่จะไม่ถูกโหลด และเส้นทางที่ถูกละเว้นจะถูกข้ามไป

## 2) เส้นทางต้นทางเฉพาะผู้ให้บริการและลำดับความสำคัญของโลคัล

## ผู้ให้บริการ `native` (`builtin.ts`)

รากการค้นหามาจากไดเรกทอรี `.xcsh`:

- โปรเจกต์: `<cwd>/.xcsh/commands/*.md`
- ผู้ใช้: `~/.xcsh/agent/commands/*.md`

`getConfigDirs()` ส่งคืนโปรเจกต์ก่อนแล้วจึงส่งคืนผู้ใช้ ดังนั้น **คำสั่ง native ของโปรเจกต์ชนะคำสั่ง native ของผู้ใช้** เมื่อชื่อชนกัน

## ผู้ให้บริการ `claude` (`claude.ts`)

โหลด:

- ผู้ใช้: `~/.claude/commands/*.md`
- โปรเจกต์: `<cwd>/.claude/commands/*.md`

ผู้ให้บริการจะดันรายการผู้ใช้ก่อนรายการโปรเจกต์ ดังนั้น **คำสั่ง Claude ของผู้ใช้ชนะคำสั่ง Claude ของโปรเจกต์** เมื่อชื่อชนกันภายในผู้ให้บริการนี้

## ผู้ให้บริการ `codex` (`codex.ts`)

โหลด:

- ผู้ใช้: `~/.codex/commands/*.md`
- โปรเจกต์: `<cwd>/.codex/commands/*.md`

ทั้งสองด้านจะถูกโหลดแล้วทำให้แบนราบในลำดับผู้ใช้ก่อน ดังนั้น **คำสั่ง Codex ของผู้ใช้ชนะคำสั่ง Codex ของโปรเจกต์** เมื่อชนกัน

เนื้อหาคำสั่ง Codex ถูกแยกวิเคราะห์ด้วยการลบ frontmatter (`parseFrontmatter`) และชื่อคำสั่งสามารถถูกแทนที่โดย frontmatter `name` หากมี มิฉะนั้นจะใช้ชื่อไฟล์

## ผู้ให้บริการ `claude-plugins` (`claude-plugins.ts`)

โหลดรากคำสั่งปลั๊กอินจาก `~/.claude/plugins/installed_plugins.json` จากนั้นสแกน `<pluginRoot>/commands/*.md`

การเรียงลำดับเป็นไปตามลำดับการวนซ้ำรีจิสทรีและลำดับรายการต่อปลั๊กอินจากข้อมูล JSON นั้น ไม่มีขั้นตอนการเรียงลำดับเพิ่มเติม

## 3) การสร้างเป็น `FileSlashCommand` สำหรับรันไทม์

`loadSlashCommands()` ใน `src/extensibility/slash-commands.ts` แปลงรายการความสามารถเป็นอ็อบเจกต์ `FileSlashCommand` ที่ใช้ในขณะแจ้งข้อความ

สำหรับแต่ละคำสั่ง:

1. แยกวิเคราะห์ frontmatter/เนื้อหา (`parseFrontmatter`)
2. แหล่งที่มาของคำอธิบาย:
   - `frontmatter.description` หากมี
   - หรือบรรทัดเนื้อหาที่ไม่ว่างบรรทัดแรก (ตัดช่องว่าง สูงสุด 60 อักขระพร้อม `...`)
3. เก็บเนื้อหาแม่แบบที่สามารถดำเนินการได้เป็นเนื้อหาแม่แบบ
4. คำนวณสตริงแหล่งที่มาสำหรับการแสดงผล เช่น `via Claude Code Project`

ความรุนแรงของการแยกวิเคราะห์ frontmatter ขึ้นอยู่กับแหล่งที่มา:

- ระดับ `native` -> ข้อผิดพลาดการแยกวิเคราะห์เป็น `fatal`
- ระดับ `user`/`project` -> ข้อผิดพลาดการแยกวิเคราะห์เป็น `warn` พร้อมการแยกวิเคราะห์สำรอง

### คำสั่งสำรองที่รวมอยู่ในตัว

หลังจากคำสั่งจากระบบไฟล์/ผู้ให้บริการ แม่แบบคำสั่งที่ฝังไว้จะถูกต่อท้าย (`EMBEDDED_COMMAND_TEMPLATES`) หากชื่อของแม่แบบยังไม่มีอยู่

ชุดที่ฝังในปัจจุบันมาจาก `src/task/commands.ts` และใช้เป็นสำรอง (`source: "bundled"`)

## 4) โหมดโต้ตอบ: แหล่งที่มาของรายการคำสั่ง

โหมดโต้ตอบรวมแหล่งคำสั่งหลายแหล่งสำหรับการเติมคำอัตโนมัติและการกำหนดเส้นทางคำสั่ง

ในช่วงการสร้างจะสร้างรายการคำสั่งที่รอดำเนินการจาก:

- คำสั่งในตัว (`BUILTIN_SLASH_COMMANDS` รวมถึงการเติมอาร์กิวเมนต์และคำแนะนำแบบอินไลน์สำหรับคำสั่งที่เลือก)
- slash command ที่ลงทะเบียนผ่านส่วนขยาย (`extensionRunner.getRegisteredCommands(...)`)
- คำสั่งที่กำหนดเองใน TypeScript (`session.customCommands`) ที่แมปกับป้ายกำกับ slash command
- คำสั่ง skill ที่เลือกได้ (`/skill:<name>`) เมื่อเปิดใช้งาน `skills.enableSkillCommands`

จากนั้น `init()` จะเรียก `refreshSlashCommandState(...)` เพื่อโหลดคำสั่งที่อิงตามไฟล์และติดตั้ง `CombinedAutocompleteProvider` หนึ่งตัวที่ประกอบด้วย:

- คำสั่งที่รอดำเนินการข้างต้น
- คำสั่งที่อิงตามไฟล์ที่ค้นพบ

`refreshSlashCommandState(...)` ยังอัปเดต `session.setSlashCommands(...)` ด้วย เพื่อให้การขยายแจ้งข้อความใช้ชุดคำสั่งไฟล์ที่ค้นพบชุดเดียวกัน

### วงจรชีวิตการรีเฟรช

สถานะ slash command จะถูกรีเฟรช:

- ระหว่างการเริ่มต้นโหมดโต้ตอบ
- หลังจาก `/move` เปลี่ยนไดเรกทอรีทำงาน (`handleMoveCommand` เรียก `resetCapabilities()` แล้วจึงเรียก `refreshSlashCommandState(newCwd)`)

ไม่มีตัวตรวจสอบไฟล์แบบต่อเนื่องสำหรับไดเรกทอรีคำสั่ง

### การแสดงผลอื่นๆ

แดชบอร์ดส่วนขยายยังโหลดความสามารถ `slash-commands` และแสดงรายการคำสั่งที่ใช้งานอยู่/ถูกซ่อน รวมถึงรายการซ้ำที่มี `_shadowed`

## 5) ตำแหน่งในไปป์ไลน์แจ้งข้อความ

ลำดับการจัดการ slash ของ `AgentSession.prompt(...)` (เมื่อ `expandPromptTemplates !== false`):

1. **คำสั่งส่วนขยาย** (`#tryExecuteExtensionCommand`)  
   หาก `/name` ตรงกับคำสั่งที่ลงทะเบียนส่วนขยาย ตัวจัดการจะดำเนินการทันทีและแจ้งข้อความส่งคืน
2. **คำสั่งที่กำหนดเองใน TypeScript** (`#tryExecuteCustomCommand`)  
   เฉพาะขอบเขต: หากตรงกัน จะดำเนินการและอาจส่งคืน:
   - `string` -> แทนที่ข้อความแจ้งข้อความด้วยสตริงนั้น
   - `void/undefined` -> ถือว่าจัดการแล้ว ไม่มีแจ้งข้อความ LLM
3. **Slash command ที่อิงตามไฟล์** (`expandSlashCommand`)  
   หากข้อความยังคงขึ้นต้นด้วย `/` ให้พยายามขยายคำสั่ง markdown
4. **แม่แบบแจ้งข้อความ** (`expandPromptTemplate`)  
   ใช้งานหลังจากการประมวลผล slash/คำสั่งที่กำหนดเอง
5. **การส่งมอบ**
   - ว่าง: แจ้งข้อความส่งไปยัง agent ทันที
   - กำลังสตรีม: แจ้งข้อความถูกจัดคิวเป็น steer/follow-up ขึ้นอยู่กับ `streamingBehavior`

นี่คือเหตุผลที่การขยาย slash command อยู่ก่อนการขยายแม่แบบแจ้งข้อความ และเหตุผลที่คำสั่งที่กำหนดเองสามารถเปลี่ยนแปลงเครื่องหมายทับนำหน้าก่อนการจับคู่คำสั่งไฟล์

## 6) ความหมายของการขยายสำหรับ slash command ที่อิงตามไฟล์

พฤติกรรมของ `expandSlashCommand(text, fileCommands)`:

- ทำงานเฉพาะเมื่อข้อความขึ้นต้นด้วย `/`
- แยกวิเคราะห์ชื่อคำสั่งจากโทเค็นแรกหลัง `/`
- แยกวิเคราะห์อาร์กิวเมนต์จากข้อความที่เหลือผ่าน `parseCommandArgs`
- ค้นหาการจับคู่ชื่อที่ตรงกันใน `fileCommands` ที่โหลดไว้
- หากตรงกัน ให้ใช้:
  - การแทนที่ตำแหน่ง: `$1`, `$2`, ...
  - การแทนที่รวม: `$ARGUMENTS` และ `$@`
  - จากนั้นการเรนเดอร์แม่แบบผ่าน `prompt.render` พร้อม `{ args, ARGUMENTS, arguments }`
- หากไม่ตรงกัน ส่งคืนข้อความเดิมโดยไม่เปลี่ยนแปลง

### ข้อควรระวังของ `parseCommandArgs`

ตัวแยกวิเคราะห์เป็นการแยกตามช่องว่างแบบรับรู้เครื่องหมายคำพูดอย่างง่าย:

- รองรับการใส่เครื่องหมายคำพูด `'single'` และ `"double"` เพื่อรักษาช่องว่าง
- ลบตัวคั่นเครื่องหมายคำพูด
- ไม่ดำเนินการกฎการ escape ด้วยแบ็กสแลช
- เครื่องหมายคำพูดที่ไม่จับคู่ไม่ถือเป็นข้อผิดพลาด ตัวแยกวิเคราะห์จะอ่านจนถึงจุดสิ้นสุด

## 7) พฤติกรรมของ `/...` ที่ไม่รู้จัก

อินพุต slash ที่ไม่รู้จักจะ **ไม่ถูกปฏิเสธ** โดยลอจิก slash หลัก

หากคำสั่งไม่ได้รับการจัดการโดยชั้น extension/custom/file `expandSlashCommand` จะส่งคืนข้อความเดิม และแจ้งข้อความ `/...` ตามตัวอักษรจะดำเนินการผ่านการขยายแม่แบบแจ้งข้อความปกติและการส่งมอบ LLM

โหมดโต้ตอบจะจัดการคำสั่งในตัวจำนวนมากใน `InputController` อย่างเข้มงวด (เช่น `/settings`, `/model`, `/mcp`, `/move`, `/exit`) คำสั่งเหล่านั้นถูกใช้ก่อน `session.prompt(...)` และด้วยเหตุนี้จึงไม่เคยไปถึงการขยายคำสั่งไฟล์ในเส้นทางนั้น

## 8) ความแตกต่างในขณะสตรีมเทียบกับโหมดว่าง

## เส้นทางว่าง

- `session.prompt("/x ...")` เรียกใช้ไปป์ไลน์คำสั่งและดำเนินการคำสั่งทันทีหรือส่งข้อความที่ขยายแล้วโดยตรง

## เส้นทางสตรีม (`session.isStreaming === true`)

- `prompt(...)` ยังคงเรียกใช้การแปลง extension/custom/file/template ก่อน
- จากนั้นต้องการ `streamingBehavior`:
  - `"steer"` -> จัดคิวข้อความขัดจังหวะ (`agent.steer`)
  - `"followUp"` -> จัดคิวข้อความหลัง turn (`agent.followUp`)
- หากละเว้น `streamingBehavior` แจ้งข้อความจะโยนข้อผิดพลาด

### พฤติกรรมการสตรีมเฉพาะคำสั่งที่สำคัญ

- คำสั่งส่วนขยายจะดำเนินการทันทีแม้ในระหว่างการสตรีม (ไม่ถูกจัดคิวเป็นข้อความ)
- เมธอดตัวช่วย `steer(...)`/`followUp(...)` ปฏิเสธคำสั่งส่วนขยาย (`#throwIfExtensionCommand`) เพื่อหลีกเลี่ยงการจัดคิวข้อความคำสั่งสำหรับตัวจัดการที่ต้องทำงานแบบซิงโครนัส
- การเล่นซ้ำคิวการ compaction ใช้ `isKnownSlashCommand(...)` เพื่อตัดสินใจว่ารายการที่จัดคิวควรเล่นซ้ำผ่าน `session.prompt(...)` (สำหรับ slash command ที่รู้จัก) หรือเมธอด steer/follow-up ดิบ

## 9) การจัดการข้อผิดพลาดและพื้นผิวความล้มเหลว

- ความล้มเหลวในการโหลดผู้ให้บริการถูกแยกออก รีจิสทรีรวบรวมคำเตือนและดำเนินต่อกับผู้ให้บริการอื่น
- รายการ slash command ที่ไม่ถูกต้อง (ไม่มีชื่อ/เส้นทาง/เนื้อหาหรือระดับที่ไม่ถูกต้อง) จะถูกลบออกโดยการตรวจสอบความสามารถ
- ความล้มเหลวในการแยกวิเคราะห์ frontmatter:
  - คำสั่ง native: ข้อผิดพลาดการแยกวิเคราะห์ที่ร้ายแรงจะปรากฏขึ้น
  - คำสั่งที่ไม่ใช่ native: คำเตือน + การแยกวิเคราะห์คีย์/ค่าสำรอง
- ข้อยกเว้นของตัวจัดการคำสั่ง extension/custom จะถูกจับและรายงานผ่านช่องทางข้อผิดพลาดส่วนขยาย (หรือ logger สำรองสำหรับคำสั่งที่กำหนดเองที่ไม่มี extension runner) และถือว่าจัดการแล้ว (ไม่มีการดำเนินการสำรองที่ไม่ตั้งใจ)
