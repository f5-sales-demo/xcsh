## 1) โมเดลการค้นพบ

Slash command เป็นความสามารถ (`id: "slash-commands"`) ที่ใช้ชื่อคำสั่งเป็นกุญแจ (`key: cmd => cmd.name`)

รีจิสทรีความสามารถโหลดผู้ให้บริการที่ลงทะเบียนทั้งหมด เรียงลำดับตามลำดับความสำคัญของผู้ให้บริการจากมากไปน้อย และลบข้อมูลซ้ำตามกุญแจด้วยหลักการ **ชนะก่อนเป็นอันดับแรก**

### ลำดับความสำคัญของผู้ให้บริการ

ผู้ให้บริการ slash-command ปัจจุบันและลำดับความสำคัญ:

1. `native` (OMP) — ลำดับความสำคัญ `100`
2. `claude` — ลำดับความสำคัญ `80`
3. `claude-plugins` — ลำดับความสำคัญ `70`
4. `codex` — ลำดับความสำคัญ `70`

พฤติกรรมกรณีเท่ากัน: ผู้ให้บริการที่มีลำดับความสำคัญเท่ากันจะรักษาลำดับการลงทะเบียน ลำดับการนำเข้าปัจจุบันลงทะเบียน `claude-plugins` ก่อน `codex` ดังนั้นคำสั่งของ plugin จึงชนะคำสั่ง codex เมื่อมีการชนกันของชื่อ

### พฤติกรรมการชนกันของชื่อ

สำหรับ `slash-commands` การชนกันจะได้รับการแก้ไขโดยการลบข้อมูลซ้ำของความสามารถอย่างเคร่งครัด:

- รายการที่มีลำดับความสำคัญสูงสุดจะถูกเก็บไว้ใน `result.items`
- รายการที่ซ้ำกันที่มีลำดับความสำคัญต่ำกว่าจะอยู่เฉพาะใน `result.all` และถูกทำเครื่องหมาย `_shadowed = true`

สิ่งนี้ใช้ทั้งกับผู้ให้บริการต่างๆ และภายในผู้ให้บริการเดียวกันหากส่งคืนชื่อที่ซ้ำกัน

### พฤติกรรมการสแกนไฟล์

ผู้ให้บริการส่วนใหญ่ใช้ `loadFilesFromDir(...)` ซึ่งปัจจุบัน:

- ค่าเริ่มต้นคือการจับคู่แบบไม่เรียกซ้ำ (`*.md`)
- ใช้ glob แบบ native ที่มี `gitignore: true`, `hidden: false`
- อ่านแต่ละไฟล์ที่จับคู่และแปลงเป็น `SlashCommand`

ดังนั้นไฟล์/ไดเรกทอรีที่ซ่อนอยู่จะไม่ถูกโหลด และเส้นทางที่ถูกละเว้นจะถูกข้ามไป

## 2) เส้นทางแหล่งที่มาเฉพาะผู้ให้บริการและลำดับความสำคัญในพื้นที่

## ผู้ให้บริการ `native` (`builtin.ts`)

รากของการค้นหามาจากไดเรกทอรี `.xcsh`:

- โปรเจกต์: `<cwd>/.xcsh/commands/*.md`
- ผู้ใช้: `~/.xcsh/agent/commands/*.md`

`getConfigDirs()` ส่งคืนโปรเจกต์ก่อนแล้วจึงส่งคืนผู้ใช้ ดังนั้น **คำสั่ง native ของโปรเจกต์ชนะคำสั่ง native ของผู้ใช้** เมื่อชื่อชนกัน

## ผู้ให้บริการ `claude` (`claude.ts`)

โหลด:

- ผู้ใช้: `~/.claude/commands/*.md`
- โปรเจกต์: `<cwd>/.claude/commands/*.md`

ผู้ให้บริการจะส่งรายการของผู้ใช้ก่อนรายการของโปรเจกต์ ดังนั้น **คำสั่ง Claude ของผู้ใช้ชนะคำสั่ง Claude ของโปรเจกต์** เมื่อชื่อชนกันภายในผู้ให้บริการนี้

## ผู้ให้บริการ `codex` (`codex.ts`)

โหลด:

- ผู้ใช้: `~/.codex/commands/*.md`
- โปรเจกต์: `<cwd>/.codex/commands/*.md`

ทั้งสองฝั่งถูกโหลดแล้วแบนราบในลำดับผู้ใช้ก่อน ดังนั้น **คำสั่ง Codex ของผู้ใช้ชนะคำสั่ง Codex ของโปรเจกต์** เมื่อมีการชนกัน

เนื้อหาของคำสั่ง Codex ถูกแยกวิเคราะห์ด้วยการตัด frontmatter (`parseFrontmatter`) และชื่อคำสั่งสามารถถูกแทนที่ด้วย frontmatter `name`; มิฉะนั้นจะใช้ชื่อไฟล์

## ผู้ให้บริการ `claude-plugins` (`claude-plugins.ts`)

โหลดรากของคำสั่ง plugin จาก `~/.claude/plugins/installed_plugins.json` แล้วสแกน `<pluginRoot>/commands/*.md`

การเรียงลำดับเป็นไปตามลำดับการวนซ้ำของรีจิสทรีและลำดับรายการต่อ plugin จากข้อมูล JSON นั้น ไม่มีขั้นตอนการเรียงลำดับเพิ่มเติม

## 3) การสร้างเป็น `FileSlashCommand` ในรันไทม์

`loadSlashCommands()` ใน `src/extensibility/slash-commands.ts` แปลงรายการความสามารถเป็นออบเจกต์ `FileSlashCommand` ที่ใช้ในช่วงเวลาพร้อมต์

สำหรับแต่ละคำสั่ง:

1. แยกวิเคราะห์ frontmatter/เนื้อหา (`parseFrontmatter`)
2. แหล่งคำอธิบาย:
   - `frontmatter.description` หากมี
   - มิฉะนั้นบรรทัดแรกของเนื้อหาที่ไม่ว่างเปล่า (ตัดแต่ง ความยาวสูงสุด 60 ตัวอักษรพร้อม `...`)
3. เก็บเนื้อหาของเทมเพลตที่สามารถดำเนินการได้ที่แยกวิเคราะห์แล้ว
4. คำนวณสตริงแหล่งที่มาสำหรับแสดงผล เช่น `via Claude Code Project`

ความรุนแรงของการแยกวิเคราะห์ Frontmatter ขึ้นอยู่กับแหล่งที่มา:

- ระดับ `native` -> ข้อผิดพลาดการแยกวิเคราะห์จะเป็น `fatal`
- ระดับ `user`/`project` -> ข้อผิดพลาดการแยกวิเคราะห์จะเป็น `warn` พร้อมการแยกวิเคราะห์สำรอง

### คำสั่งสำรองที่รวมมาด้วย

หลังจากคำสั่งจากระบบไฟล์/ผู้ให้บริการ เทมเพลตคำสั่งที่ฝังอยู่จะถูกต่อท้าย (`EMBEDDED_COMMAND_TEMPLATES`) หากชื่อของพวกเขาไม่มีอยู่แล้ว

ชุดที่ฝังอยู่ปัจจุบันมาจาก `src/task/commands.ts` และใช้เป็นทางเลือกสำรอง (`source: "bundled"`)

## 4) โหมดโต้ตอบ: รายการคำสั่งมาจากไหน

โหมดโต้ตอบรวมแหล่งคำสั่งหลายแหล่งสำหรับการเติมข้อความอัตโนมัติและการกำหนดเส้นทางคำสั่ง

ในช่วงเวลาการสร้างจะสร้างรายการคำสั่งที่รอดำเนินการจาก:

- คำสั่งในตัว (`BUILTIN_SLASH_COMMANDS` รวมถึงการเติมข้อความอาร์กิวเมนต์และคำใบ้แบบอินไลน์สำหรับคำสั่งที่เลือก)
- slash command ที่ลงทะเบียนโดยส่วนขยาย (`extensionRunner.getRegisteredCommands(...)`)
- คำสั่งกำหนดเองแบบ TypeScript (`session.customCommands`) ที่แมปไปยังป้ายชื่อ slash command
- คำสั่ง skill เพิ่มเติม (`/skill:<name>`) เมื่อ `skills.enableSkillCommands` เปิดใช้งาน

จากนั้น `init()` จะเรียก `refreshSlashCommandState(...)` เพื่อโหลดคำสั่งตามไฟล์และติดตั้ง `CombinedAutocompleteProvider` หนึ่งตัวที่ประกอบด้วย:

- คำสั่งที่รอดำเนินการข้างต้น
- คำสั่งตามไฟล์ที่ค้นพบ

`refreshSlashCommandState(...)` ยังอัปเดต `session.setSlashCommands(...)` ด้วยเพื่อให้การขยายพร้อมต์ใช้ชุดคำสั่งไฟล์ที่ค้นพบเดียวกัน

### วงจรชีวิตการรีเฟรช

สถานะ slash command จะถูกรีเฟรช:

- ระหว่างการเริ่มต้นโหมดโต้ตอบ
- หลังจากที่ `/move` เปลี่ยนไดเรกทอรีทำงาน (`handleMoveCommand` เรียก `resetCapabilities()` แล้วจึงเรียก `refreshSlashCommandState(newCwd)`)

ไม่มีตัวตรวจสอบไฟล์แบบต่อเนื่องสำหรับไดเรกทอรีคำสั่ง

### การแสดงผลอื่น

แดชบอร์ดส่วนขยายยังโหลดความสามารถ `slash-commands` และแสดงรายการคำสั่งที่ใช้งานอยู่/ที่ถูกบดบัง รวมถึงรายการที่ซ้ำกัน `_shadowed`

## 5) การวางในไปป์ไลน์พร้อมต์

ลำดับการจัดการ slash ของ `AgentSession.prompt(...)` (เมื่อ `expandPromptTemplates !== false`):

1. **คำสั่งส่วนขยาย** (`#tryExecuteExtensionCommand`)  
   หาก `/name` ตรงกับคำสั่งที่ลงทะเบียนส่วนขยาย ตัวจัดการจะดำเนินการทันทีและพร้อมต์จะส่งคืน
2. **คำสั่งกำหนดเองแบบ TypeScript** (`#tryExecuteCustomCommand`)  
   ขอบเขตเท่านั้น: หากตรงกัน จะดำเนินการและอาจส่งคืน:
   - `string` -> แทนที่ข้อความพร้อมต์ด้วยสตริงนั้น
   - `void/undefined` -> ถือว่าได้รับการจัดการ; ไม่มีพร้อมต์ LLM
3. **Slash command ตามไฟล์** (`expandSlashCommand`)  
   หากข้อความยังคงขึ้นต้นด้วย `/` ให้ลองขยายคำสั่ง markdown
4. **เทมเพลตพร้อมต์** (`expandPromptTemplate`)  
   ใช้หลังจากการประมวลผล slash/custom
5. **การส่งมอบ**
   - ไม่ได้ใช้งาน: พร้อมต์จะถูกส่งไปยัง agent ทันที
   - กำลังสตรีม: พร้อมต์จะถูกเพิ่มคิวเป็น steer/follow-up ขึ้นอยู่กับ `streamingBehavior`

นี่คือเหตุผลที่การขยาย slash command อยู่ก่อนการขยายเทมเพลตพร้อมต์ และเหตุผลที่คำสั่งกำหนดเองสามารถแปลงเครื่องหมาย slash นำหน้าออกก่อนที่จะจับคู่กับคำสั่งไฟล์

## 6) ความหมายของการขยายสำหรับ slash command ตามไฟล์

พฤติกรรม `expandSlashCommand(text, fileCommands)`:

- ทำงานเฉพาะเมื่อข้อความขึ้นต้นด้วย `/`
- แยกวิเคราะห์ชื่อคำสั่งจากโทเค็นแรกหลัง `/`
- แยกวิเคราะห์ args จากข้อความที่เหลือผ่าน `parseCommandArgs`
- ค้นหาการจับคู่ชื่อที่แน่นอนในโหลด `fileCommands`
- หากตรงกัน จะใช้:
  - การแทนที่แบบตำแหน่ง: `$1`, `$2`, ...
  - การแทนที่แบบรวม: `$ARGUMENTS` และ `$@`
  - จากนั้นการเรนเดอร์เทมเพลตผ่าน `prompt.render` ที่มี `{ args, ARGUMENTS, arguments }`
- หากไม่ตรงกัน จะส่งคืนข้อความต้นฉบับโดยไม่เปลี่ยนแปลง

### ข้อควรระวังของ `parseCommandArgs`

ตัวแยกวิเคราะห์เป็นการแยกที่รับรู้เครื่องหมายคำพูดอย่างง่าย:

- รองรับการใส่เครื่องหมายคำพูด `'single'` และ `"double"` เพื่อรักษาช่องว่าง
- ตัดตัวคั่นเครื่องหมายคำพูดออก
- ไม่ใช้กฎการ escape ด้วย backslash
- เครื่องหมายคำพูดที่ไม่มีคู่ไม่ถือเป็นข้อผิดพลาด; ตัวแยกวิเคราะห์จะบริโภคจนถึงจุดสิ้นสุด

## 7) พฤติกรรมของ `/...` ที่ไม่รู้จัก

อินพุต slash ที่ไม่รู้จัก **ไม่ถูกปฏิเสธ** โดยตรรกะ slash หลัก

หากคำสั่งไม่ได้รับการจัดการโดยเลเยอร์ส่วนขยาย/กำหนดเอง/ไฟล์ `expandSlashCommand` จะส่งคืนข้อความต้นฉบับ และพร้อมต์ `/...` ตามตัวอักษรจะดำเนินการผ่านการขยายเทมเพลตพร้อมต์ปกติและการส่งมอบ LLM

โหมดโต้ตอบจะจัดการคำสั่งในตัวหลายรายการแยกต่างหากใน `InputController` (เช่น `/settings`, `/model`, `/mcp`, `/move`, `/exit`) คำสั่งเหล่านั้นจะถูกบริโภคก่อน `session.prompt(...)` และดังนั้นจึงไม่ถึงการขยายคำสั่งไฟล์ในเส้นทางนั้น

## 8) ความแตกต่างในช่วงเวลาสตรีมเทียบกับไม่ได้ใช้งาน

## เส้นทางไม่ได้ใช้งาน

- `session.prompt("/x ...")` เรียกใช้ไปป์ไลน์คำสั่งและดำเนินการคำสั่งทันทีหรือส่งข้อความที่ขยายแล้วโดยตรง

## เส้นทางสตรีม (`session.isStreaming === true`)

- `prompt(...)` ยังคงเรียกใช้การแปลงส่วนขยาย/กำหนดเอง/ไฟล์/เทมเพลตก่อน
- จากนั้นต้องการ `streamingBehavior`:
  - `"steer"` -> เพิ่มคิวข้อความขัดจังหวะ (`agent.steer`)
  - `"followUp"` -> เพิ่มคิวข้อความหลังการหมุน (`agent.followUp`)
- หาก `streamingBehavior` ถูกละไว้ พร้อมต์จะแสดงข้อผิดพลาด

### พฤติกรรมสตรีมเฉพาะคำสั่งที่สำคัญ

- คำสั่งส่วนขยายจะดำเนินการทันทีแม้ในช่วงการสตรีม (ไม่ถูกเพิ่มคิวเป็นข้อความ)
- เมธอดตัวช่วย `steer(...)`/`followUp(...)` ปฏิเสธคำสั่งส่วนขยาย (`#throwIfExtensionCommand`) เพื่อหลีกเลี่ยงการเพิ่มคิวข้อความคำสั่งสำหรับตัวจัดการที่ต้องทำงานแบบซิงโครนัส
- การเล่นซ้ำคิว compaction ใช้ `isKnownSlashCommand(...)` เพื่อตัดสินใจว่ารายการที่เพิ่มคิวควรเล่นซ้ำผ่าน `session.prompt(...)` (สำหรับ slash command ที่รู้จัก) เทียบกับเมธอด steer/follow-up แบบดิบ

## 9) การจัดการข้อผิดพลาดและพื้นผิวความล้มเหลว

- ความล้มเหลวในการโหลดผู้ให้บริการถูกแยกออก; รีจิสทรีรวบรวมคำเตือนและดำเนินการต่อกับผู้ให้บริการอื่น
- รายการ slash command ที่ไม่ถูกต้อง (ชื่อ/เส้นทาง/เนื้อหาหายไปหรือระดับไม่ถูกต้อง) จะถูกตัดทิ้งโดยการตรวจสอบความสามารถ
- ความล้มเหลวในการแยกวิเคราะห์ Frontmatter:
  - คำสั่ง native: ข้อผิดพลาดการแยกวิเคราะห์ที่ร้ายแรงจะส่งต่อขึ้นไป
  - คำสั่งที่ไม่ใช่ native: คำเตือน + การแยกวิเคราะห์คีย์/ค่าสำรอง
- ข้อยกเว้นของตัวจัดการคำสั่งส่วนขยาย/กำหนดเองถูกจับและรายงานผ่านช่องทางข้อผิดพลาดของส่วนขยาย (หรือ logger สำรองสำหรับคำสั่งกำหนดเองที่ไม่มี extension runner) และถือว่าได้รับการจัดการ (ไม่มีการดำเนินการ fallback โดยไม่ตั้งใจ)
