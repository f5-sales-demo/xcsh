# สัญญาสถาปัตยกรรมแคชการสแกนระบบไฟล์

เอกสารนี้กำหนดสัญญาปัจจุบันสำหรับแคชการสแกนระบบไฟล์แบบใช้ร่วมกันที่พัฒนาด้วย Rust (`crates/pi-natives/src/fs_cache.rs`) และถูกใช้งานโดย API การค้นพบ/ค้นหาแบบ native ที่เปิดเผยให้กับ `packages/coding-agent`

## แคชนี้คืออะไร

แคชจัดเก็บรายการผลลัพธ์การสแกนไดเรกทอรีแบบเต็ม (`GlobMatch[]`) โดยใช้คีย์ตามขอบเขตการสแกนและนโยบายการสำรวจ จากนั้นให้การดำเนินการระดับสูงกว่า (การกรอง glob, การให้คะแนน fuzzy, การเลือกไฟล์ grep) ทำงานกับรายการที่แคชไว้เหล่านั้น

เป้าหมายหลัก:

- หลีกเลี่ยงการเดินสำรวจระบบไฟล์ซ้ำสำหรับการเรียกค้นพบ/ค้นหาที่ซ้ำกัน
- รักษาความสอดคล้องระหว่าง `glob`, `fuzzyFind` และ `grep` เมื่อใช้นโยบายการสแกนร่วมกัน
- อนุญาตการกู้คืนจากความเก่าอย่างชัดเจนสำหรับผลลัพธ์ว่าง และการทำให้ไม่ถูกต้องอย่างชัดเจนหลังการเปลี่ยนแปลงไฟล์

## ความเป็นเจ้าของและพื้นผิวสาธารณะ

- การพัฒนาแคชและนโยบาย: `crates/pi-natives/src/fs_cache.rs`
- ผู้ใช้งาน native:
  - `crates/pi-natives/src/glob.rs`
  - `crates/pi-natives/src/fd.rs` (`fuzzyFind`)
  - `crates/pi-natives/src/grep.rs`
- การเชื่อมต่อ/ส่งออก JS:
  - `packages/natives/src/glob/index.ts` (`invalidateFsScanCache`)
  - `packages/natives/src/glob/types.ts`
  - `packages/natives/src/grep/types.ts`
- ตัวช่วยทำให้ไม่ถูกต้องเมื่อมีการเปลี่ยนแปลงของ coding-agent:
  - `packages/coding-agent/src/tools/fs-cache-invalidation.ts`

## การแบ่งพาร์ทิชันคีย์แคช (สัญญาที่แน่นอน)

แต่ละรายการใช้คีย์ตาม:

- เส้นทางไดเรกทอรี `root` ที่ถูก canonicalize แล้ว
- ค่าบูลีน `include_hidden`
- ค่าบูลีน `use_gitignore`

ผลกระทบ:

- การสแกนแบบซ่อนและไม่ซ่อน **ไม่** ใช้รายการร่วมกัน
- การสแกนที่เคารพ gitignore และปิดใช้งาน ignore **ไม่** ใช้รายการร่วมกัน
- ผู้ใช้งานต้องส่งความหมายที่คงที่สำหรับพฤติกรรม hidden/gitignore; การเปลี่ยนแฟล็กใดแฟล็กหนึ่งจะสร้างพาร์ทิชันแคชที่แตกต่างกัน

การรวม `node_modules` **ไม่** อยู่ในคีย์แคช แคชจัดเก็บรายการโดยรวม `node_modules` ไว้ด้วย; การกรองตามผู้ใช้งานแต่ละรายจะถูกใช้หลังการดึงข้อมูล

## พฤติกรรมการรวบรวมการสแกน

การเติมข้อมูลแคชใช้ walker แบบ deterministic (`ignore::WalkBuilder`) ที่กำหนดค่าด้วย `include_hidden` และ `use_gitignore`:

- `follow_links(false)`
- เรียงตามเส้นทางไฟล์
- `.git` จะถูกข้ามเสมอ
- `node_modules` จะถูกรวบรวมเสมอในเวลาสแกนแคช (และสามารถกรองทีหลังได้)
- ชนิดไฟล์ของรายการ + `mtime` จะถูกจับผ่าน `symlink_metadata`

ราก (root) ของการค้นหาถูกแก้ไขโดย `resolve_search_path`:

- เส้นทางสัมพัทธ์จะถูกแก้ไขเทียบกับ cwd ปัจจุบัน
- เป้าหมายต้องเป็นไดเรกทอรีที่มีอยู่จริง
- root จะถูก canonicalize เมื่อเป็นไปได้

## นโยบายความสดใหม่และการขับออก

นโยบายทั่วไป (สามารถแทนที่ได้ผ่านตัวแปรสภาพแวดล้อม):

- `FS_SCAN_CACHE_TTL_MS` (ค่าเริ่มต้น `1000`)
- `FS_SCAN_EMPTY_RECHECK_MS` (ค่าเริ่มต้น `200`)
- `FS_SCAN_CACHE_MAX_ENTRIES` (ค่าเริ่มต้น `16`)

พฤติกรรม:

- `get_or_scan(...)`
  - หาก TTL เป็น `0`: ข้ามแคชทั้งหมด สแกนใหม่เสมอ (`cache_age_ms = 0`)
  - เมื่อ cache hit ภายใน TTL: คืนรายการที่แคชไว้ + `cache_age_ms` ที่ไม่เป็นศูนย์
  - เมื่อ hit ที่หมดอายุ: ขับออกคีย์ สแกนใหม่ จัดเก็บรายการใหม่
- การบังคับจำนวนรายการสูงสุดจะขับออกรายการเก่าสุดก่อนตาม `created_at`

## การตรวจสอบซ้ำอย่างรวดเร็วสำหรับผลลัพธ์ว่าง (แยกจาก hit ปกติ)

Cache hit ปกติ:

- cache hit ภายใน TTL จะคืนรายการที่แคชไว้และไม่ทำอะไรเพิ่มเติม

การตรวจสอบซ้ำอย่างรวดเร็วสำหรับผลลัพธ์ว่าง:

- นี่เป็นนโยบาย **ฝั่งผู้เรียก** ที่ใช้ `ScanResult.cache_age_ms`
- หากผลลัพธ์ที่กรองแล้ว/query ว่างเปล่า และอายุแคชสแกนมากกว่าหรือเท่ากับ `empty_recheck_ms()` ผู้เรียกจะทำ `force_rescan(...)` หนึ่งครั้งและลองใหม่
- มีวัตถุประสงค์เพื่อลดผลลัพธ์ลบที่เก่าเมื่อไฟล์ถูกเพิ่มเข้ามาเร็วๆ นี้แต่แคชยังอยู่ภายใน TTL

ผู้ใช้งานปัจจุบัน:

- `glob`: ตรวจสอบซ้ำเมื่อรายการที่ตรงกันหลังกรองว่างเปล่าและอายุสแกนเกินเกณฑ์
- `fuzzyFind` (`fd.rs`): ตรวจสอบซ้ำเฉพาะเมื่อ query ไม่ว่างเปล่าและรายการที่ตรงกันหลังให้คะแนนว่างเปล่า
- `grep`: ตรวจสอบซ้ำเมื่อรายการไฟล์ผู้สมัครที่เลือกว่างเปล่า

## ค่าเริ่มต้นและการใช้งานแคชของผู้ใช้งาน

แคชเป็นแบบเลือกใช้ (opt-in) ใน API ที่เปิดเผยทั้งหมด (`cache?: boolean` ค่าเริ่มต้น `false`)

ค่าเริ่มต้นปัจจุบันใน API แบบ native:

- `glob`: `hidden=false`, `gitignore=true`, `cache=false`
- `fuzzyFind`: `hidden=false`, `gitignore=true`, `cache=false`
- `grep`: `hidden=true`, `cache=false` และแคชสแกนใช้ `use_gitignore=true` เสมอ

ผู้เรียก coding-agent ในปัจจุบัน:

- การค้นพบผู้สมัครการกล่าวถึงที่มีปริมาณสูงเปิดใช้งานแคช:
  - `packages/coding-agent/src/utils/file-mentions.ts`
  - โปรไฟล์: `hidden=true`, `gitignore=true`, `includeNodeModules=true`, `cache=true`
- การรวม `grep` ระดับเครื่องมือปิดใช้งานแคชสแกนในปัจจุบัน (`cache: false`):
  - `packages/coding-agent/src/tools/grep.ts`

## สัญญาการทำให้ไม่ถูกต้อง

จุดเข้าการทำให้ไม่ถูกต้องแบบ native:

- `invalidateFsScanCache(path?: string)`
  - พร้อม `path`: ลบรายการแคชที่ root เป็น prefix ของเส้นทางเป้าหมาย
  - ไม่มี path: ล้างรายการแคชสแกนทั้งหมด

รายละเอียดการจัดการเส้นทาง:

- เส้นทางการทำให้ไม่ถูกต้องแบบสัมพัทธ์จะถูกแก้ไขเทียบกับ cwd
- การทำให้ไม่ถูกต้องพยายาม canonicalization
- หากเป้าหมายไม่มีอยู่ (เช่น ลบ) จะ fallback โดย canonicalize parent แล้วแนบชื่อไฟล์กลับเมื่อเป็นไปได้
- สิ่งนี้รักษาพฤติกรรมการทำให้ไม่ถูกต้องสำหรับการสร้าง/ลบ/เปลี่ยนชื่อที่ฝั่งใดฝั่งหนึ่งอาจไม่มีอยู่

## ความรับผิดชอบของขั้นตอนการเปลี่ยนแปลงของ coding-agent

โค้ด coding-agent ต้องทำให้ไม่ถูกต้องหลังจากการเปลี่ยนแปลงระบบไฟล์ที่สำเร็จ

ตัวช่วยกลาง:

- `invalidateFsScanAfterWrite(path)`
- `invalidateFsScanAfterDelete(path)`
- `invalidateFsScanAfterRename(oldPath, newPath)` (ทำให้ไม่ถูกต้องทั้งสองฝั่งเมื่อเส้นทางแตกต่างกัน)

ตำแหน่งเรียกใช้เครื่องมือการเปลี่ยนแปลงปัจจุบัน:

- `packages/coding-agent/src/tools/write.ts`
- `packages/coding-agent/src/patch/index.ts` (ขั้นตอน hashline/patch/replace)

กฎ: หากขั้นตอนใดเปลี่ยนแปลงเนื้อหาหรือตำแหน่งระบบไฟล์และข้ามตัวช่วยเหล่านี้ คาดว่าจะเกิดบั๊กความเก่าของแคช

## การเพิ่มผู้ใช้งานแคชใหม่อย่างปลอดภัย

เมื่อนำการใช้งานแคชมาใช้ในเส้นทางสแกนเนอร์/ค้นหาใหม่:

1. **ใช้อินพุตนโยบายการสแกนที่คงที่**
   - ตัดสินใจความหมาย hidden/gitignore ก่อน
   - ส่งอย่างสม่ำเสมอไปยัง `get_or_scan`/`force_rescan` เพื่อให้พาร์ทิชันแคชเป็นไปตามเจตนา

2. **ถือว่าข้อมูลแคชถูกกรองล่วงหน้าตามนโยบายการสำรวจเท่านั้น**
   - ใช้การกรองเฉพาะเครื่องมือ (รูปแบบ glob, ตัวกรองชนิด, กฎ node_modules) หลังการดึงข้อมูล
   - อย่าสันนิษฐานว่ารายการที่แคชไว้สะท้อนตัวกรองระดับสูงของคุณแล้ว

3. **ใช้การตรวจสอบซ้ำอย่างรวดเร็วสำหรับผลลัพธ์ว่างเฉพาะเมื่อมีความเสี่ยงผลลบที่เก่า**
   - ใช้ `scan.cache_age_ms >= empty_recheck_ms()`
   - ลองใหม่หนึ่งครั้งด้วย `force_rescan(..., store=true, ...)`
   - แยกเส้นทางนี้ออกจากตรรกะ cache-hit ปกติ

4. **เคารพโหมดไม่ใช้แคชอย่างชัดเจน**
   - เมื่อผู้เรียกปิดใช้งานแคช ให้เรียก `force_rescan(..., store=false, ...)`
   - อย่าเติมข้อมูลแคชที่ใช้ร่วมกันในเส้นทางคำขอแบบไม่ใช้แคช

5. **เชื่อมต่อการทำให้ไม่ถูกต้องจากการเปลี่ยนแปลงสำหรับเส้นทางการเขียนใหม่ทุกเส้นทาง**
   - หลังจากเขียน/แก้ไข/ลบ/เปลี่ยนชื่อสำเร็จ ให้เรียกตัวช่วยทำให้ไม่ถูกต้องของ coding-agent
   - สำหรับการเปลี่ยนชื่อ/ย้าย ให้ทำให้ไม่ถูกต้องทั้งเส้นทางเก่าและใหม่

6. **อย่าเพิ่มปุ่มปรับ TTL ต่อการเรียก**
   - สัญญาปัจจุบันเป็นนโยบายทั่วไปเท่านั้น (กำหนดค่าผ่าน env) ไม่มีการแทนที่ TTL ต่อคำขอ

## ขอบเขตที่ทราบ

- ขอบเขตแคชเป็นแบบในหน่วยความจำภายในกระบวนการ (`DashMap`) ไม่คงอยู่ข้ามการรีสตาร์ทกระบวนการ
- แคชจัดเก็บรายการสแกน ไม่ใช่ผลลัพธ์สุดท้ายของเครื่องมือ
- `glob`/`fuzzyFind`/`grep` ใช้รายการสแกนร่วมกันเฉพาะเมื่อมิติคีย์ (`root`, `hidden`, `gitignore`) ตรงกัน
- `.git` จะถูกยกเว้นเสมอในเวลารวบรวมการสแกนโดยไม่คำนึงถึงตัวเลือกของผู้เรียก
