# การปกปิดข้อมูลลับ

ป้องกันไม่ให้ค่าที่ละเอียดอ่อน (คีย์ API, โทเค็น, รหัสผ่าน) ถูกส่งไปยังผู้ให้บริการ LLM เมื่อเปิดใช้งาน ข้อมูลลับจะถูกแทนที่ด้วยตัวยึดตำแหน่งแบบกำหนดได้ก่อนออกจากกระบวนการ และจะถูกกู้คืนในอาร์กิวเมนต์การเรียกใช้เครื่องมือที่โมเดลส่งกลับมา

## การเปิดใช้งาน

เปิดใช้งานโดยค่าเริ่มต้น สลับการใช้งานผ่าน UI `/settings` หรือโดยตรงใน `config.yml`:

```yaml
secrets:
  enabled: false
```

## วิธีการทำงาน

1. เมื่อเริ่มต้นเซสชัน ข้อมูลลับจะถูกรวบรวมจากสองแหล่ง:
   - **ตัวแปรสภาพแวดล้อม** ที่ตรงกับรูปแบบข้อมูลลับทั่วไป (`*_KEY`, `*_SECRET`, `*_TOKEN`, `*_PASSWORD` เป็นต้น) โดยมีค่าที่มีความยาว >= 8 ตัวอักษร
   - **ไฟล์ `secrets.yml`** (ดูด้านล่าง)

2. ข้อความขาออกที่ส่งไปยัง LLM จะมีค่าข้อมูลลับทั้งหมดถูกแทนที่ด้วยตัวยึดตำแหน่ง เช่น `<<$env:S0>>`, `<<$env:S1>>` เป็นต้น

3. อาร์กิวเมนต์การเรียกใช้เครื่องมือที่โมเดลส่งกลับมาจะถูกตรวจสอบแบบลึก และตัวยึดตำแหน่งจะถูกกู้คืนเป็นค่าเดิมก่อนการดำเนินการ

โหมดสองแบบควบคุมสิ่งที่เกิดขึ้นกับข้อมูลลับแต่ละรายการ:

| โหมด | พฤติกรรม | สามารถย้อนกลับได้ |
|---|---|---|
| `obfuscate` (ค่าเริ่มต้น) | แทนที่ด้วยตัวยึดตำแหน่งแบบมีดัชนี `<<$env:SN>>` | ใช่ (ถอดรหัสในอาร์กิวเมนต์เครื่องมือ) |
| `replace` | แทนที่ด้วยสตริงแบบกำหนดได้ที่มีความยาวเท่ากัน | ไม่ (แบบทิศทางเดียว) |

## secrets.yml

กำหนดรายการข้อมูลลับแบบกำหนดเองใน YAML โดยมีการตรวจสอบสองตำแหน่ง:

| ระดับ | เส้นทาง | วัตถุประสงค์ |
|---|---|---|
| ทั่วโลก | `~/.xcsh/agent/secrets.yml` | ข้อมูลลับสำหรับทุกโปรเจกต์ |
| โปรเจกต์ | `<cwd>/.xcsh/secrets.yml` | ข้อมูลลับเฉพาะโปรเจกต์ |

รายการระดับโปรเจกต์จะแทนที่รายการทั่วโลกที่มี `content` ตรงกัน

### สคีมา

แต่ละรายการในอาร์เรย์มีฟิลด์ดังนี้:

| ฟิลด์ | ประเภท | จำเป็น | คำอธิบาย |
|---|---|---|---|
| `type` | `"plain"` หรือ `"regex"` | ใช่ | กลยุทธ์การจับคู่ |
| `content` | string | ใช่ | ค่าข้อมูลลับ (plain) หรือรูปแบบ regex (regex) |
| `mode` | `"obfuscate"` หรือ `"replace"` | ไม่ | ค่าเริ่มต้น: `"obfuscate"` |
| `replacement` | string | ไม่ | การแทนที่แบบกำหนดเอง (เฉพาะโหมด replace) |
| `flags` | string | ไม่ | แฟล็ก regex (เฉพาะประเภท regex) |

### ตัวอย่าง

#### ข้อมูลลับแบบ plain

```yaml
# Obfuscate a specific API key (default mode)
- type: plain
  content: sk-proj-abc123def456

# Replace a database password with a fixed string
- type: plain
  content: hunter2
  mode: replace
  replacement: "********"
```

#### ข้อมูลลับแบบ regex

```yaml
# Obfuscate any AWS-style key
- type: regex
  content: "AKIA[0-9A-Z]{16}"

# Case-insensitive match with explicit flags
- type: regex
  content: "api[_-]?key\\s*=\\s*\\w+"
  flags: "i"

# Regex literal syntax (pattern and flags in one string)
- type: regex
  content: "/bearer\\s+[a-zA-Z0-9._~+\\/=-]+/i"
```

รายการ regex จะสแกนแบบทั่วโลกเสมอ (แฟล็ก `g` จะถูกบังคับใช้โดยอัตโนมัติ) ไวยากรณ์ literal regex `/pattern/flags` รองรับเป็นทางเลือกแทนการใช้ฟิลด์ `content` + `flags` แยกกัน เครื่องหมายสแลชที่ถูก escape ภายในรูปแบบ (`\\/`) จะถูกจัดการอย่างถูกต้อง

#### โหมด replace ร่วมกับ regex

```yaml
# One-way replace connection strings (not reversible)
- type: regex
  content: "postgres://[^\\s]+"
  mode: replace
  replacement: "postgres://***"
```

## การโต้ตอบกับการตรวจจับตัวแปรสภาพแวดล้อม

ตัวแปรสภาพแวดล้อมจะถูกรวบรวมก่อนเสมอ รายการที่กำหนดในไฟล์จะถูกเพิ่มต่อท้าย ดังนั้นรายการในไฟล์จึงสามารถครอบคลุมข้อมูลลับที่ไม่ได้อยู่ในตัวแปรสภาพแวดล้อม (ไฟล์คอนฟิก, ค่าที่ฮาร์ดโค้ด เป็นต้น) หากค่าเดียวกันปรากฏในทั้งสองแหล่ง โหมดของรายการในไฟล์จะมีความสำคัญกว่า

## ไฟล์หลัก

- `src/secrets/index.ts` -- การโหลด, การรวม, การรวบรวมตัวแปรสภาพแวดล้อม
- `src/secrets/obfuscator.ts` -- คลาส `SecretObfuscator`, การสร้างตัวยึดตำแหน่ง, การปกปิดข้อความ
- `src/secrets/regex.ts` -- การแยกวิเคราะห์และการคอมไพล์ regex literal
- `src/config/settings-schema.ts` -- คำจำกัดความของการตั้งค่า `secrets.enabled`
