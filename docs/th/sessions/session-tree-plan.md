# สถาปัตยกรรมแบบต้นไม้ของเซสชัน (ปัจจุบัน)

อ้างอิง: [session.md](./session.md)

เอกสารนี้อธิบายวิธีการทำงานของการนำทางแบบต้นไม้ของเซสชันในปัจจุบัน: โมเดลต้นไม้ในหน่วยความจำ กฎการเคลื่อนที่ของลีฟ พฤติกรรมการแตกกิ่ง และการรวมส่วนขยาย/เหตุการณ์

## ระบบย่อยนี้คืออะไร

เซสชันถูกจัดเก็บเป็นบันทึกรายการแบบเพิ่มเท่านั้น (append-only) แต่พฤติกรรมขณะรันไทม์เป็นแบบต้นไม้:

- ทุกรายการที่ไม่ใช่ส่วนหัวมี `id` และ `parentId`
- ตำแหน่งที่ใช้งานอยู่คือ `leafId` ใน `SessionManager`
- การเพิ่มรายการจะสร้างไชลด์ของลีฟปัจจุบันเสมอ
- การแตกกิ่ง**ไม่**เขียนประวัติซ้ำ เพียงแค่เปลี่ยนตำแหน่งที่ลีฟชี้ก่อนการเพิ่มครั้งถัดไป

ไฟล์หลัก:

- `src/session/session-manager.ts` — โมเดลข้อมูลแบบต้นไม้ การสำรวจ การเคลื่อนที่ของลีฟ การแยกกิ่ง/เซสชัน
- `src/session/agent-session.ts` — ขั้นตอนการนำทาง `/tree` การสรุป การปล่อยฮุก/เหตุการณ์
- `src/modes/components/tree-selector.ts` — พฤติกรรม UI แบบต้นไม้แบบโต้ตอบและการกรอง
- `src/modes/controllers/selector-controller.ts` — การจัดการตัวเลือกสำหรับ `/tree` และ `/branch`
- `src/modes/controllers/input-controller.ts` — การกำหนดเส้นทางคำสั่ง (`/tree`, `/branch`, พฤติกรรม double-escape)
- `src/session/messages.ts` — การแปลงรายการ `branch_summary`, `compaction` และ `custom_message` เป็นข้อความบริบทสำหรับ LLM

## โมเดลข้อมูลแบบต้นไม้ใน `SessionManager`

ดัชนีขณะรันไทม์:

- `#byId: Map<string, SessionEntry>` — การค้นหาแบบรวดเร็วสำหรับรายการใดๆ
- `#leafId: string | null` — ตำแหน่งปัจจุบันในต้นไม้
- `#labelsById: Map<string, string>` — ป้ายกำกับที่แก้ไขแล้วตาม id ของรายการเป้าหมาย

API ของต้นไม้:

- `getBranch(fromId?)` เดินตามลิงก์พาเรนต์ไปยังรูทและส่งคืนเส้นทางจากรูท→โหนด
- `getTree()` ส่งคืน `SessionTreeNode[]` (`entry`, `children`, `label`)
  - ลิงก์พาเรนต์ถูกแปลงเป็นอาร์เรย์ของไชลด์
  - รายการที่ไม่มีพาเรนต์จะถูกถือว่าเป็นรูท
  - ไชลด์ถูกเรียงลำดับจากเก่าที่สุด→ใหม่ที่สุดตามเวลา
- `getChildren(parentId)` ส่งคืนไชลด์โดยตรง
- `getLabel(id)` แก้ไขป้ายกำกับปัจจุบันจาก `labelsById`

`getTree()` เป็นการฉายภาพขณะรันไทม์ การจัดเก็บถาวรยังคงเป็นรายการ JSONL แบบเพิ่มเท่านั้น

## ความหมายของการเคลื่อนที่ลีฟ

มีพื้นฐานการเคลื่อนที่ลีฟสามแบบ:

1. `branch(entryId)`
   - ตรวจสอบว่ารายการมีอยู่
   - ตั้งค่า `leafId = entryId`
   - ไม่มีการเขียนรายการใหม่

2. `resetLeaf()`
   - ตั้งค่า `leafId = null`
   - การเพิ่มครั้งถัดไปจะสร้างรายการรูทใหม่ (`parentId = null`)

3. `branchWithSummary(branchFromId, summary, details?, fromExtension?)`
   - รับ `branchFromId: string | null`
   - ตั้งค่า `leafId = branchFromId`
   - เพิ่มรายการ `branch_summary` เป็นไชลด์ของลีฟนั้น
   - เมื่อ `branchFromId` เป็น `null` จะบันทึก `fromId` เป็น `"root"`

## พฤติกรรมการนำทาง `/tree` (ไฟล์เซสชันเดียวกัน)

`AgentSession.navigateTree()` เป็นการนำทาง ไม่ใช่การแยกไฟล์

ขั้นตอน:

1. ตรวจสอบเป้าหมายและคำนวณเส้นทางที่ถูกทิ้ง (`collectEntriesForBranchSummary`)
2. ปล่อย `session_before_tree` พร้อม `TreePreparation`
3. สรุปรายการที่ถูกทิ้งตามต้องการ (สรุปจากฮุกหรือตัวสรุปในตัว)
4. คำนวณเป้าหมายลีฟใหม่:
   - การเลือกข้อความ **user**: ลีฟเคลื่อนที่ไปยังพาเรนต์ และข้อความถูกส่งกลับเพื่อเติมล่วงหน้าในตัวแก้ไข
   - การเลือก **custom_message**: กฎเดียวกับข้อความ user (ลีฟ = พาเรนต์ ข้อความเติมล่วงหน้าในตัวแก้ไข)
   - การเลือกรายการอื่นๆ: ลีฟ = id ของรายการที่เลือก
5. ใช้การเคลื่อนที่ลีฟ:
   - มีสรุป: `branchWithSummary(newLeafId, ...)`
   - ไม่มีสรุปและ `newLeafId === null`: `resetLeaf()`
   - กรณีอื่น: `branch(newLeafId)`
6. สร้างบริบทเอเจนต์ใหม่จากลีฟใหม่และปล่อย `session_tree`

สำคัญ: รายการสรุปถูกแนบที่**ตำแหน่งการนำทางใหม่** ไม่ใช่ที่ส่วนท้ายของกิ่งที่ถูกทิ้ง

## พฤติกรรม `/branch` (ไฟล์เซสชันใหม่)

`/branch` และ `/tree` ตั้งใจให้แตกต่างกัน:

- `/tree` นำทางภายในไฟล์เซสชันปัจจุบัน
- `/branch` สร้างไฟล์กิ่งเซสชันใหม่ (หรือแทนที่ในหน่วยความจำสำหรับโหมดไม่ถาวร)

ขั้นตอน `/branch` สำหรับผู้ใช้ (`SelectorController.showUserMessageSelector` → `AgentSession.branch`):

- แหล่งที่มาของกิ่งต้องเป็น **ข้อความ user**
- ข้อความ user ที่เลือกถูกแยกเพื่อเติมล่วงหน้าในตัวแก้ไข
- หากข้อความ user ที่เลือกเป็นรูท (`parentId === null`): เริ่มเซสชันใหม่ผ่าน `newSession({ parentSession: previousSessionFile })`
- กรณีอื่น: `createBranchedSession(selectedEntry.parentId)` เพื่อแยกประวัติจนถึงขอบเขตพรอมต์ที่เลือก

รายละเอียดของ `SessionManager.createBranchedSession(leafId)`:

- สร้างเส้นทางรูท→ลีฟผ่าน `getBranch(leafId)` โยนข้อผิดพลาดหากไม่พบ
- ไม่รวมรายการ `label` ที่มีอยู่จากเส้นทางที่คัดลอก
- สร้างรายการป้ายกำกับใหม่จาก `labelsById` ที่แก้ไขแล้วสำหรับรายการที่ยังอยู่ในเส้นทาง
- โหมดถาวร: เขียนไฟล์ JSONL ใหม่และสลับตัวจัดการไปยังไฟล์นั้น ส่งคืนเส้นทางไฟล์ใหม่
- โหมดในหน่วยความจำ: แทนที่รายการในหน่วยความจำ ส่งคืน `undefined`

## การสร้างบริบทใหม่และการรวมสรุป/custom

`buildSessionContext()` (ใน `session-manager.ts`) แก้ไขเส้นทางรูท→ลีฟที่ใช้งานอยู่และสร้างสถานะบริบท LLM ที่มีผล:

- ติดตามสถานะ thinking/model/mode/ttsr ล่าสุดบนเส้นทาง
- จัดการ compaction ล่าสุดบนเส้นทาง:
  - ปล่อยสรุป compaction ก่อน
  - เล่นซ้ำข้อความที่เก็บไว้จาก `firstKeptEntryId` ถึงจุด compaction
  - จากนั้นเล่นซ้ำข้อความหลัง compaction
- รวมรายการ `branch_summary` และ `custom_message` เป็นออบเจกต์ `AgentMessage`

จากนั้น `session/messages.ts` จะแมปประเภทข้อความเหล่านี้สำหรับอินพุตของโมเดล:

- `branchSummary` และ `compactionSummary` กลายเป็นข้อความบริบทแบบเทมเพลตบทบาท user
- `custom`/`hookMessage` กลายเป็นข้อความเนื้อหาบทบาท user

ดังนั้นการเคลื่อนที่ในต้นไม้จะเปลี่ยนบริบทโดยการเปลี่ยนเส้นทางลีฟที่ใช้งานอยู่ ไม่ใช่โดยการแก้ไขรายการเก่า

## ป้ายกำกับและพฤติกรรม UI ของต้นไม้

การจัดเก็บป้ายกำกับถาวร:

- `appendLabelChange(targetId, label?)` เขียนรายการ `label` บนสายลีฟปัจจุบัน
- `labelsById` ถูกอัปเดตทันที (ตั้งค่าหรือลบ)
- `getTree()` แก้ไขป้ายกำกับปัจจุบันบนแต่ละโหนดที่ส่งคืน

พฤติกรรมตัวเลือกต้นไม้ (`tree-selector.ts`):

- แผ่ต้นไม้สำหรับการนำทาง เก็บการเน้นเส้นทางที่ใช้งานอยู่ และให้ความสำคัญกับการแสดงกิ่งที่ใช้งานอยู่ก่อน
- รองรับโหมดกรอง: `default`, `no-tools`, `user-only`, `labeled-only`, `all`
- รองรับการค้นหาข้อความอิสระเหนือเนื้อหาเชิงความหมายที่แสดงผล
- `Shift+L` เปิดการแก้ไขป้ายกำกับแบบอินไลน์และเขียนผ่าน `appendLabelChange`

การกำหนดเส้นทางคำสั่ง:

- `/tree` เปิดตัวเลือกต้นไม้เสมอ
- `/branch` เปิดตัวเลือกข้อความ user ยกเว้น `doubleEscapeAction=tree` ซึ่งในกรณีนั้นจะใช้ UX ตัวเลือกต้นไม้เช่นกัน

## จุดสัมผัสส่วนขยายและฮุกสำหรับการดำเนินการต้นไม้

API ส่วนขยายเวลาคำสั่ง (`ExtensionCommandContext`):

- `branch(entryId)` — สร้างไฟล์เซสชันที่แยกกิ่ง
- `navigateTree(targetId, { summarize? })` — เคลื่อนที่ภายในต้นไม้/ไฟล์ปัจจุบัน

เหตุการณ์รอบการนำทางต้นไม้:

- `session_before_tree`
  - รับ `TreePreparation`:
    - `targetId`
    - `oldLeafId`
    - `commonAncestorId`
    - `entriesToSummarize`
    - `userWantsSummary`
  - สามารถยกเลิกการนำทาง
  - สามารถให้เพย์โหลดสรุปที่ใช้แทนตัวสรุปในตัว
  - รับ `signal` สำหรับการยกเลิก (เส้นทางการยกเลิกด้วย Escape)
- `session_tree`
  - ปล่อย `newLeafId`, `oldLeafId`
  - รวม `summaryEntry` เมื่อมีการสร้างสรุป
  - `fromExtension` ระบุแหล่งที่มาของสรุป

ฮุกวงจรชีวิตที่เกี่ยวข้องแต่อยู่ใกล้เคียง:

- `session_before_branch` / `session_branch` สำหรับขั้นตอน `/branch`
- `session_before_compact`, `session.compacting`, `session_compact` สำหรับรายการ compaction ที่ส่งผลต่อการสร้างบริบทแบบต้นไม้ในภายหลัง

## ข้อจำกัดจริงและเงื่อนไขขอบ

- `branch()` ไม่สามารถเป้าหมายเป็น `null` ได้ ใช้ `resetLeaf()` สำหรับสถานะรูทก่อนรายการแรก
- `branchWithSummary()` รองรับเป้าหมาย `null` และบันทึก `fromId: "root"`
- การเลือกลีฟปัจจุบันในตัวเลือกต้นไม้ไม่มีการดำเนินการ
- การสรุปต้องมีโมเดลที่ใช้งานอยู่ หากไม่มี การนำทางแบบสรุปจะล้มเหลวทันที
- หากการสรุปถูกยกเลิก การนำทางจะถูกยกเลิกและลีฟไม่เปลี่ยนแปลง
- เซสชันในหน่วยความจำไม่เคยส่งคืนเส้นทางไฟล์กิ่งจาก `createBranchedSession`

## ความเข้ากันได้แบบเก่าที่ยังคงมีอยู่

การย้ายเซสชันยังคงทำงานเมื่อโหลด:

- v1→v2 เพิ่ม `id`/`parentId` และแปลงจุดยึด compaction index เป็นจุดยึด id
- v2→v3 ย้าย `hookMessage` role แบบเก่าเป็น `custom`

พฤติกรรมรันไทม์ปัจจุบันเป็นความหมายต้นไม้เวอร์ชัน 3 หลังการย้าย
