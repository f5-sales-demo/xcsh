# أداة Python وبيئة تشغيل IPython

يصف هذا المستند مكدس تنفيذ Python الحالي في `packages/coding-agent`.
يشمل سلوك الأداة، ودورة حياة النواة/البوابة، ومعالجة البيئة، ودلالات التنفيذ، وعرض المخرجات، وأوضاع الفشل التشغيلية.

## النطاق والملفات الرئيسية

- واجهة الأداة: `src/tools/python.ts`
- تنسيق النواة لكل جلسة/استدعاء: `src/ipy/executor.ts`
- بروتوكول النواة + تكامل البوابة: `src/ipy/kernel.ts`
- منسق البوابة المحلية المشتركة: `src/ipy/gateway-coordinator.ts`
- عارض الوضع التفاعلي لتشغيل Python الذي يبادر به المستخدم: `src/modes/components/python-execution.ts`
- تصفية بيئة التشغيل وتحليل Python: `src/ipy/runtime.ts`

## ما هي أداة Python

تُنفّذ أداة `python` خلية Python واحدة أو أكثر عبر نواة مدعومة بـ Jupyter Kernel Gateway (لا عن طريق إطلاق `python -c` مباشرةً لكل خلية).

معامِلات الأداة:

```ts
{
  cells: Array<{ code: string; title?: string }>;
  timeout?: number; // بالثواني، مقيّد بين 1..600، الافتراضي 30
  cwd?: string;
  reset?: boolean; // إعادة تعيين النواة قبل الخلية الأولى فقط
}
```

الأداة `concurrency = "exclusive"` لكل جلسة، لذا لا تتداخل الاستدعاءات.

## دورة حياة البوابة

### الأوضاع

يوجد مسارَان للبوابة:

1. **بوابة خارجية** (عند ضبط `PI_PYTHON_GATEWAY_URL`)
   - تستخدم عنوان URL المُهيَّأ مباشرةً.
   - مصادقة اختيارية عبر `PI_PYTHON_GATEWAY_TOKEN`.
   - لا يُطلَق أي عملية بوابة محلية ولا يُدار أي منها.

2. **بوابة محلية مشتركة** (المسار الافتراضي)
   - تستخدم عملية مشتركة واحدة منسَّقة تحت `~/.xcsh/agent/python-gateway`.
   - ملف البيانات الوصفية: `gateway.json`
   - ملف القفل: `gateway.lock`
   - أمر الإطلاق:
     - `python -m kernel_gateway`
     - مرتبط بـ `127.0.0.1:<allocated-port>`
     - فحص صحة بدء التشغيل: `GET /api/kernelspecs`

### تنسيق البوابة المحلية المشتركة

`acquireSharedGateway()`:

- يأخذ قفل ملف (`gateway.lock`) مع نبضات قلب.
- يُعيد استخدام `gateway.json` إذا كانت عملية PID حية واجتاز فحص الصحة.
- ينظّف المعلومات/معرّفات PID القديمة عند الحاجة.
- يبدأ بوابة جديدة عند غياب بوابة سليمة.

`releaseSharedGateway()` عملية بدون تأثير حالياً (إيقاف تشغيل النواة لا يُفكّك البوابة المشتركة).

`shutdownSharedGateway()` يُنهي العملية المشتركة صراحةً ويمسح بيانات البوابة الوصفية.

### قيد مهم

`python.sharedGateway=false` مرفوض عند بدء تشغيل النواة:

- الخطأ: `Shared Python gateway required; local gateways are disabled`
- لا يوجد وضع بوابة محلية غير مشتركة لكل عملية.

## دورة حياة النواة

يستخدم كل تنفيذ نواةً مُنشأة عبر `POST /api/kernels` على البوابة المحددة.

تسلسل بدء تشغيل النواة:

1. فحص التوفر (`checkPythonKernelAvailability`)
2. إنشاء النواة (`/api/kernels`)
3. فتح WebSocket (`/api/kernels/:id/channels`)
4. تهيئة بيئة النواة (`cwd`، متغيرات البيئة، `sys.path`)
5. تنفيذ `PYTHON_PRELUDE`
6. تحميل وحدات الامتداد من:
   - المستخدم: `~/.xcsh/agent/modules/*.py`
   - المشروع: `<cwd>/.xcsh/modules/*.py` (يتجاوز وحدة المستخدم ذات الاسم المطابق)

إيقاف تشغيل النواة:

- يحذف النواة البعيدة عبر `DELETE /api/kernels/:id`
- يغلق WebSocket
- يستدعي خطّاف إصدار البوابة المشتركة (بدون تأثير حالياً)

## دلالات استمرارية الجلسة

يتحكّم `python.kernelMode` في إعادة استخدام النواة:

- `session` (الافتراضي)
  - يُعيد استخدام جلسات النواة المفهرسة بهوية الجلسة + cwd.
  - يُسلسَل التنفيذ لكل جلسة عبر قائمة انتظار.
  - تُستبعَد الجلسات الخاملة بعد 5 دقائق.
  - بحد أقصى 4 جلسات؛ تُستبعَد الأقدم عند الامتلاء.
  - تكشف فحوصات ضربات القلب النوى المتوقفة.
  - يُسمح بإعادة تشغيل تلقائية مرة واحدة؛ الأعطال المتكررة => فشل قاطع.

- `per-call`
  - يُنشئ نواة جديدة لكل طلب تنفيذ.
  - يُوقف النواة بعد انتهاء الطلب.
  - لا استمرارية للحالة عبر الاستدعاءات.

### سلوك الخلايا المتعددة في استدعاء أداة واحدة

تعمل الخلايا بالتسلسل في مثيل النواة ذاته لذلك الاستدعاء.

في حال فشل خلية وسيطة:

- تبقى حالة الخلايا السابقة في الذاكرة.
- تُعيد الأداة خطأً محدداً يُشير إلى الخلية التي فشلت.
- لا تُنفَّذ الخلايا اللاحقة.

`reset=true` ينطبق على تنفيذ الخلية الأولى فقط في ذلك الاستدعاء.

## تصفية البيئة وتحليل بيئة التشغيل

تُصفَّى البيئة قبل إطلاق بوابة/نواة بيئة التشغيل:

- تشمل القائمة البيضاء متغيرات أساسية مثل `PATH`، و`HOME`، ومتغيرات اللغة المحلية، و`VIRTUAL_ENV`، و`PYTHONPATH`، وغيرها.
- بادئات السماح: `LC_`، و`XDG_`، و`PI_`
- تحذف القائمة السوداء مفاتيح API الشائعة (OpenAI/Anthropic/Gemini/إلخ.)

ترتيب اختيار بيئة التشغيل:

1. venv نشطة/موجودة (`VIRTUAL_ENV`، ثم `<cwd>/.venv`، `<cwd>/venv`)
2. venv مُدارة في `~/.xcsh/python-env`
3. `python` أو `python3` على PATH

عند اختيار venv، يُضاف مسار bin/Scripts الخاص بها في مقدمة `PATH`.

تهيئة بيئة النواة داخل Python تقوم أيضاً بـ:

- `os.chdir(cwd)`
- حقن خريطة البيئة المُقدَّمة في `os.environ`
- ضمان وجود cwd في `sys.path`

## توفر الأداة واختيار الوضع

يتحكّم `python.toolMode` (الافتراضي `both`) مع تجاوز اختياري `PI_PY` في التعرض:

- `ipy-only`
- `bash-only`
- `both`

القيم المقبولة لـ `PI_PY`:

- `0` / `bash` -> `bash-only`
- `1` / `py` -> `ipy-only`
- `mix` / `both` -> `both`

في حال فشل الاختبار المبدئي لـ Python، تتدهور إنشاء الأداة إلى bash-only لتلك الجلسة.

## تدفق التنفيذ والإلغاء/المهلة

### مهلة مستوى الأداة

مهلة أداة `python` بالثواني، الافتراضي 30، مقيّدة بين `1..600`.

تجمع الأداة بين:

- إشارة إلغاء المُستدعي
- إشارة إلغاء المهلة

باستخدام `AbortSignal.any(...)`.

### إلغاء تنفيذ النواة

عند الإلغاء/المهلة:

- يُحدَّد التنفيذ كمُلغى.
- تُحاوَل مقاطعة النواة عبر REST (`POST /interrupt`) وقناة التحكم `interrupt_request`.
- تتضمن النتيجة `cancelled=true`.
- يُضيف مسار المهلة تعليقاً على المخرجات: `Command timed out after <n> seconds`.

### سلوك stdin

لا يدعم stdin التفاعلي.

إذا أصدرت النواة `input_request`:

- تُسجّل الأداة `stdinRequested=true`
- تُصدر نصاً توضيحياً
- ترسل `input_reply` فارغاً
- يُعامَل التنفيذ كفشل على مستوى المنفّذ

## التقاط المخرجات وعرضها

### فئات المخرجات الملتقطة

من رسائل النواة:

- `stream` -> قطع نصية عادية
- `display_data`/`execute_result` -> معالجة العرض الغني
- `error` -> نص تتبّع المكدس
- MIME مخصص `application/x-xcsh-status` -> أحداث الحالة المهيكلة

أولوية عرض MIME:

1. `text/markdown`
2. `text/plain`
3. `text/html` (يُحوَّل إلى markdown أساسي)

تُلتقط أيضاً كمخرجات مهيكلة:

- `application/json` -> بيانات شجرة JSON
- `image/png` -> حمولات الصور
- `application/x-xcsh-status` -> أحداث الحالة

### التخزين والاقتطاع

تُدفق المخرجات عبر `OutputSink` وقد تُحفظ في تخزين القطع.

يمكن أن تتضمن نتائج الأداة بيانات الاقتطاع الوصفية و`artifact://<id>` لاسترداد المخرجات الكاملة.

### سلوك العارض

- عارض الأداة (`python.ts`):
  - يعرض كتل خلايا الكود مع حالة كل خلية
  - يعتمد المعاينة المطوية افتراضياً 10 أسطر
  - يدعم وضع التوسيع للمخرجات الكاملة وتفاصيل الحالة الأكثر ثراءً
- العارض التفاعلي (`python-execution.ts`):
  - يُستخدم لتنفيذ Python الذي يبادر به المستخدم في TUI
  - يعتمد المعاينة المطوية افتراضياً 20 سطراً
  - يقيّد الأسطر الفردية الطويلة جداً بـ 4000 حرف لسلامة العرض
  - يعرض إشعارات الإلغاء/الخطأ/الاقتطاع

## دعم البوابة الخارجية

اضبط:

```bash
export PI_PYTHON_GATEWAY_URL="http://127.0.0.1:8888"
# اختياري:
export PI_PYTHON_GATEWAY_TOKEN="..."
```

الاختلافات في السلوك عن البوابة المحلية المشتركة:

- لا ملفات قفل/معلومات بوابة محلية
- لا إطلاق/إنهاء لعملية محلية
- تعمل فحوصات الصحة وعمليات CRUD للنواة على نقطة النهاية الخارجية
- تُعرض أخطاء المصادقة مع توجيه صريح للرمز المميز

## استكشاف الأخطاء التشغيلية (أوضاع الفشل الحالية)

- **أداة Python غير متاحة**
  - تحقق من `python.toolMode` / `PI_PY`.
  - في حال فشل الاختبار المبدئي، تعود بيئة التشغيل إلى bash-only.

- **أخطاء توفر النواة**
  - الوضع المحلي يتطلب إمكانية استيراد كل من `kernel_gateway` و`ipykernel` في بيئة تشغيل Python المحلولة.
  - ثبّت بـ:

    ```bash
    python -m pip install jupyter_kernel_gateway ipykernel
    ```

- **`python.sharedGateway=false` يُسبّب فشل بدء التشغيل**
  - هذا متوقع مع التطبيق الحالي.

- **أخطاء مصادقة/وصول بوابة خارجية**
  - 401/403 -> اضبط `PI_PYTHON_GATEWAY_TOKEN`.
  - مهلة/غير قابل للوصول -> تحقق من URL/الشبكة وصحة البوابة.

- **توقف التنفيذ ثم انتهاء المهلة**
  - زِد `timeout` للأداة (بحد أقصى 600 ث) إذا كان الحمل المعالج مشروعاً.
  - للكود المتوقف، يُشغّل الإلغاء مقاطعة النواة لكن كود المستخدم قد يحتاج إلى إعادة هيكلة.

- **موجّهات stdin/input في كود Python**
  - `input()` غير مدعوم بشكل تفاعلي في هذا المسار من بيئة التشغيل؛ مرّر البيانات برمجياً.

- **استنفاد الموارد (`EMFILE` / عدد كبير جداً من الملفات المفتوحة)**
  - يُشغّل مدير الجلسات استرداد البوابة المشتركة (تفكيك الجلسة + إعادة تشغيل البوابة المشتركة).

- **أخطاء دليل العمل**
  - تتحقق الأداة من وجود `cwd` وكونه دليلاً قبل التنفيذ.

## متغيرات البيئة ذات الصلة

- `PI_PY` — تجاوز تعرض الأداة (تعيين `bash-only`/`ipy-only`/`both` أعلاه)
- `PI_PYTHON_GATEWAY_URL` — استخدام بوابة خارجية
- `PI_PYTHON_GATEWAY_TOKEN` — رمز مصادقة بوابة خارجية اختياري
- `PI_PYTHON_SKIP_CHECK=1` — تجاوز فحوصات الاختبار المبدئي/الإحماء لـ Python
- `PI_PYTHON_IPC_TRACE=1` — تسجيل تتبّعات إرسال/استقبال IPC للنواة
- `PI_DEBUG_STARTUP=1` — إصدار علامات تصحيح مرحلة بدء التشغيل
