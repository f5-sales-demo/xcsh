# دليل تشغيل البناء والإصدار وتصحيح الأخطاء للملحقات الأصلية

يصف هذا الدليل كيفية إنتاج مسار بناء `@f5xc-salesdemos/pi-natives` لملحقات `.node`، وكيفية تحميل التوزيعات المُجمَّعة لها، وكيفية تصحيح أخطاء المُحمِّل/البناء.

يتبع هذا الدليل المصطلحات المعمارية من `docs/natives-architecture.md`:

- **إنتاج الحزمة في وقت البناء** (`scripts/build-native.ts`)
- **توليد بيان الملحق المُضمَّن** (`scripts/embed-native.ts`)
- **تحميل الملحق في وقت التشغيل + بوابة التحقق** (`src/native.ts`)

## ملفات التنفيذ

- `packages/natives/scripts/build-native.ts`
- `packages/natives/scripts/embed-native.ts`
- `packages/natives/package.json`
- `packages/natives/src/native.ts`
- `crates/pi-natives/Cargo.toml`

## نظرة عامة على مسار البناء

### 1) نقاط دخول البناء

سكريبتات `packages/natives/package.json`:

- `bun scripts/build-native.ts` (`build`) → بناء إصدار نهائي
- `bun scripts/build-native.ts --dev` (`dev:native`) → بناء ملف تعريف تصحيح/تطوير (نفس تسمية المخرجات)
- `bun scripts/embed-native.ts` (`embed:native`) → توليد `src/embedded-addon.ts` من الملفات المبنية

### 2) بناء حزمة Rust

يُشغِّل `build-native.ts` أداة Cargo في `crates/pi-natives`:

- الأمر الأساسي: `cargo build`
- وضع الإصدار يضيف `--release` ما لم يُمرَّر `--dev`
- الهدف المتقاطع يضيف `--target <CROSS_TARGET>`

يُعلن `crates/pi-natives/Cargo.toml` عن `crate-type = ["cdylib"]`، لذا تُصدر Cargo مكتبة مشتركة (`.so`/`.dylib`/`.dll`) يتم نسخها/إعادة تسميتها بعد ذلك إلى اسم ملف ملحق `.node`.

### 3) اكتشاف الحزمة وتثبيتها

بعد اكتمال Cargo، يفحص `build-native.ts` مجلدات المخرجات المرشحة بالترتيب:

1. `${CARGO_TARGET_DIR}` (إذا كان مُعيَّنًا)
2. `<repo>/target`
3. `crates/pi-natives/target`

لكل جذر يفحص مجلدات ملفات التعريف:

- البناء المتقاطع: `<root>/<crossTarget>/<profile>` ثم `<root>/<profile>`
- البناء الأصلي: `<root>/<profile>`

ثم يبحث عن أحد:

- `libpi_natives.so`
- `libpi_natives.dylib`
- `pi_natives.dll`
- `libpi_natives.dll`

عند اكتشافه، يُثبَّت بشكل ذري في `packages/natives/native/` باستخدام دلالات الملف المؤقت + إعادة التسمية (يعالج الحل البديل لنظام Windows صراحةً أعطال استبدال DLL المقفلة).

## نموذج الهدف/المتغير واصطلاحات التسمية

## علامة المنصة

يستخدم كلٌّ من البناء ووقت التشغيل علامة المنصة:

`<platform>-<arch>` (مثال: `darwin-arm64`، `linux-x64`)

## نموذج المتغير (x64 فقط)

يدعم x64 متغيرات المعالج:

- `modern` (مسار قادر على AVX2)
- `baseline` (احتياطي)

يستخدم غير x64 حزمة افتراضية واحدة (بدون لاحقة متغير).

### أسماء ملفات المخرجات

بناءات الإصدار النهائي:

- x64: `pi_natives.<platform>-<arch>-modern.node` أو `...-baseline.node`
- غير x64: `pi_natives.<platform>-<arch>.node`

بناء التطوير (`--dev`):

- يستخدم علامات ملف تعريف التصحيح لكنه يحافظ على تسمية المخرجات القياسية المُعلَّمة بالمنصة

ترتيب المرشحين في المُحمِّل في `native.ts`:

- مرشحو الإصدار النهائي
- الوضع المُجمَّع يُضيف مرشحي الاستخراج/التخزين المؤقت قبل الملفات المحلية للحزمة

## علامات البيئة وخيارات البناء

## علامات وقت التشغيل

- `PI_DEV` (سلوك المُحمِّل): تمكين تشخيصات المُحمِّل
- `PI_NATIVE_VARIANT` (سلوك المُحمِّل، x64 فقط): إجبار اختيار `modern` أو `baseline` في وقت التشغيل
- `PI_COMPILED` (سلوك المُحمِّل): تمكين سلوك المرشح/الاستخراج للثنائي المُجمَّع

## علامات/خيارات وقت البناء

- `--dev` (وسيطة سكريبت): بناء ملف تعريف تصحيح
- `CROSS_TARGET`: يُمرَّر إلى Cargo `--target`
- `TARGET_PLATFORM`: تجاوز تسمية علامة منصة المخرجات
- `TARGET_ARCH`: تجاوز تسمية معمارية المخرجات
- `TARGET_VARIANT` (x64 فقط): إجبار `modern` أو `baseline` لاسم ملف المخرجات وسياسة RUSTFLAGS
- `CARGO_TARGET_DIR`: جذر إضافي عند البحث في مخرجات Cargo
- `RUSTFLAGS`:
  - إذا كانت غير مُعيَّنة وغير مُجمَّعة تقاطعيًا، يُعيِّن السكريبت:
    - modern: `-C target-cpu=x86-64-v3`
    - baseline: `-C target-cpu=x86-64-v2`
    - غير x64 / لا متغير: `-C target-cpu=native`
  - إذا كانت مُعيَّنة مسبقًا، لا يتجاوزها السكريبت

## انتقالات حالة/دورة حياة البناء

### دورة حياة البناء (`build-native.ts`)

1. **التهيئة**: تحليل الوسيطات/البيئة (`--dev`، تجاوزات الهدف، علامات التقاطع)
2. **حل المتغير**:
   - غير x64 → لا متغير
   - x64 + `TARGET_VARIANT` → متغير صريح
   - بناء x64 المتقاطع بدون `TARGET_VARIANT` → خطأ صارم
   - بناء x64 المحلي بدون تجاوز → الكشف عن AVX2 للمضيف
3. **التجميع**: تشغيل Cargo مع ملف التعريف/الهدف المحلول
4. **تحديد موقع الحزمة**: فحص جذور الهدف/مجلدات ملفات التعريف/أسماء المكتبات
5. **التثبيت**: نسخ + إعادة تسمية ذرية في `packages/natives/native`
6. **الاكتمال**: الملحق جاهز لمرشحي المُحمِّل

تحدث نهايات الفشل في أي مرحلة مع نص خطأ صريح (متغير غير صالح، فشل بناء cargo، مكتبة المخرجات مفقودة، فشل التثبيت/إعادة التسمية).

### دورة حياة التضمين (`embed-native.ts`)

1. **التهيئة**: حساب علامة المنصة من `TARGET_PLATFORM`/`TARGET_ARCH` أو قيم المضيف
2. **مجموعة المرشحين**:
   - x64 يتوقع كلًّا من `modern` و`baseline`
   - غير x64 يتوقع ملفًا افتراضيًا واحدًا
3. **التحقق من التوافر** في `packages/natives/native`
4. **توليد البيان** (`src/embedded-addon.ts`) مع استيرادات `file` لـ Bun وإصدار الحزمة
5. **استخراج وقت التشغيل جاهز** للوضع المُجمَّع

`--reset` يتجاوز التحقق ويكتب بذرة بيان فارغة (`embeddedAddon = null`).

## سير عمل التطوير مقابل السلوك المشحون/المُجمَّع

## سير عمل التطوير المحلي

الحلقة المحلية النموذجية:

1. بناء الملحق:
   - إصدار نهائي: `bun --cwd=packages/natives run build`
   - ملف تعريف التصحيح: `bun --cwd=packages/natives run dev:native`
2. تعيين `PI_DEV=1` عند اختبار تشخيصات المُحمِّل
3. يحل المُحمِّل في `native.ts` مرشحات `native/` المحلية للحزمة (ومسار اليعقوبي لمجلد الملف القابل للتنفيذ)
4. يفرض `validateNative` توافق التصدير قبل استخدام الأغلفة للربط

## سير عمل الثنائي المشحون/المُجمَّع

في الوضع المُجمَّع (`PI_COMPILED` أو علامات Bun المُضمَّنة):

1. يحسب المُحمِّل مجلد التخزين المؤقت بالإصدار: `<getNativesDir()>/<packageVersion>` (تشغيليًا `~/.xcsh/natives/<version>`)
2. إذا تطابق البيان المُضمَّن مع المنصة+الإصدار الحالي، قد يستخرج المُحمِّل الملف المُضمَّن المحدد في ذلك المجلد بالإصدار
3. ترتيب المرشحين في وقت التشغيل يتضمن:
   - مجلد التخزين المؤقت بالإصدار
   - مجلد الثنائي المُجمَّع القديم (`%LOCALAPPDATA%/xcsh` على Windows، `~/.local/bin` في غيره)
   - مجلدات الحزمة/الملف القابل للتنفيذ
4. أول ملحق يُحمَّل بنجاح يجب أن يجتاز `validateNative`

هذا هو السبب في ضرورة توافق التغليف وتوقعات مُحمِّل وقت التشغيل: يجب أن تتطابق أسماء الملفات وعلامات المنصة والرموز المُصدَّرة مع ما يفحصه ويتحقق منه `native.ts`.

## تعيين JS API ↔ تصدير Rust (مجموعة فرعية من بوابة التحقق)

يتطلب `native.ts` وجود هذه التصديرات المرئية لـ JS على الملحق المُحمَّل. تُعيَّن إلى تصديرات Rust N-API في `crates/pi-natives/src`:

| اسم JS المطلوب بواسطة `validateNative` | إعلان تصدير Rust | ملف مصدر Rust |
| --- | --- | --- |
| `glob` | `#[napi] pub fn glob(...)` | `crates/pi-natives/src/glob.rs` |
| `grep` | `#[napi] pub fn grep(...)` | `crates/pi-natives/src/grep.rs` |
| `search` | `#[napi] pub fn search(...)` | `crates/pi-natives/src/grep.rs` |
| `highlightCode` | `#[napi] pub fn highlight_code(...)` | `crates/pi-natives/src/highlight.rs` |
| `getSystemInfo` | `#[napi] pub fn get_system_info(...)` | `crates/pi-natives/src/system_info.rs` |
| `getWorkProfile` | `#[napi] pub fn get_work_profile(...)` (تصدير بتحويل camel-case) | `crates/pi-natives/src/prof.rs` |
| `invalidateFsScanCache` | `#[napi] pub fn invalidate_fs_scan_cache(...)` | `crates/pi-natives/src/fs_cache.rs` |

إذا كان أي رمز مطلوب مفقودًا، يفشل المُحمِّل بسرعة مع تلميح لإعادة البناء.

## سلوك الفشل والتشخيصات

## أعطال وقت البناء

- تكوين متغير غير صالح:
  - `TARGET_VARIANT` مُعيَّن على غير x64 → خطأ فوري
  - بناء x64 المتقاطع بدون `TARGET_VARIANT` صريح → خطأ فوري
- فشل بناء Cargo:
  - يُظهر السكريبت الخروج بقيمة غير صفرية وstderr
- الحزمة غير موجودة:
  - يطبع السكريبت كل مجلد ملف تعريف تم فحصه
- فشل التثبيت:
  - رسالة صريحة؛ يتضمن Windows تلميح الملف المقفل

## أعطال مُحمِّل وقت التشغيل (`native.ts`)

- علامة منصة غير مدعومة:
  - ترمي مع قائمة المنصات المدعومة
- لا يمكن تحميل أي مرشح:
  - ترمي مع قائمة أخطاء المرشحين الكاملة وتلميحات معالجة خاصة بالوضع
- تصديرات مفقودة:
  - ترمي مع أسماء الرموز المفقودة الدقيقة وأمر إعادة البناء
- مشاكل استخراج التضمين:
  - أخطاء إنشاء المجلد/الكتابة مُسجَّلة ومُضمَّنة في التشخيصات النهائية

| --- | --- | --- | --- |
| `Native addon missing exports ... Missing: <name>` | ثنائي `.node` قديم، أو عدم تطابق اسم تصدير Rust، أو تحميل ثنائي خاطئ | التشغيل مع `PI_DEV=1` لرؤية المسار المُحمَّل؛ فحص قائمة التصدير لذلك الملف | إعادة بناء `build`؛ التأكد من أن اسم تصدير Rust `#[napi]` (أو اسم مستعار صريح عند الحاجة) يتطابق مع مفتاح JS؛ حذف الملفات القديمة المخزنة مؤقتًا/بالإصدار |
| آلة x64 تُحمِّل baseline عندما يُتوقع modern | `PI_NATIVE_VARIANT=baseline`، أو عدم اكتشاف AVX2، أو وجود ملف baseline فقط | فحص `PI_NATIVE_VARIANT`؛ فحص `native/` للملف `-modern` | بناء متغير modern (`TARGET_VARIANT=modern ... build`) والتأكد من شحن الملف |
| البناء المتقاطع يُنتج ثنائيًا غير قابل للاستخدام/مُعلَّم خاطئًا | عدم تطابق بين `CROSS_TARGET` و`TARGET_PLATFORM`/`TARGET_ARCH`، أو غياب `TARGET_VARIANT` لـ x64 | تأكيد مجموعة env واسم ملف المخرجات | إعادة التشغيل بقيم env متسقة و`TARGET_VARIANT` صريح لـ x64 |
| الثنائي المُجمَّع يفشل بعد الترقية | تخزين مؤقت مستخرج قديم (`~/.xcsh/natives/<old-or-mismatched-version>`) أو عدم تطابق البيان المُضمَّن | فحص مجلد natives بالإصدار وقائمة أخطاء المُحمِّل | حذف تخزين natives المؤقت بالإصدار لإصدار الحزمة وإعادة التشغيل؛ إعادة توليد البيان المُضمَّن أثناء التغليف |
| المُحمِّل يفحص مسارات كثيرة ولا شيء يعمل | عدم تطابق المنصة أو غياب حزمة الإصدار النهائي في `native/` الحزمة | التحقق من `platformTag` مقابل اسم/أسماء الملفات الفعلية | التأكد من أن اسم الملف المبني يتطابق تمامًا مع اصطلاح `pi_natives.<platform>-<arch>(-variant).node` وأن الحزمة تتضمن `native/` |
| فشل `embed:native` مع "Incomplete native addons" | ملفات المتغير المطلوبة غير مبنية قبل التضمين | التحقق من قائمة المتوقع مقابل الموجود في نص الخطأ | بناء الملفات المطلوبة أولًا (x64: كلٌّ من modern+baseline؛ غير x64: الافتراضي)، ثم إعادة تشغيل `embed:native` |

## الأوامر التشغيلية

```bash
# حزمة الإصدار النهائي للمضيف الحالي
bun --cwd=packages/natives run build

# بناء حزمة ملف تعريف التصحيح
bun --cwd=packages/natives run dev:native

# بناء متغيرات x64 الصريحة
TARGET_VARIANT=modern bun --cwd=packages/natives run build
TARGET_VARIANT=baseline bun --cwd=packages/natives run build

# توليد بيان الملحق المُضمَّن من الملفات الأصلية المبنية
bun --cwd=packages/natives run embed:native

# إعادة تعيين البيان المُضمَّن إلى بذرة فارغة
bun --cwd=packages/natives run embed:native -- --reset
```
