# دليل البناء والإصدار وتصحيح الأخطاء للوحدات الأصلية

يصف هذا الدليل كيف ينتج خط أنابيب بناء `@f5xc-salesdemos/pi-natives` إضافات `.node`، وكيف تقوم التوزيعات المُجمَّعة بتحميلها، وكيفية تصحيح أخطاء المُحمِّل/البناء.

يتبع هذا الدليل مصطلحات البنية من `docs/natives-architecture.md`:

- **إنتاج العناصر في وقت البناء** (`scripts/build-native.ts`)
- **توليد بيان الإضافة المُضمَّنة** (`scripts/embed-native.ts`)
- **تحميل الإضافة في وقت التشغيل + بوابة التحقق** (`src/native.ts`)

## ملفات التنفيذ

- `packages/natives/scripts/build-native.ts`
- `packages/natives/scripts/embed-native.ts`
- `packages/natives/package.json`
- `packages/natives/src/native.ts`
- `crates/pi-natives/Cargo.toml`

## نظرة عامة على خط أنابيب البناء

### 1) نقاط الدخول للبناء

سكريبتات `packages/natives/package.json`:

- `bun scripts/build-native.ts` (`build`) → بناء الإصدار
- `bun scripts/build-native.ts --dev` (`dev:native`) → بناء ملف تعريف التصحيح/التطوير (نفس تسمية المخرجات)
- `bun scripts/embed-native.ts` (`embed:native`) → توليد `src/embedded-addon.ts` من الملفات المبنية

### 2) بناء عنصر Rust

يقوم `build-native.ts` بتشغيل Cargo في `crates/pi-natives`:

- الأمر الأساسي: `cargo build`
- وضع الإصدار يضيف `--release` ما لم يتم تمرير `--dev`
- الهدف المتقاطع يضيف `--target <CROSS_TARGET>`

يُصرِّح `crates/pi-natives/Cargo.toml` عن `crate-type = ["cdylib"]`، لذلك يُصدر Cargo مكتبة مشتركة (`.so`/`.dylib`/`.dll`) يتم بعد ذلك نسخها/إعادة تسميتها إلى اسم ملف إضافة `.node`.

### 3) اكتشاف العنصر وتثبيته

بعد اكتمال Cargo، يفحص `build-native.ts` مجلدات المخرجات المُرشَّحة بالترتيب:

1. `${CARGO_TARGET_DIR}` (إذا كان مُعيَّنًا)
2. `<repo>/target`
3. `crates/pi-natives/target`

لكل جذر يتحقق من مجلدات ملف التعريف:

- البناء المتقاطع: `<root>/<crossTarget>/<profile>` ثم `<root>/<profile>`
- البناء المحلي: `<root>/<profile>`

ثم يبحث عن أحد:

- `libpi_natives.so`
- `libpi_natives.dylib`
- `pi_natives.dll`
- `libpi_natives.dll`

عند العثور عليه، يتم تثبيته بشكل ذري في `packages/natives/native/` باستخدام دلالات ملف مؤقت + إعادة تسمية (آلية Windows البديلة تتعامل مع فشل استبدال ملف DLL المقفل بشكل صريح).

## نموذج الهدف/المتغير واصطلاحات التسمية

## وسم المنصة

يستخدم كل من البناء ووقت التشغيل وسم المنصة:

`<platform>-<arch>` (مثال: `darwin-arm64`، `linux-x64`)

## نموذج المتغيرات (x64 فقط)

يدعم x64 متغيرات وحدة المعالجة المركزية:

- `modern` (مسار قادر على AVX2)
- `baseline` (احتياطي)

غير x64 يستخدم عنصرًا افتراضيًا واحدًا (بدون لاحقة متغير).

### أسماء ملفات المخرجات

بناءات الإصدار:

- x64: `pi_natives.<platform>-<arch>-modern.node` أو `...-baseline.node`
- غير x64: `pi_natives.<platform>-<arch>.node`

بناء التطوير (`--dev`):

- يستخدم علامات ملف تعريف التصحيح لكنه يحتفظ بتسمية المخرجات المُوسَّمة بالمنصة القياسية

ترتيب المُرشَّحين لمُحمِّل وقت التشغيل في `native.ts`:

- مُرشَّحو الإصدار
- الوضع المُجمَّع يُقدِّم المُرشَّحين المُستخرَجين/المُخبَّأين قبل الملفات المحلية للحزمة

## علامات البيئة وخيارات البناء

## علامات وقت التشغيل

- `PI_DEV` (سلوك المُحمِّل): تفعيل تشخيصات المُحمِّل
- `PI_NATIVE_VARIANT` (سلوك المُحمِّل، x64 فقط): فرض اختيار `modern` أو `baseline` في وقت التشغيل
- `PI_COMPILED` (سلوك المُحمِّل): تفعيل سلوك المُرشَّح/الاستخراج للثنائي المُجمَّع

## علامات/خيارات وقت البناء

- `--dev` (وسيط السكريبت): بناء ملف تعريف التصحيح
- `CROSS_TARGET`: يُمرَّر إلى `--target` في Cargo
- `TARGET_PLATFORM`: تجاوز تسمية وسم المنصة للمخرجات
- `TARGET_ARCH`: تجاوز تسمية البنية للمخرجات
- `TARGET_VARIANT` (x64 فقط): فرض `modern` أو `baseline` لاسم ملف المخرجات وسياسة RUSTFLAGS
- `CARGO_TARGET_DIR`: جذر إضافي عند البحث في مخرجات Cargo
- `RUSTFLAGS`:
  - إذا لم يكن مُعيَّنًا ولم يكن تجميعًا متقاطعًا، يُعيِّن السكريبت:
    - modern: `-C target-cpu=x86-64-v3`
    - baseline: `-C target-cpu=x86-64-v2`
    - غير x64 / بدون متغير: `-C target-cpu=native`
  - إذا كان مُعيَّنًا مسبقًا، لا يتجاوزه السكريبت

## حالات البناء/انتقالات دورة الحياة

### دورة حياة البناء (`build-native.ts`)

1. **التهيئة**: تحليل الوسائط/البيئة (`--dev`، تجاوزات الهدف، علامات التقاطع)
2. **حل المتغير**:
   - غير x64 → بدون متغير
   - x64 + `TARGET_VARIANT` → متغير صريح
   - بناء متقاطع x64 بدون `TARGET_VARIANT` → خطأ صارم
   - بناء محلي x64 بدون تجاوز → كشف AVX2 على المُضيف
3. **التجميع**: تشغيل Cargo مع ملف التعريف/الهدف المُحلَّل
4. **تحديد موقع العنصر**: فحص جذور الهدف/مجلدات ملف التعريف/أسماء المكتبات
5. **التثبيت**: نسخ + إعادة تسمية ذرية في `packages/natives/native`
6. **الاكتمال**: الإضافة جاهزة لمُرشَّحي المُحمِّل

يحدث خروج الفشل في أي مرحلة مع نص خطأ صريح (متغير غير صالح، فشل بناء cargo، مكتبة مخرجات مفقودة، فشل التثبيت/إعادة التسمية).

### دورة حياة التضمين (`embed-native.ts`)

1. **التهيئة**: حساب وسم المنصة من `TARGET_PLATFORM`/`TARGET_ARCH` أو قيم المُضيف
2. **مجموعة المُرشَّحين**:
   - x64 يتوقع كلاً من `modern` و `baseline`
   - غير x64 يتوقع ملفًا افتراضيًا واحدًا
3. **التحقق من التوفر** في `packages/natives/native`
4. **توليد البيان** (`src/embedded-addon.ts`) مع استيرادات `file` في Bun وإصدار الحزمة
5. **جاهز للاستخراج في وقت التشغيل** للوضع المُجمَّع

`--reset` يتجاوز التحقق ويكتب بيانًا فارغًا (`embeddedAddon = null`).

## سير عمل التطوير مقابل السلوك المُشحَّن/المُجمَّع

## سير عمل التطوير المحلي

الحلقة المحلية المعتادة:

1. بناء الإضافة:
   - إصدار: `bun --cwd=packages/natives run build`
   - ملف تعريف التصحيح: `bun --cwd=packages/natives run dev:native`
2. تعيين `PI_DEV=1` عند اختبار تشخيصات المُحمِّل
3. المُحمِّل في `native.ts` يحل مُرشَّحي `native/` المحلية للحزمة (والاحتياط لمجلد الملف التنفيذي)
4. `validateNative` يفرض توافق التصديرات قبل أن تستخدم الأغلفة الربط

## سير عمل الثنائي المُشحَّن/المُجمَّع

في الوضع المُجمَّع (`PI_COMPILED` أو علامات Bun المُضمَّنة):

1. يحسب المُحمِّل مجلد ذاكرة التخزين المؤقت المُسوَّم بالإصدار: `<getNativesDir()>/<packageVersion>` (عمليًا `~/.xcsh/natives/<version>`)
2. إذا تطابق البيان المُضمَّن مع المنصة+الإصدار الحاليين، قد يستخرج المُحمِّل الملف المُضمَّن المُحدَّد في ذلك المجلد المُسوَّم بالإصدار
3. ترتيب مُرشَّحي وقت التشغيل يشمل:
   - مجلد ذاكرة التخزين المؤقت المُسوَّم بالإصدار
   - مجلد الثنائي المُجمَّع القديم (`%LOCALAPPDATA%/xcsh` على Windows، `~/.local/bin` في غير ذلك)
   - مجلدات الحزمة/الملف التنفيذي
4. أول إضافة يتم تحميلها بنجاح لا تزال يجب أن تجتاز `validateNative`

لهذا السبب يجب أن تتوافق التعبئة + توقعات مُحمِّل وقت التشغيل: أسماء الملفات، ووسوم المنصات، والرموز المُصدَّرة يجب أن تتطابق مع ما يفحصه ويتحقق منه `native.ts`.

## تعيين واجهة JS API ↔ تصدير Rust (مجموعة فرعية من بوابة التحقق)

يتطلب `native.ts` وجود هذه التصديرات المرئية من JS على الإضافة المُحمَّلة. وهي تُعيَّن إلى تصديرات N-API في Rust في `crates/pi-natives/src`:

| اسم JS المطلوب بواسطة `validateNative` | تصريح تصدير Rust | ملف مصدر Rust |
| --- | --- | --- |
| `glob` | `#[napi] pub fn glob(...)` | `crates/pi-natives/src/glob.rs` |
| `grep` | `#[napi] pub fn grep(...)` | `crates/pi-natives/src/grep.rs` |
| `search` | `#[napi] pub fn search(...)` | `crates/pi-natives/src/grep.rs` |
| `highlightCode` | `#[napi] pub fn highlight_code(...)` | `crates/pi-natives/src/highlight.rs` |
| `getSystemInfo` | `#[napi] pub fn get_system_info(...)` | `crates/pi-natives/src/system_info.rs` |
| `getWorkProfile` | `#[napi] pub fn get_work_profile(...)` (تصدير بتنسيق camelCase) | `crates/pi-natives/src/prof.rs` |
| `invalidateFsScanCache` | `#[napi] pub fn invalidate_fs_scan_cache(...)` | `crates/pi-natives/src/fs_cache.rs` |

إذا كان أي رمز مطلوب مفقودًا، يفشل المُحمِّل بسرعة مع تلميح لإعادة البناء.

## سلوك الفشل والتشخيصات

## أخطاء وقت البناء

- تكوين متغير غير صالح:
  - `TARGET_VARIANT` مُعيَّن على غير x64 → خطأ فوري
  - بناء متقاطع x64 بدون `TARGET_VARIANT` صريح → خطأ فوري
- فشل بناء Cargo:
  - يعرض السكريبت رمز الخروج غير الصفري وstderr
- عنصر غير موجود:
  - يطبع السكريبت كل مجلد ملف تعريف تم التحقق منه
- فشل التثبيت:
  - رسالة صريحة؛ Windows يتضمن تلميحًا للملف المقفل

## أخطاء مُحمِّل وقت التشغيل (`native.ts`)

- وسم منصة غير مدعوم:
  - يرمي خطأ مع قائمة المنصات المدعومة
- لم يتمكن أي مُرشَّح من التحميل:
  - يرمي خطأ مع قائمة أخطاء المُرشَّحين الكاملة وتلميحات إصلاح خاصة بالوضع
- تصديرات مفقودة:
  - يرمي خطأ مع أسماء الرموز المفقودة بالضبط وأمر إعادة البناء
- مشاكل استخراج المُضمَّن:
  - أخطاء إنشاء المجلد/الكتابة تُسجَّل وتُضمَّن في التشخيصات النهائية

## مصفوفة استكشاف الأخطاء وإصلاحها

| العَرَض | السبب المُحتمل | التحقق | الإصلاح |
| --- | --- | --- | --- |
| `Native addon missing exports ... Missing: <name>` | ثنائي `.node` قديم، عدم تطابق اسم تصدير Rust، أو تحميل ثنائي خاطئ | شغِّل مع `PI_DEV=1` لرؤية المسار المُحمَّل؛ افحص قائمة التصديرات لذلك الملف | أعد بناء `build`؛ تأكد أن اسم تصدير `#[napi]` في Rust (أو الاسم المستعار الصريح عند الحاجة) يتطابق مع مفتاح JS؛ احذف الملفات المُخبَّأة/المُسوَّمة بالإصدار القديمة |
| جهاز x64 يُحمِّل baseline عندما يُتوقَّع modern | `PI_NATIVE_VARIANT=baseline`، لم يتم كشف AVX2، أو وجود ملف baseline فقط | تحقق من `PI_NATIVE_VARIANT`؛ افحص `native/` لوجود ملف `-modern` | ابنِ متغير modern (`TARGET_VARIANT=modern ... build`) وتأكد من شحن الملف |
| البناء المتقاطع ينتج ثنائيًا غير قابل للاستخدام/بتسمية خاطئة | عدم تطابق بين `CROSS_TARGET` و `TARGET_PLATFORM`/`TARGET_ARCH`، أو `TARGET_VARIANT` مفقود لـ x64 | تأكد من مجموعة متغيرات البيئة واسم ملف المخرجات | أعد التشغيل بقيم بيئة متسقة و `TARGET_VARIANT` صريح لـ x64 |
| فشل الثنائي المُجمَّع بعد الترقية | ذاكرة تخزين مؤقت مُستخرَجة قديمة (`~/.xcsh/natives/<old-or-mismatched-version>`) أو عدم تطابق البيان المُضمَّن | افحص مجلد الوحدات الأصلية المُسوَّم بالإصدار وقائمة أخطاء المُحمِّل | احذف ذاكرة التخزين المؤقت للوحدات الأصلية المُسوَّمة بإصدار الحزمة وأعد التشغيل؛ أعد توليد البيان المُضمَّن أثناء التعبئة |
| المُحمِّل يفحص مسارات عديدة ولا يعمل أي منها | عدم تطابق المنصة أو عنصر إصدار مفقود في `native/` الحزمة | تحقق من `platformTag` مقابل اسم/أسماء الملفات الفعلية | تأكد أن اسم الملف المبني يتطابق تمامًا مع اصطلاح `pi_natives.<platform>-<arch>(-variant).node` وأن الحزمة تتضمن `native/` |
| `embed:native` يفشل بـ "Incomplete native addons" | ملفات المتغيرات المطلوبة لم تُبنَ قبل التضمين | تحقق من القائمة المتوقعة مقابل الموجودة في نص الخطأ | ابنِ الملفات المطلوبة أولاً (x64: كلاً من modern+baseline؛ غير x64: الافتراضي)، ثم أعد تشغيل `embed:native` |

## الأوامر التشغيلية

```bash
# عنصر الإصدار للمُضيف الحالي
bun --cwd=packages/natives run build

# بناء عنصر ملف تعريف التصحيح
bun --cwd=packages/natives run dev:native

# بناء متغيرات x64 صريحة
TARGET_VARIANT=modern bun --cwd=packages/natives run build
TARGET_VARIANT=baseline bun --cwd=packages/natives run build

# توليد بيان الإضافة المُضمَّنة من الملفات الأصلية المبنية
bun --cwd=packages/natives run embed:native

# إعادة تعيين البيان المُضمَّن إلى بيان فارغ
bun --cwd=packages/natives run embed:native -- --reset
```
