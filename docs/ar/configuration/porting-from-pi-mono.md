# النقل من pi-mono: دليل عملي للدمج

هذا الدليل هو قائمة مراجعة قابلة للتكرار لنقل التغييرات من pi-mono إلى هذا المستودع.
استخدمه لأي عملية دمج: ملف واحد، فرع ميزة، أو مزامنة إصدار كاملة.

## آخر نقطة مزامنة

**الإيداع:** `b21b42d032919de2f2e6920a76fa9a37c3920c0a`
**التاريخ:** 2026-03-22

حدّث هذا القسم بعد كل مزامنة؛ لا تعد استخدام النطاق السابق.

عند بدء مزامنة جديدة، أنشئ الرقع من هذا الإيداع فصاعداً:

## 0) حدد النطاق

- حدد المرجع الأصلي (إيداع، وسم، أو طلب سحب).
- أدرج الحزم أو المجلدات التي تنوي التعديل عليها.
- قرر أي الميزات ضمن النطاق وأيها يتم تخطيها عمداً.

## 1) انقل الكود بأمان

- فضّل فرقاً نظيفاً ومركّزاً بدلاً من النسخ الشامل.
- تجنب نسخ الملفات المُصنّعة أو المُولّدة.
- إذا أضاف المصدر الأصلي ملفات جديدة، أضفها صراحةً وراجع محتوياتها.

## 2) طابق اصطلاحات امتدادات الاستيراد

معظم ملفات TypeScript التشغيلية تحذف `.js` في الاستيرادات الداخلية، لكن بعض نقاط الدخول للاختبارات/القياسات تحتفظ بـ `.js` للتوافق مع ESM
أثناء التشغيل. اتبع النمط الحالي للحزمة المحلية؛ لا تزل الامتدادات بشكل شامل.

- في ملفات `packages/coding-agent` التشغيلية، احتفظ بالاستيرادات الداخلية بدون امتداد ما لم تستورد أصولاً غير TS.
- في `packages/tui/test` و `packages/natives/bench`، احتفظ بـ `.js` حيث تستخدمها الملفات المحيطة بالفعل.
- احتفظ بامتدادات الملفات الحقيقية عندما تتطلبها الأدوات (مثل `.json`، `.css`، تضمينات نص `.md`).
- مثال: `import { x } from "./foo.js";` → `import { x } from "./foo";` (فقط عندما يكون اصطلاح الحزمة بدون امتداد).

## 3) استبدل نطاقات الاستيراد

يستخدم المصدر الأصلي نطاقات حزم مختلفة. استبدلها بشكل متسق.

- استبدل النطاقات القديمة بالنطاق المحلي المستخدم هنا.
- أمثلة (عدّل لتتوافق مع الحزم الفعلية التي تنقلها):
  - `@mariozechner/pi-coding-agent` → `@f5xc-salesdemos/xcsh`
  - `@mariozechner/pi-agent-core` → `@f5xc-salesdemos/pi-agent-core`
  - `@mariozechner/pi-tui` → `@f5xc-salesdemos/pi-tui`
  - `@mariozechner/pi-ai` → `@f5xc-salesdemos/pi-ai`

## 4) استخدم واجهات Bun حيث تتفوق على Node

نعمل على Bun. استبدل واجهات Node فقط عندما يوفر Bun بديلاً أفضل.

**استبدل هذه:**

- تشغيل العمليات: `child_process.spawn` → Bun Shell `$` للأوامر البسيطة، `Bun.spawn`/`Bun.spawnSync` للعمل المتدفق أو طويل الأمد
- إدخال/إخراج الملفات: `fs.readFileSync` → `Bun.file().text()` / `Bun.write()`
- عملاء HTTP: `node-fetch`، `axios` → `fetch` الأصلي
- تشفير التجزئة: `node:crypto` → Web Crypto أو `Bun.hash`
- SQLite: `better-sqlite3` → `bun:sqlite`
- تحميل المتغيرات البيئية: `dotenv` → Bun يحمل `.env` تلقائياً

**لا تستبدل هذه (تعمل بشكل جيد في Bun):**

- `os.homedir()` — لا تستبدلها بـ `Bun.env.HOME` أو `Bun.env.HOME` أو القيمة الحرفية `"~"`
- `os.tmpdir()` — لا تستبدلها بـ `Bun.env.TMPDIR || "/tmp"` أو مسارات مكتوبة يدوياً
- `fs.mkdtempSync()` — لا تستبدلها ببناء المسار يدوياً
- `path.join()`، `path.resolve()`، إلخ. — هذه تعمل بشكل جيد

**نمط الاستيراد:** استخدم البادئة `node:` مع استيرادات مساحة الأسماء فقط (لا استيرادات مسماة من `node:fs` أو `node:path`).

**اصطلاحات Bun إضافية:**

- فضّل Bun Shell `$` للأوامر القصيرة غير المتدفقة؛ استخدم `Bun.spawn` فقط عندما تحتاج إدخال/إخراج متدفق أو التحكم في العملية.
- استخدم `Bun.file()`/`Bun.write()` للملفات و `node:fs/promises` للمجلدات.
- تجنب فحوصات `Bun.file().exists()`؛ استخدم معالجة `isEnoent` في try/catch.
- فضّل `Bun.sleep(ms)` على أغلفة `setTimeout`.

**خطأ:**

```typescript
// BROKEN: env vars may be undefined, "~" is not expanded
const home = Bun.env.HOME || "~";
const tmp = Bun.env.TMPDIR || "/tmp";
```

**صحيح:**

```typescript
import * as os from "node:os";
import * as fs from "node:fs";
import * as path from "node:path";

const configDir = path.join(os.homedir(), ".config", "myapp");
const tempDir = fs.mkdtempSync(path.join(os.tmpdir(), "myapp-"));
```

## 5) فضّل تضمينات Bun (بدون نسخ)

لا تنسخ الأصول التشغيلية أو ملفات الطرف الثالث أثناء البناء.

- إذا كان المصدر الأصلي ينسخ الأصول إلى مجلد dist، استبدل بتضمينات متوافقة مع Bun.
- الموجهات هي ملفات `.md` ثابتة؛ استخدم استيرادات نص Bun (`with { type: "text" }`) وHandlebars بدلاً من سلاسل الموجهات المضمنة.
- استخدم `import.meta.dir` + `Bun.file` لتحميل الموارد غير النصية المجاورة.
- احتفظ بالأصول في المستودع ودع المُجمّع يضمنها.
- أزل سكريبتات النسخ إلا إذا طلبها المستخدم صراحةً.
- إذا كان المصدر الأصلي يقرأ ملفاً احتياطياً مُجمّعاً أثناء التشغيل، استبدل قراءات نظام الملفات باستيراد نص Bun مضمن.
  - مثال (الموجه الاحتياطي لتعليمات Codex):
    - `const FALLBACK_PROMPT_PATH = join(import.meta.dir, "codex-instructions.md");` -> تم إزالته
    - `import FALLBACK_INSTRUCTIONS from "./codex-instructions.md" with { type: "text" };`
    - استخدم `return FALLBACK_INSTRUCTIONS;` بدلاً من `readFileSync(FALLBACK_PROMPT_PATH, "utf8")`

## 6) انقل `package.json` بحذر

تعامل مع `package.json` كعقد. ادمج بشكل مقصود.

- احتفظ بالقيم الحالية لـ `name`، `version`، `type`، `exports`، و `bin` ما لم يتطلب النقل تغييرات.
- استبدل سكريبتات npm/node ببدائل Bun (مثل `bun check`، `bun test`).
- تأكد أن التبعيات تستخدم النطاق الصحيح.
- لا تخفض إصدارات التبعيات لإصلاح أخطاء الأنواع؛ قم بالترقية بدلاً من ذلك.
- تحقق من روابط حزم مساحة العمل و `peerDependencies`.

## 7) وحّد نمط الكود والأدوات

- احتفظ باصطلاحات التنسيق الحالية.
- لا تدخل `any` إلا إذا كان مطلوباً.
- تجنب الاستيرادات الديناميكية واستيرادات الأنواع المضمنة؛ استخدم الاستيرادات في المستوى الأعلى فقط.
- لا تبنِ الموجهات في الكود أبداً؛ الموجهات هي ملفات `.md` ثابتة تُعرض بـ Handlebars.
- في coding-agent، لا تستخدم `console.log`/`console.warn`/`console.error` أبداً؛ استخدم `logger` من `@f5xc-salesdemos/pi-utils`.
- استخدم `Promise.withResolvers()` بدلاً من `new Promise((resolve, reject) => ...)`.
- **لا تستخدم الكلمات المفتاحية `private`/`protected`/`public` على حقول أو دوال الفئة.** استخدم حقول ES الخاصة `#` للتغليف؛ اترك الأعضاء القابلة للوصول بدون كلمة مفتاحية. الاستثناء الوحيد هو خصائص معامل المُنشئ (`constructor(private readonly x: T)`)، حيث تكون الكلمة المفتاحية مطلوبة من TypeScript. عند نقل كود المصدر الأصلي الذي يستخدم `private foo` أو `protected bar`، حوّل إلى `#foo` (خاص) أو `bar` عارٍ (قابل للوصول).
- فضّل المساعدات والأدوات الموجودة على الكود المخصص الجديد.
- حافظ على تغييرات البنية التحتية القائمة على Bun الموجودة بالفعل في هذا المستودع:
  - بيئة التشغيل هي Bun (لا نقاط دخول Node).
  - مدير الحزم هو Bun (لا ملفات قفل npm).
  - واجهات Node الثقيلة (`child_process`، `readline`) مُستبدلة ببدائل Bun.
  - واجهات Node الخفيفة (`os.homedir`، `os.tmpdir`، `fs.mkdtempSync`، `path.*`) محتفظ بها.
  - شيبانغ CLI يستخدم `bun` (ليس `node`، وليس `tsx`).
  - الحزم تستخدم ملفات المصدر مباشرة (لا خطوة بناء TypeScript).
  - سير عمل CI يشغل Bun للتثبيت/الفحص/الاختبار.

## 8) أزل طبقات التوافق القديمة

ما لم يُطلب، أزل الحشوات التوافقية من المصدر الأصلي.

- احذف الواجهات القديمة التي تم استبدالها.
- حدّث جميع مواقع الاستدعاء إلى الواجهة الجديدة مباشرة.
- لا تحتفظ بنسخ `*_v2` أو نسخ موازية.

## 9) حدّث الوثائق والمراجع

- استبدل روابط مستودع pi-mono حيثما كان مناسباً.
- حدّث الأمثلة لاستخدام Bun ونطاقات الحزم الصحيحة.
- تأكد أن تعليمات README لا تزال تتوافق مع سلوك المستودع الحالي.

## 10) تحقق من النقل

شغّل الفحوصات القياسية بعد التغييرات:

- `bun check`

إذا كان المستودع يحتوي بالفعل على فحوصات فاشلة غير متعلقة بتغييراتك، أشر إلى ذلك.
الاختبارات تستخدم مشغّل Bun (وليس Vitest)، لكن شغّل `bun test` فقط عند الطلب صراحةً.

## 11) احمِ الميزات المحسّنة (قائمة فخ الانحدار)

إذا حسّنت سلوكاً محلياً بالفعل، تعامل مع تلك التحسينات كأمور **غير قابلة للتفاوض**. قبل النقل، دوّن
التحسينات وأضف فحوصات صريحة حتى لا تضيع في الدمج.

- **جمّد السلوك المتوقع**: أضف ملاحظة "قبل/بعد" قصيرة لكل تحسين (المدخلات، المخرجات،
  القيم الافتراضية، الحالات الحدية). هذا يمنع التراجع الصامت.
- **ارسم خريطة القديم → الجديد**: إذا أعاد المصدر الأصلي تسمية المفاهيم (hooks → extensions، custom tools → tools، إلخ)،
  تأكد أن كل نقطة دخول قديمة لا تزال متصلة. علم أو تصدير واحد مفقود يعني فقدان الوظائف.
- **تحقق من التصديرات**: افحص `exports` في `package.json`، الأنواع العامة، وملفات البرميل. عمليات النقل من المصدر الأصلي غالباً
  تنسى إعادة تصدير الإضافات المحلية.
- **غطِّ المسارات غير السعيدة**: إذا أصلحت معالجة الأخطاء، المهلات، أو منطق الاحتياط، أضف اختباراً أو
  على الأقل قائمة مراجعة يدوية تمارس تلك المسارات.
- **تحقق من القيم الافتراضية وترتيب دمج التكوين**: التحسينات غالباً تكمن في القيم الافتراضية. تأكد أن القيم الافتراضية الجديدة
  لم تتراجع (مثل أسبقية التكوين الجديدة، الميزات المعطلة، قوائم الأدوات).
- **دقق في سلوك البيئة/الصدفة**: إذا أصلحت التنفيذ أو العزل، تحقق أن المسار الجديد لا يزال يستخدم بيئتك
  المطهّرة ولا يعيد إدخال تجاوزات الاسم المستعار/الدالة.
- **أعد تشغيل عينات مستهدفة**: احتفظ بمجموعة حد أدنى من الأمثلة "المعروفة بأنها جيدة" وشغّلها بعد النقل
  (أعلام CLI، تسجيل الإضافات، تنفيذ الأدوات).

## 12) اكتشف وتعامل مع الكود المُعاد صياغته

قبل نقل ملف، تحقق مما إذا كان المصدر الأصلي قد أعاد هيكلته بشكل كبير:

```bash
# Compare the file you're about to port against what you have locally
git diff HEAD upstream/main -- path/to/file.ts
```

إذا أظهر الفرق أن الملف تم **إعادة صياغته** (ليس مجرد ترقيع):

- تجريدات جديدة، مفاهيم مُعاد تسميتها، وحدات مدمجة، تدفق بيانات متغير

فيجب عليك **قراءة التنفيذ الجديد بالكامل** قبل النقل. الدمج الأعمى للكود المُعاد صياغته يفقد الوظائف لأن:

ملاحظة: تم مؤخراً تقسيم الوضع التفاعلي إلى controllers/utils/types. عند النقل العكسي للتغييرات المتعلقة، انقل التحديثات إلى الملفات الفردية التي أنشأناها وتأكد أن ربط `interactive-mode.ts` يبقى متزامناً.

1. **القيم الافتراضية تتغير بصمت** - متغير جديد `defaultFoo = [a, b]` قد يحل محل `getAllFoo()` القديم الذي كان يُرجع `[a, b, c, d, e]`.

2. **خيارات الواجهة تُسقط** - عندما تندمج الأنظمة (مثل `hooks` + `customTools` → `extensions`)، قد لا تتصل الخيارات القديمة بالتنفيذ الجديد.

3. **مسارات الكود تصبح قديمة** - مفهوم مُعاد تسميته (مثل `hookMessage` → `custom`) يحتاج تحديثات في كل عبارة switch، حارس نوع، ومعالج — ليس فقط التعريف.

4. **السياق/القدرات تتقلص** - الواجهات القديمة ربما كانت تكشف `{ logger, typebox, pi }` التي نسيت الواجهات الجديدة تضمينها.

### عملية النقل الدلالي

عندما يعيد المصدر الأصلي صياغة وحدة:

1. **اقرأ التنفيذ القديم** - افهم ما كان يفعله، ما الخيارات التي كان يقبلها، ما الذي كان يكشفه.

2. **اقرأ التنفيذ الجديد** - افهم التجريدات الجديدة وكيف ترتبط بالسلوك القديم.

3. **تحقق من تكافؤ الميزات** - لكل قدرة في الكود القديم، تأكد أن الكود الجديد يحافظ عليها أو يزيلها صراحةً.

4. **ابحث عن المتبقيات** - ابحث عن الأسماء/المفاهيم القديمة التي ربما فُوتت في عبارات switch، المعالجات، مكونات واجهة المستخدم.

5. **اختبر الحدود** - أعلام CLI، خيارات SDK، معالجات الأحداث، القيم الافتراضية — هذه هي حيث تختبئ الانحدارات.

### فحوصات سريعة

```bash
# Find all uses of an old concept that may need updating
rg "oldConceptName" --type ts

# Compare default values between versions
git show upstream/main:path/to/file.ts | rg "default|DEFAULT"

# Check if all enum/union values have handlers
rg "case \"" path/to/file.ts
```

## 13) قائمة مراجعة التدقيق السريع

استخدم هذه كمراجعة أخيرة قبل الانتهاء:

- [ ] امتدادات الاستيراد تتبع اصطلاح الحزمة المحلية (لا إزالة شاملة لـ `.js`)
- [ ] لا واجهات خاصة بـ Node فقط في الكود الجديد/المنقول
- [ ] جميع نطاقات الحزم مُحدّثة
- [ ] سكريبتات `package.json` تستخدم Bun
- [ ] الموجهات هي استيرادات نص `.md` (لا سلاسل موجهات مضمنة)
- [ ] لا `console.*` في coding-agent (استخدم `logger`)
- [ ] الأصول تُحمّل عبر أنماط تضمين Bun (لا سكريبتات نسخ)
- [ ] الاختبارات أو الفحوصات تعمل (أو مذكور صراحةً أنها محظورة)
- [ ] لا انحدارات في الوظائف (انظر الأقسام 11-12)

## 14) صيغة رسالة الإيداع

عند إيداع نقل عكسي، اتبع صيغة المستودع `<type>(scope): <past-tense description>` واحتفظ بنطاق
الإيداع في العنوان.

```
fix(coding-agent): backported pi-mono changes (<from>..<to>)

packages/<package>:
- <type>: <description>
- <type>: <description> (#<issue> by @<contributor>)

packages/<other-package>:
- <type>: <description>
```

**مثال:**

```
fix(coding-agent): backported pi-mono changes (9f3eef65f..52532c7c0)

packages/ai:
- fix: handle "sensitive" stop reason from Anthropic API
- fix: normalize tool call IDs with special characters for Responses API
- fix: add overflow detection for Bedrock, MiniMax, Kimi providers
- fix: 429 status is rate limiting, not context overflow

packages/tui:
- fix: refactored autocomplete state tracking
- fix: file autocomplete should not trigger on empty text
- fix: configurable autocomplete max visible items
- fix: improved table column width calculation with word-aware wrapping

packages/coding-agent:
- fix: preserve external config.yml edits on save (#1046 by @nicobailonMD)
- fix: resolve macOS NFD and curly quote variants in file paths
```

**القواعد:**

- جمّع التغييرات حسب الحزمة
- استخدم أنواع الإيداع التقليدية (`fix`، `feat`، `refactor`، `perf`، `docs`)
- أضف أرقام المشاكل/طلبات السحب من المصدر الأصلي ونسب المساهم للمساهمات الخارجية
- نطاق الإيداع في العنوان يساعد في تتبع نقاط المزامنة

## 15) الاختلافات المقصودة

فرعنا لديه قرارات معمارية تختلف عن المصدر الأصلي. **لا تنقل هذه الأنماط من المصدر الأصلي:**

### معمارية واجهة المستخدم

| المصدر الأصلي                                | فرعنا                                                     | السبب                                                                 |
| ------------------------------------------- | --------------------------------------------------------- | --------------------------------------------------------------------- |
