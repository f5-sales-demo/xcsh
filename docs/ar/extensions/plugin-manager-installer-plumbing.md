# البنية الداخلية لمدير الإضافات والمُثبِّت

يصف هذا المستند كيف تُعدّل عمليات `xcsh plugin` حالة الإضافات على القرص، وكيف تصبح الإضافات المُثبَّتة قدرات وقت التشغيل (الأدوات حالياً، مع توفر دقة مسارات الخطافات/الأوامر).

## النطاق والبنية المعمارية

توجد تنفيذان لإدارة الإضافات في قاعدة الشيفرة:

1. **المسار النشط المُستخدم بواسطة أوامر CLI**: `PluginManager` (`src/extensibility/plugins/manager.ts`)
2. **وحدة مساعدة قديمة**: دوال المُثبِّت (`src/extensibility/plugins/installer.ts`)

تنفيذ أمر `xcsh plugin ...` يمر عبر `PluginManager`.

لا يزال ملف `installer.ts` يوثّق فحوصات أمان مهمة وسلوكيات نظام الملفات، لكنه ليس المسار المُستخدم بواسطة `src/commands/plugin.ts` + `src/cli/plugin-cli.ts`.

## دورة الحياة: من استدعاء CLI إلى التوفر في وقت التشغيل

```text
xcsh plugin <action> ...
  -> src/commands/plugin.ts
  -> runPluginCommand(...) in src/cli/plugin-cli.ts
  -> PluginManager method (install/list/uninstall/link/...) 
  -> mutate ~/.xcsh/plugins/{package.json,node_modules,xcsh-plugins.lock.json}
  -> runtime discovery: discoverAndLoadCustomTools(...)
  -> getAllPluginToolPaths(cwd)
  -> custom tool loader imports tool modules
```

### نقاط دخول الأوامر

- `src/commands/plugin.ts` يُعرّف الأمر/الخيارات ويُمررها إلى `runPluginCommand`.
- `src/cli/plugin-cli.ts` يربط الأوامر الفرعية بدوال `PluginManager`:
  - `install`، `uninstall`، `list`، `link`، `doctor`، `features`، `config`، `enable`، `disable`
- لا يوجد إجراء `update` صريح؛ يتم التحديث بإعادة تشغيل `install` مع مُحدِّد حزمة/إصدار جديد.

## النموذج على القرص

تقع حالة الإضافات العامة تحت `~/.xcsh/plugins`:

- `package.json` — بيان التبعيات المُستخدم بواسطة `bun install`/`bun uninstall`
- `node_modules/` — حزم الإضافات المُثبَّتة أو الروابط الرمزية
- `xcsh-plugins.lock.json` — حالة وقت التشغيل:
  - التفعيل/التعطيل لكل إضافة
  - مجموعة الميزات المُختارة لكل إضافة
  - إعدادات الإضافة المحفوظة

التجاوزات الخاصة بالمشروع تقع في:

- `<cwd>/.xcsh/plugin-overrides.json`

التجاوزات هي للقراءة فقط من منظور المدير/المُحمِّل (لا يوجد مسار كتابة هنا) ويمكنها تعطيل الإضافات أو تجاوز الميزات/الإعدادات لهذا المشروع.

## تحليل مُحدِّد الإضافة وتفسير البيانات الوصفية

## قواعد مُحدِّد التثبيت

`parsePluginSpec` (`parser.ts`) يدعم:

- `pkg` -> `features: null` (سلوك افتراضي)
- `pkg[*]` -> تفعيل جميع ميزات البيان
- `pkg[]` -> عدم تفعيل أي ميزات اختيارية
- `pkg[a,b]` -> تفعيل ميزات محددة بالاسم
- `@scope/pkg@1.2.3[feat]` -> حزمة بنطاق + إصدار مع اختيار ميزة صريح

`extractPackageName` يزيل لاحقة الإصدار للبحث عن المسار على القرص بعد التثبيت.

## مصدر البيان والحقول المطلوبة

يتم حل البيان كالتالي:

1. `package.json.xcsh`
2. البديل `package.json.pi`
3. البديل `{ version: package.version }`

الآثار المترتبة:

- لا يوجد تحقق صارم من المخطط في المدير/المُحمِّل.
- الحزمة التي تفتقد `xcsh`/`pi` لا تزال قابلة للتثبيت والإدراج.
- تحميل الإضافات في وقت التشغيل (`getEnabledPlugins`) يتخطى الحزم بدون بيان `xcsh`/`pi`.
- `manifest.version` يتم الكتابة فوقه دائماً من `version` الحزمة.

ملف `package.json` المشوه بصيغة JSON يُسبب فشلاً فورياً عند القراءة؛ شكل البيان المشوه قد يفشل لاحقاً فقط عند استهلاك حقول محددة.

## مسار التثبيت/التحديث (`PluginManager.install`)

1. تحليل صيغة الأقواس المربعة للميزات من مُحدِّد التثبيت.
2. التحقق من اسم الحزمة مقابل تعبير نمطي + قائمة حظر أحرف الصدفة الخاصة.
3. التأكد من وجود `package.json` للإضافة (`xcsh-plugins`، خريطة التبعيات الخاصة).
4. تشغيل `bun install <packageSpec>` في `~/.xcsh/plugins`.
5. قراءة `node_modules/<name>/package.json` للحزمة المُثبَّتة.
6. حل البيان وحساب `enabledFeatures`:
   - `[*]`: جميع الميزات المُعلنة (أو `null` إذا لم توجد خريطة ميزات)
   - `[a,b]`: التحقق من وجود كل ميزة في خريطة ميزات البيان
   - `[]`: قائمة ميزات فارغة
   - مُحدِّد بسيط: `null` (استخدام سياسة الافتراضي لاحقاً في المُحمِّل)
7. إدراج/تحديث حالة وقت التشغيل في ملف القفل: `{ version, enabledFeatures, enabled: true }`.

### دلالات التحديث

لأن التحديث يتم عبر التثبيت:

- `xcsh plugin install pkg@newVersion` يُحدّث التبعية وإصدار ملف القفل.
- يتم الحفاظ على الإعدادات الموجودة؛ يتم الكتابة فوق مدخل الحالة للإصدار/الميزات/التفعيل.
- لا يوجد منطق منفصل لـ "التحقق من التحديثات" أو ترحيل معاملاتي.

## مسار الإزالة (`PluginManager.uninstall`)

1. التحقق من اسم الحزمة.
2. تشغيل `bun uninstall <name>` في مجلد الإضافات.
3. إزالة حالة وقت تشغيل الإضافة من ملف القفل:
   - `config.plugins[name]`
   - `config.settings[name]`

إذا فشل أمر إلغاء التثبيت، لا تتغير حالة وقت التشغيل.

## مسار الإدراج (`PluginManager.list`)

1. قراءة خريطة تبعيات الإضافات من `~/.xcsh/plugins/package.json`.
2. تحميل تكوين وقت التشغيل من ملف القفل (ملف مفقود -> قيم افتراضية فارغة).
3. تحميل تجاوزات المشروع (`<cwd>/.xcsh/plugin-overrides.json`، أخطاء التحليل/القراءة -> كائن فارغ مع تحذير).
4. لكل تبعية لها package.json قابل للحل:
   - بناء سجل `InstalledPlugin`
   - دمج حالة الميزات/التفعيل:
     - الأساس من ملف القفل (أو الافتراضيات)
     - تجاوزات المشروع يمكنها استبدال اختيار الميزات
     - قائمة `disabled` للمشروع تُخفي الإضافة كمُعطَّلة

هذه هي الحالة الفعلية المُستخدمة بواسطة مخرجات حالة CLI وعمليات الإعدادات/الميزات.

## مسار الربط (`PluginManager.link`)

`link` يدعم تطوير الإضافات المحلية عن طريق إنشاء رابط رمزي لحزمة محلية في `~/.xcsh/plugins/node_modules/<pkg.name>`.

السلوك:

1. حل `localPath` مقابل cwd الخاص بالمدير.
2. طلب `package.json` محلي وحقل `name`.
3. التأكد من وجود مجلدات الإضافات.
4. للأسماء ذات النطاق، إنشاء مجلد النطاق.
5. إزالة المسار الموجود في موقع الرابط المستهدف.
6. إنشاء الرابط الرمزي.
7. إضافة مدخل في ملف القفل لوقت التشغيل مُفعَّل مع ميزات افتراضية (`null`).

تنبيه: `PluginManager.link` الحالي لا يفرض فحص حدود المسار `cwd` الموجود في `installer.ts` القديم (`normalizedPath.startsWith(normalizedCwd)`)، لذا المسؤولية تقع على المُستدعي.

## التحميل في وقت التشغيل: من الإضافة المُثبَّتة إلى القدرات القابلة للاستدعاء

## بوابة الاكتشاف

`getEnabledPlugins(cwd)` (`plugins/loader.ts`) يقرأ:

- بيان تبعيات الإضافة (`package.json`)
- حالة وقت التشغيل من ملف القفل
- تجاوزات المشروع عبر `getConfigDirPaths("plugin-overrides.json", { user: false, cwd })`

التصفية:

- تخطي إذا لم يوجد package.json للإضافة
- تخطي إذا كان البيان (`xcsh`/`pi`) غائباً
- تخطي إذا كانت مُعطَّلة عالمياً في ملف القفل
- تخطي إذا كانت مُعطَّلة على مستوى المشروع

## دقة مسار القدرات

لكل إضافة مُفعَّلة:

- `resolvePluginToolPaths(plugin)`
- `resolvePluginHookPaths(plugin)`
- `resolvePluginCommandPaths(plugin)`

كل مُحلِّل يتضمن مدخلات أساسية بالإضافة إلى مدخلات الميزات:

- قائمة ميزات صريحة -> الميزات المُختارة فقط
- `enabledFeatures === null` -> تفعيل الميزات المُعلَّمة بـ `default: true`

الملفات المفقودة يتم تخطيها بصمت (حارس `existsSync`).

## اختلافات الربط الحالية في وقت التشغيل

- **الأدوات مُربوطة في وقت التشغيل حالياً** عبر `discoverAndLoadCustomTools` (`custom-tools/loader.ts`)، التي تستدعي `getAllPluginToolPaths(cwd)`.
- يتم إزالة التكرارات من المسارات بواسطة المسار المطلق المحلول في اكتشاف الأدوات المخصصة (مجموعة `seen`، المسار الأول يفوز).
- **مُحلِّلات الخطافات/الأوامر موجودة** ومُصدَّرة، لكن مسار الشيفرة هذا لا يربطها حالياً بسجل وقت التشغيل بنفس طريقة ربط الأدوات.

## تفاصيل إدارة القفل/الحالة

`PluginManager` يُخزّن تكوين وقت التشغيل في الذاكرة لكل نسخة (`#runtimeConfig`) ويُحمِّله بشكل كسول مرة واحدة.

سلوك التحميل:

- ملف القفل مفقود -> `{ plugins: {}, settings: {} }`
- فشل قراءة/تحليل ملف القفل -> تحذير + نفس الافتراضيات الفارغة

سلوك الحفظ:

- يكتب JSON كامل لملف القفل بتنسيق مُرتَّب عند كل تعديل

لا يوجد قفل بين العمليات أو استراتيجية دمج؛ الكُتّاب المتزامنون يمكنهم الكتابة فوق بعضهم البعض.

## فحوصات الأمان وحدود الثقة

## التحقق من المدخلات/الحزم

مسار المدير النشط يفرض التحقق من اسم الحزمة:

- تعبير نمطي لمُحدِّدات الحزم ذات النطاق وبدونه (اختيارياً مع الإصدار)
- قائمة حظر صريحة لأحرف الصدفة الخاصة (`[;&|`$(){}[]<>\\]`)

هذا يحد من خطر حقن الأوامر عند استدعاء `bun install/uninstall`.

## حدود الثقة في نظام الملفات

- شيفرة الإضافة تُنفَّذ داخل العملية عند استيراد وحدات الأدوات المخصصة؛ لا يوجد عزل (sandboxing).
- المسارات النسبية في البيان تُضم إلى مجلد حزمة الإضافة ويتم فحص وجودها فقط.
- حزمة الإضافة نفسها هي شيفرة موثوقة بمجرد تثبيتها.

## فحوصات المُثبِّت القديم فقط

`installer.ts` يتضمن فحوصات إضافية وقت الربط غير مُنعكسة في `PluginManager.link`:

- المسار المحلي يجب أن يُحل داخل cwd المشروع
- حراسات إضافية لاسم الحزمة/اجتياز المسار لتسمية هدف الرابط الرمزي

لأن CLI يستخدم `PluginManager`، هذه الحراسات الأكثر صرامة للربط ليست حالياً على المسار الرئيسي.

## الفشل والنجاح الجزئي وسلوك التراجع

مدير الإضافات ليس معاملاتياً.

| مرحلة العملية | سلوك الفشل | التراجع |
| --- | --- | --- |
| فشل `bun install` | التثبيت يتوقف مع stderr | غير متاح (لم تتم كتابة حالة بعد) |
| نجح التثبيت، ثم فشل التحقق من البيان/الميزات | فشل الأمر | لا يوجد تراجع بإلغاء التثبيت؛ قد تبقى التبعية في `node_modules`/`package.json` |
| نجح التثبيت، ثم فشلت كتابة ملف القفل | فشل الأمر | لا يوجد تراجع للحزمة المُثبَّتة |
| نجح `bun uninstall`، فشلت كتابة ملف القفل | فشل الأمر | الحزمة أُزيلت، قد تبقى حالة وقت تشغيل قديمة |
| `link` يزيل الهدف القديم ثم يفشل إنشاء الرابط الرمزي | فشل الأمر | لا يوجد استعادة للرابط/المجلد السابق |

عملياً، `doctor --fix` يمكنه إصلاح بعض الانحرافات (`bun install`، تنظيف التكوين اليتيم، تنظيف الميزات غير الصالحة)، لكنه بأفضل جهد ممكن.

## ملخص سلوك البيان المشوه/المفقود

- حقل `xcsh`/`pi` مفقود:
  - التثبيت/الإدراج: مقبول (بيان بسيط)
  - اكتشاف الإضافات المُفعَّلة في وقت التشغيل: يتم تخطيه كغير إضافة
- ميزة مفقودة مُشار إليها بواسطة مُحدِّد التثبيت أو `features --set/--enable`: خطأ فوري مع قائمة الميزات المتاحة
- `plugin-overrides.json` غير صالح: يتم تجاهله مع الرجوع إلى `{}` في مسارات المدير والمُحمِّل
- مسارات ملفات الأدوات/الخطافات/الأوامر المفقودة المُشار إليها بواسطة البيان: يتم تجاهلها بصمت أثناء توسيع المُحلِّل؛ تُبلَّغ كأخطاء فقط بواسطة `doctor`

## اختلافات الأوضاع والأولوية

- `--dry-run` (التثبيت): يُرجع نتيجة تثبيت اصطناعية، بدون كتابة على نظام الملفات/الشبكة/الحالة.
- `--json`: تنسيق المخرجات فقط، بدون تغيير في السلوك.
- تجاوزات المشروع تأخذ الأولوية دائماً على ملف القفل العام لعرض الميزات/الإعدادات.
- التفعيل الفعلي هو `runtimeEnabled && !projectDisabled`.
