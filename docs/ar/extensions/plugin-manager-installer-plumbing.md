# البنية الداخلية لمدير الإضافات والمُثبّت

يصف هذا المستند كيف تُغيّر عمليات `xcsh plugin` حالة الإضافات على القرص وكيف تصبح الإضافات المُثبّتة قدرات وقت التشغيل (الأدوات حالياً، مع توفر تحليل مسارات الخطافات/الأوامر).

## النطاق والبنية المعمارية

يوجد تطبيقان لإدارة الإضافات في قاعدة الكود:

1. **المسار النشط المستخدم بواسطة أوامر CLI**: `PluginManager` (`src/extensibility/plugins/manager.ts`)
2. **وحدة مساعدة قديمة**: دوال المُثبّت (`src/extensibility/plugins/installer.ts`)

تنفيذ أمر `xcsh plugin ...` يمر عبر `PluginManager`.

لا يزال `installer.ts` يوثّق فحوصات أمان مهمة وسلوك نظام الملفات، لكنه ليس المسار المستخدم بواسطة `src/commands/plugin.ts` + `src/cli/plugin-cli.ts`.

## دورة الحياة: من استدعاء CLI إلى التوفر في وقت التشغيل

```text
xcsh plugin <action> ...
  -> src/commands/plugin.ts
  -> runPluginCommand(...) in src/cli/plugin-cli.ts
  -> PluginManager method (install/list/uninstall/link/...) 
  -> mutate ~/.xcsh/plugins/{package.json,node_modules,xcsh-plugins.lock.json}
  -> runtime discovery: discoverAndLoadCustomTools(...)
  -> getAllPluginToolPaths(cwd)
  -> custom tool loader imports tool modules
```

### نقاط دخول الأوامر

- `src/commands/plugin.ts` يُعرّف الأمر/الخيارات ويُحيل إلى `runPluginCommand`.
- `src/cli/plugin-cli.ts` يُعيّن الأوامر الفرعية إلى دوال `PluginManager`:
  - `install`، `uninstall`، `list`، `link`، `doctor`، `features`، `config`، `enable`، `disable`
- لا يوجد إجراء `update` صريح؛ يتم التحديث بإعادة تشغيل `install` مع مواصفة حزمة/إصدار جديدة.

## النموذج على القرص

تقع حالة الإضافات العامة تحت `~/.xcsh/plugins`:

- `package.json` — بيان التبعيات المستخدم بواسطة `bun install`/`bun uninstall`
- `node_modules/` — حزم الإضافات المُثبّتة أو الروابط الرمزية
- `xcsh-plugins.lock.json` — حالة وقت التشغيل:
  - مُفعّل/مُعطّل لكل إضافة
  - مجموعة الميزات المختارة لكل إضافة
  - إعدادات الإضافة المحفوظة

التجاوزات المحلية للمشروع توجد في:

- `<cwd>/.xcsh/plugin-overrides.json`

التجاوزات هي للقراءة فقط من منظور المدير/المُحمّل (لا يوجد مسار كتابة هنا) ويمكنها تعطيل الإضافات أو تجاوز الميزات/الإعدادات لهذا المشروع.

## تحليل مواصفات الإضافة وتفسير البيانات الوصفية

## قواعد مواصفات التثبيت

`parsePluginSpec` (`parser.ts`) يدعم:

- `pkg` -> `features: null` (سلوك افتراضي)
- `pkg[*]` -> تفعيل جميع ميزات البيان
- `pkg[]` -> عدم تفعيل أي ميزات اختيارية
- `pkg[a,b]` -> تفعيل ميزات محددة بالاسم
- `@scope/pkg@1.2.3[feat]` -> حزمة ذات نطاق + إصدار مع اختيار ميزات صريح

`extractPackageName` يزيل لاحقة الإصدار للبحث عن المسار على القرص بعد التثبيت.

## مصدر البيان والحقول المطلوبة

يتم تحليل البيان كالتالي:

1. `package.json.xcsh`
2. احتياطي `package.json.pi`
3. احتياطي `{ version: package.version }`

الآثار المترتبة:

- لا يوجد تحقق صارم من المخطط في المدير/المُحمّل.
- الحزمة التي تفتقر إلى `xcsh`/`pi` لا تزال قابلة للتثبيت والإدراج.
- تحميل الإضافات في وقت التشغيل (`getEnabledPlugins`) يتخطى الحزم بدون بيان `xcsh`/`pi`.
- `manifest.version` يُكتب فوقه دائماً من `version` الحزمة.

ملف `package.json` بتنسيق JSON تالف يُعد فشلاً حاداً عند القراءة؛ شكل البيان المشوّه قد يفشل لاحقاً فقط عند استهلاك حقول محددة.

## مسار التثبيت/التحديث (`PluginManager.install`)

1. تحليل صيغة الأقواس المربعة للميزات من مواصفة التثبيت.
2. التحقق من اسم الحزمة مقابل تعبير نمطي + قائمة رفض لأحرف الصدفة الخاصة.
3. التأكد من وجود `package.json` للإضافة (`xcsh-plugins`، خريطة التبعيات الخاصة).
4. تشغيل `bun install <packageSpec>` في `~/.xcsh/plugins`.
5. قراءة `node_modules/<name>/package.json` للحزمة المُثبّتة.
6. تحليل البيان وحساب `enabledFeatures`:
   - `[*]`: جميع الميزات المُعلنة (أو `null` إذا لم تكن هناك خريطة ميزات)
   - `[a,b]`: التحقق من وجود كل ميزة في خريطة ميزات البيان
   - `[]`: قائمة ميزات فارغة
   - مواصفة مجردة: `null` (استخدام سياسة الافتراضي لاحقاً في المُحمّل)
7. إدراج/تحديث حالة وقت التشغيل في ملف القفل: `{ version, enabledFeatures, enabled: true }`.

### دلالات التحديث

لأن التحديث مدفوع بالتثبيت:

- `xcsh plugin install pkg@newVersion` يُحدّث التبعية وإصدار ملف القفل.
- الإعدادات الحالية تُحفظ؛ يُكتب فوق مُدخل الحالة للإصدار/الميزات/التفعيل.
- لا يوجد منطق منفصل لـ "فحص التحديثات" أو ترحيل تعاملاتي.

## مسار الإزالة (`PluginManager.uninstall`)

1. التحقق من اسم الحزمة.
2. تشغيل `bun uninstall <name>` في مجلد الإضافات.
3. إزالة حالة وقت تشغيل الإضافة من ملف القفل:
   - `config.plugins[name]`
   - `config.settings[name]`

إذا فشل أمر إلغاء التثبيت، لا يتم تغيير حالة وقت التشغيل.

## مسار الإدراج (`PluginManager.list`)

1. قراءة خريطة تبعيات الإضافات من `~/.xcsh/plugins/package.json`.
2. تحميل تكوين وقت التشغيل من ملف القفل (ملف مفقود -> قيم افتراضية فارغة).
3. تحميل تجاوزات المشروع (`<cwd>/.xcsh/plugin-overrides.json`، أخطاء التحليل/القراءة -> كائن فارغ مع تحذير).
4. لكل تبعية لها package.json قابل للتحليل:
   - بناء سجل `InstalledPlugin`
   - دمج حالة الميزات/التفعيل:
     - الأساس من ملف القفل (أو القيم الافتراضية)
     - تجاوزات المشروع يمكنها استبدال اختيار الميزات
     - قائمة `disabled` في المشروع تُخفي الإضافة كمُعطّلة

هذه هي الحالة الفعلية المستخدمة بواسطة مخرجات حالة CLI وعمليات الإعدادات/الميزات.

## مسار الربط (`PluginManager.link`)

يدعم `link` تطوير الإضافات المحلية عن طريق إنشاء رابط رمزي لحزمة محلية في `~/.xcsh/plugins/node_modules/<pkg.name>`.

السلوك:

1. تحليل `localPath` بالنسبة لـ cwd الخاص بالمدير.
2. طلب `package.json` المحلي وحقل `name`.
3. التأكد من وجود مجلدات الإضافات.
4. للأسماء ذات النطاق، إنشاء مجلد النطاق.
5. إزالة المسار الحالي في موقع الرابط الهدف.
6. إنشاء الرابط الرمزي.
7. إضافة مُدخل ملف قفل وقت التشغيل مُفعّل مع ميزات افتراضية (`null`).

تحذير: `PluginManager.link` الحالي لا يفرض فحص حدود المسار `cwd` الموجود في `installer.ts` القديم (`normalizedPath.startsWith(normalizedCwd)`)، لذا الثقة تقع على عاتق المُستدعي.

## التحميل في وقت التشغيل: من إضافة مُثبّتة إلى قدرات قابلة للاستدعاء

## بوابة الاكتشاف

`getEnabledPlugins(cwd)` (`plugins/loader.ts`) يقرأ:

- بيان تبعيات الإضافة (`package.json`)
- حالة وقت التشغيل من ملف القفل
- تجاوزات المشروع عبر `getConfigDirPaths("plugin-overrides.json", { user: false, cwd })`

التصفية:

- تخطي إذا لم يكن هناك package.json للإضافة
- تخطي إذا كان البيان (`xcsh`/`pi`) غائباً
- تخطي إذا كانت مُعطّلة عالمياً في ملف القفل
- تخطي إذا كانت مُعطّلة على مستوى المشروع

## تحليل مسارات القدرات

لكل إضافة مُفعّلة:

- `resolvePluginToolPaths(plugin)`
- `resolvePluginHookPaths(plugin)`
- `resolvePluginCommandPaths(plugin)`

كل مُحلل يتضمن مُدخلات أساسية بالإضافة إلى مُدخلات الميزات:

- قائمة ميزات صريحة -> الميزات المختارة فقط
- `enabledFeatures === null` -> تفعيل الميزات المُعلّمة بـ `default: true`

الملفات المفقودة يتم تخطيها بصمت (حماية `existsSync`).

## اختلافات الربط الحالية في وقت التشغيل

- **الأدوات مربوطة بوقت التشغيل حالياً** عبر `discoverAndLoadCustomTools` (`custom-tools/loader.ts`)، التي تستدعي `getAllPluginToolPaths(cwd)`.
- يتم إزالة المسارات المكررة بواسطة المسار المطلق المُحلل في اكتشاف الأدوات المخصصة (مجموعة `seen`، المسار الأول يفوز).
- **مُحللات الخطافات/الأوامر موجودة** ومُصدّرة، لكن مسار الكود هذا لا يربطها حالياً بسجل وقت تشغيل بنفس الطريقة التي تُربط بها الأدوات.

## تفاصيل إدارة القفل/الحالة

`PluginManager` يُخزّن تكوين وقت التشغيل مؤقتاً في الذاكرة لكل نسخة (`#runtimeConfig`) ويُحمّله بتكاسل مرة واحدة.

سلوك التحميل:

- ملف القفل مفقود -> `{ plugins: {}, settings: {} }`
- فشل قراءة/تحليل ملف القفل -> تحذير + نفس القيم الافتراضية الفارغة

سلوك الحفظ:

- يكتب ملف قفل JSON كاملاً بتنسيق جميل عند كل تغيير

لا يوجد قفل بين العمليات أو استراتيجية دمج؛ الكتّاب المتزامنون يمكنهم الكتابة فوق بعضهم البعض.

## فحوصات الأمان وحدود الثقة

## التحقق من المدخلات/الحزمة

مسار المدير النشط يفرض التحقق من اسم الحزمة:

- تعبير نمطي لمواصفات الحزم ذات النطاق وبدونه (اختيارياً مع الإصدار)
- قائمة رفض صريحة لأحرف الصدفة الخاصة (`[;&|`$(){}[]<>\\]`)

هذا يُحد من خطر حقن الأوامر عند استدعاء `bun install/uninstall`.

## حدود ثقة نظام الملفات

- كود الإضافة يُنفّذ داخل العملية عند استيراد وحدات الأدوات المخصصة؛ لا يوجد عزل.
- المسارات النسبية في البيان تُضم إلى مجلد حزمة الإضافة ويتم فحص وجودها فقط.
- حزمة الإضافة نفسها تُعتبر كوداً موثوقاً بمجرد تثبيتها.

## فحوصات المُثبّت القديم فقط

`installer.ts` يتضمن فحوصات وقت الربط إضافية غير مُنعكسة في `PluginManager.link`:

- المسار المحلي يجب أن يُحلّ داخل cwd المشروع
- حماية إضافية لاسم الحزمة/اجتياز المسار لتسمية هدف الرابط الرمزي

لأن CLI يستخدم `PluginManager`، هذه الحمايات الأكثر صرامة للربط ليست حالياً على المسار الرئيسي.

## سلوك الفشل والنجاح الجزئي والتراجع

مدير الإضافات ليس تعاملاتياً.

| مرحلة العملية | سلوك الفشل | التراجع |
| --- | --- | --- |
| فشل `bun install` | التثبيت يُجهض مع stderr | غير متاح (لم تتم كتابات حالة بعد) |
| نجاح التثبيت، ثم فشل التحقق من البيان/الميزات | فشل الأمر | لا يوجد تراجع بإلغاء التثبيت؛ قد تبقى التبعية في `node_modules`/`package.json` |
| نجاح التثبيت، ثم فشل كتابة ملف القفل | فشل الأمر | لا يوجد تراجع للحزمة المُثبّتة |
| نجاح `bun uninstall`، فشل كتابة ملف القفل | فشل الأمر | الحزمة أُزيلت، قد تبقى حالة وقت تشغيل قديمة |
| `link` يزيل الهدف القديم ثم يفشل إنشاء الرابط الرمزي | فشل الأمر | لا يوجد استعادة للرابط/المجلد السابق |

عملياتياً، `doctor --fix` يمكنه إصلاح بعض الانحراف (`bun install`، تنظيف التكوين اليتيم، تنظيف الميزات غير الصالحة)، لكنه يعمل بأفضل جهد ممكن.

## ملخص سلوك البيان المشوّه/المفقود

- حقل `xcsh`/`pi` مفقود:
  - التثبيت/الإدراج: مقبول (بيان أدنى)
  - اكتشاف الإضافات المُفعّلة في وقت التشغيل: يُتخطى كغير إضافة
- ميزة مفقودة مُشار إليها بمواصفة التثبيت أو `features --set/--enable`: خطأ حاد مع قائمة الميزات المتاحة
- `plugin-overrides.json` غير صالح: يُتجاهل مع الرجوع إلى `{}` في كل من مساري المدير والمُحمّل
- مسارات ملفات الأدوات/الخطافات/الأوامر المفقودة المُشار إليها بالبيان: يُتجاهل بصمت أثناء توسيع المُحلل؛ تُعلّم كأخطاء فقط بواسطة `doctor`

## اختلافات الأوضاع والأولوية

- `--dry-run` (التثبيت): يُرجع نتيجة تثبيت اصطناعية، بدون كتابات على نظام الملفات/الشبكة/الحالة.
- `--json`: تنسيق المخرجات فقط، بدون تغيير في السلوك.
- تجاوزات المشروع تأخذ الأولوية دائماً على ملف القفل العام لعرض الميزات/الإعدادات.
- التفعيل الفعلي هو `runtimeEnabled && !projectDisabled`.
