# تحميل الإضافات (وحدات TypeScript/JavaScript)

تتناول هذه الوثيقة كيفية اكتشاف وكيل الترميز وتحميل **وحدات الإضافات** (`.ts`/`.js`) عند بدء التشغيل.

لا تتناول هذه الوثيقة إضافات ملف القائمة `gemini-extension.json` (الموثقة بشكل منفصل).

## ما الذي يقوم به هذا النظام الفرعي

يقوم نظام تحميل الإضافات ببناء قائمة من ملفات إدخال الوحدات، واستيراد كل وحدة باستخدام Bun، وتنفيذ المصنع الخاص بها، ويُعيد:

- تعريفات الإضافات المحمّلة
- أخطاء التحميل لكل مسار (دون إيقاف عملية التحميل الكاملة)
- كائن وقت تشغيل الإضافات المشترك الذي يستخدمه لاحقاً `ExtensionRunner`

## ملفات التنفيذ الأساسية

- `src/extensibility/extensions/loader.ts` — اكتشاف المسارات + الاستيراد/التنفيذ
- `src/extensibility/extensions/index.ts` — الصادرات العامة
- `src/extensibility/extensions/runner.ts` — تنفيذ وقت التشغيل/الأحداث بعد التحميل
- `src/discovery/builtin.ts` — موفر الاكتشاف التلقائي الأصلي لوحدات الإضافات
- `src/config/settings.ts` — يحمّل إعدادات `extensions` / `disabledExtensions` المدمجة

## مدخلات تحميل الإضافات

### 1) وحدات الإضافات الأصلية المكتشفة تلقائياً

تقوم `discoverAndLoadExtensions()` أولاً بالاستعلام من موفري الاكتشاف عن عناصر قدرة `extension-module`، ثم تحتفظ فقط بعناصر الموفر `native`.

المواقع الأصلية الفعلية:

- المشروع: `<cwd>/.xcsh/extensions`
- المستخدم: `~/.xcsh/agent/extensions`

تأتي جذور المسارات من الموفر الأصلي (`SOURCE_PATHS.native`).

ملاحظات:

- يعتمد الاكتشاف التلقائي الأصلي حالياً على `.xcsh`.
- لا يزال `.pi` القديم مقبولاً في مفاتيح قائمة `package.json` (‏`pi.extensions`)، لكن ليس كجذر أصلي هنا.

### 2) المسارات المُهيأة صراحةً

بعد الاكتشاف التلقائي، تُضاف المسارات المُهيأة وتُحلّ.

مصادر المسار المُهيأ في مسار بدء الجلسة الرئيسي (`sdk.ts`):

1. المسارات المُوفرة عبر سطر الأوامر (`--extension/-e`، ويُعامل `--hook` أيضاً كمسار إضافة)
2. مصفوفة `extensions` في الإعدادات (الإعدادات العامة والمشروع مدمجة)

ملف الإعدادات العامة:

- `~/.xcsh/agent/config.yml` (أو دليل الوكيل المخصص عبر `PI_CODING_AGENT_DIR`)

ملف إعدادات المشروع:

- `<cwd>/.xcsh/settings.json`

أمثلة:

```yaml
# ~/.xcsh/agent/config.yml
extensions:
  - ~/my-exts/safety.ts
  - ./local/ext-pack
```

```json
{
  "extensions": ["./.xcsh/extensions/my-extra"]
}
```

## عناصر تحكم التمكين/التعطيل

### تعطيل الاكتشاف

- سطر الأوامر: `--no-extensions`
- خيار SDK: `disableExtensionDiscovery`

تقسيم السلوك:

- SDK: عند تعيين `disableExtensionDiscovery=true`، لا يزال يحمّل `additionalExtensionPaths` عبر `loadExtensions()`.
- بناء مسار سطر الأوامر (`main.ts`) يمسح مسارات إضافات سطر الأوامر حالياً عند تعيين `--no-extensions`، لذا لا تُعاد توجيه `-e/--hook` الصريحة في هذا الوضع.

### تعطيل وحدات إضافات محددة

يقوم إعداد `disabledExtensions` بالتصفية وفق تنسيق معرّف الإضافة:

- `extension-module:<derivedName>`

يعتمد `derivedName` على مسار الإدخال (`getExtensionNameFromPath`)، على سبيل المثال:

- `/x/foo.ts` -> `foo`
- `/x/bar/index.ts` -> `bar`

مثال:

```yaml
disabledExtensions:
  - extension-module:foo
```

## حل المسارات والإدخالات

### تطبيع المسارات

للمسارات المُهيأة:

1. تطبيع المسافات unicode
2. توسيع `~`
3. إذا كان نسبياً، يُحلّ بالنسبة إلى `cwd` الحالي

### إذا كان المسار المُهيأ ملفاً

يُستخدم مباشرةً كمرشح لإدخال الوحدة.

### إذا كان المسار المُهيأ دليلاً

ترتيب الحل:

1. `package.json` في ذلك الدليل مع `xcsh.extensions` (أو `pi.extensions` القديم) -> استخدام الإدخالات المُعلنة
2. `index.ts`
3. `index.js`
4. وإلا يتم مسح مستوى واحد بحثاً عن إدخالات الإضافات:
   - `*.ts` / `*.js` المباشرة
   - `index.ts` / `index.js` في الدليل الفرعي
   - `package.json` في الدليل الفرعي مع `xcsh.extensions` / `pi.extensions`

القواعد والقيود:

- لا يوجد اكتشاف متكرر بعد مستوى دليل فرعي واحد
- تُحلّ إدخالات قائمة `extensions` المُعلنة بالنسبة إلى دليل الحزمة ذاك
- تُدرج الإدخالات المُعلنة فقط إذا كان الملف موجوداً ويسمح بالوصول إليه
- في أزواج `*/index.{ts,js}`، يُفضّل TypeScript على JavaScript
- تُعامل الروابط الرمزية كملفات/أدلة مؤهلة

### يختلف سلوك التجاهل حسب المصدر

- يستخدم الاكتشاف التلقائي الأصلي (`discoverExtensionModulePaths` في مساعدات الاكتشاف) النمط العام الأصلي مع `gitignore: true` و`hidden: false`.
- لا يُطبّق مسح الدليل المُهيأ الصريح في `loader.ts` تصفية gitignore.

## ترتيب التحميل والأسبقية

تبني `discoverAndLoadExtensions()` قائمة مرتبة واحدة ثم تستدعي `loadExtensions()`.

الترتيب:

1. الوحدات المكتشفة تلقائياً الأصلية
2. المسارات المُهيأة صراحةً (بالترتيب المُوفر)

في `sdk.ts`، الترتيب المُهيأ هو:

1. المسارات الإضافية عبر سطر الأوامر
2. إعدادات `extensions`

إزالة التكرار:

- يعتمد على المسار المطلق
- المسار الأول المرئي يفوز
- التكرارات اللاحقة تُتجاهل

الأثر: إذا كان نفس مسار الوحدة مكتشفاً تلقائياً ومُهيأً صراحةً، فإنه يُحمّل مرة واحدة في الموضع الأول (مرحلة الاكتشاف التلقائي).

## استيراد الوحدة وعقد المصنع

يتم تحميل كل مسار مرشح باستخدام الاستيراد الديناميكي:

- `await import(resolvedPath)`
- المصنع هو `module.default ?? module`
- يجب أن يكون المصنع دالة (`ExtensionFactory`)

إذا لم يكن الصادر دالة، يفشل ذلك المسار مع خطأ منظم وتستمر عملية التحميل.

## معالجة الأعطال والعزل

### أثناء التحميل

يتم التقاط الأعطال لكل مسار إضافة على شكل `{ path, error }` ولا تمنع تحميل المسارات الأخرى.

الحالات الشائعة:

- فشل الاستيراد / الملف مفقود
- صادر مصنع غير صالح (غير دالة)
- استثناء يُطرح أثناء تنفيذ المصنع

### نموذج عزل وقت التشغيل

- الإضافات **غير معزولة في صندوق رمل** (نفس العملية/وقت التشغيل).
- تشترك في `EventBus` واحد وكائن `ExtensionRuntime` واحد.
- أثناء التحميل، تُطلق عمداً أساليب إجراء وقت التشغيل `ExtensionRuntimeNotInitializedError`؛ يتم توصيل الإجراءات لاحقاً في `ExtensionRunner.initialize()`.

### بعد التحميل

عندما تمر الأحداث عبر `ExtensionRunner`، يتم التقاط استثناءات المعالج وإصدارها كأخطاء إضافات بدلاً من تعطل حلقة المشغّل.

## أمثلة تخطيط مستخدم/مشروع مبسطة

### على مستوى المستخدم

```text
~/.xcsh/agent/
  config.yml
  extensions/
    guardrails.ts
    audit/
      index.ts
```

### على مستوى المشروع

```text
<repo>/
  .xcsh/
    settings.json
    extensions/
      checks/
        package.json
      lint-gates.ts
```

`checks/package.json`:

```json
{
  "xcsh": {
    "extensions": ["./src/check-a.ts", "./src/check-b.js"]
  }
}
```

مفتاح القائمة القديم لا يزال مقبولاً:

```json
{
  "pi": {
    "extensions": ["./index.ts"]
  }
}
```
