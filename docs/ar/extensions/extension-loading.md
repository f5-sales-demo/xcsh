# تحميل الإضافات (وحدات TypeScript/JavaScript)

يتناول هذا المستند كيفية اكتشاف وكيل البرمجة وتحميل **وحدات الإضافات** (`.ts`/`.js`) عند بدء التشغيل.

لا يتناول هذا المستند إضافات ملف البيان `gemini-extension.json` (الموثقة بشكل منفصل).

## ما يقوم به هذا النظام الفرعي

يقوم تحميل الإضافات ببناء قائمة بملفات نقاط الدخول للوحدات، ثم يستورد كل وحدة باستخدام Bun، وينفذ المصنع الخاص بها، ويعيد:

- تعريفات الإضافات المحمّلة
- أخطاء التحميل لكل مسار (دون إيقاف عملية التحميل بالكامل)
- كائن وقت تشغيل الإضافات المشترك الذي يُستخدم لاحقًا بواسطة `ExtensionRunner`

## ملفات التنفيذ الرئيسية

- `src/extensibility/extensions/loader.ts` — اكتشاف المسارات + الاستيراد/التنفيذ
- `src/extensibility/extensions/index.ts` — التصديرات العامة
- `src/extensibility/extensions/runner.ts` — وقت التشغيل/تنفيذ الأحداث بعد التحميل
- `src/discovery/builtin.ts` — مزود الاكتشاف التلقائي الأصلي لوحدات الإضافات
- `src/config/settings.ts` — يحمّل إعدادات `extensions` / `disabledExtensions` المدمجة

## مدخلات تحميل الإضافات

### 1) وحدات الإضافات الأصلية المُكتشفة تلقائيًا

تقوم `discoverAndLoadExtensions()` أولاً بسؤال مزودي الاكتشاف عن عناصر القدرة `extension-module`، ثم تحتفظ فقط بعناصر المزود `native`.

المواقع الأصلية الفعّالة:

- المشروع: `<cwd>/.xcsh/extensions`
- المستخدم: `~/.xcsh/agent/extensions`

تأتي جذور المسارات من المزود الأصلي (`SOURCE_PATHS.native`).

ملاحظات:

- الاكتشاف التلقائي الأصلي يعتمد حاليًا على `.xcsh`.
- لا يزال `.pi` القديم مقبولاً في مفاتيح ملف البيان `package.json` (`pi.extensions`)، ولكن ليس كجذر أصلي هنا.

### 2) المسارات المُعدّة صراحةً

بعد الاكتشاف التلقائي، يتم إلحاق المسارات المُعدّة واستكشافها.

مصادر المسارات المُعدّة في مسار بدء الجلسة الرئيسي (`sdk.ts`):

1. المسارات المُقدمة عبر سطر الأوامر (`--extension/-e`، ويُعامل `--hook` أيضًا كمسار إضافة)
2. مصفوفة `extensions` في الإعدادات (الإعدادات العامة + إعدادات المشروع المدمجة)

ملف الإعدادات العامة:

- `~/.xcsh/agent/config.yml` (أو مجلد وكيل مخصص عبر `PI_CODING_AGENT_DIR`)

ملف إعدادات المشروع:

- `<cwd>/.xcsh/settings.json`

أمثلة:

```yaml
# ~/.xcsh/agent/config.yml
extensions:
  - ~/my-exts/safety.ts
  - ./local/ext-pack
```

```json
{
  "extensions": ["./.xcsh/extensions/my-extra"]
}
```

## عناصر التحكم في التمكين/التعطيل

### تعطيل الاكتشاف

- سطر الأوامر: `--no-extensions`
- خيار SDK: `disableExtensionDiscovery`

تقسيم السلوك:

- SDK: عندما يكون `disableExtensionDiscovery=true`، يستمر في تحميل `additionalExtensionPaths` عبر `loadExtensions()`.
- بناء مسار سطر الأوامر (`main.ts`) يقوم حاليًا بمسح مسارات إضافات سطر الأوامر عند تعيين `--no-extensions`، لذا لا يتم تمرير `-e/--hook` الصريحة في هذا الوضع.

### تعطيل وحدات إضافات محددة

إعداد `disabledExtensions` يقوم بالتصفية حسب تنسيق معرف الإضافة:

- `extension-module:<derivedName>`

يعتمد `derivedName` على مسار نقطة الدخول (`getExtensionNameFromPath`)، على سبيل المثال:

- `/x/foo.ts` -> `foo`
- `/x/bar/index.ts` -> `bar`

مثال:

```yaml
disabledExtensions:
  - extension-module:foo
```

## استكشاف المسارات ونقاط الدخول

### تطبيع المسارات

للمسارات المُعدّة:

1. تطبيع مسافات Unicode
2. توسيع `~`
3. إذا كان نسبيًا، يتم الاستكشاف مقابل `cwd` الحالي

### إذا كان المسار المُعدّ ملفًا

يُستخدم مباشرة كمرشح لنقطة دخول الوحدة.

### إذا كان المسار المُعدّ مجلدًا

ترتيب الاستكشاف:

1. `package.json` في ذلك المجلد مع `xcsh.extensions` (أو `pi.extensions` القديم) -> استخدام نقاط الدخول المُعلنة
2. `index.ts`
3. `index.js`
4. وإلا يتم مسح مستوى واحد بحثًا عن نقاط دخول الإضافات:
   - ملفات `*.ts` / `*.js` مباشرة
   - مجلد فرعي `index.ts` / `index.js`
   - مجلد فرعي `package.json` مع `xcsh.extensions` / `pi.extensions`

القواعد والقيود:

- لا يوجد اكتشاف تكراري بعد مستوى مجلد فرعي واحد
- يتم استكشاف نقاط دخول ملف البيان `extensions` المُعلنة نسبةً إلى مجلد الحزمة ذلك
- يتم تضمين النقاط المُعلنة فقط إذا كان الملف موجودًا/الوصول مسموحًا به
- في أزواج `*/index.{ts,js}`، يُفضّل TypeScript على JavaScript
- تُعامل الروابط الرمزية كملفات/مجلدات مؤهلة

### يختلف سلوك التجاهل حسب المصدر

- الاكتشاف التلقائي الأصلي (`discoverExtensionModulePaths` في مساعدات الاكتشاف) يستخدم glob أصلي مع `gitignore: true` و `hidden: false`.
- مسح المجلدات المُعدّة صراحةً في `loader.ts` يستخدم قواعد `readdir` و**لا** يطبق تصفية gitignore.

## ترتيب التحميل والأولوية

تبني `discoverAndLoadExtensions()` قائمة مرتبة واحدة ثم تستدعي `loadExtensions()`.

الترتيب:

1. الوحدات المُكتشفة تلقائيًا بشكل أصلي
2. المسارات المُعدّة صراحةً (بالترتيب المُقدم)

في `sdk.ts`، الترتيب المُعدّ هو:

1. المسارات الإضافية من سطر الأوامر
2. `extensions` من الإعدادات

إزالة التكرارات:

- تعتمد على المسار المطلق
- المسار الذي يُرى أولاً يفوز
- يتم تجاهل التكرارات اللاحقة

الدلالة: إذا كان نفس مسار الوحدة مُكتشفًا تلقائيًا ومُعدًّا صراحةً في نفس الوقت، يتم تحميله مرة واحدة في الموضع الأول (مرحلة الاكتشاف التلقائي).

## استيراد الوحدة وعقد المصنع

يتم تحميل كل مسار مرشح باستخدام الاستيراد الديناميكي:

- `await import(resolvedPath)`
- المصنع هو `module.default ?? module`
- يجب أن يكون المصنع دالة (`ExtensionFactory`)

إذا لم يكن التصدير دالة، يفشل ذلك المسار مع خطأ منظم ويستمر التحميل.

## معالجة الأخطاء والعزل

### أثناء التحميل

لكل مسار إضافة، يتم التقاط الأخطاء كـ `{ path, error }` ولا توقف المسارات الأخرى عن التحميل.

الحالات الشائعة:

- فشل الاستيراد / ملف مفقود
- تصدير مصنع غير صالح (ليس دالة)
- استثناء يُطرح أثناء تنفيذ المصنع

### نموذج عزل وقت التشغيل

- الإضافات **ليست معزولة في بيئة رملية** (نفس العملية/وقت التشغيل).
- تتشارك في `EventBus` واحد ومثيل `ExtensionRuntime` واحد.
- أثناء التحميل، تطرح أساليب إجراء وقت التشغيل عمدًا `ExtensionRuntimeNotInitializedError`؛ يحدث ربط الإجراءات لاحقًا في `ExtensionRunner.initialize()`.

### بعد التحميل

عندما تعمل الأحداث عبر `ExtensionRunner`، يتم التقاط استثناءات المعالجات وإصدارها كأخطاء إضافات بدلاً من تعطل حلقة المشغّل.

## أمثلة بسيطة لتخطيط المستخدم/المشروع

### على مستوى المستخدم

```text
~/.xcsh/agent/
  config.yml
  extensions/
    guardrails.ts
    audit/
      index.ts
```

### على مستوى المشروع

```text
<repo>/
  .xcsh/
    settings.json
    extensions/
      checks/
        package.json
      lint-gates.ts
```

`checks/package.json`:

```json
{
  "xcsh": {
    "extensions": ["./src/check-a.ts", "./src/check-b.js"]
  }
}
```

مفتاح ملف البيان القديم لا يزال مقبولاً:

```json
{
  "pi": {
    "extensions": ["./index.ts"]
  }
}
```
