# التفاصيل الداخلية لوقت تشغيل أداة الدفتر

يصف هذا المستند التنفيذ الحالي لأداة `notebook` وعلاقتها بوقت تشغيل Python المدعوم بالنواة.

التمييز الجوهري: **`notebook` هي محرر JSON للدفاتر، وليست منفذاً للدفاتر**. فهي تحرر مصادر خلايا `.ipynb` مباشرة؛ ولا تبدأ أو تتواصل مع نواة Python.

## ملفات التنفيذ

- [`src/tools/notebook.ts`](../../packages/coding-agent/src/tools/notebook.ts)
- [`src/ipy/executor.ts`](../../packages/coding-agent/src/ipy/executor.ts)
- [`src/ipy/kernel.ts`](../../packages/coding-agent/src/ipy/kernel.ts)
- [`src/session/streaming-output.ts`](../../packages/coding-agent/src/session/streaming-output.ts)
- [`src/tools/python.ts`](../../packages/coding-agent/src/tools/python.ts)

## 1) حدود وقت التشغيل: التحرير مقابل التنفيذ

## أداة `notebook` (`src/tools/notebook.ts`)

- تدعم `action: edit | insert | delete` على ملف `.ipynb`.
- تحل المسار نسبة إلى مجلد العمل الحالي للجلسة (`resolveToCwd`).
- تحمّل JSON الدفتر، وتتحقق من مصفوفة `cells`، وتتحقق من حدود `cell_index`.
- تطبق تعديلات المصدر في الذاكرة وتكتب JSON الكامل للدفتر مرة أخرى باستخدام `JSON.stringify(notebook, null, 1)`.
- تُرجع ملخصاً نصياً + `details` مهيكلة (`action`، `cellIndex`، `cellType`، `totalCells`، `cellSource`).

لا توجد دورة حياة للنواة في هذه الأداة:

- لا اكتساب بوابة
- لا معرف جلسة نواة
- لا `execute_request`
- لا أجزاء متدفقة من قنوات النواة
- لا التقاط عرض غني (`image/png`، عرض JSON، حالة MIME)

## مسار التنفيذ الشبيه بالدفتر (`src/tools/python.ts` + `src/ipy/*`)

عندما يحتاج الوكيل إلى تشغيل كود Python بأسلوب الخلايا (خلايا متتابعة، حالة مستمرة، عروض غنية)، يمر ذلك عبر **أداة `python`**، وليس `notebook`.

هذا المسار هو حيث توجد أوضاع النواة، وسلوك إعادة التشغيل/الإلغاء، والتدفق المجزأ، واقتطاع مخرجات العناصر.

## 2) دلالات معالجة خلايا الدفتر (أداة `notebook`)

## تطبيع المصدر

يُقسم `content` إلى `source: string[]` مع الحفاظ على أسطر جديدة:

- كل سطر غير نهائي يحتفظ بحرف `\n` في نهايته
- السطر الأخير لا يحتوي على سطر جديد إجباري في النهاية

هذا يحاكي اصطلاحات JSON للدفاتر ويتجنب دمج الأسطر العرضي في التعديلات اللاحقة.

## سلوك الإجراءات

- `edit`
  - يستبدل `cells[cell_index].source`
  - يحافظ على `cell_type` الموجود
- `insert`
  - يُدرج عند `[0..cellCount]`
  - `cell_type` يكون افتراضياً `code`
  - خلايا الكود تُهيئ `execution_count: null` و `outputs: []`
  - خلايا markdown تُهيئ فقط `metadata` + `source`
- `delete`
  - يحذف `cells[cell_index]`
  - يُرجع `source` المحذوف في التفاصيل لمعاينة المُصيِّر

## أسطح الأخطاء

تُطلق أخطاء صارمة في الحالات التالية:

- ملف الدفتر مفقود
- JSON غير صالح
- `cells` مفقودة أو ليست مصفوفة
- فهرس خارج النطاق (الإدراج وغير الإدراج لهما نطاقات صالحة مختلفة)
- `content` مفقود لإجراء `edit`/`insert`

تصبح هذه استجابات أداة `Error:` في المستوى الأعلى؛ يستخدم المُصيِّر مسار الدفتر + نص الخطأ المنسق.

## 3) دلالات جلسة النواة (حيث توجد فعلياً)

دلالات النواة مُنفذة في `executePython` / `PythonKernel` وتنطبق على أداة `python`.

## الأوضاع

`PythonKernelMode`:

- `session` (افتراضي)
  - الأنوية مخزنة مؤقتاً في خريطة `kernelSessions`
  - الحد الأقصى 4 جلسات؛ الأقدم يُطرد عند التجاوز
  - تنظيف الخامل/الميت كل 30 ثانية، مهلة بعد 5 دقائق
  - طابور لكل جلسة يُسلسل التنفيذ (`session.queue`)
- `per-call`
  - يُنشئ نواة للطلب
  - يُنفذ
  - يُغلق النواة دائماً في `finally`

## سلوك إعادة التعيين

أداة `python` تمرر `reset` فقط للخلية الأولى في استدعاء متعدد الخلايا؛ الخلايا اللاحقة تعمل دائماً مع `reset: false`.

## موت النواة / إعادة التشغيل / إعادة المحاولة

في وضع الجلسة (`withKernelSession`):

- يُكتشف موت النواة بواسطة نبض القلب (فحص `kernel.isAlive()` كل 5 ثوانٍ) أو فشل التنفيذ.
- حالة الموت قبل التشغيل تُفعّل `restartKernelSession`.
- مسار الانهيار أثناء التنفيذ يُعيد المحاولة مرة واحدة: إعادة تشغيل النواة، إعادة تشغيل المعالج.
- `restartCount > 1` في نفس الجلسة يُطلق `Python kernel restarted too many times in this session`.

سلوك إعادة محاولة بدء التشغيل:

- إنشاء نواة البوابة المشتركة يُعيد المحاولة مرة واحدة عند `SharedGatewayCreateError` مع HTTP 5xx.

استعادة استنفاد الموارد:

- يكتشف أخطاء نمط `EMFILE`/`ENFILE`/"Too many open files"
- يمسح الجلسات المتتبعة
- يستدعي `shutdownSharedGateway()`
- يُعيد محاولة إنشاء جلسة النواة مرة واحدة

## 4) حقن متغيرات البيئة/الجلسة

بدء تشغيل النواة يتلقى خريطة بيئة اختيارية من المنفذ:

- `PI_SESSION_FILE` (مسار ملف حالة الجلسة)
- `ARTIFACTS` (مجلد العناصر)

`PythonKernel.#initializeKernelEnvironment(...)` ثم يُشغل سكربت تهيئة داخل النواة لـ:

- `os.chdir(cwd)`
- حقن إدخالات البيئة في `os.environ`
- إضافة cwd في بداية `sys.path` إذا كان مفقوداً

الاستنتاج:

- المساعدات التمهيدية التي تقرأ سياق الجلسة أو العناصر تعتمد على متغيرات البيئة هذه في حالة عملية Python.

## 5) معالجة التدفق/الأجزاء والعرض (المسار المدعوم بالنواة)

يعالج عميل النواة رسائل بروتوكول Jupyter لكل تنفيذ:

- `stream` -> جزء نصي إلى `onChunk`
- `execute_result` / `display_data` ->
  - نص العرض يُختار حسب أولوية MIME: `text/markdown` > `text/plain` > `text/html` المُحوّل
  - المخرجات المهيكلة تُلتقط بشكل منفصل:
    - `application/json` -> `{ type: "json" }`
    - `image/png` -> `{ type: "image" }`
    - `application/x-xcsh-status` -> `{ type: "status" }` (بدون إصدار نص)
- `error` -> نص التتبع يُدفع إلى تدفق الأجزاء + بيانات وصفية للخطأ مهيكلة
- `input_request` -> يُصدر نص تحذير stdin، يُرسل `input_reply` فارغاً، يُعلّم أن stdin مطلوب
- انتظار الاكتمال ينتظر كلاً من `execute_reply` وحالة النواة `status=idle`

الإلغاء/المهلة:

- إشارة الإجهاض تُفعّل `interrupt()` (REST `/interrupt` + قناة التحكم `interrupt_request`)
- النتيجة تُعلّم `cancelled=true`
- مسار المهلة يُضيف تعليقاً توضيحياً للمخرجات بـ `Command timed out after <n> seconds`

## 6) سلوك الاقتطاع والعناصر

`OutputSink` في `src/session/streaming-output.ts` يُستخدم بواسطة مسارات تنفيذ النواة (`executeWithKernel`):

- يُنظف كل جزء (`sanitizeText`)
- يتتبع إجمالي الأسطر/المخرجات والبايتات
- ملف تفريغ عناصر اختياري (`artifactPath`، `artifactId`)
- عندما يتجاوز المخزن المؤقت في الذاكرة الحد (`DEFAULT_MAX_BYTES` ما لم يُتجاوز):
  - يُعلّم بالاقتطاع
  - يحتفظ ببايتات الذيل في الذاكرة (حد آمن لـ UTF-8)
  - يمكن تفريغ التدفق الكامل إلى حوض العناصر

`dump()` يُرجع:

- نص المخرجات المرئي (ربما مقتطع من الذيل)
- علامة الاقتطاع + العدادات
- معرف العنصر (لمراجع `artifact://<id>`)

أداة `python` تحوّل هذه البيانات الوصفية إلى إشعارات اقتطاع النتائج وتحذيرات TUI.

أداة `notebook` **لا تستخدم** `OutputSink`؛ ليس لديها خط أنابيب اقتطاع تدفق/عناصر لأنها لا تُنفذ كوداً.

## 7) افتراضات المُصيِّر والتنسيق

## مُصيِّر الدفتر (`notebookToolRenderer`)

- عرض الاستدعاء: سطر حالة مع الإجراء + مسار الدفتر + بيانات وصفية للخلية/النوع
- عرض النتيجة:
  - ملخص النجاح مُشتق من `details`
  - `cellSource` يُعرض عبر `renderCodeCell`
  - خلايا markdown تضبط تلميح اللغة `markdown`؛ الخلايا الأخرى ليس لها تجاوز لغة صريح
  - حد معاينة الكود المطوي هو `PREVIEW_LIMITS.COLLAPSED_LINES * 2`
  - يدعم الوضع الموسع عبر خيارات عرض مشتركة
  - يستخدم ذاكرة تخزين مؤقتة للعرض مفتاحها العرض + حالة التوسيع

افتراض عرض الخطأ:

- إذا بدأ محتوى النص الأول بـ `Error:`، يُنسق المُصيِّر ككتلة خطأ دفتر.

## مُصيِّر Python (لمخرجات التنفيذ الفعلية)

عرض التنفيذ المدعوم بالنواة يتوقع:

- انتقالات حالة لكل خلية (`pending/running/complete/error`)
- قسم أحداث حالة مهيكلة اختياري
- أشجار مخرجات JSON اختيارية
- تحذيرات اقتطاع + مؤشر `artifact://<id>` اختياري

سلوك هذا المُصيِّر غير مرتبط بنتائج تحرير JSON للدفتر باستثناء أن كليهما يُعيد استخدام أساسيات TUI المشتركة.

## 8) الاختلاف عن سلوك أداة Python العادية

إذا كانت "أداة Python العادية" تعني مسار تنفيذ `python`:

- `python` يُنفذ الكود في نواة، يحتفظ بالحالة حسب الوضع، يُدفق الأجزاء، يلتقط العروض الغنية، يُعالج المقاطعات/المهلات، ويدعم اقتطاع المخرجات/العناصر.
- `notebook` يُجري تحولات حتمية لـ JSON الدفتر فقط؛ لا تنفيذ، لا حالة نواة، لا تدفق أجزاء، لا مخرجات عرض، لا خط أنابيب عناصر.

إذا كان سير العمل يحتاج كليهما:

1. تحرير مصدر الدفتر باستخدام `notebook`
2. تنفيذ خلايا الكود عبر `python` (تمرير الكود يدوياً)، وليس من خلال `notebook`

التنفيذ الحالي لا يوفر أداة واحدة تُعدّل `.ipynb` وتُنفذ خلايا الدفتر من خلال سياق النواة في آن واحد.
