# البنية الداخلية لوقت تشغيل أداة الدفتر

يصف هذا المستند التنفيذ الحالي لأداة `notebook` وعلاقتها ببيئة تشغيل Python المدعومة بالنواة.

التمييز الحاسم: **`notebook` هي محرر JSON للدفاتر، وليست منفذ دفاتر**. تقوم بتحرير مصادر خلايا `.ipynb` مباشرة؛ ولا تبدأ نواة Python أو تتواصل معها.

## ملفات التنفيذ

- [`src/tools/notebook.ts`](../../packages/coding-agent/src/tools/notebook.ts)
- [`src/ipy/executor.ts`](../../packages/coding-agent/src/ipy/executor.ts)
- [`src/ipy/kernel.ts`](../../packages/coding-agent/src/ipy/kernel.ts)
- [`src/session/streaming-output.ts`](../../packages/coding-agent/src/session/streaming-output.ts)
- [`src/tools/python.ts`](../../packages/coding-agent/src/tools/python.ts)

## 1) حدود وقت التشغيل: التحرير مقابل التنفيذ

## أداة `notebook` (`src/tools/notebook.ts`)

- تدعم `action: edit | insert | delete` على ملف `.ipynb`.
- تحل المسار نسبة إلى دليل العمل الحالي للجلسة (`resolveToCwd`).
- تحمّل JSON الدفتر، وتتحقق من مصفوفة `cells`، وتتحقق من حدود `cell_index`.
- تطبق تعديلات المصدر في الذاكرة وتكتب JSON الدفتر الكامل مرة أخرى باستخدام `JSON.stringify(notebook, null, 1)`.
- تُرجع ملخصاً نصياً + `details` منظمة (`action`، `cellIndex`، `cellType`، `totalCells`، `cellSource`).

لا توجد دورة حياة نواة في هذه الأداة:

- لا اكتساب بوابة
- لا معرف جلسة نواة
- لا `execute_request`
- لا أجزاء تدفق من قنوات النواة
- لا التقاط عرض غني (`image/png`، عرض JSON، حالة MIME)

## مسار التنفيذ الشبيه بالدفتر (`src/tools/python.ts` + `src/ipy/*`)

عندما يحتاج الوكيل لتشغيل كود Python بنمط الخلايا (خلايا متتالية، حالة مستمرة، عروض غنية)، يمر ذلك عبر أداة **`python`**، وليس `notebook`.

هذا المسار هو حيث تعيش أوضاع النواة، وسلوك إعادة التشغيل/الإلغاء، وتدفق الأجزاء، واقتطاع مخرجات القطع الأثرية.

## 2) دلالات معالجة خلايا الدفتر (أداة `notebook`)

## تطبيع المصدر

يتم تقسيم `content` إلى `source: string[]` مع الحفاظ على الأسطر الجديدة:

- كل سطر غير نهائي يحتفظ بحرف `\n` الختامي
- السطر الأخير لا يحتوي على سطر جديد ختامي إجباري

هذا يعكس اتفاقيات JSON للدفاتر ويتجنب التصاق الأسطر العرضي في التعديلات اللاحقة.

## سلوك الإجراءات

- `edit`
  - يستبدل `cells[cell_index].source`
  - يحافظ على `cell_type` الحالي
- `insert`
  - يدرج في `[0..cellCount]`
  - `cell_type` افتراضياً هو `code`
  - خلايا الكود تُهيئ `execution_count: null` و `outputs: []`
  - خلايا markdown تُهيئ فقط `metadata` + `source`
- `delete`
  - يزيل `cells[cell_index]`
  - يُرجع `source` المحذوف في التفاصيل لمعاينة المُصيِّر

## أسطح الأخطاء

يتم طرح أخطاء صارمة في حالة:

- ملف دفتر مفقود
- JSON غير صالح
- `cells` مفقودة/ليست مصفوفة
- فهرس خارج النطاق (الإدراج وغير الإدراج لهما نطاقات صالحة مختلفة)
- `content` مفقود لـ `edit`/`insert`

تصبح هذه استجابات أداة `Error:` في المستوى الأعلى؛ يستخدم المُصيِّر مسار الدفتر + نص الخطأ المنسق.

## 3) دلالات جلسة النواة (حيث توجد فعلياً)

دلالات النواة مُنفذة في `executePython` / `PythonKernel` وتنطبق على أداة `python`.

## الأوضاع

`PythonKernelMode`:

- `session` (افتراضي)
  - الأنوية مخزنة مؤقتاً في خريطة `kernelSessions`
  - الحد الأقصى 4 جلسات؛ يتم إخلاء الأقدم عند التجاوز
  - تنظيف الخمول/الموت كل 30 ثانية، مهلة بعد 5 دقائق
  - طابور لكل جلسة يُسلسل التنفيذ (`session.queue`)
- `per-call`
  - ينشئ نواة لكل طلب
  - ينفذ
  - يوقف النواة دائماً في `finally`

## سلوك إعادة التعيين

أداة `python` تمرر `reset` فقط للخلية الأولى في استدعاء متعدد الخلايا؛ الخلايا اللاحقة تعمل دائماً بـ `reset: false`.

## موت النواة / إعادة التشغيل / إعادة المحاولة

في وضع الجلسة (`withKernelSession`):

- يتم اكتشاف موت النواة بواسطة نبض القلب (فحص `kernel.isAlive()` كل 5 ثوانٍ) أو فشل التنفيذ.
- حالة الموت قبل التشغيل تؤدي إلى `restartKernelSession`.
- مسار الانهيار أثناء التنفيذ يعيد المحاولة مرة واحدة: إعادة تشغيل النواة، إعادة تشغيل المعالج.
- `restartCount > 1` في نفس الجلسة يطرح `Python kernel restarted too many times in this session`.

سلوك إعادة محاولة بدء التشغيل:

- إنشاء نواة البوابة المشتركة يعيد المحاولة مرة واحدة عند `SharedGatewayCreateError` مع HTTP 5xx.

استرداد استنفاد الموارد:

- يكتشف أخطاء من نمط `EMFILE`/`ENFILE`/"Too many open files"
- يمسح الجلسات المتتبعة
- يستدعي `shutdownSharedGateway()`
- يعيد محاولة إنشاء جلسة النواة مرة واحدة

## 4) حقن متغيرات البيئة/الجلسة

بدء تشغيل النواة يستقبل خريطة بيئة اختيارية من المنفذ:

- `PI_SESSION_FILE` (مسار ملف حالة الجلسة)
- `ARTIFACTS` (دليل القطع الأثرية)

`PythonKernel.#initializeKernelEnvironment(...)` ثم يشغل سكريبت التهيئة داخل النواة لـ:

- `os.chdir(cwd)`
- حقن إدخالات البيئة في `os.environ`
- إضافة cwd في بداية `sys.path` إذا كان مفقوداً

الدلالة:

- مساعدات المقدمة التي تقرأ سياق الجلسة أو القطع الأثرية تعتمد على متغيرات البيئة هذه في حالة عملية Python.

## 5) معالجة التدفق/الأجزاء والعرض (المسار المدعوم بالنواة)

يعالج عميل النواة رسائل بروتوكول Jupyter لكل تنفيذ:

- `stream` -> جزء نصي إلى `onChunk`
- `execute_result` / `display_data` ->
  - نص العرض يُختار حسب أسبقية MIME: `text/markdown` > `text/plain` > `text/html` محوّل
  - المخرجات المنظمة تُلتقط بشكل منفصل:
    - `application/json` -> `{ type: "json" }`
    - `image/png` -> `{ type: "image" }`
    - `application/x-xcsh-status` -> `{ type: "status" }` (بدون إصدار نصي)
- `error` -> نص التتبع الخلفي يُدفع إلى تدفق الأجزاء + بيانات خطأ منظمة
- `input_request` -> يصدر نص تحذير stdin، يرسل `input_reply` فارغ، يضع علامة طلب stdin
- الاكتمال ينتظر كلاً من `execute_reply` وحالة النواة `status=idle`

الإلغاء/المهلة:

- إشارة الإجهاض تؤدي إلى `interrupt()` (REST `/interrupt` + `interrupt_request` على قناة التحكم)
- النتيجة تضع علامة `cancelled=true`
- مسار المهلة يُعلّق المخرجات بـ `Command timed out after <n> seconds`

## 6) سلوك الاقتطاع والقطع الأثرية

`OutputSink` في `src/session/streaming-output.ts` يُستخدم بواسطة مسارات تنفيذ النواة (`executeWithKernel`):

- يُعقّم كل جزء (`sanitizeText`)
- يتتبع إجمالي/مخرجات الأسطر والبايتات
- ملف تسرب اختياري للقطع الأثرية (`artifactPath`، `artifactId`)
- عندما يتجاوز المخزن المؤقت في الذاكرة الحد (`DEFAULT_MAX_BYTES` ما لم يتم تجاوزه):
  - يضع علامة مقتطع
  - يحتفظ ببايتات الذيل في الذاكرة (حدود UTF-8 آمنة)
  - يمكنه تسريب التدفق الكامل إلى مصرف القطع الأثرية

`dump()` يُرجع:

- نص المخرجات المرئي (ربما مقتطع من الذيل)
- علامة الاقتطاع + الأعداد
- معرف القطعة الأثرية (لمراجع `artifact://<id>`)

أداة `python` تحول هذه البيانات الوصفية إلى إشعارات اقتطاع النتائج وتحذيرات TUI.

أداة `notebook` **لا** تستخدم `OutputSink`؛ ليس لديها خط أنابيب اقتطاع تدفق/قطع أثرية لأنها لا تنفذ كوداً.

## 7) افتراضات المُصيِّر والتنسيق

## مُصيِّر الدفتر (`notebookToolRenderer`)

- عرض الاستدعاء: سطر حالة مع الإجراء + مسار الدفتر + بيانات الخلية/النوع الوصفية
- عرض النتيجة:
  - ملخص النجاح مشتق من `details`
  - `cellSource` يُصيَّر عبر `renderCodeCell`
  - خلايا markdown تضع تلميح اللغة `markdown`؛ الخلايا الأخرى ليس لها تجاوز لغة صريح
  - حد معاينة الكود المطوي هو `PREVIEW_LIMITS.COLLAPSED_LINES * 2`
  - يدعم الوضع الموسع عبر خيارات العرض المشتركة
  - يستخدم ذاكرة تخزين مؤقت للعرض مفتاحها العرض + حالة التوسيع

افتراض عرض الأخطاء:

- إذا بدأ محتوى النص الأول بـ `Error:`، يُنسق المُصيِّر ككتلة خطأ دفتر.

## مُصيِّر Python (لمخرجات التنفيذ الفعلي)

عرض التنفيذ المدعوم بالنواة يتوقع:

- انتقالات حالة لكل خلية (`pending/running/complete/error`)
- قسم حدث حالة منظم اختياري
- أشجار مخرجات JSON اختيارية
- تحذيرات اقتطاع + مؤشر `artifact://<id>` اختياري

سلوك هذا المُصيِّر لا علاقة له بنتائج تحرير JSON للدفتر إلا أن كليهما يعيد استخدام عناصر TUI الأساسية المشتركة.

## 8) الاختلاف عن سلوك أداة Python العادية

إذا كان المقصود بـ "أداة Python العادية" مسار تنفيذ `python`:

- `python` ينفذ الكود في نواة، يحافظ على الحالة حسب الوضع، يتدفق الأجزاء، يلتقط العروض الغنية، يعالج المقاطعات/المهلات، ويدعم اقتطاع المخرجات/القطع الأثرية.
- `notebook` يؤدي تحولات JSON حتمية للدفتر فقط؛ لا تنفيذ، لا حالة نواة، لا تدفق أجزاء، لا مخرجات عرض، لا خط أنابيب قطع أثرية.

إذا كان سير العمل يحتاج كليهما:

1. تحرير مصدر الدفتر باستخدام `notebook`
2. تنفيذ خلايا الكود عبر `python` (تمرير الكود يدوياً)، وليس من خلال `notebook`

التنفيذ الحالي لا يوفر أداة واحدة تُعدّل `.ipynb` وتنفذ خلايا الدفتر من خلال سياق النواة في آن واحد.
