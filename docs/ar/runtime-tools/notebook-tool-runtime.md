# الآليات الداخلية لوقت تشغيل أداة Notebook

يصف هذا المستند التنفيذ الحالي لأداة `notebook` وعلاقتها بوقت تشغيل Python المدعوم بالنواة.

التمييز الجوهري: **`notebook` هي محرر JSON للمفكرات، وليست منفذًا لها**. تعمل على تحرير مصادر خلايا `.ipynb` مباشرةً؛ ولا تبدأ نواة Python ولا تتواصل معها.

## ملفات التنفيذ

- [`src/tools/notebook.ts`](../../packages/coding-agent/src/tools/notebook.ts)
- [`src/ipy/executor.ts`](../../packages/coding-agent/src/ipy/executor.ts)
- [`src/ipy/kernel.ts`](../../packages/coding-agent/src/ipy/kernel.ts)
- [`src/session/streaming-output.ts`](../../packages/coding-agent/src/session/streaming-output.ts)
- [`src/tools/python.ts`](../../packages/coding-agent/src/tools/python.ts)

## 1) حدود وقت التشغيل: التحرير مقابل التنفيذ

## أداة `notebook` (`src/tools/notebook.ts`)

- تدعم `action: edit | insert | delete` على ملف `.ipynb`.
- تحل المسار بالنسبة إلى مجلد العمل الحالي للجلسة (`resolveToCwd`).
- تحمّل JSON الخاص بالمفكرة، وتتحقق من صحة مصفوفة `cells`، وتتحقق من حدود `cell_index`.
- تطبق تعديلات المصدر في الذاكرة وتكتب JSON الكامل للمفكرة مجددًا باستخدام `JSON.stringify(notebook, null, 1)`.
- تُعيد ملخصًا نصيًا بالإضافة إلى `details` منظّمة (`action`، `cellIndex`، `cellType`، `totalCells`، `cellSource`).

لا توجد دورة حياة للنواة في هذه الأداة:

- لا اكتساب لبوابة
- لا معرف جلسة للنواة
- لا `execute_request`
- لا أجزاء بثّ من قنوات النواة
- لا التقاط للعرض الغني (`image/png`، عرض JSON، حالة MIME)

## مسار التنفيذ الشبيه بالمفكرة (`src/tools/python.ts` + `src/ipy/*`)

عندما يحتاج العميل إلى تشغيل كود Python على شكل خلايا (خلايا متسلسلة، حالة مستمرة، عروض غنية)، يجري ذلك عبر **أداة `python`**، وليس `notebook`.

هذا المسار هو موضع أوضاع النواة وسلوك إعادة التشغيل/الإلغاء وبث الأجزاء واقتطاع قطع مخرجات البيانات.

## 2) دلالات معالجة خلايا المفكرة (أداة `notebook`)

## تطبيع المصدر

يُقسَّم `content` إلى `source: string[]` مع الحفاظ على أحرف السطر الجديد:

- تحتفظ كل سطر غير أخير بـ `\n` الزائدة
- السطر الأخير لا يحتوي على سطر جديد مُجبَر

يعكس هذا اصطلاحات JSON للمفكرة ويتجنب تسلسل الأسطر غير المقصود عند التعديلات اللاحقة.

## سلوك الإجراءات

- `edit`
  - يستبدل `cells[cell_index].source`
  - يحافظ على `cell_type` الموجود
- `insert`
  - يُدرج في `[0..cellCount]`
  - يأخذ `cell_type` القيمة الافتراضية `code`
  - تُهيَّأ خلايا الكود بـ `execution_count: null` و`outputs: []`
  - تُهيَّأ خلايا markdown فقط بـ `metadata` + `source`
- `delete`
  - يحذف `cells[cell_index]`
  - يُعيد `source` المحذوف في details لمعاينة المُصيِّر

## أسطح الأخطاء

تُرمى أخطاء فادحة في الحالات التالية:

- ملف المفكرة مفقود
- JSON غير صالح
- `cells` مفقود أو غير مصفوفة
- فهرس خارج النطاق (للإدراج وغير الإدراج نطاقات صالحة مختلفة)
- `content` مفقود لـ `edit`/`insert`

تتحول هذه إلى استجابات أداة `Error:` في المرحلة العليا؛ يستخدم المُصيِّر مسار المفكرة مع نص الخطأ المنسَّق.

## 3) دلالات جلسة النواة (حيث توجد فعليًا)

تُنفَّذ دلالات النواة في `executePython` / `PythonKernel` وتنطبق على أداة `python`.

## الأوضاع

`PythonKernelMode`:

- `session` (الافتراضي)
  - النوى مخزنة مؤقتًا في خريطة `kernelSessions`
  - حد أقصى 4 جلسات؛ تُزاح الأقدم عند الامتلاء
  - تنظيف الجلسات الخاملة/الميتة كل 30 ثانية، انتهاء المهلة بعد 5 دقائق
  - طابور لكل جلسة يُسلسل التنفيذ (`session.queue`)
- `per-call`
  - ينشئ نواة لكل طلب
  - ينفذ
  - يُغلق النواة دائمًا في `finally`

## سلوك إعادة الضبط

تمرر أداة `python` خيار `reset` فقط للخلية الأولى في استدعاء متعدد الخلايا؛ تعمل الخلايا اللاحقة دائمًا مع `reset: false`.

## موت النواة / إعادة التشغيل / إعادة المحاولة

في وضع الجلسة (`withKernelSession`):

- يُكتشف موت النواة عبر فحص نبضة القلب (`kernel.isAlive()` كل 5 ثوانٍ) أو فشل التنفيذ.
- الحالة الميتة قبل التشغيل تستدعي `restartKernelSession`.
- مسار الانهيار أثناء التنفيذ يُعيد المحاولة مرة واحدة: إعادة تشغيل النواة، وإعادة تشغيل المعالج.
- `restartCount > 1` في نفس الجلسة يرمي `Python kernel restarted too many times in this session`.

سلوك إعادة محاولة بدء التشغيل:

- إنشاء نواة البوابة المشتركة يُعيد المحاولة مرة واحدة على `SharedGatewayCreateError` مع HTTP 5xx.

استرداد استنفاد الموارد:

- يكتشف أنماط الفشل `EMFILE`/`ENFILE`/"Too many open files"
- يمسح الجلسات المتتبَّعة
- يستدعي `shutdownSharedGateway()`
- يُعيد المحاولة مرة واحدة لإنشاء جلسة النواة

## 4) حقن متغيرات البيئة/الجلسة

تستقبل بداية النواة خريطة بيئة اختيارية من المنفذ:

- `PI_SESSION_FILE` (مسار ملف حالة الجلسة)
- `ARTIFACTS` (مجلد البيانات)

يُشغّل `PythonKernel.#initializeKernelEnvironment(...)` بعد ذلك سكريبت التهيئة داخل النواة من أجل:

- `os.chdir(cwd)`
- حقن إدخالات البيئة في `os.environ`
- إضافة cwd إلى بداية `sys.path` إذا كان غائبًا

الاستنتاج:

- المساعدات التمهيدية التي تقرأ سياق الجلسة أو البيانات تعتمد على هذه المتغيرات البيئية في حالة عملية Python.

## 5) معالجة البث/الأجزاء والعرض (المسار المدعوم بالنواة)

يعالج عميل النواة رسائل بروتوكول Jupyter لكل تنفيذ:

- `stream` -> جزء نصي إلى `onChunk`
- `execute_result` / `display_data` ->
  - يُختار نص العرض وفق أولوية MIME: `text/markdown` > `text/plain` > `text/html` المحوَّل
  - المخرجات المنظَّمة تُلتقط بشكل منفصل:
    - `application/json` -> `{ type: "json" }`
    - `image/png` -> `{ type: "image" }`
    - `application/x-xcsh-status` -> `{ type: "status" }` (بدون إصدار نص)
- `error` -> نص تتبع الخطأ يُدفع إلى تدفق الأجزاء + بيانات الخطأ المنظَّمة
- `input_request` -> يُصدر نصًا تحذيريًا للإدخال القياسي، يُرسل `input_reply` فارغًا، يُعلّم الإدخال القياسي كمطلوب
- تنتظر إكمال كل من `execute_reply` وحالة النواة `status=idle`

الإلغاء/انتهاء المهلة:

- تُشغّل إشارة الإيقاف `interrupt()` (REST `/interrupt` + `interrupt_request` لقناة التحكم)
- تُعلَّم النتيجة بـ `cancelled=true`
- مسار انتهاء المهلة يُضيف تعليقًا على المخرجات بـ `Command timed out after <n> seconds`

## 6) سلوك الاقتطاع والبيانات

يُستخدم `OutputSink` في `src/session/streaming-output.ts` من قِبل مسارات تنفيذ النواة (`executeWithKernel`):

- يُعقّم كل جزء (`sanitizeText`)
- يتتبع إجمالي الأسطر والبايتات والمخرجات
- ملف إفراغ اختياري للبيانات (`artifactPath`، `artifactId`)
- عندما تتجاوز المخزن المؤقت في الذاكرة الحد (`DEFAULT_MAX_BYTES` ما لم يُتجاوَز):
  - يُعلَّم بالاقتطاع
  - يحتفظ ببايتات الذيل في الذاكرة (حدود آمنة لـ UTF-8)
  - يمكن إفراغ التدفق الكامل إلى حوض البيانات

تُعيد `dump()`:

- نص المخرجات المرئية (قد يكون مقتطَع الذيل)
- علامة الاقتطاع + الأعداد
- معرف البيانات (لمراجع `artifact://<id>`)

تحوّل أداة `python` هذه البيانات الوصفية إلى إشعارات اقتطاع في النتائج وتحذيرات TUI.

أداة `notebook` **لا** تستخدم `OutputSink`؛ ليس لديها خط أنابيب بث/اقتطاع بيانات لأنها لا تنفذ كودًا.

## 7) افتراضات المُصيِّر والتنسيق

## مُصيِّر المفكرة (`notebookToolRenderer`)

- عرض الاستدعاء: سطر حالة مع الإجراء + مسار المفكرة + بيانات الخلية/النوع الوصفية
- عرض النتيجة:
  - ملخص النجاح مشتق من `details`
  - `cellSource` مُصيَّرة عبر `renderCodeCell`
  - خلايا markdown تضبط تلميح اللغة على `markdown`؛ الخلايا الأخرى ليس لها تجاوز صريح للغة
  - حد معاينة الكود المطوي هو `PREVIEW_LIMITS.COLLAPSED_LINES * 2`
  - يدعم الوضع الموسع عبر خيارات العرض المشتركة
  - يستخدم ذاكرة تخزين مؤقت للعرض مفهرسة بالعرض + حالة التوسيع

افتراض عرض الأخطاء:

- إذا بدأ أول محتوى نصي بـ `Error:`، يُنسَّق المُصيِّر على شكل كتلة خطأ للمفكرة.

## مُصيِّر Python (لمخرجات التنفيذ الفعلية)

عرض التنفيذ المدعوم بالنواة يتوقع:

- انتقالات حالة لكل خلية (`pending/running/complete/error`)
- قسم حدث حالة منظَّم اختياري
- أشجار مخرجات JSON اختيارية
- تحذيرات الاقتطاع + مؤشر `artifact://<id>` اختياري

سلوك هذا المُصيِّر غير مرتبط بنتائج تحرير JSON للمفكرة، باستثناء أن كليهما يُعيد استخدام بدائيات TUI المشتركة.

## 8) الاختلاف عن سلوك أداة Python العادية

إذا كان المقصود بـ "أداة Python العادية" مسار تنفيذ `python`:

- تنفذ `python` الكود في نواة، وتحافظ على الحالة وفق الوضع، وتبث الأجزاء، وتلتقط العروض الغنية، وتعالج المقاطعات/انتهاءات المهل، وتدعم اقتطاع المخرجات/البيانات.
- تُجري `notebook` تحولات حتمية على JSON للمفكرة فحسب؛ بدون تنفيذ، ولا حالة نواة، ولا بث أجزاء، ولا مخرجات عرض، ولا خط أنابيب بيانات.

إذا احتاج سير العمل إلى كليهما:

1. حرّر مصدر المفكرة باستخدام `notebook`
2. نفّذ خلايا الكود عبر `python` (بتمرير الكود يدويًا)، وليس عبر `notebook`

لا يوفر التنفيذ الحالي أداة واحدة تُجري تحولات على `.ipynb` وتنفذ خلايا المفكرة عبر سياق النواة في الوقت ذاته.
