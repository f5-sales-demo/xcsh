# البنية الداخلية لوقت تشغيل أداة Notebook

يصف هذا المستند التنفيذ الحالي لأداة `notebook` وعلاقتها ببيئة تشغيل Python المدعومة بالنواة.

التمييز الجوهري: **`notebook` هي محرر JSON للدفاتر، وليست منفذاً للدفاتر**. فهي تحرر مصادر خلايا `.ipynb` مباشرة؛ ولا تبدأ نواة Python أو تتواصل معها.

## ملفات التنفيذ

- [`src/tools/notebook.ts`](../../packages/coding-agent/src/tools/notebook.ts)
- [`src/ipy/executor.ts`](../../packages/coding-agent/src/ipy/executor.ts)

## 1) حدود وقت التشغيل: التحرير مقابل التنفيذ

## أداة `notebook` (`src/tools/notebook.ts`)

- تدعم `action: edit | insert | delete` على ملف `.ipynb`.
- تحل المسار نسبةً إلى مجلد العمل الحالي للجلسة (`resolveToCwd`).
- تحمّل JSON الدفتر، وتتحقق من مصفوفة `cells`، وتتحقق من حدود `cell_index`.
- تطبق تعديلات المصدر في الذاكرة وتكتب JSON الدفتر بالكامل مرة أخرى باستخدام `JSON.stringify(notebook, null, 1)`.
- تُرجع ملخصاً نصياً + تفاصيل مهيكلة `details` (`action`، `cellIndex`، `cellType`، `totalCells`، `cellSource`).

لا توجد دورة حياة نواة في هذه الأداة:

- لا يوجد الحصول على بوابة
- لا يوجد معرف جلسة نواة
- لا يوجد `execute_request`
- لا توجد أجزاء متدفقة من قنوات النواة
- لا يوجد التقاط عرض غني (`image/png`، عرض JSON، حالة MIME)

## مسار التنفيذ الشبيه بالدفتر (`src/tools/python.ts` + `src/ipy/*`)

عندما يحتاج الوكيل لتشغيل كود Python بنمط الخلايا (خلايا متسلسلة، حالة مستمرة، عروض غنية)، يمر ذلك عبر أداة **`python`**، وليس `notebook`.

هذا هو المسار الذي تتواجد فيه أوضاع النواة، وسلوك إعادة التشغيل/الإلغاء، والتدفق المجزأ، واقتطاع مخرجات العناصر المنتجة.

## 2) دلالات معالجة خلايا الدفتر (أداة `notebook`)

## تطبيع المصدر

يتم تقسيم `content` إلى `source: string[]` مع الحفاظ على أسطر جديدة:

- كل سطر غير نهائي يحتفظ بـ `\n` في نهايته
- السطر الأخير لا يحتوي على سطر جديد إجباري في النهاية

يعكس هذا اصطلاحات JSON للدفاتر ويتجنب التحام الأسطر العرضي في التعديلات اللاحقة.

## سلوك الإجراءات

- `edit`
  - يستبدل `cells[cell_index].source`
  - يحافظ على `cell_type` الموجود
- `insert`
  - يدرج في `[0..cellCount]`
  - `cell_type` يكون افتراضياً `code`
  - خلايا الكود تهيئ `execution_count: null` و `outputs: []`
  - خلايا markdown تهيئ فقط `metadata` + `source`
- `delete`
  - يزيل `cells[cell_index]`
  - يُرجع `source` المحذوف في التفاصيل لمعاينة المُصيّر

## أسطح الأخطاء

يتم إلقاء أخطاء صارمة في حالة:

- ملف دفتر مفقود
- JSON غير صالح
- `cells` مفقود أو ليس مصفوفة
- فهرس خارج النطاق (الإدراج وغير الإدراج لهما نطاقات صالحة مختلفة)
- `content` مفقود لـ `edit`/`insert`

تصبح هذه استجابات أداة `Error:` في المستوى الأعلى؛ يستخدم المُصيّر مسار الدفتر + نص خطأ منسق.

## 3) دلالات جلسة النواة (حيث تتواجد فعلياً)

تُنفَّذ دلالات النواة في `executePython` / `PythonKernel` وتنطبق على أداة `python`.

## الأوضاع

`PythonKernelMode`:

- `session` (افتراضي)
  - يتم تخزين الأنوية مؤقتاً في خريطة `kernelSessions`
  - حد أقصى 4 جلسات؛ يتم إخلاء الأقدم عند التجاوز
  - تنظيف الخمول/الميت كل 30 ثانية، مهلة بعد 5 دقائق
  - تسلسل التنفيذ لكل جلسة (`session.queue`)
- `per-call`
  - ينشئ نواة للطلب
  - ينفذ
  - يوقف النواة دائماً في `finally`

## سلوك إعادة التعيين

أداة `python` تمرر `reset` فقط للخلية الأولى في استدعاء متعدد الخلايا؛ الخلايا اللاحقة تعمل دائماً مع `reset: false`.

## موت النواة / إعادة التشغيل / إعادة المحاولة

في وضع الجلسة (`withKernelSession`):

- يتم اكتشاف موت النواة عبر نبض القلب (فحص `kernel.isAlive()` كل 5 ثوانٍ) أو فشل التنفيذ.
- حالة الموت قبل التشغيل تطلق `restartKernelSession`.
- مسار الانهيار أثناء التنفيذ يعيد المحاولة مرة واحدة: إعادة تشغيل النواة، إعادة تشغيل المعالج.
- `restartCount > 1` في نفس الجلسة يلقي `Python kernel restarted too many times in this session`.

سلوك إعادة محاولة بدء التشغيل:

- إنشاء نواة البوابة المشتركة يعيد المحاولة مرة واحدة عند `SharedGatewayCreateError` مع HTTP 5xx.

استرداد استنفاد الموارد:

- يكتشف أخطاء نمط `EMFILE`/`ENFILE`/"Too many open files"
- يمسح الجلسات المتتبعة
- يستدعي `shutdownSharedGateway()`
- يعيد محاولة إنشاء جلسة النواة مرة واحدة

## 4) حقن متغيرات البيئة/الجلسة

بدء تشغيل النواة يستقبل خريطة بيئة اختيارية من المنفذ:

- `PI_SESSION_FILE` (مسار ملف حالة الجلسة)
- `ARTIFACTS` (مجلد العناصر المنتجة)

`PythonKernel.#initializeKernelEnvironment(...)` ثم يشغل سكريبت التهيئة داخل النواة لـ:

- `os.chdir(cwd)`
- حقن إدخالات البيئة في `os.environ`
- إضافة cwd في بداية `sys.path` إن كان مفقوداً

الاستنتاج:

- المساعدات التمهيدية التي تقرأ سياق الجلسة أو العناصر المنتجة تعتمد على متغيرات البيئة هذه في حالة عملية Python.

## 5) معالجة التدفق/الأجزاء والعرض (المسار المدعوم بالنواة)

يعالج عميل النواة رسائل بروتوكول Jupyter لكل تنفيذ:

- `stream` -> جزء نصي إلى `onChunk`
- `execute_result` / `display_data` ->
  - يتم اختيار نص العرض بأولوية MIME: `text/markdown` > `text/plain` > `text/html` المحوّل
  - المخرجات المهيكلة تُلتقط بشكل منفصل:
    - `application/json` -> `{ type: "json" }`
    - `image/png` -> `{ type: "image" }`
    - `application/x-xcsh-status` -> `{ type: "status" }` (بدون إرسال نص)
- `error` -> نص التتبع العكسي يُدفع إلى تدفق الأجزاء + بيانات وصفية مهيكلة للخطأ
- `input_request` -> يرسل نص تحذير الإدخال القياسي، يرسل `input_reply` فارغاً، يحدد أن الإدخال القياسي مطلوب
- الاكتمال ينتظر كلاً من `execute_reply` وحالة النواة `status=idle`

الإلغاء/المهلة:

- إشارة الإلغاء تطلق `interrupt()` (REST `/interrupt` + `interrupt_request` على قناة التحكم)
- النتيجة تحدد `cancelled=true`
- مسار المهلة يضيف تعليقاً على المخرجات بـ `Command timed out after <n> seconds`

## 6) سلوك الاقتطاع والعناصر المنتجة

`OutputSink` في `src/session/streaming-output.ts` يُستخدم بواسطة مسارات تنفيذ النواة (`executeWithKernel`):

- يعقم كل جزء (`sanitizeText`)
- يتتبع إجمالي/مخرجات الأسطر والبايتات
- ملف تسريب اختياري للعناصر المنتجة (`artifactPath`، `artifactId`)
- عندما يتجاوز المخزن المؤقت في الذاكرة الحد (`DEFAULT_MAX_BYTES` ما لم يُتجاوز):
  - يحدد علامة الاقتطاع
  - يحتفظ ببايتات النهاية في الذاكرة (حد آمن UTF-8)
  - يمكنه تسريب التدفق الكامل إلى حوض العناصر المنتجة

`dump()` يُرجع:

- نص المخرجات المرئي (ربما مقتطع من النهاية)
- علامة الاقتطاع + العدادات
- معرف العنصر المنتج (لمراجع `artifact://<id>`)

أداة `python` تحول هذه البيانات الوصفية إلى إشعارات اقتطاع النتائج وتحذيرات واجهة المستخدم النصية.

أداة `notebook` **لا** تستخدم `OutputSink`؛ ليس لديها خط أنابيب اقتطاع تدفق/عناصر منتجة لأنها لا تنفذ كوداً.

## 7) افتراضات المُصيّر والتنسيق

## مُصيّر الدفتر (`notebookToolRenderer`)

- عرض الاستدعاء: سطر الحالة مع الإجراء + مسار الدفتر + بيانات وصفية للخلية/النوع
- عرض النتيجة:
  - ملخص النجاح مشتق من `details`
  - `cellSource` يُصيّر عبر `renderCodeCell`
  - خلايا markdown تحدد تلميح اللغة `markdown`؛ الخلايا الأخرى ليس لها تجاوز صريح للغة
  - حد معاينة الكود المطوي هو `PREVIEW_LIMITS.COLLAPSED_LINES * 2`
  - يدعم الوضع الموسع عبر خيارات التصيير المشتركة
  - يستخدم ذاكرة تصيير مخبأة مفتاحها العرض + حالة التوسيع

افتراض تصيير الخطأ:

- إذا بدأ محتوى النص الأول بـ `Error:`، يُنسّقه المُصيّر ككتلة خطأ دفتر.

## مُصيّر Python (لمخرجات التنفيذ الفعلية)

تصيير التنفيذ المدعوم بالنواة يتوقع:

- انتقالات حالة لكل خلية (`pending/running/complete/error`)
- قسم أحداث حالة مهيكلة اختياري
- أشجار مخرجات JSON اختيارية
- تحذيرات اقتطاع + مؤشر `artifact://<id>` اختياري

سلوك هذا المُصيّر غير مرتبط بنتائج تحرير JSON لـ `notebook` إلا أن كليهما يعيد استخدام عناصر واجهة المستخدم النصية المشتركة.

## 8) الاختلاف عن سلوك أداة Python العادية

إذا كان المقصود بـ "أداة Python العادية" مسار تنفيذ `python`:

- `python` تنفذ الكود في نواة، وتحتفظ بالحالة حسب الوضع، وتدفق الأجزاء، وتلتقط العروض الغنية، وتتعامل مع المقاطعات/المهلات، وتدعم اقتطاع المخرجات/العناصر المنتجة.
- `notebook` تنفذ تحولات JSON حتمية للدفتر فقط؛ بدون تنفيذ، بدون حالة نواة، بدون تدفق أجزاء، بدون مخرجات عرض، بدون خط أنابيب عناصر منتجة.

إذا كان سير العمل يحتاج كليهما:

1. تحرير مصدر الدفتر باستخدام `notebook`
2. تنفيذ خلايا الكود عبر `python` (بتمرير الكود يدوياً)، وليس عبر `notebook`

التنفيذ الحالي لا يوفر أداة واحدة تقوم بتعديل `.ipynb` وتنفيذ خلايا الدفتر من خلال سياق النواة في آن واحد.
