# بروتوكول MCP وآليات النقل الداخلية

يصف هذا المستند كيفية تنفيذ coding-agent لرسائل MCP JSON-RPC وكيفية فصل مسؤوليات البروتوكول عن مسؤوليات النقل.

## النطاق

يغطي:

- تدفق طلب/استجابة JSON-RPC والإشعارات
- ربط الطلبات ودورة حياتها لوسائل نقل stdio وHTTP/SSE
- سلوك المهلة الزمنية والإلغاء
- نشر الأخطاء ومعالجة الحمولات غير الصالحة
- حدود اختيار وسيلة النقل (`stdio` مقابل `http`/`sse`)
- مسؤوليات إعادة الاتصال/المحاولة على مستوى النقل مقابل مستوى المدير

لا يغطي تجربة المستخدم لتأليف الإضافات أو واجهة الأوامر.

## حدود الطبقات

### طبقة البروتوكول (JSON-RPC + أساليب MCP)

- يتم تعريف أشكال الرسائل في `types.ts` (`JsonRpcRequest`، `JsonRpcNotification`، `JsonRpcResponse`، `JsonRpcMessage`).
- منطق عميل MCP (`client.ts`) يحدد ترتيب الأساليب ومصافحة الجلسة:
  1. طلب `initialize`
  2. إشعار `notifications/initialized`
  3. استدعاءات الأساليب مثل `tools/list`، `tools/call`

### طبقة النقل (`MCPTransport`)

يجرد `MCPTransport` عملية التسليم ودورة الحياة:

- `request(method, params, options?) -> Promise<T>`
- `notify(method, params?) -> Promise<void>`
- `close()`
- `connected`
- دوال رد الاتصال الاختيارية: `onClose`، `onError`، `onNotification`

تمتلك تنفيذات النقل تفاصيل التأطير والإدخال/الإخراج:

- `StdioTransport`: JSON محدد بأسطر جديدة عبر stdio للعملية الفرعية
- `HttpTransport`: JSON-RPC عبر HTTP POST، مع استجابات/استماع SSE اختيارية

### تحذير مهم حالي

دوال رد اتصال النقل (`onClose`، `onError`، `onNotification`) مُنفَّذة، لكن تدفقات `MCPClient`/`MCPManager` الحالية لا تربط منطق إعادة الاتصال بهذه الدوال. يتم استهلاك الإشعارات فقط إذا سجّل المستدعي معالجات.

## اختيار وسيلة النقل

`client.ts:createTransport()` يختار وسيلة النقل من التكوين:

- `type` محذوف أو `"stdio"` -> `createStdioTransport`
- `"http"` أو `"sse"` -> `createHttpTransport`

يُعامل `"sse"` كمتغير لنقل HTTP (نفس الفئة)، وليس كتنفيذ نقل منفصل.

## تدفق رسائل JSON-RPC والربط

## معرّفات الطلبات

ينشئ كل ناقل معرّفات لكل طلب (سلسلة `Math.random` + طابع زمني). المعرّفات هي رموز ربط محلية للنقل.

## مسار ربط Stdio

- يتم تسلسل الطلب الصادر ككائن JSON واحد + `\n`.
- `#pendingRequests: Map<id, {resolve,reject}>` يخزن الطلبات قيد التنفيذ.
- حلقة القراءة تحلل JSONL من stdout وتستدعي `#handleMessage`.
- إذا كانت الرسالة الواردة تحتوي على `id` مطابق، يتم حل/رفض الطلب.
- إذا كانت الرسالة الواردة تحتوي على `method` بدون `id`، تُعامل كإشعار وتُرسل إلى `onNotification`.

يتم تجاهل المعرّفات غير المعروفة (بدون رفض، بدون دالة رد خطأ).

## مسار ربط HTTP

- الطلب الصادر هو HTTP `POST` بجسم JSON ومعرّف مُنشأ.
- مسار الاستجابة غير SSE: يحلل استجابة JSON-RPC واحدة ويعيد `result`/يرمي خطأ عند `error`.
- مسار استجابة SSE (`Content-Type: text/event-stream`): يبث الأحداث، ويعيد أول رسالة يتطابق `id` الخاص بها مع معرّف الطلب المتوقع وتحتوي على `result` أو `error`.
- رسائل SSE التي تحتوي على `method` بدون `id` تُعامل كإشعارات.

إذا انتهى بث SSE قبل الاستجابة المطابقة، يفشل الطلب برسالة `No response received for request ID ...`.

## الإشعارات

يُصدر العميل إشعارات JSON-RPC عبر `transport.notify(...)`.

- Stdio: يكتب إطار الإشعار إلى stdin (`jsonrpc`، `method`، `params` اختياري) مع سطر جديد.
- HTTP: يرسل جسم POST بدون `id`؛ النجاح يقبل `2xx` أو `202 Accepted`.

الإشعارات المبادرة من الخادم تظهر فقط من خلال `onNotification` الخاص بالنقل؛ لا يوجد مشترك عام افتراضي في المدير/العميل.

## آليات نقل Stdio الداخلية

## دورة الحياة وتحولات الحالة

- الحالة الأولية: `connected=false`، `process=null`، خريطة الطلبات المعلقة فارغة
- `connect()`:
  - إنشاء عملية فرعية بالأمر/المعاملات/البيئة/المسار المُعدّ
  - تمييز الاتصال كنشط
  - بدء حلقة قراءة stdout (`readJsonl`)
  - بدء حلقة stderr (قراءة/تجاهل؛ صامتة حالياً)
- `close()`:
  - تمييز الاتصال كمنقطع
  - رفض جميع الطلبات المعلقة (`Transport closed`)
  - إنهاء العملية الفرعية
  - انتظار إيقاف حلقة القراءة
  - إطلاق `onClose`

إذا خرجت حلقة القراءة بشكل غير متوقع، يُشغّل `finally` دالة `#handleClose()` التي تؤدي نفس عملية رفض الطلبات المعلقة ودالة رد الإغلاق.

## المهلة الزمنية والإلغاء

لكل طلب:

- المهلة الافتراضية `config.timeout ?? 30000`
- `AbortSignal` اختياري من المستدعي
- كل من الإلغاء والمهلة يرفضان الوعد المعلق وينظفان مدخل الخريطة

الإلغاء محلي فقط: لا يرسل النقل إشعار إلغاء على مستوى البروتوكول إلى الخادم.

## معالجة الحمولات غير الصالحة

في حلقة القراءة:

- كل سطر JSONL محلل يُمرر إلى `#handleMessage` داخل `try/catch`
- استثناءات معالجة الرسائل غير الصالحة/الخاطئة يتم تجاهلها (تعليق `Skip malformed lines`)
- تستمر الحلقة، لذا رسالة سيئة واحدة لا تقتل الاتصال

إذا رمى محلل البث الأساسي استثناءً، يتم استدعاء `onError` (عندما يكون الاتصال نشطاً)، ثم يُغلق الاتصال.

## سلوك انقطاع الاتصال/الفشل

عند خروج العملية أو إغلاق البث:

- يتم رفض جميع الطلبات قيد التنفيذ برسالة `Transport closed`
- لا يوجد إعادة تشغيل أو إعادة اتصال تلقائية
- يجب على الطبقات الأعلى إعادة الاتصال بإنشاء ناقل جديد

## ملاحظات حول الضغط العكسي/البث

- الكتابات الصادرة تستخدم `stdin.write()` + `flush()` بدون انتظار دلالات التصريف.
- لا توجد إدارة صريحة للطابور أو علامة المستوى العالي في النقل.
- المعالجة الواردة مدفوعة بالبث (`for await` عبر `readJsonl`)، رسالة محللة واحدة في كل مرة.

## آليات نقل HTTP/SSE الداخلية

## دورة الحياة ودلالات الاتصال

يمتلك نقل HTTP حالة اتصال منطقية، لكن مسار الطلب عديم الحالة لكل استدعاء HTTP:

- `connect()` يضبط `connected=true` (بدون مصافحة مقبس/جلسة)
- تتبع جلسة الخادم اختياري عبر ترويسة `Mcp-Session-Id`
- `close()` يرسل اختيارياً `DELETE` مع `Mcp-Session-Id`، يلغي مستمع SSE، يطلق `onClose`

لذا `connected` يعني "النقل قابل للاستخدام"، وليس "تم إنشاء بث مستمر".

## سلوك ترويسة الجلسة

- عند استجابة POST، إذا كانت ترويسة `Mcp-Session-Id` موجودة، يخزنها النقل.
- الطلبات/الإشعارات اللاحقة تتضمن `Mcp-Session-Id`.
- `close()` يحاول إنهاء جلسة الخادم بـ HTTP DELETE؛ يتم تجاهل فشل الإنهاء.

## المهلة الزمنية والإلغاء

لكل من `request()` و`notify()`:

- المهلة تستخدم `AbortController` (`config.timeout ?? 30000`)
- الإشارة الخارجية، إذا وُجدت، تُدمج عبر `AbortSignal.any([...])`
- معالجة AbortError تميز بين إلغاء المستدعي والمهلة

الأخطاء المرمية:

- المهلة: `Request timeout after ...ms` (أو `SSE response timeout ...`، `Notify timeout ...`)
- إلغاء المستدعي: يُعاد رمي AbortError الأصلي عندما تكون الإشارة الخارجية ملغاة بالفعل

## نشر أخطاء HTTP

عند استجابة غير ناجحة:

- يُضمّن نص الاستجابة في الخطأ المرمي (`HTTP <status>: <text>`)
- إذا وُجدت، تُلحق تلميحات المصادقة من `WWW-Authenticate` و`Mcp-Auth-Server`

عند كائن خطأ JSON-RPC:

- يرمي `MCP error <code>: <message>`

فشل تحليل جسم JSON (`response.json()`) ينتشر كاستثناء تحليل.

## سلوك SSE وأنماطه

يوجد مساران لـ SSE:

1. **استجابة SSE لكل طلب** (`#parseSSEResponse`)
   - تُستخدم عندما يكون نوع محتوى استجابة POST هو `text/event-stream`
   - تستهلك البث حتى العثور على معرّف الاستجابة المطابق
   - يمكنها معالجة الإشعارات المتداخلة خلال نفس البث

2. **مستمع SSE في الخلفية** (`startSSEListener()`)
   - مستمع GET اختياري للإشعارات المبادرة من الخادم
   - لا يُبدأ تلقائياً بواسطة مدير/عميل MCP حالياً
   - إذا أعاد GET خطأ `405`، يُعطّل المستمع نفسه بصمت (الخادم لا يدعم هذا الوضع)

## معالجة الحمولات غير الصالحة وانقطاع الاتصال

أخطاء تحليل JSON في SSE تنبثق من `readSseJson` وترفض الطلب/المستمع.

- أخطاء تحليل SSE للطلب ترفض الطلب النشط.
- أخطاء المستمع في الخلفية تُشغّل `onError` (باستثناء AbortError).
- لا يوجد إعادة اتصال تلقائية للمستمع في الخلفية.

## أداة `json-rpc.ts` المساعدة مقابل تجريد النقل

`src/mcp/json-rpc.ts` يوفر دوال `callMCP()` و`parseSSE()` المساعدة لاستدعاءات HTTP MCP المباشرة (تُستخدم بواسطة تكامل Exa)، وليس تجريد `MCPTransport` المستخدم بواسطة `MCPClient`/`MCPManager`.

الاختلافات الملحوظة عن `HttpTransport`:

- يحلل نص الاستجابة الكامل أولاً، ثم يستخرج أول سطر `data:` (`parseSSE`)، مع بديل JSON
- بدون إدارة مهلة للطلب، بدون واجهة إلغاء، بدون معالجة session-id، بدون دورة حياة للنقل
- يعيد كائن مغلف JSON-RPC الخام

هذا المسار خفيف لكنه أقل متانة من تنفيذ النقل الكامل.

## مسؤوليات إعادة المحاولة/إعادة الاتصال

## على مستوى النقل

تنفيذات النقل الحالية **لا** تقوم بـ:

- إعادة محاولة الطلبات الفاشلة
- إعادة الاتصال بعد خروج عملية stdio
- إعادة اتصال مستمعي SSE
- إعادة إرسال الطلبات قيد التنفيذ بعد انقطاع الاتصال

تفشل بسرعة وتنشر الأخطاء.

## على مستوى المدير/العميل

`MCPManager` يتعامل مع اكتشاف/تنسيق الاتصال الأولي ويمكنه إعادة الاتصال فقط عن طريق تشغيل تدفقات الاتصال مرة أخرى (مسارات `connectToServer`/`discoverAndConnect`). لا يُصلح تلقائياً نقلاً متصلاً بالفعل عند فشل وقت التشغيل عبر دوال رد الاتصال.

يمتلك `MCPManager` سلوك بديل عند بدء التشغيل للخوادم البطيئة (أدوات مؤجلة من ذاكرة التخزين المؤقت)، لكن ذلك بديل لتوفر الأدوات وليس إعادة محاولة للنقل.

## ملخص سيناريوهات الفشل

- **سطر رسالة stdio غير صالح**: يتم تجاهله؛ يستمر البث.
- **انتهاء بث/عملية Stdio**: يُغلق النقل؛ تُرفض الطلبات المعلقة كـ `Transport closed`.
- **HTTP غير 2xx**: الطلب/الإشعار يرمي خطأ HTTP.
- **استجابة JSON غير صالحة**: ينتشر استثناء التحليل.
- **انتهاء SSE بدون معرّف مطابق**: يفشل الطلب برسالة `No response received for request ID ...`.
- **المهلة الزمنية**: خطأ مهلة خاص بالنقل.
- **إلغاء المستدعي**: ينتشر AbortError/السبب من إشارة المستدعي.

## قاعدة الحدود العملية

إذا كان الشأن يتعلق بشكل الرسالة، أو ربط المعرّفات، أو ترتيب أساليب MCP، فهو ينتمي إلى منطق البروتوكول/العميل.

إذا كان الشأن يتعلق بالتأطير (JSONL مقابل HTTP/SSE)، أو تحليل البث، أو دورة حياة fetch/spawn، أو ساعات المهلة، أو تفكيك الاتصال، فهو ينتمي إلى تنفيذ النقل.
