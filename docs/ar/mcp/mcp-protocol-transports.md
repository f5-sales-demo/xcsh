# بروتوكول MCP والبنية الداخلية لطبقة النقل

يصف هذا المستند كيفية تنفيذ coding-agent لرسائل MCP JSON-RPC وكيفية فصل شؤون البروتوكول عن شؤون النقل.

## النطاق

يغطي:

- تدفق طلبات/استجابات JSON-RPC والإشعارات
- ربط الطلبات ودورة حياتها لطبقات نقل stdio وHTTP/SSE
- سلوك المهلة الزمنية والإلغاء
- انتشار الأخطاء ومعالجة الحمولات المشوهة
- حدود اختيار طبقة النقل (`stdio` مقابل `http`/`sse`)
- المسؤوليات المتعلقة بإعادة الاتصال/المحاولة على مستوى النقل مقابل مستوى المدير

لا يغطي تجربة المستخدم في تأليف الإضافات أو واجهة الأوامر.

## حدود الطبقات

### طبقة البروتوكول (JSON-RPC + أساليب MCP)

- أشكال الرسائل معرّفة في `types.ts` (`JsonRpcRequest`، `JsonRpcNotification`، `JsonRpcResponse`، `JsonRpcMessage`).
- منطق عميل MCP (`client.ts`) يحدد ترتيب الأساليب ومصافحة الجلسة:
  1. طلب `initialize`
  2. إشعار `notifications/initialized`
  3. استدعاءات الأساليب مثل `tools/list`، `tools/call`

### طبقة النقل (`MCPTransport`)

`MCPTransport` يجرّد التسليم ودورة الحياة:

- `request(method, params, options?) -> Promise<T>`
- `notify(method, params?) -> Promise<void>`
- `close()`
- `connected`
- دوال رد الاتصال الاختيارية: `onClose`، `onError`، `onNotification`

تنفيذات النقل تمتلك تفاصيل التأطير والإدخال/الإخراج:

- `StdioTransport`: JSON محدد بأسطر جديدة عبر stdio للعملية الفرعية
- `HttpTransport`: JSON-RPC عبر HTTP POST، مع استجابات/استماع SSE اختيارية

### تحذير مهم حالي

دوال رد الاتصال في النقل (`onClose`، `onError`، `onNotification`) مُنفَّذة، لكن تدفقات `MCPClient`/`MCPManager` الحالية لا تربط منطق إعادة الاتصال بهذه الدوال. يتم استهلاك الإشعارات فقط إذا سجّل المستدعي معالجات.

## اختيار طبقة النقل

`client.ts:createTransport()` يختار النقل من التكوين:

- `type` محذوف أو `"stdio"` -> `createStdioTransport`
- `"http"` أو `"sse"` -> `createHttpTransport`

`"sse"` يُعامَل كمتغير لنقل HTTP (نفس الفئة)، وليس كتنفيذ نقل منفصل.

## تدفق رسائل JSON-RPC والربط

## معرّفات الطلبات

كل طبقة نقل تولّد معرّفات لكل طلب (سلسلة `Math.random` + طابع زمني). المعرّفات هي رموز ربط محلية للنقل.

## مسار ربط Stdio

- الطلب الصادر يُسلسَل كـ كائن JSON واحد + `\n`.
- `#pendingRequests: Map<id, {resolve,reject}>` يخزّن الطلبات قيد التنفيذ.
- حلقة القراءة تحلل JSONL من stdout وتستدعي `#handleMessage`.
- إذا كانت الرسالة الواردة تحتوي على `id` مطابق، يُحَل الطلب أو يُرفَض.
- إذا كانت الرسالة الواردة تحتوي على `method` بدون `id`، تُعامَل كإشعار وتُرسَل إلى `onNotification`.

المعرّفات غير المعروفة تُتجاهَل (لا رفض، لا دالة رد خطأ).

## مسار ربط HTTP

- الطلب الصادر هو HTTP `POST` بجسم JSON ومعرّف مولَّد.
- مسار الاستجابة غير SSE: تحليل استجابة JSON-RPC واحدة وإرجاع `result`/طرح عند `error`.
- مسار استجابة SSE (`Content-Type: text/event-stream`): بث الأحداث، إرجاع أول رسالة يطابق `id` الخاص بها معرّف الطلب المتوقع وتحتوي على `result` أو `error`.
- رسائل SSE التي تحتوي على `method` بدون `id` تُعامَل كإشعارات.

إذا انتهى بث SSE قبل الاستجابة المطابقة، يفشل الطلب بـ `No response received for request ID ...`.

## الإشعارات

العميل يصدر إشعارات JSON-RPC عبر `transport.notify(...)`.

- Stdio: يكتب إطار الإشعار إلى stdin (`jsonrpc`، `method`، `params` اختياري) بالإضافة إلى سطر جديد.
- HTTP: يرسل جسم POST بدون `id`؛ النجاح يقبل `2xx` أو `202 Accepted`.

الإشعارات المبادرة من الخادم تُعرَض فقط من خلال `onNotification` في النقل؛ لا يوجد مشترك عام افتراضي في المدير/العميل.

## البنية الداخلية لنقل Stdio

## دورة الحياة وتحولات الحالة

- الحالة الأولية: `connected=false`، `process=null`، خريطة المعلقات فارغة
- `connect()`:
  - إنشاء عملية فرعية بالأمر/المعاملات/البيئة/المسار المُكوَّن
  - تعليم الحالة كمتصل
  - بدء حلقة قراءة stdout (`readJsonl`)
  - بدء حلقة stderr (قراءة/تجاهل؛ صامتة حالياً)
- `close()`:
  - تعليم الحالة كمنفصل
  - رفض جميع الطلبات المعلقة (`Transport closed`)
  - إنهاء العملية الفرعية
  - انتظار إيقاف حلقة القراءة
  - إصدار `onClose`

إذا خرجت حلقة القراءة بشكل غير متوقع، يُشغّل `finally` دالة `#handleClose()` التي تؤدي نفس رفض الطلبات المعلقة ودالة رد الإغلاق.

## المهلة الزمنية والإلغاء

لكل طلب:

- المهلة الافتراضية `config.timeout ?? 30000`
- `AbortSignal` اختياري من المستدعي
- كل من الإلغاء والمهلة يرفضان الوعد المعلق وينظفان إدخال الخريطة

الإلغاء محلي فقط: لا ترسل طبقة النقل إشعار إلغاء على مستوى البروتوكول إلى الخادم.

## معالجة الحمولات المشوهة

في حلقة القراءة:

- كل سطر JSONL محلل يُمرَّر إلى `#handleMessage` داخل `try/catch`
- استثناءات معالجة الرسائل المشوهة/غير الصالحة تُسقَط (تعليق `Skip malformed lines`)
- تستمر الحلقة، لذا رسالة سيئة واحدة لا تقتل الاتصال

إذا طرح محلل البث الأساسي استثناء، يُستدعى `onError` (عندما يكون لا يزال متصلاً)، ثم يُغلَق الاتصال.

## سلوك قطع الاتصال/الفشل

عند خروج العملية أو إغلاق البث:

- جميع الطلبات قيد التنفيذ تُرفَض بـ `Transport closed`
- لا إعادة تشغيل أو اتصال تلقائي
- يجب على الطبقات الأعلى إعادة الاتصال بإنشاء نقل جديد

## ملاحظات حول الضغط العكسي/البث

- الكتابة الصادرة تستخدم `stdin.write()` + `flush()` بدون انتظار دلالات التصريف.
- لا توجد إدارة صريحة لقائمة الانتظار أو علامة المستوى العالي في النقل.
- المعالجة الواردة مدفوعة بالبث (`for await` عبر `readJsonl`)، رسالة محللة واحدة في كل مرة.

## البنية الداخلية لنقل HTTP/SSE

## دورة الحياة ودلالات الاتصال

نقل HTTP له حالة اتصال منطقية، لكن مسار الطلب عديم الحالة لكل استدعاء HTTP:

- `connect()` يضبط `connected=true` (لا مصافحة مقبس/جلسة)
- تتبع جلسة الخادم اختياري عبر ترويسة `Mcp-Session-Id`
- `close()` يرسل اختيارياً `DELETE` مع `Mcp-Session-Id`، يلغي مستمع SSE، يصدر `onClose`

لذا `connected` تعني "النقل قابل للاستخدام"، وليس "بث مستمر مُنشأ".

## سلوك ترويسة الجلسة

- عند استجابة POST، إذا كانت ترويسة `Mcp-Session-Id` موجودة، يخزنها النقل.
- الطلبات/الإشعارات اللاحقة تتضمن `Mcp-Session-Id`.
- `close()` يحاول إنهاء جلسة الخادم بـ HTTP DELETE؛ يتم تجاهل فشل الإنهاء.

## المهلة الزمنية والإلغاء

لكل من `request()` و`notify()`:

- المهلة تستخدم `AbortController` (`config.timeout ?? 30000`)
- الإشارة الخارجية، إن وُجدت، تُدمج عبر `AbortSignal.any([...])`
- معالجة AbortError تميز بين إلغاء المستدعي والمهلة

الأخطاء المطروحة:

- المهلة: `Request timeout after ...ms` (أو `SSE response timeout ...`، `Notify timeout ...`)
- إلغاء المستدعي: يُعاد طرح AbortError الأصلي عندما تكون الإشارة الخارجية ملغاة بالفعل

## انتشار أخطاء HTTP

عند استجابة غير OK:

- نص الاستجابة يُضمَّن في الخطأ المطروح (`HTTP <status>: <text>`)
- إن وُجدت، تلميحات المصادقة من `WWW-Authenticate` و`Mcp-Auth-Server` تُلحَق

عند كائن خطأ JSON-RPC:

- يطرح `MCP error <code>: <message>`

فشل تحليل جسم JSON (`response.json()`) ينتشر كاستثناء تحليل.

## سلوك SSE وأوضاعه

يوجد مساران لـ SSE:

1. **استجابة SSE لكل طلب** (`#parseSSEResponse`)
   - تُستخدم عندما يكون نوع محتوى استجابة POST هو `text/event-stream`
   - تستهلك البث حتى إيجاد معرّف الاستجابة المطابق
   - يمكنها معالجة الإشعارات المتداخلة خلال نفس البث

2. **مستمع SSE في الخلفية** (`startSSEListener()`)
   - مستمع GET اختياري للإشعارات المبادرة من الخادم
   - حالياً لا يُبدأ تلقائياً بواسطة مدير/عميل MCP
   - إذا أرجع GET `405`، يعطّل المستمع نفسه بصمت (الخادم لا يدعم هذا الوضع)

## معالجة الحمولات المشوهة وقطع الاتصال

أخطاء تحليل JSON في SSE تخرج من `readSseJson` وترفض الطلب/المستمع.

- أخطاء تحليل SSE للطلب ترفض الطلب النشط.
- أخطاء المستمع في الخلفية تُشغّل `onError` (باستثناء AbortError).
- لا إعادة اتصال تلقائية للمستمع في الخلفية.

## أداة `json-rpc.ts` مقابل تجريد النقل

`src/mcp/json-rpc.ts` يوفر مساعدات `callMCP()` و`parseSSE()` لاستدعاءات MCP HTTP المباشرة (تُستخدم بواسطة تكامل Exa)، وليس تجريد `MCPTransport` المستخدم بواسطة `MCPClient`/`MCPManager`.

اختلافات ملحوظة عن `HttpTransport`:

- يحلل نص الاستجابة بالكامل أولاً، ثم يستخرج أول سطر `data:` (`parseSSE`)، مع الرجوع لـ JSON
- لا إدارة مهلة زمنية للطلب، لا واجهة إلغاء، لا معالجة session-id، لا دورة حياة نقل
- يُرجع كائن مغلف JSON-RPC الخام

هذا المسار خفيف الوزن لكنه أقل متانة من تنفيذ النقل الكامل.

## مسؤوليات إعادة المحاولة/الاتصال

## مستوى النقل

تنفيذات النقل الحالية **لا** تقوم بـ:

- إعادة محاولة الطلبات الفاشلة
- إعادة الاتصال بعد خروج عملية stdio
- إعادة اتصال مستمعي SSE
- إعادة إرسال الطلبات قيد التنفيذ بعد قطع الاتصال

تفشل بسرعة وتنشر الأخطاء.

## مستوى المدير/العميل

`MCPManager` يتولى اكتشاف/تنسيق الاتصال الأولي ويمكنه إعادة الاتصال فقط بتشغيل تدفقات الاتصال مرة أخرى (مسارات `connectToServer`/`discoverAndConnect`). لا يعالج تلقائياً نقل متصل بالفعل عند دوال رد الفشل أثناء التشغيل.

`MCPManager` لديه سلوك احتياطي عند بدء التشغيل للخوادم البطيئة (أدوات مؤجلة من ذاكرة التخزين المؤقت)، لكن ذلك احتياطي لتوفر الأدوات، وليس إعادة محاولة نقل.

## ملخص سيناريوهات الفشل

- **سطر رسالة stdio مشوه**: يُسقَط؛ يستمر البث.
- **انتهاء بث/عملية Stdio**: يُغلَق النقل؛ الطلبات المعلقة تُرفَض بـ `Transport closed`.
- **HTTP غير 2xx**: الطلب/الإشعار يطرح خطأ HTTP.
- **استجابة JSON غير صالحة**: استثناء التحليل ينتشر.
- **انتهاء SSE بدون معرّف مطابق**: يفشل الطلب بـ `No response received for request ID ...`.
- **المهلة الزمنية**: خطأ مهلة خاص بالنقل.
- **إلغاء المستدعي**: AbortError/السبب ينتشر من إشارة المستدعي.

## قاعدة الحدود العملية

إذا كان الشأن متعلقاً بشكل الرسالة، أو ربط المعرّفات، أو ترتيب أساليب MCP، فهو ينتمي إلى منطق البروتوكول/العميل.

إذا كان الشأن متعلقاً بالتأطير (JSONL مقابل HTTP/SSE)، أو تحليل البث، أو دورة حياة fetch/spawn، أو ساعات المهلة، أو تفكيك الاتصال، فهو ينتمي إلى تنفيذ النقل.
