# دورة حياة تشغيل MCP

يصف هذا المستند كيفية اكتشاف خوادم MCP والاتصال بها وعرضها كأدوات وتحديثها وإيقافها في بيئة تشغيل وكيل البرمجة.

## نظرة عامة على دورة الحياة

1. **بدء تشغيل SDK** يستدعي `discoverAndLoadMCPTools()` (ما لم يكن MCP معطلاً).
2. **الاكتشاف** (`loadAllMCPConfigs`) يحل تكوينات خوادم MCP من مصادر القدرات، ويُصفّي المدخلات المعطلة/الخاصة بالمشروع/Exa، ويحتفظ ببيانات المصدر الوصفية.
3. **مرحلة اتصال المدير** (`MCPManager.connectServers`) تبدأ عملية الاتصال لكل خادم + `tools/list` بشكل متوازٍ.
4. **بوابة البدء السريع** تنتظر حتى 250 مللي ثانية، ثم قد تُعيد:
   - أدوات `MCPTool` مُحمّلة بالكامل،
   - حالات فشل لكل خادم،
   - أو أدوات `DeferredMCPTool` مُخزنة مؤقتًا للخوادم التي لا تزال قيد الانتظار.
5. **ربط SDK** يدمج أدوات MCP في سجل أدوات وقت التشغيل للجلسة.
6. **الجلسة الحية** يمكنها تحديث أدوات MCP عبر مسارات `/mcp` (`disconnectAll` + إعادة اكتشاف + `session.refreshMCPTools`).
7. **الإيقاف** يحدث عندما يستدعي المُستدعون `disconnectServer`/`disconnectAll`؛ كما يُزيل المدير تسجيلات أدوات MCP للخوادم المنفصلة.

## مرحلة الاكتشاف والتحميل

### مسار الدخول من SDK

`createAgentSession()` في `src/sdk.ts` يُنفذ بدء تشغيل MCP عندما يكون `enableMCP` مُفعّلاً (الافتراضي):

- يستدعي `discoverAndLoadMCPTools(cwd, { ... })`,
- يُمرر `authStorage`، وتخزين ذاكرة التخزين المؤقت، وإعداد `mcp.enableProjectConfig`,
- يضبط دائمًا `filterExa: true`,
- يُسجل أخطاء التحميل/الاتصال لكل خادم,
- يُخزن المدير المُعاد في `toolSession.mcpManager` ونتيجة الجلسة.

إذا كان `enableMCP` معطلاً، يتم تخطي اكتشاف MCP بالكامل.

### اكتشاف التكوين والتصفية

`loadAllMCPConfigs()` (`src/mcp/config.ts`) يُحمّل عناصر خادم MCP القياسية من خلال اكتشاف القدرات، ثم يُحوّلها إلى `MCPServerConfig` القديم.

سلوك التصفية:

- `enableProjectConfig: false` يُزيل المدخلات على مستوى المشروع (`_source.level === "project"`).
- الخوادم ذات `enabled: false` يتم تخطيها قبل محاولات الاتصال.
- خوادم Exa تُصفّى افتراضيًا ويتم استخراج مفاتيح API لتكامل أداة Exa الأصلية.

تتضمن النتيجة كلاً من `configs` و`sources` (بيانات وصفية تُستخدم لاحقًا لتسمية المزود).

### سلوك الفشل على مستوى الاكتشاف

`discoverAndLoadMCPTools()` يُميز بين فئتين من الفشل:

- **فشل صلب في الاكتشاف** (استثناء من `manager.discoverAndConnect`، عادةً من اكتشاف التكوين): يُعيد مجموعة أدوات فارغة وخطأ اصطناعي واحد `{ path: ".mcp.json", error }`.
- **فشل تشغيل/اتصال لكل خادم**: يُعيد المدير نجاحًا جزئيًا مع خريطة `errors`؛ تستمر الخوادم الأخرى.

لذا فإن بدء التشغيل لا يُفشل جلسة الوكيل بأكملها عند فشل خوادم MCP فردية.

## نموذج حالة المدير

`MCPManager` يتتبع دورة حياة التشغيل بسجلات منفصلة:

- `#connections: Map<string, MCPServerConnection>` — الخوادم المتصلة بالكامل.
- `#pendingConnections: Map<string, Promise<MCPServerConnection>>` — المصافحة قيد التنفيذ.
- `#pendingToolLoads: Map<string, Promise<{ connection, serverTools }>>` — متصل لكن الأدوات لا تزال قيد التحميل.
- `#tools: CustomTool[]` — عرض أدوات MCP الحالي المكشوف للمُستدعين.
- `#sources: Map<string, SourceMeta>` — بيانات المزود/المصدر الوصفية حتى قبل اكتمال الاتصال.

`getConnectionStatus(name)` يستنتج الحالة من هذه الخرائط:

- `connected` إذا كان في `#connections`,
- `connecting` إذا كان في انتظار الاتصال أو انتظار تحميل الأدوات,
- `disconnected` خلاف ذلك.

## إنشاء الاتصال وتوقيت البدء

## خط أنابيب الاتصال لكل خادم

لكل خادم مُكتشف في `connectServers()`:

1. تخزين/تحديث بيانات المصدر الوصفية,
2. تخطي إذا كان متصلاً بالفعل/في الانتظار,
3. التحقق من صحة حقول النقل (`validateServerConfig`),
4. حل استبدالات المصادقة/الغلاف (`#resolveAuthConfig`),
5. استدعاء `connectToServer(name, resolvedConfig)`,
6. استدعاء `listTools(connection)`,
7. تخزين تعريفات الأدوات مؤقتًا (`MCPToolCache.set`) بأفضل جهد.

سلوك `connectToServer()` (`src/mcp/client.ts`):

- يُنشئ نقل stdio أو HTTP/SSE,
- يُنفذ `initialize` + `notifications/initialized` لـ MCP,
- يستخدم مهلة زمنية (`config.timeout` أو 30 ثانية افتراضيًا),
- يُغلق النقل عند فشل التهيئة.

### بوابة البدء السريع + الاحتياطي المؤجل

`connectServers()` ينتظر سباقًا بين:

- اكتمال جميع مهام الاتصال/تحميل الأدوات, و
- `STARTUP_TIMEOUT_MS = 250`.

بعد 250 مللي ثانية:

- المهام المُنجزة تصبح أدوات `MCPTool` حية,
- المهام المرفوضة تُنتج أخطاء لكل خادم,
- المهام التي لا تزال قيد الانتظار:
  - تستخدم تعريفات الأدوات المُخزنة مؤقتًا إن وُجدت (`MCPToolCache.get`) لإنشاء `DeferredMCPTool`,
  - وإلا تنتظر حتى اكتمال تلك المهام المعلقة.

هذا نموذج بدء هجين: إرجاع سريع عند توفر ذاكرة التخزين المؤقت، وانتظار للدقة عندما لا تكون متوفرة.

### سلوك الإكمال في الخلفية

كل `toolsPromise` معلق لديه أيضًا استمرار في الخلفية يقوم في النهاية بـ:

- استبدال شريحة أدوات ذلك الخادم في حالة المدير عبر `#replaceServerTools`,
- كتابة ذاكرة التخزين المؤقت,
- تسجيل حالات الفشل المتأخرة فقط بعد بدء التشغيل (`allowBackgroundLogging`).

## عرض الأدوات وتوفرها في الجلسة الحية

### التسجيل عند بدء التشغيل

`discoverAndLoadMCPTools()` يُحوّل أدوات المدير إلى `LoadedCustomTool[]` ويُزيّن المسارات (`mcp:<server> via <providerName>` عند المعرفة).

`createAgentSession()` ثم يدفع هذه الأدوات إلى `customTools`، التي يتم تغليفها وإضافتها إلى سجل أدوات وقت التشغيل بأسماء مثل `mcp_<server>_<tool>`.

### استدعاءات الأدوات

- `MCPTool` يستدعي الأدوات من خلال `MCPServerConnection` متصل بالفعل.
- `DeferredMCPTool` ينتظر `waitForConnection(server)` قبل الاستدعاء؛ هذا يسمح للأدوات المُخزنة مؤقتًا بالوجود قبل أن يكون الاتصال جاهزًا.

كلاهما يُعيد مخرجات أداة مُهيكلة ويُحوّل أخطاء النقل/الأداة إلى محتوى أداة `MCP error: ...` (الإلغاء يبقى إلغاءً).

## مسارات التحديث/إعادة التحميل (بدء التشغيل مقابل إعادة التحميل الحية)

### مسار بدء التشغيل الأولي

- اكتشاف/تحميل لمرة واحدة في `sdk.ts`,
- يتم تسجيل الأدوات في سجل أدوات الجلسة الأولية.

### مسار إعادة التحميل التفاعلي

مسار `/mcp reload` (`src/modes/controllers/mcp-command-controller.ts`) يقوم بـ:

1. `mcpManager.disconnectAll()`,
2. `mcpManager.discoverAndConnect()`,
3. `session.refreshMCPTools(mcpManager.getTools())`.

`session.refreshMCPTools()` (`src/session/agent-session.ts`) يُزيل جميع أدوات `mcp_`، ويُعيد تغليف أحدث أدوات MCP، ويُعيد تفعيل مجموعة الأدوات حتى تُطبّق تغييرات MCP دون إعادة تشغيل الجلسة.

هناك أيضًا مسار متابعة للاتصالات المتأخرة: بعد انتظار خادم معين، إذا أصبحت الحالة `connected`، يُعيد تشغيل `session.refreshMCPTools(...)` حتى تُربط الأدوات المتاحة حديثًا في الجلسة.

## الصحة وإعادة الاتصال وسلوك الفشل الجزئي

سلوك وقت التشغيل الحالي بسيط عمدًا:

- **لا يوجد مراقب صحة مستقل** في المدير/العميل.
- **لا توجد حلقة إعادة اتصال تلقائية** عند انقطاع النقل.
- المدير لا يشترك في `onClose`/`onError` للنقل؛ الحالة مدفوعة بالسجل.
- إعادة الاتصال صريحة: مسار إعادة التحميل أو استدعاء `connectServers()` مباشرة.

عمليًا:

- فشل خادم واحد لا يُزيل أدوات الخوادم السليمة,
- حالات فشل الاتصال/القائمة معزولة لكل خادم,
- ذاكرة التخزين المؤقت للأدوات والتحديثات في الخلفية تتم بأفضل جهد (يتم تسجيل التحذيرات/الأخطاء، بدون توقف صلب).

## دلالات الإيقاف

### إيقاف على مستوى الخادم

`disconnectServer(name)`:

- يُزيل المدخلات المعلقة/بيانات المصدر الوصفية,
- يُغلق النقل إذا كان متصلاً,
- يُزيل أدوات `mcp_` الخاصة بذلك الخادم من حالة المدير.

### الإيقاف الشامل

`disconnectAll()`:

- يُغلق جميع عمليات النقل النشطة باستخدام `Promise.allSettled`,
- يُمسح الخرائط المعلقة والمصادر والاتصالات وقائمة أدوات المدير.

في الربط الحالي، يُستخدم الإيقاف الصريح في مسارات أوامر MCP (لإعادة التحميل/الإزالة/التعطيل). لا يوجد خطاف تخلص تلقائي منفصل للمدير في مسار بدء التشغيل نفسه؛ المُستدعون مسؤولون عن استدعاء طرق فصل المدير عندما يحتاجون إلى إيقاف MCP بشكل محدد.

## أنماط الفشل والضمانات

| السيناريو | السلوك | فشل صلب مقابل أفضل جهد |
| --- | --- | --- |
| الاكتشاف يطرح استثناء (مسار تحميل القدرات/التكوين) | المُحمّل يُعيد أدوات فارغة + خطأ اصطناعي `.mcp.json` | أفضل جهد لبدء تشغيل الجلسة |
| تكوين خادم غير صالح | يتم تخطي الخادم مع إدخال خطأ التحقق | أفضل جهد لكل خادم |
| مهلة الاتصال/فشل التهيئة | يتم تسجيل خطأ الخادم؛ تستمر الخوادم الأخرى | أفضل جهد لكل خادم |
| `tools/list` لا يزال قيد الانتظار عند بدء التشغيل مع وجود ذاكرة تخزين مؤقت | يتم إرجاع أدوات مؤجلة فورًا | أفضل جهد لبدء تشغيل سريع |
| `tools/list` لا يزال قيد الانتظار عند بدء التشغيل بدون ذاكرة تخزين مؤقت | بدء التشغيل ينتظر اكتمال المعلقات | انتظار صلب للدقة |
| فشل تحميل أدوات متأخر في الخلفية | يتم التسجيل بعد بوابة البدء | أفضل جهد للتسجيل |
| انقطاع النقل أثناء التشغيل | لا إعادة اتصال تلقائية؛ الاستدعاءات المستقبلية تفشل حتى إعادة الاتصال/التحميل | أفضل جهد للاستعادة عبر إجراء يدوي |

## واجهة API العامة

`src/mcp/index.ts` يُعيد تصدير واجهات المُحمّل/المدير/العميل للمُستدعين الخارجيين. `src/sdk.ts` يكشف `discoverMCPServers()` كغلاف مُيسّر يُعيد نفس شكل نتيجة المُحمّل.
