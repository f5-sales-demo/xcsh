# دورة حياة MCP أثناء التشغيل

يصف هذا المستند كيفية اكتشاف خوادم MCP والاتصال بها وعرضها كأدوات وتحديثها وإيقافها في بيئة تشغيل وكيل البرمجة.

## نظرة عامة على دورة الحياة

1. **بدء تشغيل SDK** يستدعي `discoverAndLoadMCPTools()` (ما لم يكن MCP معطلاً).
2. **الاكتشاف** (`loadAllMCPConfigs`) يحل تكوينات خوادم MCP من مصادر القدرات، ويرشح الإدخالات المعطلة/المشاريع/Exa، ويحافظ على بيانات المصدر الوصفية.
3. **مرحلة اتصال المدير** (`MCPManager.connectServers`) تبدأ عملية الاتصال لكل خادم + `tools/list` بشكل متوازٍ.
4. **بوابة البدء السريع** تنتظر حتى 250 مللي ثانية، ثم قد تُرجع:
   - أدوات `MCPTool` محملة بالكامل،
   - حالات فشل لكل خادم،
   - أو أدوات `DeferredMCPTool` مخزنة مؤقتاً للخوادم التي لا تزال قيد الانتظار.
5. **ربط SDK** يدمج أدوات MCP في سجل أدوات وقت التشغيل للجلسة.
6. **الجلسة المباشرة** يمكنها تحديث أدوات MCP عبر تدفقات `/mcp` (`disconnectAll` + إعادة اكتشاف + `session.refreshMCPTools`).
7. **الإيقاف** يحدث عندما يستدعي المستدعون `disconnectServer`/`disconnectAll`؛ كما يمسح المدير تسجيلات أدوات MCP للخوادم المنفصلة.

## مرحلة الاكتشاف والتحميل

### مسار الدخول من SDK

`createAgentSession()` في `src/sdk.ts` يُنفذ بدء تشغيل MCP عندما يكون `enableMCP` مفعلاً (الافتراضي):

- يستدعي `discoverAndLoadMCPTools(cwd, { ... })`،
- يمرر `authStorage`، وتخزين ذاكرة التخزين المؤقت، وإعداد `mcp.enableProjectConfig`،
- يضبط دائماً `filterExa: true`،
- يسجل أخطاء التحميل/الاتصال لكل خادم،
- يخزن المدير المُرجع في `toolSession.mcpManager` ونتيجة الجلسة.

إذا كان `enableMCP` معطلاً، يتم تخطي اكتشاف MCP بالكامل.

### اكتشاف التكوين والترشيح

`loadAllMCPConfigs()` (`src/mcp/config.ts`) يحمل عناصر خوادم MCP القياسية من خلال اكتشاف القدرات، ثم يحولها إلى `MCPServerConfig` القديم.

سلوك الترشيح:

- `enableProjectConfig: false` يزيل إدخالات مستوى المشروع (`_source.level === "project"`).
- الخوادم ذات `enabled: false` يتم تخطيها قبل محاولات الاتصال.
- خوادم Exa تُرشح افتراضياً ويتم استخراج مفاتيح API لتكامل أداة Exa الأصلية.

تتضمن النتيجة كلاً من `configs` و`sources` (بيانات وصفية تُستخدم لاحقاً لتسمية الموفر).

### سلوك الفشل على مستوى الاكتشاف

`discoverAndLoadMCPTools()` يميز بين فئتين من الفشل:

- **فشل اكتشاف حاد** (استثناء من `manager.discoverAndConnect`، عادةً من اكتشاف التكوين): يُرجع مجموعة أدوات فارغة وخطأ اصطناعي واحد `{ path: ".mcp.json", error }`.
- **فشل تشغيل/اتصال لكل خادم**: يُرجع المدير نجاحاً جزئياً مع خريطة `errors`؛ تستمر الخوادم الأخرى.

لذا لا يفشل بدء التشغيل جلسة الوكيل بالكامل عندما تفشل خوادم MCP فردية.

## نموذج حالة المدير

`MCPManager` يتتبع دورة حياة التشغيل بسجلات منفصلة:

- `#connections: Map<string, MCPServerConnection>` — خوادم متصلة بالكامل.
- `#pendingConnections: Map<string, Promise<MCPServerConnection>>` — مصافحة قيد التنفيذ.
- `#pendingToolLoads: Map<string, Promise<{ connection, serverTools }>>` — متصلة لكن الأدوات لا تزال قيد التحميل.
- `#tools: CustomTool[]` — عرض أدوات MCP الحالي المكشوف للمستدعين.
- `#sources: Map<string, SourceMeta>` — بيانات وصفية للموفر/المصدر حتى قبل اكتمال الاتصال.

`getConnectionStatus(name)` يشتق الحالة من هذه الخرائط:

- `connected` إذا كان في `#connections`،
- `connecting` إذا كان في حالة اتصال معلق أو تحميل أدوات معلق،
- `disconnected` خلاف ذلك.

## إنشاء الاتصال وتوقيت البدء

## خط أنابيب الاتصال لكل خادم

لكل خادم مُكتشف في `connectServers()`:

1. تخزين/تحديث بيانات المصدر الوصفية،
2. تخطي إذا كان متصلاً/معلقاً بالفعل،
3. التحقق من حقول النقل (`validateServerConfig`)،
4. حل استبدالات المصادقة/الصدفة (`#resolveAuthConfig`)،
5. استدعاء `connectToServer(name, resolvedConfig)`،
6. استدعاء `listTools(connection)`،
7. تخزين تعريفات الأدوات مؤقتاً (`MCPToolCache.set`) بأفضل جهد.

سلوك `connectToServer()` (`src/mcp/client.ts`):

- ينشئ نقل stdio أو HTTP/SSE،
- ينفذ MCP `initialize` + `notifications/initialized`،
- يستخدم مهلة زمنية (`config.timeout` أو 30 ثانية افتراضياً)،
- يغلق النقل عند فشل التهيئة.

### بوابة البدء السريع + الرجوع المؤجل

`connectServers()` تنتظر على سباق بين:

- تسوية جميع مهام الاتصال/تحميل الأدوات، و
- `STARTUP_TIMEOUT_MS = 250`.

بعد 250 مللي ثانية:

- المهام المكتملة تصبح أدوات `MCPTool` حية،
- المهام المرفوضة تنتج أخطاء لكل خادم،
- المهام المعلقة:
  - تستخدم تعريفات الأدوات المخزنة مؤقتاً إن كانت متاحة (`MCPToolCache.get`) لإنشاء أدوات `DeferredMCPTool`،
  - وإلا تنتظر حتى تتم تسوية تلك المهام المعلقة.

هذا نموذج بدء هجين: إرجاع سريع عندما تكون ذاكرة التخزين المؤقت متاحة، وانتظار صحيح عندما لا تكون كذلك.

### سلوك الإكمال في الخلفية

كل `toolsPromise` معلق لديه أيضاً استمرار في الخلفية يقوم في النهاية بـ:

- استبدال شريحة أدوات ذلك الخادم في حالة المدير عبر `#replaceServerTools`،
- كتابة ذاكرة التخزين المؤقت،
- تسجيل حالات الفشل المتأخرة فقط بعد بدء التشغيل (`allowBackgroundLogging`).

## عرض الأدوات والتوفر في الجلسة المباشرة

### التسجيل عند البدء

`discoverAndLoadMCPTools()` يحول أدوات المدير إلى `LoadedCustomTool[]` ويزين المسارات (`mcp:<server> via <providerName>` عند المعرفة).

`createAgentSession()` ثم يدفع هذه الأدوات إلى `customTools`، التي تُغلف وتُضاف إلى سجل أدوات وقت التشغيل بأسماء مثل `mcp_<server>_<tool>`.

### استدعاءات الأدوات

- `MCPTool` يستدعي الأدوات من خلال `MCPServerConnection` متصل بالفعل.
- `DeferredMCPTool` ينتظر `waitForConnection(server)` قبل الاستدعاء؛ هذا يسمح للأدوات المخزنة مؤقتاً بالوجود قبل أن يكون الاتصال جاهزاً.

كلاهما يُرجع مخرجات أداة منظمة ويحول أخطاء النقل/الأداة إلى محتوى أداة `MCP error: ...` (الإلغاء يبقى إلغاءً).

## مسارات التحديث/إعادة التحميل (بدء التشغيل مقابل إعادة التحميل المباشرة)

### مسار البدء الأولي

- اكتشاف/تحميل لمرة واحدة في `sdk.ts`،
- تُسجل الأدوات في سجل أدوات الجلسة الأولي.

### مسار إعادة التحميل التفاعلي

مسار `/mcp reload` (`src/modes/controllers/mcp-command-controller.ts`) يقوم بـ:

1. `mcpManager.disconnectAll()`،
2. `mcpManager.discoverAndConnect()`،
3. `session.refreshMCPTools(mcpManager.getTools())`.

`session.refreshMCPTools()` (`src/session/agent-session.ts`) يزيل جميع أدوات `mcp_`، ويعيد تغليف أحدث أدوات MCP، ويعيد تنشيط مجموعة الأدوات حتى تُطبق تغييرات MCP دون إعادة تشغيل الجلسة.

هناك أيضاً مسار متابعة للاتصالات المتأخرة: بعد انتظار خادم محدد، إذا أصبحت الحالة `connected`، يُعيد تشغيل `session.refreshMCPTools(...)` حتى تُربط الأدوات المتاحة حديثاً في الجلسة.

## الصحة وإعادة الاتصال وسلوك الفشل الجزئي

سلوك وقت التشغيل الحالي بسيط عمداً:

- **لا يوجد مراقب صحة مستقل** في المدير/العميل.
- **لا توجد حلقة إعادة اتصال تلقائية** عند انقطاع النقل.
- المدير لا يشترك في `onClose`/`onError` للنقل؛ الحالة مدفوعة بالسجل.
- إعادة الاتصال صريحة: تدفق إعادة التحميل أو استدعاء `connectServers()` مباشرة.

من الناحية التشغيلية:

- فشل خادم واحد لا يزيل الأدوات من الخوادم السليمة،
- حالات فشل الاتصال/القائمة معزولة لكل خادم،
- ذاكرة التخزين المؤقت للأدوات والتحديثات الخلفية تعمل بأفضل جهد (تُسجل التحذيرات/الأخطاء، بدون توقف حاد).

## دلالات الإيقاف

### إيقاف على مستوى الخادم

`disconnectServer(name)`:

- يزيل الإدخالات المعلقة/بيانات المصدر الوصفية،
- يغلق النقل إذا كان متصلاً،
- يزيل أدوات `mcp_` لذلك الخادم من حالة المدير.

### إيقاف شامل

`disconnectAll()`:

- يغلق جميع عمليات النقل النشطة باستخدام `Promise.allSettled`،
- يمسح الخرائط المعلقة والمصادر والاتصالات وقائمة أدوات المدير.

في الربط الحالي، يُستخدم الإيقاف الصريح في تدفقات أوامر MCP (لإعادة التحميل/الإزالة/التعطيل). لا يوجد خطاف تخلص تلقائي منفصل للمدير في مسار البدء نفسه؛ المستدعون مسؤولون عن استدعاء طرق فصل المدير عندما يحتاجون إلى إيقاف MCP حتمي.

## أوضاع الفشل والضمانات

| السيناريو | السلوك | فشل حاد مقابل أفضل جهد |
| --- | --- | --- |
| فشل الاكتشاف (مسار تحميل القدرات/التكوين) | يُرجع المُحمّل أدوات فارغة + خطأ اصطناعي `.mcp.json` | بدء جلسة بأفضل جهد |
| تكوين خادم غير صالح | يُتخطى الخادم مع إدخال خطأ التحقق | أفضل جهد لكل خادم |
| مهلة اتصال/فشل تهيئة | يُسجل خطأ الخادم؛ تستمر الخوادم الأخرى | أفضل جهد لكل خادم |
| `tools/list` لا يزال معلقاً عند البدء مع وجود ذاكرة تخزين مؤقت | تُرجع أدوات مؤجلة فوراً | بدء سريع بأفضل جهد |
| `tools/list` لا يزال معلقاً عند البدء بدون ذاكرة تخزين مؤقت | ينتظر البدء حتى تتم تسوية المعلق | انتظار حاد للصحة |
| فشل تحميل أدوات خلفي متأخر | يُسجل بعد بوابة البدء | تسجيل بأفضل جهد |
| انقطاع نقل أثناء التشغيل | لا إعادة اتصال تلقائية؛ تفشل الاستدعاءات المستقبلية حتى إعادة الاتصال/التحميل | استرداد بأفضل جهد عبر إجراء يدوي |

## واجهة API العامة

`src/mcp/index.ts` يُعيد تصدير واجهات المُحمّل/المدير/العميل للمستدعين الخارجيين. `src/sdk.ts` يكشف `discoverMCPServers()` كغلاف ملائم يُرجع نفس شكل نتيجة المُحمّل.
